## 一、为什么要再次调整

`docs/20260423_01` 附录 A 的模型是**preset-first**：

```
用户选 preset（项目类型）→ traits 一次性填满 → 加载对应 SOP yaml → skill/agent/mcp 按 trait 过滤
         ↑ 必选入口，锁定流程
```

讨论中用户提出：

> "新建项目时应该是多个关键词的组合，而不是先就定死了项目类型。在组装不同关键词的过程中，能自动组装出来工作流，后面的 agent skill mcp 这些。"

**问题定性**：preset-first 把"项目类型"当成必选入口，但真实需求是**累积关键词**（例如"多人 + 在线 + 对战 + Godot + PC + 原型"），这些关键词**交叉维度 ≥ 6 个**，不是枚举能覆盖的。

**范式反转**：

```
用户对话  →  traits 逐个累积  →  AI 实时展示"当前组装结果"  →  用户确认
                    ↓
              SOP / skill / agent / mcp 都是 trait 的派生
                    ↓
              preset 降级为"已保存的 trait 组合快捷方式"（可选，非必选路径）
```

差异**不是加字段**，是**控制流反转**：配置从"一次定死"变成"按需组装 + 可回溯调整"。

## 二、三个核心设计变更

### 2.1 SOP：从"按 preset 挑 yaml"到"base + fragments 组合"

**旧**：每个 preset 一份 yaml：`backend/sop/web.yaml`、`ue5-game.yaml`、`godot-game.yaml` ...

**新**：基础 SOP + Stage Fragments 组合装配。

```yaml
# backend/sop/_core.yaml  ← 所有项目共用的基础骨架
base_stages:
  - design:        { agent: ProductAgent,   action: DesignAction }
  - architecture:  { agent: ArchitectAgent, action: DesignArchitectureAction }
  - dev:           { agent: DevAgent,       action: WriteCodeAction }
  - review:        { agent: ReviewAgent,    action: CodeReviewAction }
  - acceptance:    { agent: ProductAgent,   action: AcceptanceReviewAction }
  - deploy:        { agent: DeployAgent,    action: DeployAction }

# backend/sop/fragments/  ← 每份一个条件性阶段
smoke_test.yaml:
  insert_after: dev
  priority: 100
  required_traits: { any_of: [category:service, category:app] }
  stage: { agent: TestAgent, action: SmokeTestAction }

engine_compile.yaml:
  insert_after: dev
  priority: 90
  required_traits: { all_of: [engine:ue5] }
  stage: { agent: DevAgent, action: UECompileCheckAction }

asset_gen.yaml:
  insert_before: dev
  priority: 100
  required_traits: { all_of: [category:game] }
  stage: { agent: ArtistAgent, action: GenerateAssetsAction }

play_test.yaml:
  insert_after: dev
  priority: 90
  required_traits: { all_of: [category:game] }
  stage: { agent: TestAgent, action: PlaytestAction }

multiplayer_stress.yaml:
  insert_after: play_test
  priority: 50
  required_traits: { all_of: [multiplayer:true] }
  stage: { agent: TestAgent, action: NetworkStressTestAction }

i18n_check.yaml:
  insert_after: dev
  priority: 80
  required_traits: { any_of: [i18n:required] }
  stage: { agent: DevAgent, action: I18nCompletenessCheckAction }
```

**Runtime 组装流程**：
1. 读 `_core.yaml` 拿 `base_stages`
2. 扫 `fragments/` 目录所有 yaml
3. 按项目 `traits` 过滤出符合 `required_traits` 的 fragments
4. 按 `insert_after` / `insert_before` 插入 base（同插入点按 `priority` 降序）
5. 产出 effective SOP 给 orchestrator

**收益**：加一个新 trait（如 `monetization:iap`）→ 只需新增一份 fragment yaml → 所有声明了此 trait 的项目自动多出 IAP 相关阶段，**无需修改任何 preset/yaml**。

**代价**：
- Fragment 依赖关系（`multiplayer_stress` 依赖 `play_test` 先存在）需要通过 `insert_after` 显式表达
- 同插入点冲突用 `priority` 消歧（明确优先级比隐式合并更好调试）
- 不做复杂 merge 语义（附录 A 曾采纳的 `include: _base` 也被此方案替代）

### 2.2 Preset 从"项目类型"降级为"快捷组合"

**旧定义**：preset = 项目类型（ue5-game / webapp），每个 preset 对应一份 SOP yaml，用户建项目必须选。

**新定义**：preset = **已知好用的 trait 组合的命名快捷方式**，不对应任何 yaml，不是必选路径。

```yaml
# backend/skills/rules/presets.yaml
presets:
  # 系统内置，常见组合
  ue5-game:
    label: "UE5 客户端游戏"
    traits: [platform:desktop, category:game, engine:ue5, lang:cpp]

  web-app:
    label: "网页应用"
    traits: [platform:web, category:app]

  wechat-miniapp:
    label: "微信小程序"
    traits: [platform:wechat, platform:web, category:app]

  # 用户保存的个人常用
  my-mp-godot:
    label: "我的多人 Godot 模板"
    traits: [platform:desktop, category:game, engine:godot4, multiplayer:true]
    source: user_saved
```

**preset 的三种使用方式**：
- **LLM 建议**：对话中识别到用户描述跟某 preset 高度匹配 → 询问"要不要一键用 ue5-game 这组"
- **UI 快捷按钮**：建项目卡片里放常用 preset 作为"一键填充"
- **用户保存**：在卡片上点"保存为我的模板"，未来自己或队友可快速复用

preset **不再决定 SOP / agent / skill / mcp**，这些全部从 traits 派生。

### 2.3 UX 核心：实时展示"traits → 组装结果"

trait-first 最大的 UX 价值是**让配置决策过程可见**：

```
User: 帮我做一个多人在线对战游戏

      🏷 multiplayer:true
      🏷 realtime:true

      还需要定：
      1️⃣ 平台？（网页 / PC / 手机 / 小程序）
      2️⃣ 引擎？（UE5 / Godot / Unity / 自研 canvas）

User: PC 端，Godot 引擎，先做原型
AI  : 明白。当前 traits（6 个）：
      🏷 category:game
      🏷 multiplayer:true
      🏷 realtime:true
      🏷 platform:desktop    ← 新
      🏷 engine:godot4       ← 新
      🏷 lifecycle:prototype ← 新

      📋 SOP 会组装成 7 阶段：
        design → architecture → asset_gen → dev →
        engine_compile(godot) → play_test → multiplayer_stress → deploy

      🤖 启用 Agents：
        ProductAgent / ArchitectAgent / DevAgent /
        ArtistAgent(新) / TestAgent / ReviewAgent / DeployAgent

      🎓 注入 Skills：
        game-design (any:game)
        godot-dev (engine:godot4)
        godot-multiplayer (engine:godot4 + multiplayer:true)
        godot-export (engine:godot4 + lifecycle:prototype)

      🔌 启用 MCP：
        godot-mcp（若已装）

      Git 仓库给一下？
User: https://github.com/me/mp-game.git

      [✅ 确认创建] [➕ 加关键词] [✗ 取消]
```

每加/减一个 trait，AI **立刻重新计算**组装结果。这要求后端提供 **preview API**（§5）供前端实时查询。

## 三、Skill 方案需要的 7 个调整

范式从 preset-first 变成 trait-first 之后，附录 A 的 skill 方案有 7 个点要改。分三类：

### 3.A 加载/匹配实现层（实现级小改）

#### 3.A.1 Skill 计算从"一次性"变"按需"

现 `SkillLoader._agent_prompt_cache` 按 `agent_type` 单维度缓存（Agent 启动时算一次）。trait-first 下 traits 会变，缓存 key 改成复合：

```python
# 旧
def build_prompt_for_agent(self, agent_type: str) -> str:
    if agent_type in self._agent_prompt_cache:
        return self._agent_prompt_cache[agent_type]

# 新
def build_prompt_for_agent(self, agent_type: str, traits: list = None) -> str:
    cache_key = (agent_type, tuple(sorted(traits or [])))
    if cache_key in self._agent_prompt_cache:
        return self._agent_prompt_cache[cache_key]
```

读 md 文件本身不贵，缓存主要是防重复正则匹配；traits 变动时自然失效。

#### 3.A.2 Action 的 traits 校验从"启动期"改"fragment 激活期"

附录 A §A.3.2 说"SOP loader 加载 yaml 时校验"。trait-first 下 SOP 是 fragments 组合，不是静态 yaml。校验改成：

- **运行期**：fragment 自带 `required_traits`，traits 不匹配 → fragment 不激活 → 不会引用到 Action，不需要运行期 assert
- **构建期 lint**：每个 fragment 的 `required_traits` 应当是它引用的 Action 的 `available_for_traits` **的子集或交集**，否则 lint warning（不阻塞，但提示作者可能写错）

#### 3.A.3 Skill 注入时机：BaseAgent 拿 traits 的路径

`BaseAgent.run_action` 前当前只拿 `self.agent_type`。要改成：

```python
# BaseAgent.run_action 改
async def run_action(self, action_name, context):
    traits = await self._get_project_traits(context.get("project_id"))
    skills_prompt = skill_loader.build_prompt_for_agent(self.agent_type, traits=traits)
    token = _current_skills.set(skills_prompt)
    try:
        return await self._actions[action_name].run(context)
    finally:
        _current_skills.reset(token)
```

工单运行时的 traits 要做**快照**（开始执行时的值 snapshot 到工单），避免跑一半 traits 变了导致行为跳动。

### 3.B Skill 内容和分类层（设计级重想）

#### 3.B.1 traits_match 的最佳实践要写进规范

附录 A 只给了 `all_of/any_of/none_of` 三元组语法，没说什么时候用哪个。trait-first 下使用场景多样，必须明确约定：

| Trait 类型 | 示例 | 推荐 matcher |
|---|---|---|
| 技术栈硬约束 | `engine:ue5`、`framework:fastapi` | **all_of** — 不是 UE5 就别注入 UE skill |
| 功能特性 | `multiplayer:true`、`i18n:required` | **any_of** — 项目只要有这个特性就需要 |
| 排斥条件 | "Playwright 不适用非 web" | **none_of: [platform:desktop, platform:wechat]** |

**放进 `trait_taxonomy.yaml` 注释** 和 SkillLoader 文档里，否则 skill 作者乱用（比如把 `multiplayer` 写成 all_of，其他功能都得在才注入，太严格）。

#### 3.B.2 现有 4 个 skill 归位重新评估

附录 A §A.1.1 那张归位表在 trait-first 下要收一收：

| skill | 附录 A 结论 | 本文修正 |
|---|---|---|
| `git-workflow` | 提升到 `rules/git.md` (alwaysApply) | ⚠️ **修正**：单机 UE 项目用 P4 不用 git，提升到 alwaysApply 会污染非 git 项目。改成 **skill 保留**，加 `traits_match: any_of: [vcs:git]`，新增 `vcs:*` 维度（`vcs:git` / `vcs:p4` / `vcs:svn`）|
| `react-dev` | traits_match + paths | ✅ 不变 |
| `fastapi-dev` | traits_match | ✅ 不变 |
| `playwright-e2e` | traits_match: platform:web | ✅ 不变 |
| （新增）"语言一致性/命名/文档规范" | - | 🆕 **这才是真正的 `rules/global.md`**，跟技术栈完全正交，alwaysApply |

**原则调整**：rules 层**只放跟技术栈彻底无关的全局约束**（语言一致性、命名规范、文档要求、保密/安全红线）。"用了 git 才成立"的规范降回 skill 层，引入新维度表达。

#### 3.B.3 新增功能性 trait 带来的 skill 缺口

附录 B 扩充的 trait 维度（`multiplayer`、`realtime`、`i18n`、`offline-first`、`a11y`、`in-app-purchase`...）激活时应当注入的 skill 目前**都不存在**：

- `multiplayer-networking`（TCP/UDP vs websocket、同步策略、延迟补偿）
- `i18n-patterns`（字符串提取、RTL 布局、日期格式）
- `offline-first-patterns`（缓存策略、同步冲突解决）
- `realtime-ui`（帧率管理、输入延迟、动画）
- `a11y-patterns`（ARIA、键盘导航、屏幕阅读器）
- `iap-integration`（平台差异、验证、订阅管理）

**决策**：本次不做，但 `trait_taxonomy.yaml` 要**承诺"声明了这个维度就会有对应 skill 填上"** —— 在 TODO.md 里记一笔，每个维度对应一个待开发 skill。避免 trait 声明了但 LLM 实际得不到相关知识的"承诺空转"。

#### 3.B.4 Skill 的"特化压制通用"机制（新需求）

trait-first 下会出现**多个 skill 同时命中但应当互斥**的情况：

- `multiplayer:true` + `engine:godot4` 命中 → 应该注入 `godot-multiplayer`
- `multiplayer:true` 单独命中 → 应该注入 `multiplayer-networking`（通用）
- 两者**不应同时注入**（Godot 特化版已覆盖通用知识）

附录 A 的 priority 字段只是排序用，不做压制。需要引入 **skill 分组**：

```yaml
godot-multiplayer:
  group: multiplayer-knowledge  # 同组内只留 priority 最高的
  priority: 100
  traits_match:
    all_of: [engine:godot4, multiplayer:true]

multiplayer-networking:
  group: multiplayer-knowledge
  priority: 50
  traits_match:
    any_of: [multiplayer:true]
```

Loader 聚合时：同 `group` 下只激活 priority 最高的一个 skill，其他被压制（但会在 preview 里显示"这个 skill 被 X 压制了"供调试）。

### 3.C 新增能力层（trait-first 引入的新工程）

#### 3.C.1 Preview Assembly API

前端要实时展示"当前 traits → 组装结果"需要一个只读端点：

```http
POST /api/projects/preview-assembly
Body: { "traits": ["platform:web", "category:game", "multiplayer:true", ...] }

Response: {
  "sop": {
    "stages": [
      {"name": "design", "agent": "ProductAgent", "action": "DesignAction", "from": "base"},
      {"name": "asset_gen", "agent": "ArtistAgent", "action": "GenerateAssetsAction",
       "from": "fragment:asset_gen", "required_traits": "all_of:[category:game]"},
      ...
    ],
    "fragments_activated": ["asset_gen", "play_test"],
    "fragments_excluded": ["engine_compile", "multiplayer_stress"]
  },
  "agents": {
    "active": ["ProductAgent", "ArchitectAgent", "DevAgent", "ArtistAgent", ...],
    "excluded": []
  },
  "skills_by_agent": {
    "DevAgent": [
      {"id": "react-dev", "matched": "any_of:[framework:react]"},
      {"id": "global.md", "matched": "alwaysApply (rule)"}
    ],
    "ProductAgent": [{"id": "game-design", "matched": "any_of:[category:game]"}]
  },
  "mcps": {
    "enabled": ["filesystem"],
    "pending": [
      {"id": "godot-mcp", "reason": "需要装 Godot MCP Server 才可启用"}
    ]
  },
  "warnings": [
    "缺少 engine:* trait，asset_gen 阶段不知道调用哪个引擎 MCP"
  ]
}
```

**实现**：复用 `skill_loader` / `sop_loader` / `agent_filter` / `mcp_filter` 的匹配逻辑，拼成一个 preview payload。**纯只读**，不落库不触发任何副作用。

#### 3.C.2 Preview 调用时机

- **LLM 调 `confirm_project(traits=...)` 后**：前端渲染确认卡片前，先打 preview API → 在卡片里嵌入组装预览
- **用户在「项目特征」页改 traits 时**：每次保存后立刻打 preview 刷新展示
- **LLM 对话中调 `analyze_traits(traits_delta=...)` 探索性工具**（可选）：让 LLM 自己能查"加这个 trait 会改变什么"再做决策，不是盲推

第三条是可选的提升，不做也可以。

## 四、Trait Taxonomy 扩展

trait-first 下 taxonomy 要**从枚举扩到功能维度**：

```yaml
# backend/skills/rules/trait_taxonomy.yaml

# === 原有维度（结构性识别） ===
platform:
  - web / wechat / desktop / mobile / server / cli

category:
  - app / game / service / library / doc

engine:
  - ue5 / ue4 / godot4 / godot3 / unity / cocos / none

lang:
  - python / javascript / typescript / cpp / csharp / gdscript / rust / go

framework:
  - react / vue / svelte / fastapi / django / flask / express / ...

# === 新增维度（功能性识别，这是 trait-first 的关键） ===

features:        # 功能特性
  - multiplayer
  - realtime
  - offline-first
  - p2p
  - turn-based
  - ai-opponent
  - leaderboard
  - in-app-purchase
  - push-notification

compliance:      # 合规要求
  - enterprise
  - wechat-review
  - ios-appstore
  - gdpr
  - hipaa

quality:         # 质量约束
  - i18n            # 国际化必需
  - a11y            # 无障碍
  - offline         # 可离线运行
  - pwa             # PWA

lifecycle:       # 生命周期阶段
  - prototype
  - mvp
  - production
  - maintenance

vcs:             # ← 新加的版本控制维度（解决 3.B.2 的 git-workflow 归位）
  - git
  - p4
  - svn
  - hg

# === 约束规则 ===
constraints:
  # 推荐至少有 category（用来决定主流程），其他都可推迟累积
  - soft_required: [category]

  # 若 category=game，推荐有 engine
  - if_any_of: [category:game]
    recommend: [engine:*]

  # 若 category=service，推荐有 framework
  - if_any_of: [category:service]
    recommend: [framework:*, lang:*]
```

**约束松绑**：附录 A 曾说"platform + category 必填"。trait-first 下改成**推荐至少有 category**，其他都可以在对话里逐步补齐，或建完项目后再加。

## 五、Preview Assembly API 详细设计

### 5.1 接口契约

```
POST /api/projects/preview-assembly
Content-Type: application/json

Request:
{
  "traits": string[],                // required
  "include_rules": boolean,          // default true, 预览里要不要列出 always-apply 的 rules
  "highlight_changes": string[] | null, // 可选，列出相比上次哪些 trait 是新加的，UI 好高亮
}

Response 200:
{
  "effective_config": {
    "sop": { ... },
    "agents": { ... },
    "skills_by_agent": { ... },
    "mcps": { ... },
  },
  "warnings": string[],              // 缺什么 / 冲突
  "suggestions": [                   // AI 推荐下一步问什么
    { "dimension": "engine", "reason": "category:game 但缺 engine:*" }
  ]
}
```

### 5.2 后端实现草图

```python
@router.post("/preview-assembly")
async def preview_assembly(req: PreviewRequest):
    traits = list(req.traits)

    # 1. SOP 组装
    sop = sop_loader.compose(traits)

    # 2. Agent 过滤
    agents = [
        a for a in orchestrator.all_agents.values()
        if _traits_match(a.available_for_traits, traits)
    ]

    # 3. Skill 组装（按 Agent 分别计算）
    skills_by_agent = {}
    for agent in agents:
        skills_by_agent[agent.agent_type] = skill_loader.list_applicable_skills(
            agent.agent_type, traits
        )

    # 4. MCP 过滤
    mcps = mcp_client.list_enabled_for_traits(traits)

    # 5. 警告 & 建议
    warnings = _compute_warnings(traits, sop, agents)
    suggestions = _compute_suggestions(traits)

    return {
        "effective_config": {
            "sop": sop.to_dict(),
            "agents": {"active": [a.agent_type for a in agents]},
            "skills_by_agent": skills_by_agent,
            "mcps": mcps,
        },
        "warnings": warnings,
        "suggestions": suggestions,
    }
```

### 5.3 前端消费

`doConfirmProject` 收到 `action.type === 'confirm_project'` 时：

```javascript
async function renderProjectConfirmCard(action) {
    // 调 preview API 拿组装结果
    const preview = await api('/projects/preview-assembly', {
        method: 'POST',
        body: { traits: action.traits, include_rules: true }
    });

    // 渲染卡片: 上半部分 traits chips（可点击删除）
```

## 九、对照 MagicAI 实现的跨 kernel 启示

> 参考源：`D:\A_Works\MagicAI\docs\20260423_01_AI对话驱动项目创建与流水线配置深度分析.md`
> MagicAI 是一个已在生产的 Godot 游戏 AI 平台，基于 CodeBuddy SDK；对话驱动项目创建 + 流水线配置的链路已跑通。本节从对照中提取对我们 trait-first 方案的具体借鉴。

### 9.1 总体结论：方向验证 + 三个精确借鉴

**方向验证**：我们已经比 MagicAI 走远一代：

| 维度 | MagicAI | 我们 trait-first |
|---|---|---|
| Skill 机制 | `inject_to: [agent]` 单维度 | 三层过滤（Agent/project/file）+ 动态组装 |
| SOP | 硬编码 6 Phase 写死在 game_director.md | trait 驱动的 fragments 组合器 |
| 项目类型 | 4 个 Godot 模板，硬分类 | 多维 traits，表达 DAG 继承关系 |
| 配置透明度 | 黑盒 | Preview Assembly API，组装过程可见 |

**三个精确借鉴**：他们的 3 项成熟实现可直接并入我们的 roadmap。

### 9.2 借鉴 1：模板匹配算法 → preset 推荐落地（⭐⭐⭐）

**来源**：`match_template_by_description()` (`godot_models.py:286-373`)

**他们做什么**：根据用户自然语言对 4 个 Godot 模板评分 → 选最匹配的那个。

```
用户："做一个马里奥风格的横版平台跳跃游戏"
评分：
  - 2d_platformer: "平台"+5 "跳跃"+5 "马里奥"+5 "横版"+5
                   + 模板名分词命中 "平台跳跃"+6 = 26 分
  - 2d_default:    0 分
→ 选中 2d_platformer

维度冲突惩罚：用户说"3D"但模板是 2D → -20（避免模糊推荐）
阈值：≥5 才返回，否则 fallback 到 2d_default
```

**对我们的价值**：正是 `20260423_02` 里 preset 降级后"怎么给用户推荐 preset"的空白。

**采纳方案**：
1. `presets.yaml` 每个 preset 加 `keywords: [...]` 字段
2. 新工具 `match_preset_by_message(user_text) -> (preset_id, score, reasons)`
3. ChatAssistant 对话里识别意图时调此工具，如命中就提 preset 建议："我看你的描述跟 ue5-game 匹配度 26 分，要不要用这个 preset？"
4. **维度冲突惩罚要保留** —— "3D" 撞 "engine:godot4 + preset 是 2D-only" 类型的 -20 扣分机制

**落点**：Phase A 扩 0.5 天。

### 9.3 借鉴 2：MODIFY vs NEW_PROJECT 模式 → 引入 `ticket_type:*` trait（⭐⭐⭐）

**来源**：`agent_session.py:203-231` + `agent_session_config.py:60-124`

**他们做什么**：按用户消息意图（新建 vs 修改）和关键词（速度/参数/美术/颜色/...）**只激活相关 Agent**，而不是每次都跑全部流水线。

```python
modify_indicators = ["修改","调整","添加","删除","bug","speed","size","color",...]
new_indicators    = ["新游戏","新项目","创建","做一个","new game","build a",...]

if has_modify and not has_new:
    # 按关键词打分激活 developer/artist/designer/qa/publisher 中相关的 1-2 个
    _agent_filter = infer_agents_for_modify(message)
```

**对我们的价值**：trait-first 的 SOP fragments **只解决了"项目初建时的流程"**，但"日常维护小改"场景没覆盖。对话说"调整一下这个按钮颜色"不应触发整条 SOP。

**采纳方案**：**工单类型也是 trait 的一个维度**。

```yaml
# 扩充 trait_taxonomy.yaml
ticket_type:
  - feature       # 新功能 → 走完整 SOP
  - bug-fix       # 只调 DevAgent + TestAgent
  - refactor      # 只调 DevAgent + ReviewAgent
  - polish        # 只调 DevAgent（小调整，skip review）
  - rollback      # 只调 DeployAgent
```

SOP fragments 不仅 `required_traits` 过滤（项目级），也 `required_ticket_type` 过滤（工单级）：

```yaml
# backend/sop/fragments/review.yaml
required_ticket_type: { any_of: [feature, refactor] }   # bug-fix/polish 跳过 review
required_traits: { ... }
```

ChatAssistant 识别用户意图（通过类似 MagicAI 的 indicator keyword 词表）→ 给工单打 `ticket_type` trait → orchestrator 组装时按此过滤 fragments。

**落点**：Phase C 扩 0.5 天（trait_taxonomy 加维度 + sop_loader.compose 参数扩充）。

### 9.4 借鉴 3：历史压缩两级策略 → 对话上下文收紧（⭐⭐⭐）

**来源**：`build_conversation_history_section()` (`godot_prompt_config.py:271+`)

**他们做什么**：Windows `CreateProcess()` 参数上限 ~32K 字符，SDK 把 system_prompt 当 CLI 参数传，超了就 `[WinError 206]`。两级压缩策略：

```
keep_recent_n=6             最近 6 条全文保留（每条 ≤ max_msg_chars=4000）
compress_older=True         更早的压缩到 ≤ older_msg_chars=800 / 条（去代码块保语义）
max_history_chars=8000      历史段总硬上限
current_prompt_size > 20K   动态收紧：系统 prompt 大了历史预算自动减小
```

**对我们的价值**：我们走 Anthropic API 有 200K context，不会撞 32K 硬边界，但：
- v0.16.4 的 `content.replace is not a function` bug 本质就是没处理好 tool_use 消息
- 复杂多轮对话的 token 费用成正比于历史长度，粗暴 `history[-20:]` 浪费
- v0.16.5 上线的全局聊天同样问题

**采纳方案**：把 MagicAI 这套两级策略直接移植到 `ChatAssistantAgent._assemble_messages` 里，参数可配置：

```python
@staticmethod
def _compress_history(
    history: List[Dict],
    keep_recent_n: int = 6,
    max_recent_chars: int = 4000,
    older_msg_chars: int = 800,
    max_history_chars: int = 8000,
) -> List[Dict]:
    """两级压缩对话历史。"""
    # 1. 后 N 条：截断到 max_recent_chars
    # 2. 前面的：摘要到 older_msg_chars（去代码块、去工具消息 detail）
    # 3. 总长超 max_history_chars → 继续删最老的
```

附加价值：这个 helper 可以同时用在 `ChatAssistantAgent.chat()` 和 `chat_global()`，全局聊天也受益。

**落点**：Phase A 扩 0.5 天，跟 confirm_project schema 改动一起做。

## 十一、TL;DR

**问题**：preset-first 模型把"项目类型"当必选入口，锁死用户路径。真实需求是**累积关键词**（traits），组合维度 ≥ 6 个，无法枚举。

**方案**：**trait-first + 动态组装**
- 用户通过对话累积 traits，不选项目类型
- SOP 从"按 preset 挑 yaml"改为**base + fragments 动态组合**（`_core.yaml` + `fragments/*.yaml` 按 required_traits 过滤插入）
- preset 降级为**命名的 trait 组合快捷方式**（可选，非必选）
- 新增 **Preview Assembly API** 让前端实时展示"traits → 组装结果"

**Skill 方案的 7 个调整**：
- A 类（实现）：缓存 key 改 traits hash；校验时机从启动期改 fragment 激活期；注入路径拿 traits
- B 类（设计）：traits_match 最佳实践写规范；`git-workflow` 加 vcs 维度不升 rules；新增功能性 trait 对应的 skill 缺口在 TODO 列出；Skill 分组 + 特化压制（Godot 多人压制通用多人）
- C 类（新能力）：Preview Assembly API

**工程估时**：~14.5 天（A/B/C/C' 一批做 11.5 天跑通核心垂直切片，D/E/F 二批 3 天补全），并入了 MagicAI 案例启发的 3 条精确借鉴（preset 关键词推荐 + ticket_type 维度 + 历史压缩）

**对历史的覆盖**：`20260423_01` 附录 A 的 SOP 多 yaml 方案整块作废；本文是最终形态。跟 MagicAI 同期方案对照确认 trait-first 方向比他们先进一代（动态组装 vs 硬编码 6 Phase）。
