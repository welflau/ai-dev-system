# DevAgent UE 项目自测方案

> 日期：2026-04-26
> 起因：用户观察到 UE 项目 DevAgent 写完代码后的自测环节几乎是空的；TestFPS 实战暴露了 5 个全静态可发现的错都没被拦住
> 版本上下文：v0.19.x（CI/CD 合并改造刚完成，Phase A-E 30/30 通过）

---

## 1. 现状实事求是

### 1.1 DevAgent 当前的自测流程

`backend/actions/self_test.py` 对不同项目类型的覆盖：

| 项目类型 | 当前自测内容 | 有用度 |
|---|---|---|
| Web（platform:web） | ESLint + Playwright 截图 + DOM 断言 | ★★★★ |
| 服务（category:service） | pytest + 语法 compile + HTTP smoke | ★★★ |
| **UE（engine:ue5/4）** | **只做磁盘落地验证**（文件写进去了没 + Git add 成功） | ★ |
| 其他 | 基本同服务 | ★★ |

### 1.2 DevAgent UE 自测空转的代价

v0.18 架构决策是把"UBT 编译"挪到独立 `engine_compile` stage，这在 SOP 层面合理（单一职责），但 **DevAgent 内部** 对 UE 代码的语义完全无认知：

```
当前流程：
DevAgent.develop (写 C++ + 磁盘检查) ──────▶ development_done
                  ↓ SOP 推进
          engine_compile stage (跑 UBT 3-5 min) ──▶ failed
                  ↓ reject_goto
          DevAgent.fix_issues (Reflexion) ──▶ 重写 ──▶ 再 UBT 3-5 min ──▶ ...
```

后果：**一次新手错误 → 至少 5-10 分钟 + 2 次 LLM 调用 + 可能的循环**。

### 1.3 TestFPS 实战反哺的具体 bug 清单

今天排查 TestFPS UBT 编译，一次性暴露 5 个错，**全部是静态可发现的**：

| # | 错误 | 现象 | 是否静态可查 |
|---|---|---|---|
| 1 | `UFUNCTION()` 写在 override 之上 | UHT Error: "Override of UFUNCTION 'OnRep_Score' in parent 'APlayerState' cannot have a UFUNCTION() declaration above it" | ✅ 父类 OnRep_* 白名单比对 |
| 2 | `IncludeOrderVersion.Unreal5_1` 废弃 | warning CS0618（5.4 会删） | ✅ Target.cs 版本 vs 引擎版本对照表 |
| 3 | `#include "FPSCharacterBase.h"` 缺路径前缀 | fatal C1083: 无法打开包括文件 | ✅ 文件名在本模块 Source/ 里全路径搜 |
| 4 | `HandleMatchIsWaitingToStart/HasStarted/HasEnded` 标 `override` 但父类 `AGameModeBase` 没这组函数 | error C3668 ×3 | ✅ 父类 virtual 方法白名单 |
| 5 | 用 `UCapsuleComponent*` 但没 `#include "Components/CapsuleComponent.h"` | error C2027 undefined + C2664 | ✅ UE 常用类型 → 必需 header 映射表 |

**零例外——5/5 都能用纯静态规则抓。**

## 2. 三层自测架构设计

### 2.1 分层原则

按"**耗时 × 覆盖率**"权衡：

| 层 | 耗时 | 抓错覆盖率 | 成本 |
|---|---|---|---|
| **Layer 1 静态预检** | < 1 秒 | 80%（新手错）| 零成本，纯 Python |
| **Layer 2 UBT SingleFile 预编译** | 30-90 秒 | 95%（加上类型错 / 重载错 / 模板实例化错）| 跑 UBT 子集 |
| **Layer 3 Full engine_compile + play_test** | 3-5 分钟 + 3-5 分钟 | 100% | 完整跑 |

**设计立场**：
- Layer 1 **在 DevAgent 内部强制跑**，SelfTestAction 默认开启
- Layer 2 **默认开**但 SOP config 可关（fast 模式 / 改动文件极小）
- Layer 3 **留在独立 SOP stage**，不下沉到 DevAgent

这样常见新手错在 Layer 1 被秒拦，Layer 2 顶住剩余的静态不可达问题，Layer 3 做最后兜底。

### 2.2 数据流

```
DevAgent.develop
  │
  ├─ write_code (LLM 产出 .h/.cpp/Build.cs)
  ├─ write_files_to_disk (git add)
  │
  └─ SelfTestAction.run():
       │
       ├─ Layer 1 (UE 分支触发条件: traits 含 engine:ue*)
       │    ├─ _ue_lint_uclass_genbody()
       │    ├─ _ue_lint_ufunction_override()
       │    ├─ _ue_lint_include_paths()
       │    ├─ _ue_lint_build_cs_deps()
       │    ├─ _ue_lint_uproject_modules()
       │    ├─ _ue_lint_target_cs_include_order()
       │    └─ _ue_lint_common_type_headers()
       │    → 合并成 issues[] 列表（blocking / warning 两类）
       │
       ├─ blocking issues > 0 ? → 立即返回 fail，进入 fix_issues 循环（不花 UBT 时间）
       │
       └─ Layer 2 (SOP config.ue_precompile = true)
            └─ UECompileCheckAction.run(context + single_files=changed_cpp_h)
              → UBT -SingleFile 快速预编
              → status != success → fail
```

## 3. Layer 1 详细规则（7 条）

### 3.1 规则 R1：UCLASS / USTRUCT / UINTERFACE 必须有 `GENERATED_BODY()`

正则扫 `.h`，抓每个 `UCLASS(...)` / `USTRUCT(...)` 块，往下 5 行内必须见 `GENERATED_BODY()` 或 `GENERATED_UCLASS_BODY()`。

```python
_RE_UCLASS = re.compile(r"^\s*(UCLASS|USTRUCT|UINTERFACE)\s*\(", re.MULTILINE)
_RE_GEN_BODY = re.compile(r"GENERATED_(BODY|UCLASS_BODY|USTRUCT_BODY|IINTERFACE_BODY)\s*\(")
```

误报预防：`UCLASS` 可能跨行（多个 meta），扫完后在下一个 `class` 声明前匹配 GENERATED_BODY 即可。

### 3.2 规则 R2：子类 OnRep_* 不能有 UFUNCTION() 宏

**核心白名单**（UE 5.3 APlayerState / AActor / UActorComponent 等常用父类的 OnRep 方法）：

```python
UE_PARENT_ONREP_METHODS = {
    "APlayerState": [
        "OnRep_Score", "OnRep_PlayerName", "OnRep_PlayerId",
        "OnRep_UniqueId", "OnRep_bIsInactive",
        "OnRep_PingReplicationTimestamp",
    ],
    "AActor": ["OnRep_Owner", "OnRep_Instigator", "OnRep_ReplicatedMovement"],
    "ACharacter": ["OnRep_IsCrouched", "OnRep_ReplicatedBasedMovement"],
    "APawn": ["OnRep_PlayerState"],
    # ... 其他常见父类
}
```

规则：子类 `.h` 里的 `UFUNCTION()` 下一行是 `void OnRep_XXX()` 时，若 `XXX` 出现在继承链上某父类的 OnRep 方法列表里 → `blocking=True`，建议"去掉 UFUNCTION()，加 virtual/override"。

实现细节：
- 解析子类声明 `class TESTFPS_API AFoo : public AParent` 找到 `AParent`
- 按继承链（`AParent` 自己 + 向上查——但一层足够因为 UE 的 OnRep 都声明在直接父类）

### 3.3 规则 R3：#include 路径可定位

扫每个 `.cpp` 和 `.h` 的 `#include "<relative>.h"`（双引号版本，尖括号的跳过）：
- 路径必须能在以下位置找到：
  - 本模块 `Public/` 和 `Private/`（含所有子目录递归）
  - 其他本项目模块的 `Public/`
  - UE Engine 头文件路径（需维护一张常见 header 列表，如 `CoreMinimal.h` / `Engine/World.h` 都合法）
- 找不到但在**同模块内某个子目录**存在的 → `blocking=True`，给出建议路径

```python
# 伪代码
for include in cpp_includes:
    if _find_in_engine_known_headers(include):
        continue
    if _find_in_module_include_paths(include, module_path):
        continue
    # 在本模块 Source 下全路径搜
    matches = list(module_path.rglob(Path(include).name))
    if matches:
        rel = matches[0].relative_to(public_or_private_root)
        issues.append({
            "rule": "R3-include-path",
            "blocking": True,
            "msg": f'#include "{include}" 找不到；应为 "{rel.as_posix()}"',
        })
    else:
        issues.append({"rule": "R3-include-path", "blocking": True,
                       "msg": f'#include "{include}" 找不到头文件'})
```

### 3.4 规则 R4：Build.cs 模块依赖有效

维护一张 UE 5.x 常见 **200+ 内置模块** 白名单（可以扫一次引擎 `Engine/Source/Runtime/*/` 直接列全，或维护精选列表）：

```python
UE_BUILTIN_MODULES_5_3 = {
    "Core", "CoreUObject", "Engine", "InputCore", "EnhancedInput",
    "Slate", "SlateCore", "UMG", "AIModule", "GameplayTasks",
    "NavigationSystem", "UnrealEd", "ToolMenus", "EditorStyle",
    "PropertyEditor", "KismetCompiler", "BlueprintGraph",
    # ... 200+
}
```

Build.cs 里的 `PublicDependencyModuleNames` / `PrivateDependencyModuleNames`：
- 每个字符串要么在白名单里，要么是项目本地模块（Source/ 下有同名 `<Name>.Build.cs`）
- 否则 `blocking=True`，提示"模块 X 不存在或未启用插件"

### 3.5 规则 R5：.uproject Modules ⊇ Source/ 模块

扫 `Source/` 下所有 `<Name>.Build.cs`，跟 `.uproject` 的 `"Modules"` 数组比对：

- Source/ 有 FPSGame.Build.cs 但 .uproject Modules 没声明 FPSGame → `blocking=True`
- 反之 .uproject 声明了但 Source/ 没对应 Build.cs → `warning`（可能是插件引用）

对应今天 TestFPS 里 FPSGame / WeaponSystem 两个未注册模块的坑（虽然 UBT 在 UHT 先挂没跑到这步暴露，但它确实是雷）。

### 3.6 规则 R6：Target.cs IncludeOrderVersion 兼容

维护 UE 版本 → 可用 `IncludeOrderVersion` 的映射：

```python
UE_INCLUDE_ORDER_VERSION = {
    "5.0": ["Unreal5_0"],
    "5.1": ["Unreal5_0", "Unreal5_1"],
    "5.2": ["Unreal5_0", "Unreal5_1", "Unreal5_2"],
    "5.3": ["Unreal5_1", "Unreal5_2", "Unreal5_3"],  # 5.3 里 5_0 被 deprecate
    "5.4": ["Unreal5_2", "Unreal5_3", "Unreal5_4"],  # 5.4 里 5_1 被 remove
}
```

- 版本不在列表 → `blocking=True`（引擎会拒绝编译）
- 被 deprecated 但还能编 → `warning`

### 3.7 规则 R7：常用类型必需 header

维护"类型名 → 必需 `#include`"映射表：

```python
UE_TYPE_REQUIRED_HEADERS = {
    "UCapsuleComponent": "Components/CapsuleComponent.h",
    "UStaticMeshComponent": "Components/StaticMeshComponent.h",
    "USkeletalMeshComponent": "Components/SkeletalMeshComponent.h",
    "UCharacterMovementComponent": "GameFramework/CharacterMovementComponent.h",
    "USpringArmComponent": "GameFramework/SpringArmComponent.h",
    "UCameraComponent": "Camera/CameraComponent.h",
    "UInputComponent": "Components/InputComponent.h",
    "UEnhancedInputComponent": "EnhancedInputComponent.h",
    "UInputAction": "InputAction.h",
    "UInputMappingContext": "InputMappingContext.h",
    # ... 常用 50+
}
```

规则：`.cpp` 里用到某类型（不是 forward decl） 但没对应 `#include` → `blocking=True`。

"用到"的判定：
- `NewObject<TypeName>` / `CreateDefaultSubobject<TypeName>` / `Cast<TypeName>`
- `TypeName* Var` 的**实际成员方法调用**（`Var->SomeMethod()` 或 `Var->SomeField`）

`.h` 里只做 forward decl（`class UCapsuleComponent;`）不触发，只要 `.cpp` include 了就 OK。

### 3.8 规则扩展性

规则存 `backend/actions/ue_lint/rules.py` 每条独立函数；主入口 `_ue_static_lint(files, repo_path)` 依次调，收集 issues 列表。

未来补新规则成本低：加一个函数 + 注册进 dispatch 表。

## 4. Layer 2 UBT SingleFile 预编译

### 4.1 UBT 命令

```
UnrealBuildTool.exe <TargetEditor> Win64 Development <.uproject>
  -SingleFile=<DevAgent 改过的某个 .cpp 绝对路径>
  -NoHotReload -waitmutex
```

`-SingleFile` 是 UBT 内置参数，跑"只编这个 .cpp + 它的依赖头" 的子集。首次仍需 shader 编（如果没 cache），后续改动亚 2 分钟。

多个文件改动：UBT 支持多次 `-SingleFile` 或把"改动集"合并成最小编译单元，实现时选第一个 .cpp 就够（抓主要错即可，不求覆盖全改动）。

### 4.2 复用 UECompileCheckAction

加一个参数：

```python
# actions/ue_compile_check.py
async def run(self, context):
    single_files = context.get("single_files") or []
    # ... 原有逻辑
    cmd = [str(ubt), target_name, platform, config_name, str(up), "-waitmutex", "-NoHotReload"]
    for sf in single_files[:1]:  # 只取第一个，避免 -SingleFile 多次参数
        cmd.append(f"-SingleFile={sf}")
    # ... 剩余逻辑不变
```

SelfTestAction 调用时：

```python
changed_cpp = [f for f in files_written if f.endswith((".cpp", ".h"))]
if changed_cpp:
    result = await UECompileCheckAction().run({
        **context,
        "single_files": [str(repo_path / changed_cpp[0])],
        "timeout_seconds": 120,  # 单文件 2 分钟足够
    })
```

### 4.3 SOP config 开关

```yaml
# base SOP

  config:
    ...
    ue_precompile: true          # Layer 2 开关，默认开
    ue_precompile_timeout: 120
```

## 5. SelfTestAction 的 UE 分支实现

`backend/actions/self_test.py` 加新分支：

```python
async def _self_test_ue(self, context):
    """v0.19.x+: UE 项目自测"""
    files_written = context.get("files_written") or []
    repo_path = Path(context.get("repo_path") or "")
    traits = context.get("traits") or []
    sop_cfg = context.get("sop_config") or {}

    issues: List[Dict] = []

    # Layer 1 静态预检（必跑）
    from actions.ue_lint.rules import run_all_rules
    l1_issues = run_all_rules(files_written, repo_path, context)
    issues.extend(l1_issues)

    blocking = [i for i in issues if i.get("blocking")]
    warnings = [i for i in issues if not i.get("blocking")]

    # 有 blocking 立即 fail（省 UBT 时间）
    if blocking:
        return {
            "status": "fail",
            "summary": f"UE 自测失败：{len(blocking)} blocking / {len(warnings)} warning",
            "issues": issues,
            "phase": "layer1_static",
        }

    # Layer 2（SOP 开关）
    if sop_cfg.get("ue_precompile", True):
        changed_cpp = [f for f in files_written if f.endswith((".cpp", ".h"))]
        if changed_cpp:
            from actions.ue_compile_check import UECompileCheckAction
            r = await UECompileCheckAction().run({
                **context,
                "single_files": [str(repo_path / changed_cpp[0])],
                "timeout_seconds": sop_cfg.get("ue_precompile_timeout", 120),
            })
            d = r.data or {}
            if d.get("status") != "success":
                return {
                    "status": "fail",
                    "summary": f"UE SingleFile 预编译失败（{len(d.get('errors') or [])} errors）",
                    "issues": issues,
                    "precompile": d,
                    "phase": "layer2_precompile",
                }

    return {
        "status": "pass",
        "summary": f"UE 自测通过 ({len(warnings)} warnings)",
        "issues": issues,
        "phase": "passed",
    }
```

主 `run()` 按 traits 分流到 `_self_test_ue` / `_self_test_web` / `_self_test_generic`。

## 6. 与 Reflexion / 现有流程的协同

Layer 1 / Layer 2 失败的处理一致：
- 返回 `status=fail` + 结构化 `issues` 列表
- DevAgent.develop 捕获 self_test.status=fail → ticket.status 不转 development_done，而是转 self_test_failed
- SOP 派生 `self_test_failed → DevAgent.fix_issues`（复用 v0.18 reject_goto 派生链）
- ReflectionAction 新增 `self_test_failed` 分支，prompt 里把 `issues` 结构化塞进去：

```
UE 自测失败（Layer 1 静态预检 / Layer 2 SingleFile 预编译）：

【Blocking errors】（必须修）
  - [R2-ufunction-override] TestFPSPlayerState.h:106
      子类 OnRep_Score 重写父类 APlayerState::OnRep_Score，
      不能加 UFUNCTION() 宏。建议改成: virtual void OnRep_Score() override;

      #include "FPSCharacterBase.h" 找不到；应为 "Character/FPSCharacterBase.h"
  - ...

  - [R6-include-order] TestFPS.Target.cs:9
      Unreal5_1 在 UE 5.3 已 deprecated（5.4 会移除），建议改 Unreal5_3
  - ...

请 DevAgent 修这些问题后重写代码。
```

## 8. 风险与考虑

### 8.1 UE 版本多样性

规则里硬编码了 UE 5.3 的父类 OnRep 列表、Builtin Modules 列表、Type → Header 表。**UE 5.4 / 5.5 / 5.6 会变**。

应对：
- 每张表加 `UE_VERSION_COMPAT` 标注，按项目引擎版本动态加载
- 首版只覆盖 5.3（TestFPS 当前版本），5.x 其他版本后续补
- 留 `_skip_unknown_version` 开关，遇到没覆盖的版本降级到纯语法检查（不做跨版本白名单比对）

### 8.2 规则误报

静态规则天然会有误报场景：
- Epic 的某些特殊 UE 类型（如 `USomething`）真的 forward decl 就够（成员方法都在模板里）
- 自定义父类的 OnRep 方法不在我们的白名单内

应对：
- issue 分 `blocking` / `warning`，误报宁可当 warning
- 每条规则支持项目级 `sop_config.ue_lint_skip_rules: [R2, R5]` 关单条
- 规则内置 escape hatch：`// @ue-lint-skip R2` 注释行

### 8.3 Layer 2 首次跑慢

UBT `-SingleFile` 首次仍要 build shared PCH、link shader。首次 2-5 min，后续 30-90s。

应对：
- SOP config `ue_precompile_on_first_dev: false` —— 首次 develop 跳过 Layer 2，走到 engine_compile stage 时顺便做了 PCH
- SSE 推进度 log 避免用户以为挂了

### 8.4 规则跟 Skill pack 同步

Skill pack `unreal_cpp_dev/prompt.md` 里写的"禁止事项"要跟规则语义一致，否则"DevAgent 写完觉得对 → 自测挂"这种循环会很影响体验。

应对：每条规则的 `hint`/`suggest` 字段跟 skill pack 里的说明同一出处（规则代码里定义常量，skill pack 从常量生成——或 skill pack markdown 里直接 include 规则说明）。

## 9. 预期效果

**TestFPS 历史错误 vs 规则抓取率**：

| TestFPS 错误 | 规则 | 抓不抓 |
|---|---|---|
| UFUNCTION + override 冲突 | R2 | ✅ |
| Unreal5_1 废弃 | R6 | ✅ warning |
| FPSCharacterBase.h 路径错 | R3 | ✅ + 建议正确路径 |
| HandleMatch* override 父类错 | R4（父类 virtual 白名单的延伸） | ✅ |
| UCapsuleComponent 未 include | R7 | ✅ |

**5/5 抓到**。预计 TestFPS 这类"新手错误"的回合数从 3-5 次压到 0-1 次。

对 LLM 成本的影响：每次 Layer 1 预检省掉一次 3-5 min UBT + 一次 LLM fix_issues 反思（约 3-5 万 token）。按 AI Dev System 跑 10 个 UE 工单估，**月度省 10-20 美元 + 1-2 小时等待**。

## 10. 一句话总结

**当前 DevAgent UE 自测几乎是空的，全靠下游 stage 兜底**。引入 7 条静态规则（Layer 1）+ UBT `-SingleFile` 预编译（Layer 2），把 80-95% 新手错在 DevAgent 内部拦下，避免 3-5 分钟 UBT 跑完才发现的循环。~2 天落地，优先级高于 v0.20 UCP 等未来项。
