# Git 仓库管理流程与分支策略

> 日期: 2026-04-16 | 版本: v0.15.0

---

## 一、分支模型

```
main (生产)
  ↑ CI/CD 自动合并（develop 构建通过后）
  ↑ 禁止 Agent 直接提交
  │
develop (测试)
  ↑ 自动合并（需求所有工单完成后）
  ↑ CI 每 60s 检测新提交
  │
feat/YYYYMMDD-req-XXXXXX (开发)
  ↑ Agent 在此分支工作
  ↑ 每个需求一个独立分支
```

### 分支职责

| 分支 | 用途 | 谁写入 | 保护机制 |
|------|------|--------|---------|
| `main` | 生产代码 | 仅 CI/CD 合并 | Agent push 拦截 + 切换失败阻止执行 |
| `develop` | 集成测试 | 需求完成两步合并（develop→feat→develop） | CI 构建通过才合并到 main |
| `feat/YYYYMMDD-req-*` | 功能开发 | Agent（ArchitectAgent, DevAgent 等） | 每个需求隔离 |

## 二、完整生命周期

### 2.1 分支创建

```
用户提交需求
  ↓
ProductAgent 拆单
  ↓
Orchestrator 创建分支:
  1. 检测主分支名（main 或 master）
  2. 确保 develop 存在（从主分支创建）
  3. 切到 develop
  4. 创建 feat/YYYYMMDD-req-XXXXXX
  5. 保存 branch_name 到 requirements 表
```

**分支命名**: `feat/{日期}-req-{需求ID后6位}`
- 示例: `feat/20260416-req-75cfbf`

### 2.2 Agent 开发（feat 分支）

```
工单进入处理 → Orchestrator:
  1. 查 requirements.branch_name
  2. git clean -fd（清理 .pyc 等）
  3. git checkout feat/xxx
  4. 如果切换失败且在 main → 🛑 阻止执行
  5. 构建 context（读取已有代码）
  6. 调 Agent.execute()

Agent 完成 → write_and_commit:
  1. 写文件到仓库
  2. git add . → git commit
  3. git push origin feat/xxx
  4. push 时检查：如果在 main/master → 🛑 阻止 push

自动部署 dev 环境（端口 base+0）
```

### 2.3 需求完成 → feat 合并到 develop（两步合并规范）

```
所有工单 TESTING_DONE
  ↓
Orchestrator._check_requirement_completion:
  1. 检查所有工单状态
  2. 确保 develop 存在

  Step 1: merge develop → feat（先拉最新到 feat）
    ├─ 成功 → 继续 Step 2
    └─ 冲突 → 停止，记录错误日志，等手动解决
         （冲突在 feat 分支上解决，不污染 develop）

  Step 2: merge feat → develop（功能合入）
    ├─ 成功 → push origin develop → 标记需求 completed
    └─ 冲突 → 停止，记录错误（理论上 Step 1 已解决冲突，极少发生）
```

**为什么两步合并**：
- 直接 `feat→develop` 可能因为其他需求已经改了 develop 而冲突
- 先拉 develop 最新到 feat，在 feat 上解决冲突
- 确认 feat 包含最新 develop 内容后，再合回 develop，此时一定无冲突
- 保证 develop 始终干净可用

**合并策略**: `--no-ff`（保留合并历史，不快进）

### 2.4 CI/CD: develop → main

```
CI 调度器（每 60s）:
  1. 检测 develop HEAD 是否有新提交
  2. 有新提交 → 触发 develop_build

develop_build:
  1. checkout develop
  2. 语法检查（JS node --check / Python compile）
  3. 冒烟测试（文件存在 / 入口文件）
  4. 单元测试（pytest）
  5. 全部通过 → git merge develop → main
  6. 自动部署 test 环境（端口 base+100）

master_build（每 24h）:
  1. checkout main
  2. 集成测试
  3. 回归测试
  4. 全部通过 → 触发部署
  5. 自动部署 prod 环境（端口 base+200）
```

## 三、保护机制

### 3.1 Agent Push 拦截

```python
# git_manager.py push()
if branch in ("main", "master"):
    # 检查调用栈：如果来自 write_and_commit / _handle_git_files
    stack = traceback.format_stack()
    is_agent_push = any("write_and_commit" in f or "_handle_git_files" in f for f in stack)
    if is_agent_push:
        logger.warning("🛑 阻止 Agent 直接 push 到 %s", branch)
        return False  # 静默失败
```

**原理**: 通过调用栈检测 push 来源。Agent 的 push 经过 `write_and_commit`，被拦截。CI/CD 的 push 经过 `merge_branch`，放行。

### 3.2 分支切换失败保护

```python
# orchestrator.py process_ticket()
ok = await git_manager.switch_branch(project_id, feat_branch)
if not ok:
    current = await git_manager.get_current_branch(project_id)
    if current in ("main", "master"):
        logger.error("🛑 阻止 Agent 执行（当前在主分支）")
        return  # 不执行 Agent
```

**原理**: 切换 feat 分支失败时，如果当前在主分支，直接终止工单处理。

### 3.3 分支切换清理

```python
# git_manager.py switch_branch()
await self._run_git(repo_dir, "clean", "-fd", "--exclude=.env")  # 清理未跟踪文件
await self._run_git(repo_dir, "checkout", ".")                    # 重置工作区
await self._run_git(repo_dir, "checkout", branch_name)            # 切换分支
```

**原理**: 切换前清理 `.pyc` 等缓存文件，防止冲突。

### 3.4 合并冲突处理

```python
# git_manager.py merge_branch()
rc, out, err = await self._run_git(repo_dir, "merge", source, "--no-ff", "-m", msg)
if rc != 0:
    await self._run_git(repo_dir, "merge", "--abort")  # 冲突时回滚
    return {"success": False, "error": f"合并冲突: {err}"}
```

## 四、环境与分支对应

| 环境 | 分支 | 端口 | 部署目录 | 触发时机 |
|------|------|------|----------|---------|
| dev | feat/* (当前) | base+0 | 仓库原目录 | Agent 提交代码后 |
| test | develop | base+100 | `{repo}_test/` | develop 构建通过后 |
| prod | main | base+200 | `{repo}_prod/` | master 构建通过后 |

**端口计算**: `base = 9000 + (hash(project_id) % 100)`
- 项目 A: dev=9042, test=9142, prod=9242
- 项目 B: dev=9073, test=9173, prod=9273

**目录隔离**: test/prod 环境通过 `git clone --branch` 到独立目录

## 六、关键配置参数

| 参数 | 值 | 位置 |
|------|------|------|

| 分支命名 | `feat/{YYYYMMDD}-req-{6位ID}` | `orchestrator.py` |

| dev 端口偏移 | +0 | `deploy.py` |
| test 端口偏移 | +100 | `deploy.py` |
| prod 端口偏移 | +200 | `deploy.py` |

| push 保护 | Agent 不能 push 到 main/master | `git_manager.py` |
| 切换保护 | 在主分支时阻止 Agent 执行 | `orchestrator.py` |

## 七、异常场景处理

| 场景 | 处理 |
|------|------|
| 分支切换失败（文件冲突） | git clean -fd + git checkout . → 重试 |
| 分支切换失败且在 main | 阻止 Agent 执行，工单不处理 |
| Agent 意外在 main 上 push | push() 检测调用栈，阻止并返回 False |
| 合并冲突 | merge --abort 回滚，记录错误日志 |
| develop 构建失败 | 不合并到 main，记录失败原因 |
| master 构建失败 | 不部署 prod，记录失败原因 |
| deploy 端口被占用 | 记录警告，跳过部署 |
