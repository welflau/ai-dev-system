# ADS 项目目录规范：`.ads/` 目录设计方案

> 日期：2026-05-19  
> 背景：参考腾讯游戏 KM 文章《驾驭 AI Coding：一份面向团队的 Harness Engineering 落地规范》  
> 对标：Claude Code 的 `.claude/` 目录约定  
> 状态：设计方案，待实现

---

## 一、问题背景

### 现状：项目配置数据分散

```
项目配置数据存放位置（现在）
├── SQLite DB（ai_dev_system_new.db）
│   ├── projects.traits          ← 项目特征（trait 标签）
│   ├── project_skills           ← 项目级 Skill 开关
│   └── agent_memory             ← 项目记忆（按 project_id 隔离）
│
└── {项目仓库}/
    └── .Agent/skills/           ← 项目本地 Skill（已实现）
        └── {skill-name}/
            └── SKILL.md
```

**存在问题**：
1. DB 里的项目配置跟着 ADS 服务走，项目仓库迁移/克隆后配置丢失
2. `.Agent/` 只支持 Skills，没有 rules、config 等扩展点
3. 没有项目级 Rules，无法为单个项目定义专属编码规范
4. 记忆只在 DB，换机器或分享项目时需要手动导出
5. **没有"用户全局"层**：全局配置混在 `backend/skills/`（服务端代码），多用户场景无法隔离

### 类比参考

| 工具 | 系统内置 | 用户全局 | 项目级 | 本地覆盖 |
|---|---|---|---|---|
| Claude Code | 编译进 CLI | `~/.claude/` | `.claude/` in repo | `CLAUDE.local.md` |
| CodeBuddy | 内置包 | `~/.codebuddy/` | `.codebuddy/` in repo | `CODEBUDDY.local.md` |
| **ADS（目标）** | `backend/skills/packs/` | `~/.ads/` | `.ads/` in repo | `.ads/config.local.json` |

## 二、四层配置架构

### 完整层级

```
优先级（低 → 高）

┌─────────────────────────────────────────────────────────────────┐
│ 层级 1：系统内置（System）                                        │
│ 位置：backend/skills/packs/ + backend/skills/rules/global.md    │
│ 内容：ADS 随服务发布的内置 Skills 和全局规则                      │
│ 权限：只读，升级 ADS 才能变更                                     │
└─────────────────────────────────────────────────────────────────┘
                           ↓ 被以下层覆盖

┌─────────────────────────────────────────────────────────────────┐
│ 层级 2：用户全局（User Global）                                   │
│ 位置：~/.ads/                                                   │
│ 内容：个人全局 Rules、常用 Skills、跨项目记忆、用户偏好配置         │
│ 权限：当前用户私有，不提交 git，不影响其他用户                      │
│ 状态：❌ 尚未实现                                                 │
└─────────────────────────────────────────────────────────────────┘
                           ↓ 被以下层覆盖

┌─────────────────────────────────────────────────────────────────┐
│ 层级 3：项目级（Project）                                         │
│ 位置：{项目仓库}/.ads/                                           │
│ 内容：项目 Rules、项目 Skills、项目记忆快照、项目配置              │
│ 权限：提交 git，团队共享                                           │
│ 状态：⚠️ 设计中（本文档）                                         │
└─────────────────────────────────────────────────────────────────┘
                           ↓ 被以下层覆盖

┌─────────────────────────────────────────────────────────────────┐
│ 层级 4：本地覆盖（Local Override）                                │
│ 位置：{项目仓库}/.ads/config.local.json                          │
│ 内容：本机路径、个人调试开关等（gitignore）                        │
│ 权限：本机私有，不提交 git                                         │
│ 状态：❌ 尚未实现                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 三、`~/.ads/` 用户全局目录规范

```
~/.ads/
├── rules/                       ← 用户个人全局规则（跨所有项目生效）
│   └── personal-style.md        ← 个人编码偏好（优先级低于项目级）
│
├── skills/                      ← 用户全局已安装 Skills（当前 use_skills/ 迁移目标）
│   ├── playwright-e2e/
│   └── godot-data-driven/
│
├── memory.md                    ← 跨项目个人记忆（用户偏好、习惯、常用引用）
│
└── config.json                  ← 用户级默认配置
    {
      "default_model": "claude-sonnet-4-5",
      "default_thinking_mode": "adaptive",
      "language": "zh-CN"
    }
```

**注意**：`~/.ads/` 是用户私有目录，不随项目仓库提交。ADS 服务启动时自动扫描并加载。

## 五、各子目录详细规范

### 3.1 `.ads/rules/` — 项目级规则（P1）

**格式**：Markdown + YAML frontmatter，与 `backend/skills/rules/global.md` 一致。

```yaml
---
alwaysApply: true           # true = 对本项目所有 Agent 强制生效
priority: high              # high / medium / low
description: ThunderStrike 项目编码规范
---

# ThunderStrike 项目规范

## 波次系统
- 波次配置必须使用独立 JSON 文件，不得硬编码
- 每个波次 Actor 必须继承 BP_WaveActorBase

## 命名
- 所有 Blueprint 前缀：BP_TS_（区别于通用前缀）
```

**注入时机**：
- `ChatAssistantAgent._build_system_prompt()` — 每次对话
- `BaseAgent._resolve_skills_prompt()` — 每次 Agent 执行
- 注入位置：rules 注入在 system prompt 最前面，优先级仅次于全局 `global.md`

**优先级层次**：
```
1. backend/skills/rules/global.md      ← 全局（最高，alwaysApply）
2. .ads/rules/*.md                     ← 项目级（次之）
3. backend/skills/rules/{trait}.md     ← 按 trait 条件（最低）
```

---

### 3.2 `.ads/skills/` — 项目级 Skill（P2）

与现有 `.Agent/skills/` 格式完全相同，逐步迁移：

```
.ads/skills/{skill-name}/
└── SKILL.md     ← 与系统 use_skills 格式一致（含 YAML frontmatter）
```

**过渡策略**：
- 新建项目：使用 `.ads/skills/`
- 已有项目：`.Agent/skills/` 继续兼容读取，不强制迁移
- 加载优先级：`.ads/skills/` > `.Agent/skills/` > `use_skills/`（全局）

---

### 3.3 `.ads/memory.md` — 项目记忆快照（P3）

**格式**：MEMORY.md 索引样式（与 Claude Code 对齐）

```markdown
# 项目记忆（ThunderStrike）

> 最后导出：2026-05-19

- [📁 项目] **波次系统使用独立配置文件**（2026-05-14）
  参考 WaveConfig.json，不要硬编码波次参数

- [💬 反馈] **不要用 UE 的 Blueprint 做性能敏感逻辑**（2026-05-15）
  之前 Tick 里做 Actor 查找导致帧率下降，改用 C++ 缓存引用

- [🔗 外部] **美术资产规范文档**
  https://km.woa.com/xxx
```

**用途**：
- 项目仓库 check in，团队共享关键记忆
- ADS 启动时可选择性导入到 `agent_memory` 表
- 定期由 AI 助手自动生成/更新（`/memory export` 命令）

---

### 3.4 `.ads/config.json` — 项目配置声明（P4）

```json
{
  "project_name": "ThunderStrike",
  "traits": ["engine:ue5", "category:game", "lang:cpp", "feature:wave-spawner"],
  "ue_engine_path": "F:/Epic Games/UE_5.5",
  "description": "像素风纵向卷轴飞行射击游戏"
}
```

**用途**：
- 项目 traits 可以在这里声明，无需每次在 UI 里设置
- UE 引擎路径等本地配置
- ADS 导入项目时自动读取，减少初始化步骤

## 六、与现有系统的关系

### 数据流向（目标状态）

```
项目仓库
  └── .ads/
      ├── rules/*.md    → SkillLoader 加载 → 注入 system prompt
      ├── skills/       → SkillLoader 加载 → 注入 system prompt
      ├── memory.md     → 可导入 agent_memory 表
      └── config.json   → 创建/导入项目时读取 traits

SQLite DB（运行时状态）
  ├── agent_memory      ← 运行时产生的记忆（可导出到 memory.md）
  ├── project_skills    ← UI 操作的 Skill 开关
  └── projects.traits   ← 运行时的 traits（可与 config.json 同步）
```

### `.gitignore` 建议

**ADS 生成的项目**：`.ads/` 不在忽略列表里，默认会被 git 追踪，无需额外操作。

**接入已有项目时需注意**：如果项目已有 `.gitignore` 含 `.*` 通配符（常见于 UE 项目），需要显式排除：

```gitignore
# UE 项目通常有 .*，需要显式允许 .ads/
!.ads/
!.ads/**
```

`.ads/` 内应该提交 git 的内容：

```
.ads/rules/       ← ✅ 提交（团队共享规范）
.ads/skills/      ← ✅ 提交（团队共享 Skill）
.ads/memory.md    ← ✅ 提交（团队共享关键记忆）
.ads/config.json  ← ✅ 提交（项目配置声明）
```

不应提交的内容（加入 `.gitignore`）：

```gitignore
.ads/config.local.json   # 本地路径、个人调试开关（机器相关）
```

## 七、实现路线图

### P1：`.ads/rules/` 支持（约 2 天）

**改动文件**：
- `backend/skills/loader.py`：`load_project_rules(repo_path)` 方法
- `backend/agents/base.py`：`_resolve_skills_prompt` 加载项目 rules
- `backend/agents/chat_assistant.py`：`_build_system_prompt` 加载项目 rules

**核心逻辑**：
```python
# SkillLoader 新增方法

    rules_dir = Path(repo_path) / ".ads" / "rules"
    if not rules_dir.exists():
        return []
    result = []
    for md_file in sorted(rules_dir.glob("*.md")):
        # 解析 frontmatter + content，注入 system prompt
        ...
    return result
```

### P2：`.ads/skills/` 支持（约 1 天）

- `load_skill.py`：`_get_project_agent_skills_dir` 优先读 `.ads/skills/`，再降级 `.Agent/skills/`

### P3：`memory.md` 导入/导出（约 2 天）

- `/memory export` 命令：把 `agent_memory` 导出为 `.ads/memory.md`
- `/memory import` 命令：把 `.ads/memory.md` 导入到 `agent_memory` 表
- 项目导入时自动读取 `.ads/memory.md`

### P4：`config.json` 支持（约 1 天）

- 创建/导入项目时读取 `.ads/config.json`
- `traits` 字段自动同步到 `projects.traits`

### P5：`~/.ads/` 用户全局目录支持（约 3 天）

**目标**：让 `use_skills/`、`global_skill_settings` 等全局配置也有文件系统存储层。

**改动**：
- ADS 服务启动时扫描 `~/.ads/rules/`，注入全局 rules
- `~/.ads/skills/` 作为 `use_skills/` 的文件系统镜像
- `~/.ads/memory.md` 对应 `project_id='__global__'` 的跨项目记忆
- `~/.ads/config.json` 存储用户级默认配置（LLM 模型、语言偏好等）

**优先级关系**（覆盖顺序）：
```
系统内置(backend/skills/)
  ← 用户全局(~/.ads/)
      ← 项目级({repo}/.ads/)
          ← 本地覆盖({repo}/.ads/config.local.json)
```

## 八、与 Harness Engineering 规范的对应关系

| Harness 支柱 | ADS 实现 | `.ads/` 贡献 |
|---|---|---|
| 上下文管理 | skills.json + knowledge_index | `.ads/rules/` + `.ads/skills/` |
| 工具系统 | MCP + Skills + 知识库 | `.ads/skills/` |
| 状态与记忆 | agent_memory 表 | `.ads/memory.md`（可移植）|
| 约束（Guardrails）| global.md rules | `.ads/rules/`（项目级）|
| SOP | backend/sop/fragments/ | `.ads/sop/`（项目定制，P4）|
