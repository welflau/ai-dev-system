## 一、三篇核心内容速览

### 第一篇：规则系统 v1.0 → v3.0（基础搭建）

核心是通过 **Rule System（精心编排的 Prompt 工程）** 让 AI 扮演一个完整游戏开发团队：

| 设计要点 | 内容 |
|---|---|
| **8 个 Agent 角色** | 制作人、项目管理、策划、UX、主程、程序、美术、QA |
| **9 种需求类型** | FEATURE/FEATURE_UI/OPTIMIZE/BUGFIX/CONFIG/TEST/DOC/REVIEW/RESEARCH，各走不同流水线 |
| **禁止越权原则** | 每个 Agent 有明确"禁止做的事"——"禁止"比"允许"更能约束 AI 行为 |
| **主从 Agent 架构** | 复杂任务拆子任务，子 Agent 独立上下文，只返回 ≤200 字摘要 |
| **双层架构** | Rule 文件夹（跨项目通用"操作系统"）vs GameDev 文件夹（当前项目"工作空间"）|
| **5Why 自我迭代** | AI 发现问题 → 分析根因 → 直接更新对应 Agent 文件，规则系统自己进化 |

关键经验：**"禁止"比"允许"更能约束 AI；先复杂后简化是正常路径；每条防御性规则背后都是一次真实事故。**

### 第二篇：v3.0 → v5.0（借鉴 BMAD / SuperClaude / ECC 三次升级）

**BMAD 借鉴（底层架构）**：
- **微文件架构**：Agent 文件拆为"入口+步骤+模板"三层，节省 76% token
- **质量门禁**：3 个关键流转节点检查内容质量（不只是"文件存在"）
- **对抗性代码审查**：程序 Agent 必须找自己代码的 ≥3 个问题
- **Smart Probe 需求探测**：5 维度清晰度评分（≥20 直接做 / 15-19 自动补假设 / <15 提问）
- **规模自适应**：XS/S/M/L/XL 分级流程，避免"用大炮打蚊子"
- **记忆层 Frontmatter**：每个需求文档头部 YAML 状态块，跨会话恢复

**SuperClaude 借鉴（使用效率）**：
- **/gd: 快捷命令体系**：22 个快捷命令（/gd:feature、/gd:bugfix、/gd:status 等）
- **Agent 智能路由**：三级识别 + 置信度标注（🟢高/🟡中/🔴低）
- **行为模式标签**：7 种 [MODE:xxx] 标签（EXPLORE/DESIGN/IMPLEMENT/REVIEW/DEBUG/VERIFY/REFLECT）
- **三级内省机制**：阶段级（1min）/ 需求级（5min）/ 周期级（10min）
- **上下文压缩**：知识库按关键词精准加载，只取相关章节

**ECC 借鉴（防护纵深）**：
- **事件驱动防护**：事前 Guard（操作前自动拦截）+ 事后 Guard（完成后自动检查）
- **Skills 技能包体系**：独立可复用技能文件
- **规则分层与不可变性**：三层分级，核心规则不可被 AI 修改
- **置信度驱动的经验应用**：高置信度自动应用，中置信度提示应用

### 三个缺失 Agent 的具体分析

#### ❌ 策划 Agent（缺失，PRD 专职）

**现状**：ProductAgent 把"需求分类 + 写 PRD + 拆单"三件事全揉在一起。

**问题**：没有独立的"策划审查"环节——没有人在进入架构阶段之前专门检查"PRD 本身是否合理"，导致：
- 验收标准常缺失或模糊（"显示正常"不是可测量标准）
- 技术方案和策划方案经常互相污染
- 策划发现问题只能在开发后期返回

**独立后的职责**：
- 产出完整 PRD（功能描述、交互说明、验收标准、边界条件）
- 验收标准必须可量化（❌"界面美观" / ✅"点击后 300ms 内弹出弹窗"）
- 不写任何技术实现细节
- 策划完成后设质量门禁（策划 → 架构的关卡）

---

#### ❌ UX Agent（缺失，交互设计专职）

**适用范围**：网页、小程序、移动端、桌面端、游戏 UI——**所有有界面的项目都需要**。

**现状**：DevAgent 直接按 PRD 文字描述写代码，交互逻辑全靠 DevAgent 自己理解和发挥。

**问题**：
- 交互状态缺失（没想到空状态、错误状态、加载状态）
- 页面跳转逻辑混乱（开发后期才发现流程有问题）
- 响应式/适配问题到开发完才暴露

**独立后的职责**：
- 产出交互流程图（页面间跳转关系，用 Mermaid 表示）
- 产出组件线框说明（布局层级、关键组件位置）
- 定义所有交互状态（正常/hover/active/disabled/loading/empty/error）
- 定义响应式规则（Web：断点规则 / 小程序：rpx 换算规则）
- **禁止写代码、不关心视觉样式**（那是美术的事）

**SOP 位置**：策划 → **UX** → 美术 → 架构 → 开发

---

#### ❌ 美术/设计 Agent（缺失，视觉规范专职）

**适用范围**：同 UX，**所有有界面的项目**（网页、小程序、游戏 UI）。

**现状**：没有视觉规范约束，DevAgent 随意选颜色、字号、间距，最终产出风格混乱。

**独立后的职责**：
- 产出 Design Token（颜色系统/字体层级/间距规范）
- 产出组件视觉说明（按钮、卡片、弹窗的视觉样式规格）
- 产出 asset_manifest.yaml（资产清单，技美并行的契约）
- 检查是否与已有 design system 一致（或定义新的）
- 游戏项目额外输出：精灵图规格、UI Atlas 布局方案、粒子效果参数
- **对每个资源判断类型和生产路径**（见下）

**禁止**：写代码、写策划逻辑、写交互流程（那是 UX 的事）

**SOP 位置**：UX → **美术** → 与 ArchitectAgent+DevAgent 并行 → 资产合并校验

#### 美术资源生产路径（扩展版）

| 路径 | 适用资源类型 | 实现方式 | 同步/异步 |
|---|---|---|---|
| **AI 生图** | Banner、插画、游戏场景、角色形象、背景图 | 发给图片处理 Agent，调生图大模型 | 异步，占位先行 |
| **图标库检索** | 标准 UI 图标（导航、操作、状态）| 调 Iconify / Material Icons API，关键词搜索下载 SVG | 同步，秒级完成 |
| **CSS/SVG 代码生成** | 渐变背景、纯色块、简单几何形状、Loading 动画 | 美术 Agent 直接输出代码，DevAgent 内联使用 | 同步，无外部依赖 |
| **占位兜底** | 以上均失败 / API 未配置 / 暂无条件生产 | 按规格生成纯色块 + 尺寸 + 用途文字标注 | 同步，始终可用 |

决策逻辑：
```
资产类型 == illustration / banner / game_sprite / hero_image
    → AI 生图（ImageAgent 异步）

资产类型 == icon AND 风格 == 标准UI
    → 图标库 API（找不到则降级 AI 生图）

资产类型 == background / gradient / simple_shape / loading
    → 输出 CSS/SVG 代码

其他 / API 不可用
    → 生成占位图（颜色+尺寸+用途描述）
```

这样设计的好处：**图标库和 CSS 生成是同步的，不阻塞并行流水线；AI 生图是异步的，通过占位符机制对主流程透明。**

#### 美术资产库（Asset Library）

资产库是整个美术体系的基础设施，所有来源的资产都汇入同一个库，下次直接复用。

**资产生产完整优先级（从快到慢，从省到贵）**：

```
资产请求（来自 Art Agent / 任意 Agent）
    ↓
① 查本地资产库（local DB + 文件存储）
   按 tags / type / 风格 / 尺寸匹配，命中直接复用
    ↓ 未命中
② 网络搜索（图标/图片素材站 API）
   - 图标：Iconify API（免费，10万+ 图标）
   - 图片：Unsplash API（免费，高清摄影）
   - 插画：unDraw（免费 SVG 插画库）
   - 游戏资源：OpenGameArt（免费游戏素材）
   找到 → 下载 → **自动入库** → 使用
    ↓ 未找到
③ AI 生图（ImageAgent 异步）
   生成完成 → **自动入库** → 替换占位符
    ↓ API 未配置 / 超时
④ CSS/SVG 代码生成（纯逻辑资源）
⑤ 占位图兜底（纯色块 + 文字标注）
```

**资产库数据结构**：

```sql
CREATE TABLE art_assets (
    id          TEXT PRIMARY KEY,    -- AST-xxxxxxxx
    name        TEXT,                -- 资源名称
    tags        TEXT,                -- JSON 数组：["icon","navigation","home"]
    type        TEXT,                -- icon / photo / illustration / background / svg_code
    style       TEXT,                -- flat / outline / filled / realistic / cartoon
    source      TEXT,                -- iconify / unsplash / undraw / ai_generated / css_code
    source_url  TEXT,                -- 原始来源 URL（网络搜索类）
    file_path   TEXT,                -- 本地存储路径 art_assets/icons/xxx.svg
    width       INTEGER,
    height      INTEGER,
    format      TEXT,                -- svg / png / jpg / css
    project_id  TEXT,                -- NULL = 全局共享；有值 = 项目私有
    used_count  INTEGER DEFAULT 0,   -- 被引用次数（越高越优先返回）
    created_at  TEXT
);
```

**核心价值**：
- **复用优先**：同一图标/插画跨需求、跨项目直接拿，不重复生成
- **成本节省**：避免对同类资源重复调用 AI 生图 API
- **风格一致**：同项目第一次用了 Feather Icons 风格，后续搜索优先返回同风格
- **离线可用**：网络搜索结果持久化，不依赖外网二次访问

**与知识库的关系**：
- 知识库（`knowledge_index`）存的是**文本**（文档、代码经验、开发笔记）
- 资产库（`art_assets`）存的是**二进制文件**（图片、图标、SVG）
- 两者都支持关键词检索，但资产库额外有视觉属性（尺寸、格式、风格）过滤

#### 资产库用独立 Git 仓库管理（全局共享）

资产库是**全局共用**的，不属于任何单个项目——用独立 Git 仓库管理是最合适的方式。

**为什么用 Git**：
- 资产是文件，和代码一样需要版本历史（知道谁加的、什么时候加的、为什么加）
- 不同机器/实例部署时可以 pull 同一个资产仓库，真正全局共享
- 资产变更可追溯（图标从 flat 换成 outline 也有 commit 记录）

**仓库结构**：
```
art-assets-global/           ← 独立 Git 仓库（与项目仓库分开）
├── icons/
│   ├── navigation/          home.svg / search.svg / back.svg
│   ├── action/              edit.svg / delete.svg / share.svg
│   └── status/              success.svg / error.svg / loading.svg
├── illustrations/
│   ├── empty-state/
│   └── onboarding/
├── photos/
│   ├── backgrounds/
│   └── heroes/
├── game/
│   ├── ui/                  game_button.png / health_bar.png
│   └── sprites/
└── manifest.json            ← 资产索引（DB 的来源，人可直接阅读）
```

**Git 仓库与 DB 的分工**：

```
Git 仓库（art-assets-global）← 持久化存储 + 版本管理，文件真正存在这里
    ↕ 同步
manifest.json（资产元数据，可 diff，人可读）
    ↓ 系统启动时 / 新增资产时同步
art_assets DB 表 ← 搜索索引（从 manifest.json 构建，可随时重建）
    ↓
Art Agent / 任意 Agent 快速检索复用
```

**新增资产的完整写入流程**：
```
找到/生成一个新资源
    ↓
写入 art-assets-global/对应目录
更新 manifest.json（追加条目）
    ↓
git commit -m "add: home_icon.svg (source: iconify, tags: icon,navigation)"
git push  ← 其他实例可 pull 获取
    ↓
同步更新 art_assets DB 表（本地搜索索引）
```

**API 接口**：
```
GET  /api/art-assets?tags=icon,home&style=flat&format=svg   # 搜索
POST /api/art-assets                                         # 上传/入库（自动 git commit）
GET  /api/art-assets/{id}/file                               # 下载文件
POST /api/art-assets/search-web?query=home+icon&style=flat  # 网络搜索+入库+git commit
POST /api/art-assets/sync                                    # 从远端 pull 最新资产库
```

#### ✳️ 图片处理 Agent（新增，异步服务型）

这是一个与上面三个"流水线型"Agent 完全不同的角色——它是一个**异步服务**，不在 SOP 主流水线里，而是作为一个可随时调用的后台能力。

**设计动机**：
- AI 生图耗时不定（短则 5 秒、长则 60 秒以上），如果放在主流水线里是硬等，会严重阻塞
- 任何 Agent 在任何阶段都可能需要图片（ProductAgent 写 PRD、美术 Agent 生成 UI 效果图、DevAgent 生成 placeholder 图）
- 图片生成完成后需要"回调"到原来的文档或上下文里

**核心设计：异步消息 + 占位符替换**

```
1. 请求方（任意 Agent）→ 发送生图指令给 ImageAgent
   指令包含：{描述, 尺寸, 风格, 回调位置(文档路径+行号)}

2. 立即返回占位符：[IMAGE_PENDING:img_001]
   请求方继续其他工作，不等待

3. ImageAgent 后台队列处理：
   → 调用生图大模型 API（Flux / DALL-E 3 / Stable Diffusion）
   → 图片保存到项目 assets/ 目录
   → 在回调文档中将 [IMAGE_PENDING:img_001] 替换为 ![desc](assets/img_001.png)
   → 推送 SSE 事件通知前端刷新

4. 如果生图失败或超时：
   → 保留 ASCII 艺术作为最终产出
   → 记录失败原因到 ticket_logs
```

**ASCII 艺术兜底机制**：

如果生图 API 不可用或超时，ImageAgent 调用 LLM 生成 ASCII 图作为替代：

```
┌─────────────────────┐
│   [Logo区域]        │
│   ████████████      │
├─────────────────────┤
│  导航栏             │
│  首页 | 功能 | 关于  │
└─────────────────────┘
```

**技术架构**：

```
image_requests 表
├── id           # IMG-xxxxx
├── project_id
├── requester    # 请求的 Agent 类型
├── prompt       # 生图描述
├── style        # realistic/cartoon/ui-mockup/icon
├── size         # 512x512 / 1024x768 等
├── callback_doc # 需要替换占位符的文档路径
├── callback_tag # [IMAGE_PENDING:id]
├── status       # pending / processing / done / failed / ascii_fallback
├── result_path  # 生成后的图片路径
└── created_at

配置项（.env）：
IMAGE_API_URL=       # 生图 API 地址（留空则始终用 ASCII 兜底）
IMAGE_API_KEY=
IMAGE_TIMEOUT=60     # 超时后触发 ASCII 兜底
```

**触发方式（其他 Agent 的调用接口）**：

```python
# 任意 Agent 发送生图请求
await request_image(
    project_id=project_id,
    ticket_id=ticket_id,
    prompt="一个现代风格的游戏背包界面，深色主题，包含物品格子",
    style="ui-mockup",
    size="800x600",
    callback_doc="docs/UX设计.md",
    callback_tag="[IMAGE_PENDING:img_001]",
)
# 立即返回，不等待
```

**与 SOP 的关系**：
- 不占用 SOP 的 stage 位置，是横切关注点
- 美术 Agent 效果图、ProductAgent PRD 插图、UX Agent 线框效果图都可以调用
- `IMAGE_API_URL` 未配置时全程透明降级为 ASCII，不影响流水线

#### 关键设计：asset_manifest.yaml（技美契约）

UX Agent 阶段结束后产出，是技术和美术双方的共同契约：

```yaml
# asset_manifest.yaml（由 UX Agent 产出，双方遵守）
assets:
  - id: btn_primary_icon
    path: assets/icons/btn_primary.png
    size: 32x32
    format: png
    usage: 主操作按钮图标
    placeholder: 纯蓝色 32x32 方块

  - id: home_banner
    path: assets/images/home_banner.jpg
    size: 750x300
    format: jpg
    usage: 首页顶部横幅
    placeholder: 灰色矩形 + 文字"BANNER"

  - id: nav_icons
    path: assets/icons/nav_*.png
    size: 24x24
    format: png
    usage: 底部导航栏图标（首页/发现/我的）
    placeholder: 简单几何形状
```

**DevAgent**：按此清单在对应路径放占位文件，代码里直接引用这个路径。  
**美术 Agent**：按此清单产出真实资源，存到相同路径，直接覆盖占位文件。  
**资产合并校验**：核对清单中每一项都有真实文件存在，才触发测试阶段。

#### 技美并行的前提

并行才有价值，需要满足：
1. **UX 先完成**，产出资产清单，明确路径和规格——这是并行的起点
2. **路径和尺寸提前约定**——DevAgent 引用的路径就是美术最终交付的路径，美术覆盖即生效
3. **功能验收 ≠ 视觉验收**——TestAgent 需要等资产合并后再做最终验收，但功能逻辑可以先跑通

#### 按 traits 决定触发哪些阶段

| platform | 策划 | UX | 美术 | 技美并行 | 图片 Agent |
|---|---|---|---|---|---|
| web / wechat / mobile / desktop | ✅ | ✅ | ✅ | ✅ | ✅ |
| server / cli | ✅ | ❌ 跳过 | ❌ 跳过 | ❌ | ❌ |
| category:game | ✅ | ✅ | ✅（游戏规范）| ✅ | ✅（重度）|

#### SOP YAML 扩展（新增字段）

```yaml
stages:
  - id: ux_design
    agent: UXAgent
    outputs: [asset_manifest.yaml]   # 产出物，作为后续阶段的输入

  # 美术和技术并行
  - id: art_production
    agent: ArtAgent
    parallel_with: development       # 与 development stage 并行
    inputs: [asset_manifest.yaml]

  - id: development
    agent: [ArchitectAgent, DevAgent]
    parallel_with: art_production
    inputs: [asset_manifest.yaml]
```

实现上复用现有 SOP fragment 机制：新增 `ux_design.yaml`、`art_design.yaml`、`asset_parallel.yaml` fragment，对 `platform:web`、`platform:wechat` 等 traits 自动注入。图片 Agent 作为独立后台服务，不在 SOP 中配置。

## 三、可借鉴方案（按实施难度排序）

### P1：高价值、低成本

#### 1. 行为模式标签 [MODE:xxx]（1 天）

**问题**：DevAgent 在"编码实现"和"对抗审查"两个阶段用同一种心态，审查不够狠。

**方案**：在 Agent 的 system prompt 中不同阶段注入行为模式标签：

```yaml
# SOP stage 配置
stages:
  - id: write_code
    agent: DevAgent
    mode: IMPLEMENT   # 专注、严谨、遵循规范
  - id: self_review  
    agent: DevAgent
    mode: REVIEW      # 对抗、怀疑、主动找问题
```

DevAgent 的 prompt 中根据 `mode` 切换系统提示词开头：
```
[MODE: REVIEW] 你现在是严苛的代码审查者，目标是找出自己代码中的问题，
不允许"看起来不错"的结论，必须至少指出 3 个具体问题…
```

**改动**：`SOP loader` + `DevAgent prompt` 加 mode 注入，改动小。

#### 2. 对抗性代码审查强制要求（0.5 天）

**问题**：ReviewAgent 存在但没有"必须找 ≥3 个问题"的硬性要求，常流于形式。

**方案**：在 ReviewAgent/DevAgent 的 system prompt 和 actions 里加：
- 必须输出 `🔴 严重 / 🟡 警告 / 🟢 建议` 三级问题列表
- 严重级别问题必须修复才能流转
- 禁止"代码质量良好"类的敷衍结论

**改动**：更新 `review_code.py` + ReviewAgent system prompt。

#### 3. Smart Probe 需求清晰度评分（1 天）

**问题**：模糊需求直接进入流水线，做到一半发现关键信息缺失，大量返工。

**方案**：在 ProductAgent 拆单前加 5 维度评分：

| 维度 | 满分 |
|---|---|
| 用户价值（做这个是为了什么）| 5 |
| 范围边界（哪些做、哪些不做）| 5 |
| 交互方式（如何触发、操作流程）| 5 |
| 数值参数（关键数字有预期吗）| 5 |
| 关联影响（与现有功能关系）| 5 |

- ≥20：直接开发
- 15-19：AI 自动补假设，在策划产出中标注 `[假设]`
- <15：暂停，提 ≤3 个关键问题（附默认假设，用户可说"按默认来"跳过）

**改动**：`DecomposeAction` 前加 `SmartProbeAction`，ChatAssistant system prompt 更新。

#### 4. 需求规模自适应流程（0.5 天）

**问题**：一个改颜色的小需求和实现新战斗系统走同一套 9 阶段流水线，浪费大量时间。

**方案**：在 SOP 里加 `complexity` 字段，根据 DecomposeAction 判断的复杂度走不同路径：

| 规模 | 条件 | 跳过的阶段 |
|---|---|---|
| XS（极小改动）| 1 个工单、< 50 行代码 | 跳过 ArchitectAgent（直接 Dev）|
| S（小功能）| 1-2 个工单 | ArchitectAgent 轻量过（只产出接口定义）|
| M（标准）| 3-5 个工单 | 标准流水线 |
| L/XL（大功能）| >5 个工单 | 触发主从模式，拆子需求 |

#### 补充：Agent 间通信机制差距（文件 vs DB JSON）

**现状（我们）**：Agent 间通过 DB 的 `artifacts.content` JSON 字段传递产出，`path` 字段为 `null`，没有实际文件。

```python
# _build_context() 中
context["architecture"] = json.loads(art["content"])  # DB JSON，无文件
```

**KM 的做法**：每个 Agent 产出一个 Markdown 文件写入 Git 仓库，下一个 Agent 直接读文件。

```
策划 Agent → docs/{req}/PRD.md
主程 Agent → docs/{req}/架构设计.md
程序 Agent → src/ 代码文件
QA Agent   → docs/{req}/测试报告.md
```

**差距对比**：

| | KM 系统 | 我们系统 |
|---|---|---|
| 通信媒介 | 文件（Markdown/JSON）| DB 的 JSON 字段 |
| 人工可读 | 直接打开文件查看/修改 | 需查 DB |
| Git 追踪 | 自动进版本历史 | `path: null`，无文件 |
| 大内容传递 | 文件无大小限制 | DB 字段有截断风险 |
| 知识沉淀 | 文件自动索引进知识库 | 需额外处理 |

**注意**：代码文件是例外——DevAgent 写的代码已经通过 `git_manager.write_file()` 存到 Git 仓库。但 PRD、架构设计、UX 设计等**中间产出物**只存在 DB，没有对应文件。

**改进方案（1 天）**：每个 Agent 完成时，同步把产出写一份文件到 Git 仓库：

```python
# Orchestrator 每个 Agent 阶段完成后
docs_prefix = f"docs/{req_id[-6:]}/{ticket_id[-6:]}/"
await git_manager.write_file(project_id,
    f"{docs_prefix}PRD.md", prd_content)         # ProductAgent
    f"{docs_prefix}架构设计.md", arch_content)    # ArchitectAgent
    f"{docs_prefix}UX设计.md", ux_content)        # UX Agent（新增）
    f"{docs_prefix}视觉规范.md", art_content)     # 美术 Agent（新增）
```

三个收益：
1. 人可以打开文件查看/修改中间产出，修改后影响下游 Agent
2. 需求的完整决策链路进 Git 历史，可追溯可回滚
3. 文件自动被 knowledge_index 索引，同类需求的历史方案变成可检索经验

## 四、最有启发的三个洞察

### 洞察 1："禁止"比"允许"更能约束 AI 行为

KM 文章中反复强调：明确告诉 AI "不要做什么"远比告诉它"要做什么"更有效。

**对我们的启示**：每个 Agent 的 system prompt 里应该有明确的"禁止清单"。目前我们的 ChatAssistant 已有类似设计（"不要调工具"、"不要调 git_read_file"），但 DevAgent / ArchitectAgent 的 prompt 里禁止规则还不够完备。

---

### 洞察 2：规则系统自身也需要迭代，而且是一个演化过程

KM 从 v1.0 迭代到 v5.0，每个版本号背后都是一次"踩坑→反思→改进"。他们的核心经验是：
- 不要一开始设计过于复杂的机制，简单直接往往更有效
- 每条"防御性规则"都是一次真实事故的产物
- AI 很擅长找"合理理由"来简化流程，需要提前预判并明确禁止

**对我们的启示**：我们的 system prompt 和 SOP 规则应该有版本记录（类似 rule_changelog.md），每次发现 AI "偷懒"或"越权"的行为都应该立即固化为规则，而不是依靠每次对话重新提醒。

---

### 洞察 3：工程化的本质是"把软约束变成硬约束"

KM 三篇的演化主线：文档规则（软约束）→ Prompt 系统（半硬）→ CLI 工具链（硬约束）。

每一步工程化都是把"AI 自觉遵守"变成"系统强制执行"。

**对我们的启示**：
- 阶段流转检查现在是通过 LLM 判断的（软约束），可以改为代码层面的状态检查（硬约束）
- Insight 注入现在是"注入到 prompt"（AI 可能忽略），可以改为"如果 Insight 匹配到高置信度经验，直接添加到 DevAgent 的 files_to_read 列表"
- 工单依赖关系现在要靠 LLM 拆单时输出（刚修复的），可以增加依赖关系的 UI 管理入口
