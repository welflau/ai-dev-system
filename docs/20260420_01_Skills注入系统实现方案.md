# Skills 注入系统 — 实现方案

> 日期: 2026-04-20 | 对标: MagicAI `src/agents/game_agent/utils/skill_loader.py` + `config/skills.json` + `skills/<id>/prompt.md`

---

## 一、背景

### 1.1 当前痛点

各 Agent 的领域知识（React 模式、FastAPI 规范、Playwright 用法、Git 工作流等）散在两个地方：

1. **Action 的 prompt 里硬编码**——想给 DevAgent 补充"用 Hooks 不要用 class"这种指导，得去改 `write_code.py` / `plan_code_change.py` 的 prompt 字符串。每次调整一点都要动源码。
2. **靠 LLM 常识**——实际很多项目级约束（单文件部署、英文文件名、内联 script）LLM 并不知道，每次开发要在 ticket_description 里反复提。

结果：
- 新需求常常要重复提醒 LLM 同样的规则，token 浪费
- "访问计数"需求被打回 5 次 → force_pass 的核心原因之一是 DevAgent 不知道"页面必须有可见 DOM"这条隐含约束

### 1.2 MagicAI 的方案

MagicAI 把这类"Agent 专业知识"外化成 **Skill**：

```
skills/
├── godot_dev/           ← skill_id
│   ├── prompt.md        ← LLM 看到的提示文本
│   └── skill.json       ← 元数据
├── pixel_art/
│   ...
config/skills.json       ← 主配置：每个 skill 挂给哪些 Agent
```

Agent 启动时读 skills.json，把 `inject_to` 包含自己的所有 skill 的 prompt.md 拼起来，注入自己的 system prompt。

### 1.3 本项目落地目标

1. **一个 `.md` 文件 = 一个 Skill**；一条 JSON 配置决定挂给哪些 Agent
2. **多对多**：一个 Agent 可挂 N 个 Skill；一个 Skill 可挂给 N 个 Agent
3. **MagicAI Skill 包可直接 drop-in 复用**（目录结构与 MagicAI 对齐）
4. **现有 Action 代码零改动**（用 ContextVar 透明注入）

## 三、配置格式（与 MagicAI 兼容）

### 3.1 `backend/skills/skills.json`

```json
{
  "react-dev": {
    "name": "React/HTML 开发",
    "description": "Hooks、组件命名、内联到 index.html 的模式",
    "prompt_file": "packs/react_dev/prompt.md",
    "inject_to": ["DevAgent"],
    "priority": "high",
    "enabled": true
  },
  "fastapi-dev": {
    "name": "FastAPI 后端开发",
    "prompt_file": "packs/fastapi_dev/prompt.md",
    "inject_to": ["DevAgent"],
    "priority": "high",
    "enabled": true
  },
  "playwright-e2e": {
    "name": "Playwright 端到端测试",
    "prompt_file": "packs/playwright_e2e/prompt.md",
    "inject_to": ["TestAgent", "ProductAgent"],
    "priority": "high",
    "enabled": true
  },
  "git-workflow": {
    "name": "Git 工程化",
    "prompt_file": "packs/git_workflow/prompt.md",
    "inject_to": ["DevAgent", "DeployAgent"],
    "priority": "medium",
    "enabled": true
  }
}
```

### 3.2 字段说明

| 字段 | 类型 | 说明 |
|---|---|---|
| `name` | str | 显示名 |
| `description` | str | 简介（供 `get_skills` API / 前端展示） |
| `prompt_file` | str | 相对 `backend/skills/` 的路径，LLM 会看到的 prompt 内容 |
| `inject_to` | list[str] | 挂到哪些 Agent（`agent_type` 字段值） |
| `priority` | str | `high` / `medium` / `low` — 同一 Agent 有多个 skill 时的排序 |
| `enabled` | bool | false 时完全忽略 |

### 3.3 可用 `inject_to` 值

| Agent | agent_type |
|---|---|
| 产品 Agent | `ProductAgent` |
| 架构 Agent | `ArchitectAgent` |
| 开发 Agent | `DevAgent` |
| 测试 Agent | `TestAgent` |
| 代码审查 Agent | `ReviewAgent` |
| 部署 Agent | `DeployAgent` |
| 智能助手 | `ChatAssistant` |

## 四、注入机制（ContextVar）

### 4.1 为什么选 ContextVar 而不是显式字段

| 维度 | ContextVar | 显式 `context["skills_prompt"]` |
|---|---|---|
| 改动文件数 | **2 个**（base.py + action_node.py） | 7+ 个（所有 Action） |
| 新增 Action 要不要记着加 skills？ | 不用（自动注入） | 要（容易漏） |
| 和 reflection 模式一致 | 不一致 | 一致 |
| asyncio 并发安全 | ✅ 原生支持 | ✅（ctx 是局部变量） |

**决策理由**：
1. Skills 语义是"给整个 Agent 的通用知识"，不是"某个 Action 的特殊参数"——全局注入语义更准
2. 本项目 Action 数量还会长（TODO.md 里挂了 12 个待开发），零改动的优势越往后越明显
3. reflection 用显式字段是因为**只在重试场景注入**（有条件），skills 是**常驻**（无条件），两者语义本来就不同

### 4.2 数据流

```
进程启动
  └─ skills/__init__.py 创建 skill_loader 单例 → _load_config() 读 skills.json

BaseAgent.__init__()
  └─ self._skills_prompt = skill_loader.build_prompt_for_agent(self.agent_type)
      # 按 inject_to 过滤 → 按 priority 排序 → 读取所有 prompt.md → 拼成一段文本

Orchestrator 触发 Agent 执行
  └─ BaseAgent.run_action(action_name, context)
       └─ _current_skills.set(self._skills_prompt)   # ContextVar 入栈
            └─ await action.run(context)
                 └─ action 内部调 ActionNode.fill(req, llm)
                      └─ ActionNode._compile(req)
                           └─ skills = _current_skills.get()   # 读到 agent 的 skills
                           └─ return f"{skills_block}{instruction}{req}{schema}"
       └─ _current_skills.reset(token)               # ContextVar 出栈
```

### 4.3 ChatAssistant 特殊处理

ChatAssistant 不走 ActionNode（直接用 Anthropic tool_use + 自建 system prompt），所以在 `_build_system_prompt()` 里直接读 `self._skills_prompt` 拼接。

## 五、改动清单

### 5.1 新增文件

| 文件 | 作用 |
|---|---|
| `backend/skills/__init__.py` | 导出 `skill_loader` 单例 + `_current_skills` ContextVar |
| `backend/skills/loader.py` | `SkillLoader` 类（`_load_config` / `build_prompt_for_agent` / `get_enabled_skills` / `print_status` / `reload`） |
| `backend/skills/skills.json` | 主配置（4 条初始 skill） |
| `backend/skills/packs/{react_dev,fastapi_dev,playwright_e2e,git_workflow}/prompt.md` | 4 个 skill 正文 |
| `backend/skills/packs/<id>/skill.json` | 每个 skill 元数据（可选，便于从 MagicAI drop-in） |
| `backend/_test_skills.py` | 单测（8 用例） |
| `docs/20260420_01_Skills注入系统实现方案.md` | 本文档 |

### 5.2 修改文件

**`backend/agents/base.py`**
- `BaseAgent.__init__()`：加载 `self._skills_prompt`（try-except 兜底，skills 系统失败不阻塞 Agent 初始化）
- `BaseAgent.run_action()`：调 Action 前 `_current_skills.set(...)`，finally 里 reset
- `_react_by_order()` / `_react_with_think()`：同样用 ContextVar 包裹（这两个路径不走 run_action）

**`backend/actions/action_node.py`**
- `_compile()`：从 ContextVar 读 skills，非空则 prepend 到 prompt 头部，格式：
  ```
  ## 专业技能 (Skills)
  <skills 文本>

  ---

  <原 instruction>
  <context>
  <JSON schema>
  ```

**`backend/agents/chat_assistant.py`**
- `_build_system_prompt()`：在"产出物列表"段之后、"你的能力"段之前插入 `## 专业技能 (Skills)` 段

## 六、4 个初始 Skill 一览

### 6.1 `react-dev` → DevAgent（priority: high）

核心内容：
- **单文件优先原则**——默认所有前端代码写到 `index.html` 内联 `<style>` + `<script>`，不产生构建配置
- **React Hooks 规范**——只在顶层调用、依赖数组完整、惰性初始化
- **命名约定**——PascalCase / camelCase / SCREAMING_SNAKE_CASE
- **可见 DOM 是验收硬门槛**——针对"访问计数被打回 5 次"的历史教训，反复强调
- **原生 CSS 优先**——不引 Tailwind / styled-components
- **安全要点**——textContent 防 XSS、fetch 加 try/catch

### 6.2 `fastapi-dev` → DevAgent（priority: high）

核心内容：
- **async 全栈**——所有路由 `async def`，IO 用 aiosqlite / httpx.AsyncClient
- **Pydantic v2 Schema**——请求/响应都定义 model，不用裸 dict
- **错误处理**——`HTTPException` / `logger.exception` / 全局 exception_handler
- **依赖注入**——用 `Depends()` 复用 DB、user、分页
- **启动规范**——监听 `0.0.0.0`、端口从 env 读、`if __name__ == "__main__"`
- **性能**——SQLite busy_timeout、BackgroundTasks、StreamingResponse

### 6.3 `playwright-e2e` → TestAgent + ProductAgent（priority: high）

核心内容：
- **验收看截图不看 console.log**——ProductAgent 是基于截图判断，硬规则
- **等待策略**——永远不 sleep，用 `wait_for_selector` / `wait_until="networkidle"`
- **选择器优先级**——`get_by_role` > `get_by_test_id` > text > id > class > XPath
- **截图规范**——本项目约定 `backend/screenshots/<req_id>/<name>.png`
- **Playwright `expect` 断言**——元素可见、文本、递增
- **集成模式代码片段**——可直接 copy
- **常见坑**——浏览器未装、端口没起、SPA 动态加载、字体差异

### 6.4 `git-workflow` → DevAgent + DeployAgent（priority: medium）

核心内容：
- **三层分支模型**——`feat/TICKET-<id>-<slug>` → `develop` → `main`
- **Conventional Commits**——`feat` / `fix` / `refactor` / `ui` / `docs` / `chore` / `test`
- **禁止操作清单**——force push、reset --hard、--no-verify、改 global config、直推 main
- **DevAgent 典型流程**——5 步命令示例
- **合并策略**——feat→develop squash、develop→main merge commit
- **文件粒度**——具体 add、忽略 `.env` 等

## 七、操作手册

### 7.1 新增一个 Skill

1. 在 `backend/skills/packs/` 下建目录，例：`mysql_dev/`
2. 写 `packs/mysql_dev/prompt.md`（内容即 LLM 会看到的 markdown 文本）
3. （可选）写 `packs/mysql_dev/skill.json` 元数据
4. 在 `skills.json` 加一条记录：
   ```json
   "mysql-dev": {
     "name": "MySQL 数据库开发",
     "prompt_file": "packs/mysql_dev/prompt.md",
     "inject_to": ["DevAgent"],
     "priority": "high",
     "enabled": true
   }
   ```
5. 重启后端服务（`run_server.bat`）

### 7.2 禁用一个 Skill

改 `skills.json` 里对应条目的 `"enabled": false`，重启。文件不用删。

### 7.3 从 MagicAI drop-in 一个 Skill

MagicAI 的 `src/agents/game_agent/skills/godot_dev/` 目录结构与本项目 `packs/` 下完全一致，操作：

```bash
cp -r D:/A_Works/MagicAI/src/agents/game_agent/skills/godot_dev \
      D:/A_Works/ai-dev-system/backend/skills/packs/
```

然后在 `backend/skills/skills.json` 加一条：

```json
"godot-dev": {
  "name": "Godot 游戏开发",
  "prompt_file": "packs/godot_dev/prompt.md",
  "inject_to": ["DevAgent"],   // 按需改，MagicAI 里挂的是 developer
  "priority": "high",
  "enabled": true
}
```

重启即可。注意 MagicAI 用小写 `developer`，本项目用 `DevAgent`——`inject_to` 字段要改。

### 7.4 查看当前启用的 Skills

```python
# Python REPL
from skills import skill_loader
print(skill_loader.print_status())
```

或查日志：启动时 Agent 会打印 `🎓 DevAgent 加载 3 个 Skill: react-dev, fastapi-dev, git-workflow`。

## 九、未来扩展

### 9.1 运行时 API（目前未做）

可以加一个 `GET /api/skills` 只读接口，让 ChatAssistant 能回答"当前 DevAgent 注入了哪些技能"。

改动点：新增 `backend/api/skills.py` 路由，调用 `skill_loader.get_all_skills_status()` 返回 JSON。

### 9.2 前端管理页（目前未做）

类似现在的 SOP 编辑器，做一个 `/skills` 页面：
- 列出所有 skill 卡片（名称、描述、挂给哪些 Agent、启用/禁用 toggle）
- 在线编辑 prompt.md（Markdown 编辑器 + preview）
- 改完 PUT `/api/skills/<id>`，后端更新 skills.json + reload

工作量：和 SOP 编辑器类似，约 2-3 天。

### 9.3 动态热加载

目前 `SkillLoader.reload()` 方法已经写好，只是没挂到 API。可以加一个 `POST /api/skills/reload`，改 skills.json 后不用重启服务就能生效。

### 9.4 Action 级别 opt-out

如果某些 Action（比如 `ReflectionAction` 自己）不想吃 skills，可以加一个类属性：

```python
class ReflectionAction(ActionBase):
    skip_skills = True  # 反思本身不需要领域 skill
```

然后在 `BaseAgent.run_action()` 里：

```python
skills_to_inject = "" if getattr(action, "skip_skills", False) else self._skills_prompt
```

## 十、与 MagicAI 的差异

| 特性 | MagicAI | 本方案 |
|---|---|---|
| 配置格式 | `config/skills.json` | `skills/skills.json`（单文件） |
| Skill 目录 | `skills/<id>/` | `skills/packs/<id>/`（避免和 `skills/skills.json` 混淆） |
| 注入方式 | 启动时拼到 prompt | ContextVar 运行时透明注入 |
| 对 LLM 可见 | system prompt 末尾 | user prompt 开头（ActionNode 约束） |
| 运行时 API | 无 | 预留 `get_all_skills_status()`，未挂 API |
| 多 Agent 绑定 | `inject_to: [agent_name_lowercase]` | `inject_to: [AgentType]`（大小写敏感） |

---

## 十一、一句话总结

> 把 Agent 的领域知识从硬编码 prompt 里解放成**独立的 `.md` 文件 + JSON 配置**；MagicAI Skill 包可直接 drop-in；用 ContextVar 做透明注入，7+ 个 Action 一行代码不用改；新增/禁用/切换只改文件不动代码。
