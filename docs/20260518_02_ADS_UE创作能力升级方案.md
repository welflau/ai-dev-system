# ADS UE 创作能力升级方案

> 日期：2026-05-18  
> 背景：ECC（everything-claude-code）已验证完整的 UE 自动创作流水线，ADS 只有编译/测试/部署，缺少「从无到有生成 UE 内容」的能力。  
> 目标：把 ECC 的 UE Python 桥接 + 内容生成能力移植进 ADS，让工单不只能修改代码，也能生成 BP / 关卡 / 资产。

---

## 一、现状差距

| 能力 | ECC | ADS | 差距 |
|---|:---:|:---:|---|
| UE Python 桥接（发送代码到 Editor）| ✅ | ❌ | 核心缺失 |
| Blueprint 生成 | ✅ | ❌ | 无法新建 BP |
| 关卡自动布置 | ✅ | ❌ | 无法生成关卡 |
| C++ 骨架生成 | ✅ | ❌ | 需要手写模板 |
| GDD 生成 | ✅ | ⚠️ | ADS 有 PM Agent 但不产 GDD |
| 资产扩展（/ue-extend）| ✅ | ❌ | 无 Python API 扩展 |
| 引擎编译 | ✅ | ✅ | 已有 |
| 自动化测试 | ✅ | ✅ | 已有 |
| 打包（RunUAT）| ❌ | ✅ | ADS 更强 |
| 引擎自动发现（注册表）| ⚠️ | ✅ | ADS 更强 |
| 工单状态机 | ❌ | ✅ | ADS 独有 |

## 三、实现方案（按优先级）

### P0：移植 ue_python_bridge（基础设施，1-2天）

**来源**：`ECC/scripts/ue_python.py`（~250行，已验证）

**目标位置**：`ADS/backend/engines/ue_python_bridge.py`

**改造点**：
- 移除 ECC 的文件读取入口点（`__main__`）
- 改为 Python 函数 `async def run_python(code: str, project_id: str) -> dict`
- 复用 ADS 已有的 `ue_resolver.py` 提供引擎路径
- 返回 `{success, stdout, stderr, exit_code}`

```python
# 调用示例
from engines.ue_python_bridge import run_python

result = await run_python("""
import unreal
bp = unreal.EditorAssetLibrary.load_asset('/Game/BP_Enemy')
print(f'已加载：{bp}')
""", project_id="PRJ-xxx")
```

**新增 Action**：`actions/chat/ue_call.py` 已有 UCP 协议版，新增 `ue_run_python` Action 走 Python 桥。

---

### P1：BlueprintGenAction（BP 生成，3-5天）

**能力**：AI 根据需求描述生成 Blueprint，通过 Python 桥写入 UE Editor。

**参照**：`ECC/commands/ue-bp-gen.md` + `ECC/agents/ue-blueprint-coder`

**实现**：

```python
# backend/actions/ue_blueprint_gen.py
class BlueprintGenAction(ActionBase):
    """
    LLM 生成 Blueprint Python 脚本 → ue_python_bridge 写入 Editor
    """
    async def run(self, context):
        description = context.get("description")
        project_id  = context.get("project_id")
        
        # 1. LLM 生成 UE Python 代码（使用 ue-blueprint-patterns skill）
        py_code = await self._generate_bp_code(description)
        
        # 2. 通过 Python 桥发送到 UE Editor
        result = await run_python(py_code, project_id)
        
        # 3. 编译验证
        compile_code = "unreal.EditorAssetLibrary.save_all_dirty_assets()"
        await run_python(compile_code, project_id)
        
        return ActionResult(success=result["success"], ...)
```

**SOP 集成**：新增 `sop/fragments/ue_content_gen.yaml`，在 `architecture_done` 后插入，trait = `engine:ue5`。

---

### P1：LevelGenAction（关卡布置，3-5天）

**能力**：根据关卡设计文档（来自 PlannerAgent），通过 Python 桥在 UE Editor 里布置场景。

**参照**：`ECC/commands/ue-level.md`

**实现步骤**：
1. PlannerAgent 输出关卡描述（地形/灯光/出生点/NavMesh 要求）
2. LevelGenAction 调 LLM 生成 UE Python 脚本
3. 通过 `run_python` 执行：spawn actor、set transform、build NavMesh
4. `ue_screenshot.py`（已有）截图验收

---

### P2：UE Scaffold（C++ 骨架生成，5-7天）

**能力**：ADS DevAgent 生成 C++ 骨架文件，写入本地工程目录，然后触发 UBT 编译。

**参照**：`ECC/commands/ue-scaffold.md`

**与 ADS 现有流程的区别**：
- 现在 DevAgent 只修改已有代码（`plan_code_change`）
- 新增 `write_ue_module`：创建新 Module / 新类文件 → UBT 编译 → 自动修错（最多3轮）

---

### P3：ue-extend 集成（Python API 扩展，3天）

**能力**：当标准 UE Python API 不够用时，生成 C++ 扩展脚本（BlueprintFunctionLibrary）并热加载。

**价值**：BlueprintGen / LevelGen 遇到限制时自动扩展，不中断流程。

## 四、工单流程集成

新的 UE 工单流转路径（在已有编译/测试/部署之前插入内容生成阶段）：

```
pending
  ↓ (traits: engine:ue5)
ue_content_pending      ← 新增
  ↓ UEEditorAgent
ue_content_done
  ↓
development_done        ← 传统编码（C++ 修改）
  ↓
engine_compile_pending  ← 已有
  ↓
play_test_pending       ← 已有
  ↓
deploy_pending          ← 已有
  ↓
deployed
```

**关键点**：
- `ue_content_pending` 阶段专门处理「需要写入 UE Editor 的内容」
- 纯代码修改的工单继续走原有 development → compile 路径
- 通过工单 `type` 字段区分（`type=ue_content` vs `type=feature`）

## 六、技术风险与应对

| 风险 | 说明 | 应对 |
|---|---|---|
| Python 桥连接不稳定 | UDP 多播在复杂网络环境下可能失败 | 增加重试机制 + 连接超时检测 |
| LLM 生成的 UE Python 代码有错 | UE API 调用复杂，容易出错 | 移植 ECC 的 ue-blueprint-patterns skill 作为 few-shot |
| 多个 Editor 实例 | 可能连接到错误的项目 | 复用 ECC 的项目匹配逻辑（按 project_root 过滤） |
| 大规模关卡生成超时 | 关卡对象多时 Python 执行慢 | 分批执行 + 超时控制（600s）|
