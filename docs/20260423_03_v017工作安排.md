## 四、阶段 2 — v0.17 Trait-First（~14.5 天，**高强度**）

### 4.1 第一批：核心垂直切片（~11.5 天）

跑通"对话 → traits 累积 → 组装预览 → 建项目 → SOP/Skill/Agent 按 traits 生效"完整链路。

| Phase | 工作块详情 | 预计天数 | 关键产出 |
|---|---|---|---|
| **A 基础** | ① DB 迁移（`projects` 表加 `traits` / `traits_confidence` / `preset_id`）<br>② `backend/skills/rules/trait_taxonomy.yaml`（含 ticket_type / vcs / 功能性 trait 三组新维度）<br>③ `backend/skills/rules/presets.yaml`（每 preset 带 `keywords:`，支持 §9.2 匹配推荐）<br>④ **对话历史两级压缩**（借鉴 MagicAI §9.4，最近 N 全文 + 更早压缩 + 硬上限）<br>⑤ **preset 关键词匹配推荐工具**（§9.2，评分 + 维度冲突惩罚）<br>⑥ `confirm_project` schema 加 `traits` 必填 + `preset_id` 可选<br>⑦ ChatAssistant system prompt 加"反问规则" | **2.5 天** | 用户对话里能累积 traits；LLM 能按关键词推荐 preset；对话历史智能压缩 |
| **B Skill 重构** | ① `SkillLoader` 三层过滤（inject_to + traits_match + paths），缓存 key 改 (agent_type, traits_hash)<br>② `backend/skills/rules/global.md`（**只放技术栈无关规则**：语言一致性/命名/文档/保密）<br>③ 4 个现有 skill 重构：`fastapi-dev` / `react-dev` / `playwright-e2e` 加 `traits_match`；`git-workflow` 加 `vcs:git` trait（保留为 skill **不升 rules**）<br>④ **Skill 分组特化压制机制**（§3.B.4：Godot 多人压制通用多人）<br>⑤ `traits_match` 最佳实践文档化（all_of/any_of/none_of 何时用哪个） | **4.5 天** | skill 按 traits 精准注入，不再一锅端 |
| **C SOP/Agent/Action** | ① `backend/sop/_core.yaml`（通用 6 阶段骨架）<br>② `backend/sop/fragments/*.yaml`（smoke_test / engine_compile / asset_gen / play_test / multiplayer_stress / i18n_check...）<br>③ `sop_loader.compose(traits, ticket_type)` 组合器 + `insert_after/insert_before` + priority 合并<br>④ **ticket_type 维度过滤**（§9.3，fragments 加 `required_ticket_type`）<br>⑤ `ActionBase.available_for_traits` + `BaseAgent.available_for_traits`<br>⑥ `orchestrator.get_agents_for_project(project_id)` 按 trait 过滤<br>⑦ Fragments lint 规则（required_traits 应是 action available_for_traits 的子集）| **3.5 天** | 改 traits 就能改流水线；feature / bug-fix / polish 走不同子集 SOP |
| **C' Preview API** | ① `POST /api/projects/preview-assembly` 只读端点<br>② 返回结构：`{sop, agents, skills_by_agent, mcps, warnings, suggestions}`<br>③ 前端 `doConfirmProject` 卡片打 preview 渲染组装预览 | **1 天** | 用户每改一个 trait 都能实时看到流水线变化，决策透明 |

## 八、灵活性说明

以下顺序**不是硬死的**：

- **观察期可以压缩**：如果你想直接动 v0.17，2 个 P2 UI（分支树、diff 优化）可以**放到 v0.17 之后**做，不 block 主线
- **Phase A/B/C 可以部分重叠**：
  - Phase A 的 DB 迁移完成后，Phase B 的 SkillLoader 三层过滤 + Phase C 的 Agent 加 `available_for_traits` 可以并行起手（不同文件，合并冲突小）
  - Preview API (C') 依赖 B/C 的 loader 改造完成，无法前置
- **清理可以随时做**：legacy 文本协议清理不一定要等 v0.17 完整结束，Phase B 结束后就可以做（反正它们已经不被 default 路径使用了）
- **第二批可以插队**：如果 Phase A+B+C 跑通后发现某个 P2 UI 更紧急（比如用户要求），可以中断插队做

## 附：风险清单

开工前心里有数的几个不确定性：

1. **v0.16.5 观察期可能发现问题** —— 如果 Agent 新路径 fallback 率 > 10%，要先修了再做 v0.17（不要带病扩架构）
2. **SOP fragments 组合器的依赖排序** —— 同一个 `insert_after: dev` 的多个 fragment 按 priority 排没问题，但跨 fragment 的隐式依赖（如 `multiplayer_stress` 依赖 `play_test` 先存在）要 lint 检查出来
3. **Preview API 性能** —— 每次用户改一个 trait 都要完整跑一遍 skill/sop/agent/mcp 过滤，如果后期 skill 数量 ≥ 50 要考虑增量计算
4. **旧 Web 项目回归** —— 既有 6 个项目的 traits 要回填（`source: file_detected`），跑一次 batch script 不能破坏任何现有数据
5. **Trait 分类法设计偏差** —— 第一版 taxonomy 可能漏维度或维度粒度不对，要预留 1-2 天给"用了发现需要补维度"
