# 工单面板「当前阶段进度」区设计

> 日期：2026-04-26
> 起因：用户反馈"工单属性面板上没有自测进度/编译进度条"；当前过程是黑箱
> 版本上下文：v0.19.x（CI/CD 合并完成 + UE 自测方案已写）
> 决策范围：实现方案 A「工单面板 + 当前阶段进度」；方案 B（自测分层进度条）/ C（操作日志按工单过滤）单独提（见 `20260426_01_DevAgent_UE项目自测方案.md` §讨论）

---

## 1. 要解决的痛点

### 1.1 当前用户看到什么

工单属性面板（drawer）显示的是「**事件点**」时间线：

```
[14:01] DevAgent  develop  开始
[14:03] DevAgent  develop  自测 5/5 通过
[14:03] DevAgent           → development_done
[14:03] DevAgent  run_engine_compile  开始
[14:06] DevAgent           UBT 编译失败 → engine_compile_failed      ← 等 3 分钟才出
[14:06] DevAgent  fix_issues  开始
...
```

中间的 **3 分钟是黑箱**：UBT 有没有在跑？卡在哪一步？还要等多久？用户只能等最后的结果行出现。

### 1.2 SSE 日志去哪了

`ue_compile_log` / `ue_playtest_log` 等事件**有**，但推到的是**项目级「操作日志」**（右侧独立面板），不跟工单绑定。

**问题链**：
1. **并发工单会混** —— v0.15 并发调度里多个工单同跑，操作日志混一起没法区分
2. **不知道当前工单跑到哪** —— 得自己脑补"刚才 14:03 那个开始的 UBT 现在应该跑了几分钟"
3. **切工单查日志要滚屏** —— 要看另一个工单的 UBT 日志得手动过滤

### 1.3 为什么不是"进度条"

UBT / RunUAT / UnrealEditor-Cmd **原生都不输出百分比**（不像 apt-get / pytest）。做不了"28% / 45%"这类精确进度。能做的是：
- **阶段级进度**（当前跑到哪个 SOP stage / 哪个 action）
- **心跳活性**（有 stdout → 绿，60s 无新输出 → 黄，行动失败 → 红）
- **最新一行摘要**（显示 UBT / Automation 最近一行关键日志，方便一眼看"还没挂呢"）

这三个一起比纯进度条更有用。

---

## 2. 设计目标

1. 工单面板顶部加「📍 当前进度」区，任何工单处于 `*_pending` / `*_in_progress` / 各种 *_running 状态时都显示
2. 显示 4 个核心字段：**当前 action** / **已用时** / **活性状态** / **最新一行 log 摘要**
3. 点「查看操作日志 →」跳转到操作日志面板 + 自动过滤当前工单（依赖方案 C 的 ticket_id 过滤，不在本方案实现）
4. 工单完成 / 失败 / BLOCKED 时自动收起进度区
5. 复用现有 SSE 事件（`ue_compile_log` / `ue_playtest_log` / `ci_build_log`），不新增事件 topic

---

## 3. 数据模型

### 3.1 tickets 表新增 4 列

```sql
-- v0.19.x 迁移
ALTER TABLE tickets ADD COLUMN current_action TEXT;             -- 如 "DevAgent.run_engine_compile"
ALTER TABLE tickets ADD COLUMN current_action_started_at TEXT;  -- ISO 时间戳
ALTER TABLE tickets ADD COLUMN current_action_latest_log TEXT;  -- 最近一行关键输出（≤200 字符）
ALTER TABLE tickets ADD COLUMN current_action_updated_at TEXT;  -- 最近一次 log 心跳时间（判活性）
```

这 4 列走 `_auto_migrate`（跟 v0.19.1 `action_state` 同套路）。

**为什么 4 列而不是 1 列 JSON**：
- `started_at` / `updated_at` 查询时要算"已用时""静默时长"，放 JSON 里每次 parse 麻烦
- `current_action` 未来可能做统计（看 agent 平均跑多久），纯列好聚合

### 3.2 状态语义

| 字段组合 | 含义 |
|---|---|
| `current_action = NULL` | 工单处于"等状态"（pending / passed / BLOCKED / done 等），没有 action 正在跑 |
| `current_action 有值` + `updated_at` 在 60s 内 | 🟢 活跃中 |
| `current_action 有值` + `updated_at` 60s-300s 前 | 🟡 静默中（可能卡 UBT shader 编译 / 可能挂起） |
| `current_action 有值` + `updated_at` > 5min 前 | 🔴 僵尸（后端可能崩了但没来得及清字段） |
| `updated_at = NULL` + `started_at 有值` | ⚪ 刚开始，还没 log 回调 |

这些判活性**全在前端算**，不在后端写死——后端只负责写字段。

## 4. 后端改造

### 4.1 Orchestrator：action 开始 / 结束时更新字段

`backend/orchestrator.py` 里每次分派 action 前后：

```python
async def _dispatch_action(self, ticket, agent, action_name, context):
    # 开始
    await db.update("tickets", {
        "current_action": f"{agent.agent_type}.{action_name}",
        "current_action_started_at": now_iso(),
        "current_action_latest_log": None,
        "current_action_updated_at": None,
    }, "id = ?", (ticket["id"],))

    try:
        result = await agent.execute(action_name, context)
    finally:
        # 结束——清空字段
        await db.update("tickets", {
            "current_action": None,
            "current_action_started_at": None,
            "current_action_latest_log": None,
            "current_action_updated_at": None,
        }, "id = ?", (ticket["id"],))

    return result
```

### 4.2 心跳：所有 log_callback 同时写 DB

现有 UE Actions（`UECompileCheckAction` / `UEPlaytestAction` / `UEPackageAction`）已经有 `log_callback` 参数，走 SSE。扩展**再多做一件事**：过滤出"关键行"（error / warning / 阶段性标志）写到 `tickets.current_action_latest_log`。

**过滤规则**（避免 200 MB/min 写爆 SQLite）：
- 包含 `error` / `warning` / `fatal` / `compiling` / `linking` / `cooking` / `staging` / `result` 关键字
- 以 `[ubt]` / `[playtest]` / `[package]` / `[engine]` 等我们自己的里程碑前缀开头
- 去重：跟上次一样就不写
- **最少 5 秒 throttle**：`updated_at` 距今 < 5s 就只更新内存不落 DB

实现：在 orchestrator 分派前把 `_update_ticket_heartbeat` 包进 `log_callback` chain：

```python
# orchestrator._build_context 或 action 层
async def _ticket_heartbeat_cb(project_id, ticket_id):
    last_write = {"t": 0, "line": ""}
    _IMPORTANT = re.compile(r"(error|warning|fatal|\[ubt\]|\[playtest\]|\[package\]|compiling|linking|cooking)", re.I)

    async def cb(line):
        if not _IMPORTANT.search(line):
            return
        now = time.time()
        if line == last_write["line"]:
            return
        if now - last_write["t"] < 5:
            return
        last_write["t"] = now
        last_write["line"] = line
        try:
            await db.update("tickets", {
                "current_action_latest_log": line[:200],
                "current_action_updated_at": now_iso(),
            }, "id = ?", (ticket_id,))
        except Exception:
            pass
    return cb

# 在 dispatch 时套壳
original_log_cb = context.get("log_callback")
heartbeat = await _ticket_heartbeat_cb(project_id, ticket_id)
async def combined_cb(line):
    if original_log_cb: await original_log_cb(line)
    await heartbeat(line)
context["log_callback"] = combined_cb
```

### 4.3 API：查询当前进度

新增 `GET /api/projects/{pid}/tickets/{tid}/current-action`:

```python
{
    "current_action": "DevAgent.run_engine_compile",
    "started_at": "2026-04-26T14:03:12+08:00",
    "elapsed_ms": 187000,                  # 后端算好，前端不用自己算
    "latest_log": "[ubt] Running Internal UnrealHeaderTool ...",
    "latest_log_at": "2026-04-26T14:06:15+08:00",
    "silence_ms": 4200,                     # 距上次 log 多久
    "health": "active" | "silent" | "zombie" | "starting" | null
}
```

`health` 字段后端算好下发，前端直接染色。

为什么单独端点而不是 `GET /tickets/{tid}` 带这些字段：
- drawer 打开后**轮询频次高**（1-2s），单独 endpoint 避免拉完整工单
- 也支持批量（看板页面多工单同时显示"⚡ 运行中"）

**批量端点** `GET /api/projects/{pid}/tickets/current-actions?ids=T1,T2,T3` 返回 `{T1: {...}, T2: {...}}`。

### 4.4 SSE 事件：ticket_action_progress

drawer 打开时通过 SSE 实时收进度，不靠轮询。新增事件（复用 `event_manager`）：

```python
# 在 _ticket_heartbeat_cb 里，写 DB 的同时推 SSE
await event_manager.publish_to_project(project_id, "ticket_action_progress", {
    "ticket_id": ticket_id,
    "current_action": current_action,
    "latest_log": line[:200],
    "silence_ms": 0,
    "health": "active",
})
```

前端 drawer 订阅该事件 + 过滤 `ticket_id` 匹配当前打开的工单。

## 5. 前端改造

### 5.1 Drawer 顶部新增进度区

位置：drawer 现有「基本信息」卡之后、「历次日志流」之前。

初始打开时拉 `GET /current-action`；之后通过 SSE `ticket_action_progress` 事件实时更新。

HTML 骨架：

```html
<div id="ticketActionProgress" class="ticket-action-progress" style="display:none;">
  <div class="progress-head">
    <span class="progress-icon">⚡</span>
    <span class="progress-title">当前进度</span>
    <span class="progress-health"></span>  <!-- 🟢/🟡/🔴/⚪ -->
  </div>
  <div class="progress-action-line">
    <code class="progress-action-name">—</code>
    <span class="progress-elapsed">—</span>
  </div>
  <div class="progress-latest-log">—</div>
  <div class="progress-actions">
    <a class="action-link" onclick="scrollToLogPanel()">查看操作日志 →</a>
  </div>
</div>
```

5 秒 ticker 更新"已用时 / 静默时长"显示（纯前端计时，不重拉 API）。

### 5.2 Health 配色

```js
const HEALTH = {
    active:   { icon: '🟢', label: '运行中',     color: 'var(--success)' },
    silent:   { icon: '🟡', label: '静默中',     color: 'var(--warning)' },
    zombie:   { icon: '🔴', label: '疑似挂起',   color: 'var(--danger)'  },
    starting: { icon: '⚪', label: '刚启动',     color: 'var(--text-muted)' },
};
```

静默/僵尸 时 tooltip 给用户下一步动作提示：
- 🟡 silent：`UE 首次 shader 编译可能 2-5 分钟无输出，正常`
- 🔴 zombie：`进程可能已挂；可「查看操作日志」排查或在管理员界面手动重置工单`

### 5.3 UE 项目特化

对 `traits` 含 `engine:ue*` 的工单，进度区多显示一条"预期耗时":

| action | 预期耗时（首次 / 增量） |
|---|---|
| `run_engine_compile` | 3-5 min / 30-90 s |
| `run_playtest` | 3-5 min / 1-2 min（shader cache 命中）|
| `run_ci_deploy` (package_client) | 10-30 min / 3-5 min |
| `run_self_test` (v0.19.x+ Layer 2) | 30-90 s |

预期耗时表存前端常量，后端不下发（避免污染模型层）。

### 5.4 与"操作日志"面板联动

点「查看操作日志 →」：
1. 跳转到主视图底部的操作日志面板（滚动定位）
2. 若未来做了方案 C（ticket_id 过滤），自动把过滤器切到当前工单

## 7. 风险与考虑

### 7.1 写 DB 频次

即使 5s throttle，多工单并发时 `tickets` 表会变"热写表"。

应对：
- `current_action_latest_log` 有字符长度上限（200 字符 → 每行 SQLite 单次写 ~1KB）
- 5s throttle 实测可再加大到 10s（取决于 UBT 输出速率）
- 若成为瓶颈，下沉到内存 dict（key=ticket_id）+ 定时批量 flush，但**优先级低**（目前 SQLite write 10/s 是 easy mode）

### 7.2 僵尸字段清理

进程 crash 可能留下 `current_action 有值 + started_at > 1h 前` 的僵尸工单。

应对：
- 启动时 orchestrator 扫一遍，`started_at > 30min` 且 `status` 不是 running 类 → 清字段
- 前端 health 判 zombie 时允许用户手动「重置当前进度」按钮（只清字段，不改工单 status）

### 7.3 不是真进度条

用户可能习惯看 xx% 百分比。本方案只有"心跳 + 已用时 + 最新 log"，需 UI 明确标注"**无精确进度**（UE 工具链不提供百分比）"，避免预期错位。

可在进度区右上角加小标签 `无精确 %`，点开 tooltip 解释：`UE 构建工具链（UBT / RunUAT / UnrealEditor-Cmd）不输出百分比。我们只显示最新一行日志 + 活性状态。点"查看操作日志"看详细输出。`

### 7.4 权限

drawer 如果同时打开多个（tab），SSE 事件会被所有打开的 drawer 收到——这其实是想要的（同步多窗口），不用特别处理。

## 8. 预期效果

用户**从此**在工单面板能看到：

```
┌─ 工单 T-042 ─────────────────────────┐
│ 实现玩家重生逻辑                     │
│ 状态: 🔄 engine_compile_pending       │
├──────────────────────────────────────┤
│ 📍 当前进度            🟢 运行中      │
│ DevAgent.run_engine_compile          │
│ ⏱ 已用时 2m 34s · 🔄 4s 前有输出     │
│ [ubt] Compiling TestFPSGameMode.cpp  │
│ [查看操作日志 →]   无精确 %ⓘ        │
├──────────────────────────────────────┤
│ 📜 历次日志                          │
│ 14:01 DevAgent develop  开始         │
│ ...                                  │
└──────────────────────────────────────┘
```

**价值**：
- **有感等待**：编译 3-5 分钟不再是黑箱，用户看到"正在编哪个文件 + 刚才 4 秒还有输出" → 不焦虑不误以为挂了
- **异常感知**：🟡 静默 60s + / 🔴 僵尸 5min+ → 用户一眼看出"这次可能真挂了"
- **并发不混**：多工单同开，每个 drawer 只显示自己工单的进度

## 9. 跟 UE 自测方案（`20260426_01`）的协同

两个方案**互补**：
- UE 自测方案负责"**更早发现错**"（Layer 1 亚秒拦 80%）
- 本方案负责"**错的过程更透明**"（剩余 20% 要跑 UBT 时能看到活性 + 最新 log）

推荐顺序：
1. 先做 UE 自测 Phase ①-② （~1 天）—— 建立静态规则基础
2. 再做本方案 Phase ①-⑥ （~1 天）—— drawer 进度区
3. UE 自测 Phase ③ UBT SingleFile 预编译（~0.5 天）—— 跟本方案 ⑤-⑥ 同步验收

也可以反过来，**先做本方案** —— 这样 UE 自测 Layer 2 的 30-90s 就天然有活性反馈了。

---

## 10. 一句话总结

工单面板顶部加 4 字段进度区（action / 已用时 / 活性 health / 最新 log），复用现有 SSE 事件做活性心跳。不做百分比进度条（UE 工具链不给）。**~1.5 天**落地，让 3-5 分钟的 UBT / Package 等待从黑箱变有感。
