## 一、MetaGPT 核心架构

### 设计哲学

**Code = SOP(Team)** — 将标准操作流程（SOP）物化为 AI Agent 团队的协作规则。

### 核心公式

```
Agent  = LLM + Observation + Thought + Action + Memory
Multi  = Agents + Environment + SOP + Communication + Economy
```

### Agent 定义方式

MetaGPT 用 Python 类定义 Agent，核心是 **Role + Action** 模式：

```python
# 定义 Action（能力单元）
class WriteCode(Action):
    async def run(self, instruction: str):
        prompt = f"Write code for: {instruction}"
        return await self._aask(prompt)  # 调用 LLM

# 定义 Role（Agent 角色）
class Coder(Role):
    def __init__(self):
        self.set_actions([WriteCode])
        self._watch([DesignArchitecture])  # 监听上游 Action
    
    async def _act(self) -> Message:
        msg = self.get_memories(k=1)[0]    # 获取上游消息
        result = await self.rc.todo.run(msg.content)
        return Message(content=result, cause_by=WriteCode)
```

### 协作机制：Watch/Subscribe

```
UserRequirement
      ↓ (SimpleCoder watches UserRequirement)
SimpleWriteCode
      ↓ (SimpleTester watches SimpleWriteCode)
SimpleWriteTest
      ↓ (SimpleReviewer watches SimpleWriteTest)
SimpleReviewCode
```

Agent 通过 `_watch([ActionType])` 声明依赖，Environment 自动路由消息。

### 团队组装

```python
team = Team()
team.hire([ProductManager(), Architect(), Engineer(), QA()])
team.run_project("Create a 2048 game")
```

## 二、逐维度对比

### 2.1 Agent 角色

## 三、MetaGPT 的优势（我们可借鉴）

### 3.1 ⭐ SOP 可配置化

MetaGPT 的 Agent 流程通过 **Watch 订阅 + Action 类** 动态组装，而非硬编码。

```python
# MetaGPT: 声明式定义流程
class Coder(Role):
    self._watch([DesignArchitecture])  # 监听架构完成

class Tester(Role):
    self._watch([WriteCode])           # 监听代码完成
```

```python
# 我们: 硬编码在 transition_rules
transition_rules = {
    "architecture_done": {"agent": "DevAgent", "action": "develop"},
    "development_done": {"agent": "ProductAgent", "action": "acceptance_review"},
}
```

**改进方向**：将 transition_rules 改为可配置的 SOP 定义文件（JSON/YAML），前端可视化编辑流程。

### 3.2 ⭐ 事件驱动 vs 轮询

MetaGPT 是**事件驱动**（Agent 监听消息自动触发），我们是**轮询驱动**（10s 扫一次数据库）。

**改进方向**：引入内部事件总线，工单状态变更时直接触发下一个 Agent，减少延迟。

### 3.3 ⭐ Agent 抽象层

MetaGPT 的 Role + Action 分离设计更清晰：
- **Role**：角色身份（谁）
- **Action**：具体能力（做什么）
- 一个 Role 可以有多个 Action

我们的 Agent 是 Role 和 Action 混在一起的：
```python
class DevAgent:
    async def develop(self, context):  # Action 1
    async def rework(self, context):   # Action 2
    async def fix_issues(self, context): # Action 3
```

**改进方向**：抽离 Action 层，让 Agent 能力可组合、可扩展。

### 3.4 ⭐ Memory 系统

MetaGPT 有结构化的 Memory：
```python
msg = self.get_memories(k=1)[0]  # 获取最近 k 条记忆
```

我们的 "记忆" 分散在 artifacts/ticket_logs/llm_conversations 中，没有统一接口。

**改进方向**：封装统一的 Agent Memory 接口，整合历史产物、对话记录、上下文。

### 3.5 ⭐ 多 LLM 支持

MetaGPT 支持 GPT-4/Azure/Ollama/Groq/本地模型，我们目前只支持 Anthropic + OpenAI 兼容。

**改进方向**：增加 Ollama 本地模型支持，降低 API 成本。

## 四、我们的优势（MetaGPT 缺失）

### 4.1 ✅ Harness 工程化

MetaGPT 的理念是"模拟软件公司"，但**没有工程约束层**：
- 没有质量门禁（生成完就结束）
- 没有降级兜底（LLM 失败即崩溃）
- 没有增量开发（每次从零生成）

我们的 Harness 模式提供了 7 层约束，确保 AI 的不确定性不会直接传导到产出物。

### 4.2 ✅ 持续运营能力

MetaGPT 是"一次性生成"工具——输入需求，输出代码，结束。

我们支持**项目全生命周期管理**：
- 创建项目 → 持续提需求 → 迭代开发 → 持续部署
- 需求可重新执行、工单可打回返工
- 项目历史可追溯

### 4.3 ✅ Git 原生集成

MetaGPT 把代码写到文件夹就结束了，我们有完整的 Git 工作流：
- 每个需求独立分支
- 有意义的 commit message
- 自动合并（feat → develop → main）
- GitHub 远程推送

### 4.4 ✅ CI/CD + 环境隔离

MetaGPT 不管部署。我们有：
- 自动构建流水线
- dev/test/prod 三环境隔离
- 自动部署 + 预览

### 4.5 ✅ 全链路可追溯

MetaGPT 运行完内存释放，无法追溯。我们 7 张表记录全过程：
- 从部署的代码 → 倒推到用户的一句话
- 每次 LLM 调用的完整 prompt + response 都有记录

### 4.6 ✅ Web 可视化

MetaGPT 是 CLI 工具。我们有完整的 Web UI：
- 看板（Kanban）、甘特图、依赖关系图
- AI 聊天面板（自然语言操控）
- 实时日志 + SSE 推送
- 仓库文件浏览器

## 六、总结

```
MetaGPT 的核心价值：SOP 抽象 + Agent 可组合 + 事件驱动
AI Dev System 的核心价值：Harness 工程化 + 持续运营 + 全链路追溯

两者互补：
  MetaGPT 解决 "如何让 AI 协作"
  AI Dev System 解决 "如何让 AI 协作的结果可靠地投产"
```

**最值得借鉴的 3 件事**：
1. SOP 配置化（从硬编码到声明式）
2. 事件驱动（从轮询到消息触发）
3. Role + Action 分离（能力可组合）

**我们独有的护城河**：
1. Harness 7 层约束（质量门禁 + 降级兜底）
2. Git 原生集成 + CI/CD + 三环境部署
3. 7 张表全链路审计追溯
4. Web 可视化管理 + AI 聊天面板
