# AI 自动开发系统 — 技术架构文档

> 版本: v0.12.1 | 日期: 2026-04-13 | 总代码量: ~20,500 行

---

## 一、系统概述

AI 自动开发系统是一个**工单驱动的 AI 自动开发管理平台**，通过多 Agent 协作，实现从需求分析、架构设计、代码开发、测试验收到部署上线的全自动化软件开发流程。

**核心能力**：
- 需求自动拆解为可执行工单
- 6 个 AI Agent 协作完成软件开发全流程
- Git 仓库集成（分支管理、代码提交、合并）
- CI/CD 自动构建部署（dev/test/prod 三环境隔离）

---

## 一.五、核心架构思想：Harness 模式

### 为什么需要 Harness

AI（LLM）本身**不可靠**：会幻觉、会截断、会生成废代码、会忽略已有代码从零重写。直接让 AI 写代码并部署，等于把不确定性直接推到生产环境。

**Harness（驾驭套件）** 的核心理念：**不信任 AI 的每一步输出，而是构建工程化的约束层去驾驭、验证、兜底。**

这与 Claude Code 的 Harness 思想一脉相承——AI 是引擎，Harness 是方向盘和刹车。

### 架构全景

```
┌─────────────────────────────────────────────────────────┐
│                   AI Dev Harness                         │
│                                                          │
│   ┌─────────────┐    约束规则     ┌─────────────┐       │
│   │   LLM AI    │ ─────────────→ │   Agent     │       │
│   │  (能力层)    │ ← 上下文注入 ── │  (执行层)    │       │
│   └─────────────┘                └─────────────┘       │
│         ↑                              ↓               │
│    Prompt 工程                   结构化输出              │
│    (精简/JSON约束)               (files dict)           │
│         ↑                              ↓               │
│   ┌─────────────┐                ┌─────────────┐       │
│   │  知识注入    │                │  质量门禁    │       │
│   │ (已有代码    │                │ (自测→验收   │       │
│   │  架构上下文  │                │  →5层测试)   │       │
│   │  同需求工单) │                └─────────────┘       │
│   └─────────────┘                      ↓               │
│                          ┌─────────────────────┐       │
│                          │   编排引擎           │       │
│                          │  (Orchestrator)      │       │
│                          │  状态机 + 10s 轮询   │       │
│                          │  依赖检查 + 僵尸检测  │       │
│                          └─────────────────────┘       │
│                                    ↓                    │
│              ┌──────────────────────────────┐           │
│              │      工程规范层               │           │
│              │  Git 分支隔离 + CI/CD 流水线  │           │
│              │  dev/test/prod 环境隔离       │           │
│              │  降级兜底 + 自动重试          │           │
│              └──────────────────────────────┘           │
└─────────────────────────────────────────────────────────┘
```

### 7 层 Harness 详解

#### 1. Prompt Harness — 约束 LLM 输出

| 约束 | 手段 | 效果 |
|------|------|------|
| 输出格式 | 强制 JSON 返回 + 自动解析 + 二次提取 | 避免 Markdown/自由文本导致解析失败 |
| 输入精简 | 架构只保留关键字段、代码只取入口文件前 2000 字符 | input tokens 控制在合理范围，减少截断 |
| 角色锁定 | 每个 Agent 有专属 system prompt | 防止 LLM "跑偏"做不属于它的事 |

#### 2. Context Harness — 注入已有知识

| 注入内容 | 来源 | 目的 |
|---------|------|------|
| 已有文件列表 | `git_manager.get_file_tree()` | Agent 知道项目里有什么 |
| 入口文件代码 | `git_manager.get_file_content()` | 在现有代码上修改，不从零重写 |
| 同需求工单 | `tickets WHERE requirement_id` | 知道别的工单做了什么，不重复 |
| 架构设计 | `artifacts WHERE type=architecture` | 开发遵循架构而非自由发挥 |

**解决的问题**：没有 Context Harness 时，每个工单独立生成代码 → 后面工单覆盖前面工单 → 互相冲突。

#### 3. Agent Harness — 单一职责 + 状态机

```
每个 Agent 只做一件事：
  ProductAgent  → 只拆单/只验收
  ArchitectAgent → 只设计架构
  DevAgent      → 只写代码
  TestAgent     → 只测试

状态机严格控制流转：
  pending → architecture → development → review → testing → done
  不允许跳过步骤、不允许逆向流转（除打回返工）
```

**解决的问题**：如果让一个 Agent 做所有事（分析+设计+开发+测试），它会顾此失彼，质量不可控。

#### 4. Quality Gate Harness — 多层质量门禁

```
                    DevAgent 自测（5 项检查）
                          ↓ 通过
                    ProductAgent 验收
                      ↓ 不通过 → 打回返工
                    TestAgent 5 层测试
                      ↓ 通过率 < 60% → 打回修复
                    CI/CD 构建测试
                      ↓ 失败 → 阻止合并
```

每一层都是独立的质量关卡，任何一层失败都会阻止代码进入下一阶段。

#### 5. Git Harness — 版本安全

| 机制 | 作用 |
|------|------|
| 每个需求一个 feat 分支 | 需求间代码隔离，不互相影响 |
| Agent 提交前切到正确分支 | 防止代码提交到 main 分支 |
| switch_branch 前 git clean | 清理 .pyc 等冲突文件 |
| 有意义的 commit message | `[DevAgent] 开发: 排行榜功能 (index.html)` |
| feat → develop → main 逐级合并 | 代码经过测试才能到生产 |

#### 6. Environment Harness — 部署隔离

```
dev  (端口 base+0)    ← Agent 每次提交自动部署，快速预览
test (端口 base+100)  ← develop 构建通过后部署，集成验证
prod (端口 base+200)  ← main 构建通过后部署，正式上线

各环境独立目录、独立端口、独立分支，互不干扰。
```

#### 7. Fallback Harness — 降级兜底

| 失败场景 | 降级策略 |
|---------|---------|
| LLM 超时/不可用 | 规则引擎生成可运行的模板代码 |
| JSON 解析失败 | 二次提取（找 `{` 到 `}` 子串） |
| Agent 执行异常 | 记录错误日志，工单状态不丢失 |
| 工单卡住 60 秒 | 僵尸检测自动重置状态 |
| 分支切换失败 | 继续在当前分支工作（非致命） |

**核心原则**：系统永远不会因为 AI 的一次失败而停摆，每层都有兜底。

### Harness 与 Agent 的关系

```
没有 Harness 的 AI：
  用户 → LLM → 代码 → 直接部署 → 💥 灾难

有 Harness 的 AI：
  用户 → [需求约束] → LLM → [格式约束] → [上下文注入] → Agent 
       → [自测] → [验收] → [5层测试] → [Git分支] → [CI/CD] 
       → [环境隔离] → ✅ 安全上线
```

**Harness 不是限制 AI 的能力，而是把 AI 的不确定性转化为工程的确定性。**
- AI 聊天面板（自然语言操控项目）
- 实时 SSE 事件推送 + 看板可视化

## 五、Agent 系统

### 6 个 AI Agent

| Agent | 职责 | 输入 | 输出 |
|-------|------|------|------|
| **ProductAgent** | 需求分析、拆单、验收 | 需求描述 | PRD + 工单列表 + 依赖关系 |
| **ArchitectAgent** | 增量架构设计 | 工单 + 已有代码 | 架构方案 + architecture.md |
| **DevAgent** | 编写代码 + 自测 | 架构 + 已有代码 | 代码文件 + dev-notes.md + 自测结果 |
| **TestAgent** | 5 层测试 + 报告 | 代码产物 | test-report.md + 测试用例 + 截图 |
| **ReviewAgent** | 代码审查 | 代码文件 | 评分 + 问题 + 建议 |
| **DeployAgent** | 环境部署 | 项目 + 环境类型 | 预览 URL + Dockerfile |

### 状态转换规则

```
工单状态                    → 下一步 Agent              → 新状态
────────────────────────────────────────────────────────────────
pending                    → ArchitectAgent.设计架构     → architecture_in_progress
architecture_done          → DevAgent.开发              → development_in_progress
development_done           → ProductAgent.验收          → (passed/rejected)
acceptance_passed          → TestAgent.测试             → testing_in_progress
acceptance_rejected        → DevAgent.返工              → development_in_progress
testing_failed             → DevAgent.修复              → development_in_progress
testing_done               → (终态，CI/CD 接管)
```

### 增量开发机制

1. **_build_context()** 在 Agent 执行前读取仓库已有文件
2. **ArchitectAgent** 看到已有代码后设计增量架构（不推翻重来）
3. **DevAgent** 在现有 index.html 基础上添加功能（不从零生成）
4. **自测**：DevAgent 完成后自动运行 5 项检查（文件/入口/语法/规范/非空）

### TestAgent 5 层测试

| 层级 | 内容 | 前端 | 后端 |
|------|------|------|------|
| 1. 静态分析 | 文件完整性 + 语法 + 入口 | JS node --check | Python compile |
| 2. 代码审查 | LLM 评分 1-10 | ✓ | ✓ |
| 3. 功能测试 | 按类型分发 | HTTP server + 页面内容检查 | Python import 检查 |
| 4. 测试用例生成 | LLM 生成 pytest | ✓ | ✓ |
| 5. pytest 执行 | 运行生成的测试 | ✓ | ✓ |

通过率 < 60% → **testing_failed** → DevAgent 自动返工

## 六、核心流程

### 需求到部署全流程

```
┌──────────────────────┐
│  用户提交需求          │  前端表单 / AI 聊天对话
└──────────┬───────────┘
           ↓
┌──────────────────────┐
│  ProductAgent 拆单    │  分析需求 → 生成 PRD → 拆分为工单
│  创建 feat 分支       │  feat/YYYYMMDD-req-XXXXXX
└──────────┬───────────┘
           ↓
┌──────────────────────┐
│  编排器 10s 轮询      │  扫描 pending 工单 → 检查依赖
└──────────┬───────────┘
           ↓
┌──────────────────────┐
│  ArchitectAgent      │  读取已有代码 → 增量架构设计
└──────────┬───────────┘
           ↓
┌──────────────────────┐
│  DevAgent 开发+自测   │  在已有代码基础上修改 → 5 项自测
│  代码提交到 feat 分支  │  commit: "[DevAgent] 开发: 排行榜功能 (index.html)"
└──────────┬───────────┘
           ↓ 自动部署 dev 环境
┌──────────────────────┐
│  ProductAgent 验收    │  审查代码是否符合需求
│  不通过 → 打回返工     │
└──────────┬───────────┘
           ↓
┌──────────────────────┐
│  TestAgent 测试       │  5 层测试 → 生成报告 + 截图
│  不通过 → 打回修复     │  通过率 < 60% 自动打回
└──────────┬───────────┘
           ↓
┌──────────────────────┐
│  需求完成             │  所有工单 testing_done
│  feat → develop 合并  │  自动合并 + 部署 test 环境
└──────────┬───────────┘
           ↓
┌──────────────────────┐
│  CI/CD Pipeline      │  develop 构建 → merge main → master 构建
│  自动部署 prod 环境    │  端口隔离：dev+0, test+100, prod+200
└──────────────────────┘
```

### Git 分支策略

```
main (生产)  ←──── develop (测试) ←──── feat/YYYYMMDD-req-XXXXXX (开发)
     ↑                  ↑                          ↑
  prod 部署          test 部署              dev 部署 + Agent 工作
```
