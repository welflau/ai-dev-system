## 一、背景

### 1.1 现状：Reflexion 只在"同一工单内"共享反思

Reflexion（P0 已落地）让 DevAgent 失败→反思→重试，本工单内最多累积 3 条反思注入 prompt：

```python
# backend/agents/dev.py:_enrich_retry_context
refl_logs = await db.fetch_all(
    """SELECT detail FROM ticket_logs
       WHERE ticket_id = ? AND action = 'reflection'
       ORDER BY created_at DESC LIMIT 3""",
    (ticket_id,),
)
```

**问题**：这些反思**永远只在该工单生命周期内被读取**。工单 A "加访问计数"踩坑 3 次学到"必须改 index.html 加可见 DOM"；工单 B "加点击计数"来了还会再踩同样的坑——因为 ticket_id 不同，B 检索不到 A 的反思。

### 1.2 真实案例

实测（docs/20260419 里的反思方案提到）："加访问计数"需求被打回 5 次 → force_pass。其中第 2-5 次的反思里反复指出"DevAgent 自测只跑 pytest，没查前端 DOM"。如果有 Failure Library，后续出现任何"加 XX 计数器"类需求，第 1 次就能看到这条历史反思。

### 1.3 目标

**跨工单反思学习**。DevAgent 失败 → 反思存档 → 将来相似工单检索到历史案例 → 注入 ReflectionAction 的 prompt 让它借鉴。

## 二、核心设计

### 2.1 数据流

```
【写入】
DevAgent 失败 → ReflectionAction 输出反思 → orchestrator._log(action='reflection')
            → failure_library.record()  ← 新增
            → failure_cases 表

【解决标记】
ticket 转 acceptance_passed / testing_done
            → failure_library.mark_resolved(ticket_id)
            → UPDATE resolved=1

【检索】
DevAgent rework → _enrich_retry_context
            → failure_library.search_similar(agent_type, module, failure_type, desc, ...)
            → context['similar_failures'] = [case1, case2, case3]
            → ReflectionAction.run() 渲染「历史相似失败」段
            → LLM 产出 strategy_change 时借鉴历史已解决策略
```

### 2.2 表结构 `failure_cases`

```sql
CREATE TABLE IF NOT EXISTS failure_cases (
  id                   TEXT PRIMARY KEY,
  project_id           TEXT,                -- 项目作用域；NULL = 全局（预留）
  requirement_id       TEXT,
  ticket_id            TEXT,                -- 反向追溯
  agent_type           TEXT NOT NULL,       -- DevAgent（M1 范围）
  module               TEXT,                -- frontend/backend/api/...
  failure_type         TEXT NOT NULL,       -- acceptance_rejected / testing_failed
  ticket_title         TEXT,
  ticket_description   TEXT,
  root_cause           TEXT NOT NULL,
  missed_requirements  TEXT,                -- JSON array
  strategy_change      TEXT,
  specific_changes     TEXT,                -- JSON array
  confidence           REAL DEFAULT 0.5,
  keywords             TEXT,                -- 派生：空格分隔关键词，LIKE 检索
  resolved             INTEGER DEFAULT 0,   -- 0=失败未解决，1=后续策略成功
  resolved_at          TEXT,
  created_at           TEXT NOT NULL,
  updated_at           TEXT NOT NULL
);

CREATE INDEX idx_failure_cases_lookup
  ON failure_cases(agent_type, module, failure_type, resolved);
CREATE INDEX idx_failure_cases_ticket ON failure_cases(ticket_id);
CREATE INDEX idx_failure_cases_project ON failure_cases(project_id);
```

### 2.3 检索策略（无 embedding / 无 FTS）

当前本项目无 FTS5 / embedding / vector 基础设施（探索代码确认）。M1 用**关键词 LIKE 匹配**实现：

**硬过滤**（SQL WHERE）：
- `agent_type`（精确）
- `failure_type`（精确）
- `module`（精确，若 None 则跳过）
- `project_id = :self OR project_id IS NULL`（项目作用域 + 全局预留）
- `ticket_id != :current`（排除自己）

**软匹配**（至少命中一个）：
- 从当前工单描述提取关键词集合
- `keywords LIKE '%kw1%' OR keywords LIKE '%kw2%' OR ...`（最多 8 个 LIKE，防 SQL 超长）

**排序**：
```
ORDER BY resolved DESC,    -- 已验证成功的优先
         confidence DESC,  -- 当时反思置信度高的优先
         created_at DESC   -- 新的优先
LIMIT 3
```

SQLite 对 UTF-8 中文 LIKE 是原生支持的，无需额外配置。

### 2.6 resolved 标记

在 `_handle_agent_result` 里两个成功分支加：

- `ProductAgent` 判定 `acceptance_passed` → `mark_resolved(ticket_id)`
- `TestAgent` 判定 `testing_done` → `mark_resolved(ticket_id)`

两处都用 try/except 降级。

**语义**：mark_resolved 把该 ticket 下所有 failure_cases 的 resolved 从 0 翻转为 1。表示"历次反思策略最终奏效"——后续检索时优先展示这些已解决案例的成功策略。

### 2.7 注入 Reflexion prompt

`backend/agents/dev.py:_enrich_retry_context` 追加：

```python
context["similar_failures"] = await failure_library.search_similar(
    agent_type=self.agent_type,
    failure_type=context.get("failure_type"),
    module=context.get("module"),
    project_id=context.get("project_id"),
    ticket_description=context.get("ticket_description", ""),
    current_ticket_id=ticket_id,
    limit=3,
)
```

`backend/actions/reflection.py::ReflectionAction.run()` 在 `user_prompt` 里加段落：

```
## 历史相似失败（跨工单，供参考避免重复踩坑）

[案例 1 — ✅已解决 | module=frontend] 工单「加访问计数」
  根因: 上次没在 index.html 加计数器 DOM
  策略: 直接改 index.html 的 body 插入 div + script
  具体修改:
    - 在 </body> 前加 <div id="counter">
    - 加 <script> 用 localStorage 计数

[案例 2 — ❌未解决 | module=frontend] 工单「加点击统计」
  根因: ...

⚠️ 若上面有 ✅已解决 策略，优先借鉴；❌未解决 策略则要说清本次为什么能规避。
```

同时在 `_SYSTEM_PROMPT` 里追加硬要求：
> 如果用户消息给出了"历史相似失败（跨工单）"且有标记为 ✅已解决 的案例，本次 strategy_change **优先借鉴**该成功策略；对 ❌未解决 的案例要明确说明这次为什么能避开同样的坑。

### 2.8 一次性回灌（bootstrap）

`backend/_migrate_failure_library.py`：

```bash
cd backend && PYTHONIOENCODING=utf-8 python _migrate_failure_library.py
# 📦 回灌 17 条历史反思到 failure_cases
```

扫描 `ticket_logs WHERE action='reflection'`，解析 detail JSON 的 reflection 字段，逐条调 `record()`。**幂等**：按 `(ticket_id, root_cause)` 去重，重复跑不会产生冗余。

启动时**不**自动运行（避免每次启动扫全表）。用户按需手工调用。

## 五、未来扩展

### 5.1 FTS5 升级（中等工作量）

当 failure_cases 行数 > 1000 时，LIKE 匹配效果会变差。升级方案：
- 加 `CREATE VIRTUAL TABLE failure_cases_fts USING fts5(root_cause, strategy_change, keywords)`
- 同步 trigger 维护
- `search_similar` 改用 `MATCH` 查询

预计 1 天工作量。

### 5.2 Embedding 检索（大工作量）

调 `text-embedding-3-small` 把 `root_cause + ticket_description` 向量化存表，检索时算 cosine similarity。

- 成本：每条案例约 $0.00002 + 存 1536 维向量
- 效果：能捕获"加计数器"和"添加访问统计"这类同义匹配
- 实现：新列 `embedding BLOB`，新方法 `search_similar_vector`

预计 2-3 天工作量（含向量化、余弦 SQL、降级到 LIKE）。

### 5.3 扩展到其他 Agent

M1 只有 DevAgent 触发反思。未来若 TestAgent / ArchitectAgent 也引入 Reflexion，只需：
1. 在它们的 `_enrich_retry_context` 等价位置调 `search_similar(agent_type=self.agent_type)`
2. 写入逻辑已经按 agent_name 区分，无需改

### 5.4 策略有效性统计

每条 `failure_cases` 的 `resolved` 字段天然可以统计"哪些 strategy_change 最常导致成功"。加一个只读 API `GET /api/failure_cases/stats` 返回：
- 总案例数 / 已解决率
- Top N `strategy_change` 关键词（用于优化后续 Reflexion prompt）

### 5.5 前端展示

前端加一个"项目失败档案"Tab，列出本项目的 failure_cases：
- 表格：时间 / 工单 / 根因 / 策略 / 状态
- 点击进入详情看完整 reflection
- 让人工 reviewer 能手动精选"全局案例"（`project_id=NULL`）

## 六、风险与降级

| 风险 | 缓解 |
|---|---|
| record() 写入失败阻塞 orchestrator | try/except 包裹，warning log，不影响主流程 |
| search_similar 查询慢 | 已加复合索引 `(agent_type, module, failure_type, resolved)`；LIKE 数 ≤ 8 |
| 关键词提取误伤（中文分词粗糙） | 正则 2+ 字片段够用；精度不足时再上 jieba |
| 跨工单 prompt 膨胀 | 只取 top 3，每字段截断 ≤ 200 字；反思段落总计约 500 token |
| 回灌脚本重复跑重复写 | 按 `(ticket_id, root_cause)` 去重 |
| 老数据库无 failure_cases 表 | SCHEMA_SQL `IF NOT EXISTS`；首次启动自动建 |

## 七、与 MagicAI 的差异

MagicAI 的 `src/reflection/README.md` 规划了 Failure Library（✅ 完整实现），但**实际代码没写**——只是 README 级别的 API 签名。本方案是精简版实现：

| 特性 | MagicAI README | 本方案（M1） |
|---|---|---|
| 存储 | Weaviate/Neo4j（L3） | SQLite 表 |
| 检索 | 语义检索（embedding） | LIKE 关键词 + 结构化过滤 |
| 触发范围 | 所有 Agent | 仅 DevAgent rework/fix_issues |
| 解决追踪 | 未提 | 有 `resolved` + `mark_resolved` |
| 回灌机制 | 未提 | `_migrate_failure_library.py` 一次性 |

---

## 八、一句话总结

> Reflexion 把单工单反思变成结构化数据；Failure Library 把这些结构化数据**沉淀到 DB 并提供跨工单检索**。让"访问计数被打回 5 次"这种教训只需要被真实学一次——下次相似需求第 1 次就能从 prompt 里看到前人的成功策略。
