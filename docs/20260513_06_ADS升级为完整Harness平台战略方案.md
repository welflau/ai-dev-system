# ADS 升级为完整 Harness 平台战略方案

> 日期：2026-05-13  
> 基于：Claude Code 深度分析 + SuperPower 对比 + ADS 现状评估  
> 定位：架构决策文档，面向技术负责人

---

## 一、背景与结论

经过对 Claude Code（1884 个 TS 文件）、SuperPower（纯 Markdown 行为框架）和 ADS（AI 自动开发系统）三个项目的系统性分析，得出如下定性结论：

| 项目 | 本质定位 | Harness 特性含量 |
|------|---------|----------------|
| **Claude Code** | AI 执行基础设施（Harness 本身） | 100% |
| **ADS** | 领域专用应用平台（内建部分 Harness） | 58% |
| **SuperPower** | 行为注入框架（寄生于 Harness） | 0%（借用宿主）|

**核心判断**：ADS 处于独特的"中间层"位置——向上已经完成了 Claude Code 完全没有的应用逻辑（工单体系、SOP 工作流、多角色 Agent），向下自己实现了部分 Harness 能力（工具引擎、状态持久化、技能注入）。

58% 不是失败，是有意取舍的结果。但剩余 42% 中有几个特性会成为 ADS 规模化时的瓶颈。

---

## 二、Harness 思想与三层圈层模型

Claude Code 将 Harness 思想实现为三个同心圈层，从外到内依次收紧：

```
┌──────────────────────────────────────────────────────┐
│  组织策略层 (policyLimits / remoteManagedSettings)    │ ← 企业管控，不可逾越
│  ┌────────────────────────────────────────────────┐  │
│  │  用户权限层 (settings.json + hooks)             │  │ ← 可配置
│  │  ┌──────────────────────────────────────────┐  │  │
│  │  │  工具声明层 (Tool.checkPermissions)      │  │  │ ← 工具自报能力
│  │  │  ┌────────────────────────────────────┐  │  │  │
│  │  │  │  Claude 模型 (QueryEngine 包裹)    │  │  │  │ ← 被服务的核心
│  │  │  └────────────────────────────────────┘  │  │  │
│  │  └──────────────────────────────────────────┘  │  │
│  └────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────┘
```

**Harness 的本质**：Claude 模型本身不拥有任何能力，它只能"请求"执行工具。是否真正执行，由外层框架根据声明、规则、Hooks 和组织策略共同决定。模型是被服务的对象，Harness 是让这种服务安全、可审计、可控制的基础设施。

Claude Code 的权限决策链体现了这一思想的极致：

```
工具调用请求
  └─► Step 1：alwaysAllow / alwaysDeny 规则
        └─► Step 2：身份路由
              ├─► CoordinatorHandler（后台 Worker，等待自动化）
              ├─► SwarmWorkerHandler（子 Agent，转发给 Leader）
              └─► InteractiveHandler（主线程，展示对话框）
                    └─► Step 3：Speculative Classifier（2秒快速判断）
                          └─► Step 4：外部 Hooks 脚本
                                └─► Step 5：用户对话框（人工最终确认）
```

## 三、ADS 现有 Harness 能力盘点（已有的 58%）

### 3.1 工具执行引擎（★★★ 完善）

- **Chat Tool**：38+ 个 Action，遵循 Anthropic `tool_use` 格式
- **Action 类结构**：每个 Action 有 `tool_schema()` + `run(context)` + `available_for_traits`
- **Orchestrator Actions**：10 个 Agent Role 各自封装 ActionNode 执行链
- **双模式降级**：新路径（tool_use）+ 旧路径（文本协议）互为兜底

### 3.2 跨轮次状态持久化（★★★ 超越 Claude Code）

- **15+ 张表**：projects / tickets / requirements / chat_messages / agent_memory / failure_library
- **WAL 模式**：SQLite 支持并发读写，进程重启不丢失状态
- **相比 Claude Code**：Claude Code 依赖内存 cache，进程重启即失；ADS 的持久化更彻底

### 3.3 多 Agent 编排（★★ 中等）

```
ProductAgent → PlannerAgent → ArchitectAgent
                                    ↓
TestAgent ← DevAgent ← UXAgent
    ↓
ReviewAgent → DeployAgent
```

- **YAML SOP 状态机**：热重载，Trait-based 条件分支
- **不足**：顺序调度，Agent 不能在执行中自主派发子任务

### 3.4 三层 Skill 注入（★★★ 超越 SuperPower）

```
inject_to（Agent 粒度）
  → traits_match（项目类型粒度）
    → paths（文件路径粒度）
      → 主动触发（load_skill，按需加载）
```

比 SuperPower 的 Markdown Skill 更细粒度，支持动态触发而非全量注入。

### 3.5 规则型权限控制（★★ 中等）

- `_INTERNAL_ONLY_ACTIONS`：屏蔽 LLM 直接访问敏感 Action
- Shell 白名单：`git / npm / python / pytest / docker`
- Trait-based Access：按项目类型过滤可用工具和 MCP 服务

## 四、缺口分析（缺失的 42%）

### 4.1 特性分类

| 缺失特性 | 类型 | 缺口严重性 | 优先级 |
|---------|------|----------|--------|
| Pre/Post Tool Hooks | 可补全型 | 中——运维无依托点 | P0 |
| 预算/配额约束 | 可补全型 | 中——失控风险 | P0 |
| QueryEngine 抽象 | 可补全型 | 高——技术债积累 | P1 |
| 异步权限审批 | 设计适配型 | 中——高风险操作无人工确认 | P2 |
| 递归子 Agent | 架构升级型 | 低——当前顺序调度够用 | P3 |

### 4.2 为什么这些缺口没有更早出现

ADS 在单机单项目场景下，这些缺口不会触发。但在以下场景会暴露：
- **规模化**：同时跑 50+ 工单，无预算约束会失控
- **高风险操作**：Agent 删文件、force-push，无审批链兜底
- **故障溯源**：没有 Pre/Post Hooks，工具调用错误难以追踪根因
- **架构演进**：`api/chat.py` 88KB 的大文件成为所有扩展的阻力

## 五、补全方案详述

### Phase 0：理解 ADS 与 Claude Code 的关系

在动手之前，有一个重要的战略判断：

**ADS 不应该成为 Claude Code 的克隆，而应该成为运行在 Claude Code 之上的应用层。**

目前 ADS 自己实现了工具执行、LLM 调用等基础设施。更理想的形态是：
- **底层**：Claude Code 提供 Harness 基础设施（工具执行、权限、Hooks、QueryEngine）
- **上层**：ADS 提供应用逻辑（工单、SOP、角色 Agent、前端界面）

但由于 Claude Code 是 CLI 工具而非可嵌入的库，短期内直接集成难度较高。因此当前策略是：**ADS 自己补全 Harness 缺口，同时保持未来集成 Claude Code Hooks 体系的可能性**（通过兼容 settings.json 的 Hooks 格式）。

---

### Phase 1：Pre/Post Tool Hooks（2-3 天，P0）

**目标**：在每次 Action 执行前后提供可注册的钩子，实现审计日志、限流、错误追踪等横切逻辑。

#### 核心设计

```python
# backend/hooks/types.py
class HookEvent(str, Enum):
    PRE_TOOL_USE  = "PreToolUse"
    POST_TOOL_USE = "PostToolUse"
    TOOL_ERROR    = "ToolError"
    SESSION_START = "SessionStart"
    SESSION_END   = "SessionEnd"

@dataclass
class ToolHookContext:
    event:       HookEvent
    tool_name:   str
    input:       dict
    output:      Any | None    = None   # PostToolUse 才有
    error:       Exception     = None   # ToolError 才有
    duration_ms: float | None  = None
    project_id:  str | None    = None
    ticket_id:   str | None    = None
    agent_type:  str | None    = None
```

#### 执行包装器

```python
# backend/actions/executor.py
async def run_action(action, context: dict) -> Any:
    # 1. PreToolUse
    await hook_registry.emit(ToolHookContext(
        event=HookEvent.PRE_TOOL_USE, tool_name=..., ...))

    start = time.monotonic()
    try:
        result = await action.run(context)
        duration_ms = (time.monotonic() - start) * 1000

        # 2. PostToolUse
        await hook_registry.emit(ToolHookContext(
            event=HookEvent.POST_TOOL_USE, output=result,
            duration_ms=duration_ms, ...))
        return result
    except Exception as e:
        # 3. ToolError
        await hook_registry.emit(ToolHookContext(
            event=HookEvent.TOOL_ERROR, error=e, ...))
        raise
```

#### 内置三个高价值 Hook

| Hook | 触发时机 | 作用 |
|------|---------|------|
| `audit_log_hook` | POST_TOOL_USE | 写 `tool_audit_log` 表，支持工具使用统计 |
| `shell_rate_limit_hook` | PRE_TOOL_USE | 防止 Shell Action 被无限调用（每 Ticket 上限 50 次）|
| `failure_library_hook` | TOOL_ERROR | 工具错误自动写入 `failure_library` |

#### 与 Claude Code 格式兼容

```json
// settings.json（未来可复用 Claude Code 的 Hook 配置格式）
{
  "hooks": [
    { "event": "PreToolUse",  "if": "ShellAction", "shell": "audit-shell.sh" },
    { "event": "PostToolUse", "if": "*",           "shell": "telemetry.sh"   }
  ]
}
```

### Phase 1：预算/配额约束（1 天，P0）

**目标**：防止 Agent 失控——无限循环、Token 超支、执行超时。

```python
# backend/query_engine/budget.py
@dataclass
class Budget:
    max_tokens:  int   = 200_000
    max_turns:   int   = 50
    max_seconds: float = 600.0

    # 检查是否超限，返回原因字符串或 None
    def check(self) -> str | None: ...
```

**配置化**：在 `.env` 中暴露 `AGENT_MAX_TOKENS`、`CHAT_MAX_TURNS` 等参数，不同场景使用不同预算。

**效果**：5 行代码预防最严重的失控场景（无限工具调用循环）。

---

### Phase 2：QueryEngine 抽象（3-5 天，P1）

**目标**：将散落在 `api/chat.py`（88KB）中的工具调用循环提取为独立类，让 Chat 接口和 Orchestrator 共走同一条执行路径。

#### 问题诊断

```
现状：
  api/chat.py ──────────────────────────────► 88KB 大文件
  orchestrator.py → agent.execute() → 各自实现

目标：
  api/chat.py → QueryEngine.run() ──┐
                                     ├─► 统一：预算、Hooks、审计
  orchestrator.py → QueryEngine.run()─┘
```

#### QueryEngine 核心职责

```python
class QueryEngine:
    def __init__(self, llm_client, action_registry, budget=None):
        ...

    async def run(self, messages, tools, context) -> AsyncIterator[dict]:
        """
        主循环：
          LLM 流式返回 → 检查预算 → 执行工具（经过 Hooks）→ 结果回填 → 继续
        yield 规范化消息块供调用方消费（SSE / WebSocket / Orchestrator）
        """
```

#### 收益

- `api/chat.py` 从 88KB 缩减到 ~20KB（只负责 HTTP 层）
- Orchestrator 和 Chat 共享预算检查、审计 Hooks
- 未来接 Claude Code 的 QueryEngine 只需替换这一层

---

### Phase 2：异步权限审批系统（5-7 天，P2）

**目标**：为高风险操作（删文件、force-push、DROP TABLE）提供人工审批链，同时不破坏无人值守的自动化模式。

**关键设计：将实时弹窗改为异步审批**

```
高风险工具调用
  └─► PermissionGate.check()
        ├─► 低风险 → 直接放行
        └─► 高风险 → 写 permission_requests 表 + 暂停协程
                        └─► 前端展示待审批列表
                              └─► 用户点击批准/拒绝
                                    └─► WebSocket 通知 → future.set_result() → 恢复
              超时 5 分钟未审批 → 自动拒绝（fail-closed）
```

**新增前端页**：`/permissions` 展示所有待审批请求，类似 GitHub 的 PR 审批界面。

---

### Phase 3：子任务派发（有限递归 Agent）（3-4 天，P3）

**目标**：不重构整个 Orchestrator，给 Agent 增加一个 `dispatch_subtask` Action，允许 Agent 在执行中自主创建子 Ticket 交给另一个 Agent 处理。

```python
class DispatchSubtaskAction(BaseAction):
    """
    Agent 在执行中可以派发子任务：
    1. 创建子 Ticket（关联父 Ticket）
    2. 写入 PENDING 状态
    3. Orchestrator 下一轮 poll 自动拾取执行
    4. 父 Ticket 等待子 Ticket 全部完成
    """
```

这是一种"有限递归"——不是真正的递归 Agent，而是通过工单体系实现任务分解。

## 八、战略判断

### ADS 的独特竞争优势

ADS 拥有 Claude Code 完全没有的能力：

| 能力 | 说明 |
|------|------|
| **工单体系** | 需求→拆单→执行→验收完整闭环 |
| **SOP 配置** | YAML 热重载工作流，Trait-based 条件分支 |
| **失败库** | 跨工单经验积累，Reflexion 自进化 |
| **前端看板** | 实时可视化，不需要终端 |
| **知识库** | FTS5 全文检索，跨会话记忆 |
| **UE/游戏集成** | UCP 编辑器控制，专用 Skill 生态 |

这些是 Claude Code 作为通用工具所不打算提供的，是 ADS 的真正护城河。

### 建议的发展方向

**不要**：把 ADS 改造成 Claude Code 的 Python 克隆  
**应该**：让 ADS 成为"用 Claude Code 的 Harness 思想武装的领域应用平台"

具体说：补全 Pre/Post Hooks、QueryEngine 抽象和预算约束（Phase 1-2），让 ADS 的基础设施层达到工业级可靠性；同时继续深化应用层能力（更好的 Agent 协作、更丰富的 Skill 生态、更完善的前端体验）——这是 Claude Code 追不上的。
