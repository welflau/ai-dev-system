## 一、参考背景（KM 三篇核心要点）

### 第一篇：规则系统 v1.0 → v3.0

| 设计要点 | 内容 |
|---|---|
| **8 个 Agent 角色** | 制作人、项目管理、策划、UX、主程、程序、美术、QA |
| **9 种需求类型** | FEATURE/FEATURE_UI/OPTIMIZE/BUGFIX 等，各走不同流水线 |
| **禁止越权原则** | 每个 Agent 有明确"禁止做的事"——"禁止"比"允许"更有效 |
| **主从 Agent 架构** | 复杂任务拆子任务，子 Agent 独立上下文，只返回 ≤200 字摘要 |
| **双层架构** | Rule 文件夹（跨项目通用）vs GameDev 文件夹（项目专用）|
| **5Why 自我迭代** | AI 发现问题 → 分析根因 → 直接更新规则文件，系统自己进化 |

### 第二篇：v3.0 → v5.0（借鉴 BMAD / SuperClaude / ECC）

**BMAD 借鉴**：微文件架构（节省 76% token）、质量门禁、对抗性代码审查（≥3 个问题）、Smart Probe 需求探测（5 维度评分）、规模自适应（XS-XL）

**SuperClaude 借鉴**：/gd: 快捷命令体系、Agent 智能路由（三级识别 + 置信度）、行为模式标签（7 种 [MODE:xxx]）、三级内省机制

**ECC 借鉴**：事件驱动防护规则（事前拦截 + 事后检查）、Skills 技能包体系、规则分层不可变性

### 第三篇：工程化落地

vibecli CLI 工具链、.vibecli/ 目录结构（context.yaml / tasks.json / workflows/）、Workflow YAML 编排、HTML 原型验证管线（先验证玩法循环再进引擎）、Insight 经验系统（结构化 + 置信度 + 自动匹配）、三层验收体系（GM 命令 + P0 日志 + E2E 测试）

## 二、现有系统差距分析

### 相同/相近之处

| 维度 | KM 系统 | 我们的系统 |
|---|---|---|
| 多 Agent 角色分工 | 8 个 Agent | 7 个（Product/Architect/Dev/Test/Review/Deploy/ChatAssistant）|
| 流水线流转 | SOP 规则系统 | SOP YAML + 状态机 ✅ |
| 需求类型分类 | 9 种 | SOP fragments 按 traits 派生（部分实现）|
| 知识库搜索 | 知识库关键词匹配 | FTS5 全文搜索 ✅ |
| Insight 主动注入 | vibecli insight search | DevAgent 编码前自动调 SearchKnowledgeAction ✅ |
| 工作流 YAML 编排 | Workflow YAML | SOP YAML ✅（本质相同）|
| 跨会话记忆 | context.yaml + Frontmatter | 计划中（Memory 持久化）|

### 核心差距

1. **Agent 结构**：缺少策划/UX/美术/图片处理 4 个 Agent（详见第三章）
2. **通信机制**：中间产出物只存 DB JSON，不写文件，无法 Git 追踪（详见第六章）
3. **流程优化**：行为模式标签、Smart Probe、对抗性审查等未实现（详见第七章）
4. **资产体系**：无美术资产库、无策划知识库（详见第五章）

### 3.2 新增 Agent 详析

#### 策划 Agent（PRD 专职）

**问题**：ProductAgent 把"需求入口 + 写 PRD + 拆单"三件事合并，没有独立的 PRD 质量审查，验收标准常缺失或不可量化。

**职责**：
- 产出完整 PRD（功能描述、交互说明、验收标准、边界条件）
- 验收标准必须可量化（❌"界面美观" / ✅"点击后 300ms 内弹出弹窗"）
- PRD 末尾固定输出"资产需求线索"章节（为美术 Agent 识别资产提供输入）
- 不写任何技术实现细节
- 完成后设质量门禁（策划 → 架构的关卡检查）

**适用范围**：所有项目类型

---

#### UX Agent（交互设计专职）

**问题**：DevAgent 直接按 PRD 文字描述写代码，交互逻辑全靠自己理解，交互状态缺失、页面跳转逻辑混乱、响应式问题到开发完才暴露。

**职责**：
- 产出交互流程图（页面跳转关系，Mermaid 格式）
- 产出组件线框说明（布局层级、关键组件位置）
- 定义所有交互状态（正常/hover/active/disabled/loading/empty/error）
- 定义响应式规则（Web：断点 / 小程序：rpx 换算规则）
- 禁止写代码、不关心视觉样式

**适用范围**：web / wechat / mobile / desktop / category:game

---

#### 美术 Agent（视觉规范 + 资产管理）

**问题**：没有视觉规范约束，DevAgent 随意选颜色、字号、间距，最终风格混乱；图标从不同来源随意引用，风格不一致。

**职责**：
- 产出 Design Token（颜色系统/字体层级/间距规范）→ DevAgent 直接引用
- 产出组件视觉规范（按钮/卡片/弹窗的视觉样式）
- 产出 asset_manifest.yaml（资产清单，技美并行的契约）
- 对每个资源判断类型，路由到合适的生产方式
- 游戏项目额外：精灵图规格、UI Atlas 方案、粒子特效参数

**禁止**：写代码、写策划逻辑、写交互流程

**产出物两类性质**：

| 规范类（文字/结构化）| 资产类（直接可用文件）|
|---|---|
| Design Token（JSON/CSS）| 图标 SVG（来自 Iconify）|
| 组件视觉规范（Markdown）| 插画/图片（来自 AI 生图或素材库）|
| asset_manifest.yaml（YAML）| CSS/SVG 代码（直接生成）|
| 精灵图规格（Markdown）| 音频（来自 Freesound/Suno）|
| UI Atlas 方案（YAML）| 3D 材质（来自 Poly Haven）|
| 粒子特效参数（JSON）| Lottie 动效（来自 LottieFiles）|

**完整资产类型全景**：

| 大类 | 子类型 | 格式 | AI 可自动化 | 主要来源 |

#### 图片处理 Agent（异步服务型）

**核心设计**：任意 Agent 在任意阶段可以发送生图请求，立即返回占位符继续工作，图片生成完成后异步回填文档。

```
请求方发出生图指令 → 立即返回 [IMG_PENDING:img_001]（继续其他工作）
                                    ↓（后台并行）
                    ImageAgent 调用生图 API（Flux/DALL-E 3 等）
                                    ↓（完成后）
                    替换文档中的占位符 + 推送 SSE 通知前端刷新
                    生成结果自动入 G_ArtRes 资产库
```

**ASCII 兜底**：生图 API 未配置或超时时，自动降级为 LLM 生成 ASCII 图，主流水线不感知。

**技术表**：`image_requests` 表（id/project_id/prompt/style/size/callback_doc/status/result_path）

### 3.3 资产识别责任链

资产需求分散在多个阶段的产出里，需要逐层提取、最终由美术 Agent 正式汇总。

```
策划 Agent（PRD）
    ↓ 隐式包含资产线索
    "战斗时背景音乐切换"  → 音频资产
    "技能释放有特效"      → 粒子特效
    "角色有四种形态"      → 角色精灵图 × 4
    "关卡结束弹出奖励面板"→ UI 面板 + 图标
    ↓ PRD 末尾固定输出"资产需求线索"章节

UX Agent（交互流程）
    ↓ 从交互角度识别 UI 层资产
    "首页有 5 个导航图标"  → 5 个 icon
    "空状态显示插画"       → 1 张 illustration
    ↓ 产出组件清单时附带资产说明

美术 Agent ← 正式负责资产识别
    读取 PRD（资产线索章节）+ UX 设计（组件清单）
    提取所有类型资产需求 → 产出完整 asset_manifest.yaml
```

**美术 Agent 识别逻辑**：

```python
def identify_assets(prd: str, ux_design: str, traits: List[str]) -> AssetManifest:
    assets = []
    # 1. 从 UX 设计提取 UI 类资产（图标/插画/UI贴图）
    assets.extend(extract_ui_assets(ux_design))
    # 2. 从 PRD 资产线索章节提取功能类资产
    if "category:game" in traits:
        # 游戏项目额外识别音频/特效/骨骼动画/3D资产
        assets.extend(extract_game_assets(prd))
    # 3. 通用资产（加载动画/错误提示图/成功反馈图）
    assets.extend(extract_universal_assets(ux_design))
    return AssetManifest(assets=assets)
```

**PRD 末尾固定的"资产需求线索"章节模板**：

```markdown
## 资产需求线索（策划填写，非正式，供美术 Agent 参考）

- 音频：战斗开始/结束音效，关卡背景音乐 2 首
- 动画：角色攻击/死亡骨骼动画，UI 面板弹出动效
- 图片：角色立绘 × 4，关卡缩略图 × 8，Banner 图 × 1
- 特效：技能释放粒子特效 × 3
- 其他：暂无
```

**关键点**：如果识别有遗漏，asset_manifest.yaml 对人可见，用户可在技美并行阶段手动补充。

## 四、工作流重设计

### 4.1 完整工作流图

```
ProductAgent（需求分析）
    ↓
策划 Agent（PRD + 验收标准 + 资产需求线索）
    ↓
UX Agent（交互流程 + 组件清单）
    ↓
  生成 asset_manifest.yaml（技美并行的契约）
    ↓
    ┌─────────────────────────────────────────┐
    ↓（并行开工）                             ↓（并行开工）
美术 Agent                             ArchitectAgent → DevAgent
产出最终视觉资源（按资产清单）           用占位资源开发（按路径和规格）
真实图标/视觉规范/音效/动效等            功能跑通即可，代码引用约定路径
    ↓                                        ↓
    └────── 双方完成 → 资产合并校验 ──────────┘
              验证清单里每项都有真实文件覆盖
              （美术资源直接覆盖占位文件，代码零修改）
                         ↓
              ┌──────────────────────┐
              ↓（并行）             ↓（并行）
          TestAgent           ReviewAgent
          （功能+视觉验收）     （代码审查）
              ↓                    ↓
              └────── 都通过 ───────┘
                         ↓
                    DeployAgent

────────────────────────────────────────────────
图片处理 Agent（横切异步服务，任意阶段按需调用）
策划需要插图 → [IMG_PENDING:001]
UX 需要效果图 → [IMG_PENDING:002]  →→ 后台处理 →→ 替换占位
美术需要示意图 → [IMG_PENDING:003]
```

### 4.2 技美并行的前提

1. **UX 先完成**，产出 asset_manifest.yaml（路径和规格约定）
2. **路径提前约定**——DevAgent 引用的路径就是美术最终交付的路径
3. **功能验收 ≠ 视觉验收**——TestAgent 等资产合并后再做最终验收

### 4.3 按 traits 触发哪些阶段

| platform | 策划 | UX | 美术 | 技美并行 | 图片 Agent |
|---|---|---|---|---|---|
| web / wechat / mobile / desktop | ✅ | ✅ | ✅ | ✅ | ✅ |
| server / cli | ✅ | ❌ | ❌ | ❌ | ❌ |
| category:game | ✅ | ✅ | ✅（游戏规范）| ✅ | ✅（重度）|

### 5.2 策划知识库（G_DesignKnowledge）

**定位**：全局共享的领域设计知识，与项目知识库互补。

| | 项目知识库（现有）| 策划知识库（新建）|
|---|---|---|
| 内容 | 本项目踩过的坑、历史 PRD | 行业通用设计知识、设计模式 |
| 范围 | 项目私有 | **全局共享，跨所有项目** |

## 六、Agent 间通信机制改进

### 现状：DB JSON，无文件

```python
# _build_context() 中——当前实现
context["architecture"] = json.loads(art["content"])  # DB JSON，path: null
```

| | KM 系统 | 我们系统 |
|---|---|---|
| 通信媒介 | 文件（Markdown/YAML）| DB 的 JSON 字段 |
| 人工可读 | 直接打开文件查看/修改 | 需查 DB |
| Git 追踪 | 自动进版本历史 | `path: null`，无文件 |
| 大内容传递 | 文件无大小限制 | DB 字段有截断风险 |
| 知识沉淀 | 文件自动索引进知识库 | 需额外处理 |

注意：代码文件是例外——DevAgent 写的代码已通过 `git_manager.write_file()` 存到 Git。但 PRD、架构设计、UX 设计等中间产出物只在 DB。

### 改进方案（1 天）

每个 Agent 完成时，同步把产出写一份文件到 Git 仓库：

```python
# Orchestrator 每个 Agent 阶段完成后
docs_prefix = f"docs/Reqs/{req_id[-6:]}/{ticket_id[-6:]}/"
# ProductAgent → PRD.md
# ArchitectAgent → 架构设计.md
# UX Agent → UX设计.md（新增）
# 美术 Agent → 视觉规范.md + asset_manifest.yaml（新增）
```

**三个收益**：
1. 人可以打开文件查看/修改中间产出，修改后影响下游
2. 需求完整决策链路进 Git 历史，可追溯可回滚
3. 文件自动被 knowledge_index 索引，历史方案变成可检索经验

## 七、流程优化方案（来自 KM 借鉴）

### 7.1 P1：高价值、低成本

#### 行为模式标签 [MODE:xxx]（1 天）

**问题**：DevAgent 在"编码实现"和"对抗审查"用同一种心态，审查不够狠。

**方案**：SOP stage 里加 `mode` 字段，Agent prompt 根据 mode 切换开头：

```yaml
stages:
  - id: write_code
    agent: DevAgent
    mode: IMPLEMENT  # 专注、严谨
  - id: self_review
    agent: DevAgent
    mode: REVIEW     # 对抗、怀疑、主动找问题
```

7 种模式：EXPLORE / DESIGN / IMPLEMENT / REVIEW / DEBUG / VERIFY / REFLECT

---

#### 对抗性代码审查强制要求（0.5 天）

**方案**：ReviewAgent 必须输出 🔴严重 / 🟡警告 / 🟢建议 三级列表，严重问题必须修复，禁止"代码质量良好"类敷衍结论。

---

#### Smart Probe 需求清晰度评分（1 天）

**方案**：策划 Agent 生成 PRD 前，先对需求做 5 维度评分：

| 维度 | 满分 |
|---|---|
| 用户价值 | 5 |
| 范围边界 | 5 |
| 交互方式 | 5 |
| 数值参数 | 5 |
| 关联影响 | 5 |

- ≥20：直接开发
- 15-19：AI 自动补假设，标注 `[假设]`
- <15：暂停，提 ≤3 个关键问题（附默认假设，用户可说"按默认来"跳过）

---

#### 需求规模自适应流程（0.5 天）

| 规模 | 条件 | 处理 |
|---|---|---|
| XS | 1工单、<50行代码 | 跳过 ArchitectAgent |
| S | 1-2 个工单 | ArchitectAgent 轻量过 |
| M | 3-5 个工单 | 标准流水线 |
| L/XL | >5 个工单 | 触发主从模式 |

## 八、三个核心洞察

### 洞察 1：「禁止」比「允许」更能约束 AI 行为

KM 文章反复强调：明确告诉 AI "不要做什么"远比"要做什么"更有效。每个 Agent 应该有明确的禁止清单，每条禁止规则背后都是一次真实事故。

**启示**：DevAgent / ArchitectAgent 的 prompt 里禁止规则还不够完备，需要补全。

### 洞察 2：规则系统自身也需要迭代

KM 从 v1.0 迭代到 v5.0，每个版本号背后都是"踩坑→反思→改进"。不要一开始设计过于复杂的机制，简单直接更有效。AI 很擅长找"合理理由"简化流程，需要提前预判并明确禁止。

**启示**：system prompt 和 SOP 规则应该有版本记录（rule_changelog），每次发现 AI "偷懒"立即固化为规则。

### 洞察 3：工程化的本质是把软约束变成硬约束

KM 演化主线：文档规则（软约束）→ Prompt 系统（半硬）→ CLI 工具链（硬约束）。

**启示**：
- 阶段流转检查现在靠 LLM 判断 → 改为代码状态检查
- Insight 注入靠 AI 自觉 → 高置信度经验直接加入 files_to_read 列表
- 工单依赖靠 LLM 拆单输出 → 增加依赖关系 UI 管理入口
