# Hermes Agent 对标分析与 AI Dev System 知识库增强方案

> 日期：2026-04-27  
> 背景：调研 Hermes Agent（Nous Research v0.11.0）的知识管理设计，提取可移植的机制，制定知识库功能的分阶段增强路线

---

## 一、Hermes 核心知识管理机制解析

Hermes 放弃了"传统 RAG（向量化 + 检索）"路线，选择了**5层分治**策略，每层独立管理、互不耦合：

```
Layer 1  会话历史检索    SQLite FTS5 全文索引 + 轻量 LLM 摘要
Layer 2  持久记忆        冻结快照 Markdown（§ 分隔，字符上限）
Layer 3  @ 引用          运行时动态展开文件/目录/URL/git diff
Layer 4  技能知识库      SKILL.md 结构化文档，Agent 自动写入/更新
Layer 5  项目上下文文件  自动向上遍历 git root 找 HERMES.md / AGENTS.md
```

### 1.1 Layer 5 — 项目上下文文件自动发现

**机制**（`agent/prompt_builder.py`）：

```python
_HERMES_MD_NAMES = (".hermes.md", "HERMES.md")

def _find_hermes_md(cwd: Path) -> Optional[Path]:
    stop_at = _find_git_root(cwd)           # 找 .git 目录，确定 git 根
    current = cwd.resolve()
    for directory in [current, *current.parents]:
        for name in _HERMES_MD_NAMES:
            candidate = directory / name
            if candidate.is_file():
                return candidate
        if stop_at and directory == stop_at:
            break                            # 到 git root 停止，不污染全系统
    return None
```

- 向上遍历到 git root，找到第一个匹配文件即用
- 内容安全扫描：注入检测（`ignore previous instructions`）、凭证外泄（`curl + SECRET`）、SSH 后门（`authorized_keys`）
- YAML frontmatter 自动剥离，只注入正文
- 单文件上限 20,000 字符

**对我们的启发**：扫描项目 git 仓库的 `docs/`、`Design/`、`Spec/` 等目录并同步到知识库，逻辑完全类似。

### 1.2 Layer 4 — 技能自进化（最有价值的机制）

**触发条件**（`run_agent.py`）：
- 累积 ≥ 10 个工具迭代后，当前轮对话完成时，后台异步触发
- 配置：`agent.skills.creation_nudge_interval = 10`

**后台子 Agent 配置**：

**对我们的启发**：工单完成后，后台子 Agent 审查 → 自动写/更新项目知识库文档（如 `FAQ.md`、`已知问题.md`）。

### 1.3 Layer 1 — FTS5 会话历史检索

**SQLite FTS5 建立**（`hermes_state.py`）：
```sql
CREATE VIRTUAL TABLE messages_fts USING fts5(
    content,
    content=messages,        -- 外部内容表（零数据重复）
    content_rowid=id
);

-- 触发器自动同步
CREATE TRIGGER messages_fts_insert AFTER INSERT ON messages BEGIN
    INSERT INTO messages_fts(rowid, content) VALUES (new.id, new.content);
END;
```

**双模式搜索**（`tools/session_search_tool.py`）：

| 模式 | 触发 | 成本 | 返回 |
|---|---|---|---|
| 最近会话 | query 为空 | 0（纯 DB 查询） | 元数据列表（标题/时间/预览） |
| 语义搜索 | 有 query | LLM 摘要 × N 个会话 | 结构化摘要 JSON |

**FTS5 支持语法**：
- `"exact phrase"` 精确短语
- `docker OR kubernetes` 布尔 OR
- `python NOT java` 排除
- `deploy*` 前缀匹配

**摘要并发**：
```python
semaphore = asyncio.Semaphore(max_concurrency=3)  # 并发限制
# 摘要模型：Gemini Flash（快 + 便宜），temperature=0.1
```

**对我们的启发**：工单历史已有大量 `ticket_logs` 数据，加 FTS5 索引后，DevAgent 遇到问题可主动搜索历史工单的解决方案。

### 1.4 Layer 3 — @ 引用动态展开

**支持的引用类型**：
```
@file:"path"               整个文件
@file:"path":10-50         行号范围（1-based）
@folder:"path"             目录文件树列表

@staged                    git 暂存区改动

```

**安全阻止列表**：
```python

```

**对我们的启发**：DevAgent/ArchitectAgent 在 prompt 中可以引用当前工单关联的具体文件，而不是全量注入整个知识库。

### 1.5 持久记忆的冻结快照机制

**核心问题**：Anthropic prefix caching TTL = 5 分钟。若 system prompt 在会话中途改变，缓存失效，每次 API 调用都要重传完整 system prompt（成本 × N）。

**Hermes 的解法**：
```python
def load_from_disk(self):
    self.memory_entries = read_file(MEMORY_MD)
    # 一次性捕获快照，会话内不再变
    self._system_prompt_snapshot = {
        "memory": render_block(self.memory_entries),
    }

def system_prompt_block(self) -> str:
    return self._system_prompt_snapshot["memory"]  # 永远返回快照
    
def handle_tool_call(self, action, args):
    # 工具调用确实更新了磁盘
    # 但返回的是实时状态（让 Agent 知道最新情况）
    # 而 system prompt 不变（维持 prefix cache）
    return f"已更新，当前状态: {render_live_state()}"
```

**对我们的启发**：我们现在每次工单执行都重新读取 docs/ 目录（orchestrator._load_knowledge_docs），这会导致不同工单 system prompt 不同，prefix cache 命中率低。可以在项目级别缓存知识库内容，只在内容变更时刷新。

### 1.6 上下文压缩

**触发阈值**：`token_count >= context_length × 50%`（默认）

**5阶段压缩**：
1. 工具结果剪枝（无 LLM，廉价）：去重 + 简短摘要替换旧工具输出
2. 边界确定：保护前 N 条（system prompt）+ 后 N 条（最近交互）
3. LLM 生成结构化摘要（目标 20% 大小）
4. 消息重组
5. tool_call/result 对完整性修复

**结构化摘要模板**（防止信息丢失）：
```
## Active Task      当前任务（最重要）
## Goal             总体目标
## Constraints      用户偏好/约束
## Completed Actions 已完成操作（编号列表）
## Active State     当前工作目录/文件/进程
## In Progress      进行中的工作
## Blocked          阻塞项
## Key Decisions    重要技术决策
## Critical Context 关键上下文（变量值/错误消息）
```

**对我们的启发**：工单 context 超长时，可以在 fix_issues 循环内自动压缩上下文，保留结构化摘要而非截断。

## 二、现状 vs 目标 Gap 分析

### 2.1 当前知识库能力

```
用户手动写 docs/*.md
        ↓
orchestrator._load_knowledge_docs()  每次工单执行
        ↓
全量拼接（全局 2000 字 + 项目 3000 字，超出截断）
        ↓
塞进所有 Agent 的 system prompt（被动注入，Agent 无法主动查询）
```

| 维度 | 现状 | 问题 |
|---|---|---|
| 内容来源 | 用户手动写 | 更新成本高，容易过时 |
| 注入方式 | 全量被动 | 5000 字上限，大项目不够用 |
| 路径配置 | 硬编码 | 项目 docs 在 git 仓库里但读不到 |
| Agent 查询 | 无法主动搜索 | 不知道历史怎么解决过类似问题 |
| 知识更新 | 手动 | 解决方案不会自动沉淀 |
| 安全扫描 | 无 | 上传内容可能含 prompt injection |

### 2.2 目标能力

```
工单完成 → 后台子 Agent 审查 → 自动写/更新 project/docs/FAQ.md
git 仓库 docs/ 目录 → 一键同步 + 路径配置 → 知识库
                                                ↓
                                       FTS5 索引（SQLite）
                                                ↓
DevAgent 遇到问题 → tool_use: search_knowledge("UCapsuleComponent 头文件")
                                                ↓
                                    返回相关文档片段 + 来源
```

**同步逻辑**（`api/knowledge.py` 或新 `actions/sync_knowledge.py`）：

```python
async def sync_knowledge_from_repo(project_id: str):
    project = await db.fetch_one(...)
    repo_path = Path(project["git_repo_path"])
    scan_paths = json.loads(project.get("knowledge_scan_paths") or '[]')
    
    # 默认扫描路径（若未配置）
    if not scan_paths:
        scan_paths = [
            {"path": "docs",   "enabled": True},
            {"path": "Design", "enabled": True},
            {"path": "Spec",   "enabled": True},
        ]
    
    synced, skipped = [], []
    for entry in scan_paths:
        if not entry.get("enabled", True):
            continue
        scan_dir = repo_path / entry["path"]
        if not scan_dir.is_dir():
            continue
        
        for md_file in scan_dir.rglob("*.md"):
            content = md_file.read_text(encoding="utf-8", errors="ignore")
            
            # 安全扫描（借鉴 Hermes）
            if _contains_injection_patterns(content):
                skipped.append(str(md_file))
                continue
            
            # 相对路径作为知识库文件名
            rel = md_file.relative_to(repo_path)
            dest = get_project_docs_dir(project_id) / str(rel).replace("/", "__")
            dest.parent.mkdir(parents=True, exist_ok=True)
            dest.write_text(content, encoding="utf-8")
            synced.append(str(rel))
    
    await db.update("projects", {"knowledge_sync_at": now_iso()}, "id = ?", (project_id,))
    return {"synced": synced, "skipped": skipped}
```

**注入安全扫描**（借鉴 Hermes `_CONTEXT_THREAT_PATTERNS`）：
```python
_INJECTION_PATTERNS = [
    re.compile(r'ignore\s+(previous|all|above|prior)\s+instructions', re.I),
    re.compile(r'you\s+are\s+now\s+a', re.I),
    re.compile(r'disregard\s+(all|your|previous)', re.I),
    re.compile(r'curl\s+[^\n]*\$\{?\w*(KEY|TOKEN|SECRET|PASSWORD)', re.I),
    re.compile(r'authorized_keys', re.I),
]

def _contains_injection_patterns(content: str) -> bool:
    return any(p.search(content) for p in _INJECTION_PATTERNS)
```

#### B2. SearchKnowledgeAction

新建 `backend/actions/chat/search_knowledge.py`：

```python
class SearchKnowledgeAction:
    """Agent 主动搜索知识库的 tool_use 动作"""
    
    name = "search_knowledge"
    description = (
        "搜索项目知识库文档。当遇到不确定的 API 用法、已知错误、架构约束时使用。\n"
        "返回相关文档片段 + 文件来源，而非完整文档。\n"
        "使用场景：编译错误 / API 不确定 / 历史解决方案 / 架构规范查询"
    )
    
    async def run(self, context: dict) -> ActionResult:
        query = context.get("query", "")
        project_id = context.get("project_id", "")
        limit = context.get("limit", 3)
        
        if not query:
            return ActionResult(status="fail", data={"error": "query 不能为空"})
        
        # FTS5 搜索
        rows = await db.fetch_all("""
            SELECT k.filename, k.scope, k.content,
                   snippet(knowledge_fts, 0, '**', '**', '...', 32) AS snippet
            FROM knowledge_fts
            JOIN knowledge_index k ON knowledge_fts.rowid = k.id
            WHERE knowledge_fts MATCH ?
              AND (k.project_id = ? OR k.scope = 'global')
            ORDER BY rank
            LIMIT ?
        """, (query, project_id, limit))
        
        if not rows:
            return ActionResult(status="success", data={
                "results": [],
                "message": f"知识库中未找到与「{query}」相关的内容"
            })
        
        results = [{
            "filename": r["filename"],
            "scope": r["scope"],
            "snippet": r["snippet"],
            "content_preview": r["content"][:500],
        } for r in rows]
        
        return ActionResult(status="success", data={
            "query": query,
            "results": results,
            "count": len(results),
        })
```

### Phase C：知识自进化（~2.5 天）

**解决的核心问题**：工单里积累了大量解决方案，但不会自动沉淀到知识库。

#### C1. 触发机制

工单状态变为 `development_done` / `acceptance_passed` 时，在 `orchestrator.py` 中后台触发：

```python
async def _trigger_knowledge_distill(self, ticket_id: str, project_id: str):
    """工单完成后，异步蒸馏知识到知识库"""
    asyncio.create_task(self._run_knowledge_distill(ticket_id, project_id))

async def _run_knowledge_distill(self, ticket_id: str, project_id: str):
    """后台运行，不阻塞主流程"""
    try:
        from actions.knowledge_distill import KnowledgeDistillAction
        result = await KnowledgeDistillAction().run({
            "ticket_id": ticket_id,
            "project_id": project_id,
        })
        if result.data.get("updated"):
            logger.info("[知识蒸馏] 工单 %s → %s", ticket_id, result.data.get("filename"))
    except Exception as e:
        logger.warning("[知识蒸馏] 失败（忽略）: %s", e)
```

#### C2. KnowledgeDistillAction

```python
class KnowledgeDistillAction:
    """工单完成后，分析解决过程，更新/创建知识库文档"""
    
    async def run(self, context):
        ticket_id = context["ticket_id"]
        project_id = context["project_id"]
        
        # 获取工单完整信息
        ticket = await db.fetch_one("SELECT * FROM tickets WHERE id = ?", (ticket_id,))
        logs = await db.fetch_all(
            "SELECT * FROM ticket_logs WHERE ticket_id = ? ORDER BY created_at",
            (ticket_id,)
        )
        
        # 构建分析 prompt（借鉴 Hermes _SKILL_REVIEW_PROMPT 的 class-first 策略）
        prompt = f"""
分析以下已完成工单，判断是否应写入项目知识库。

工单标题：{ticket['title']}
工单类型：{ticket['ticket_type']}
关联需求：{ticket.get('requirement_id', '无')}

执行过程摘要（最近日志）：
{self._format_logs(logs[-20:])}  # 最近 20 条

工作要求（严格遵守）：
1. CLASS-FIRST：识别问题的通用类别，不记录单次具体实现
2. 只记录"下次遇到类似问题时有用的知识"
3. 优先更新已有文档，不无限新建
4. 如果没有值得记录的通用知识，返回 {{skip: true}}

若值得记录，返回 JSON：
{{
  "skip": false,
  "filename": "已知问题/UE编译错误.md",   // 建议文件名（相对于 docs/）
  "section_title": "## UFUNCTION override 冲突",  // 新增的章节标题
  "content": "...",  // 新增的 Markdown 内容（简洁，<500字）
  "action": "append" | "create"  // append=追加到已有文件，create=新建
}}
"""
        
        response = await llm_client.call(prompt, max_tokens=1000)
        data = json.loads(response)
        
        if data.get("skip"):
            return ActionResult(status="success", data={"updated": False})
        
        # 写入知识库
        filename = data["filename"]
        docs_dir = get_project_docs_dir(project_id)
        file_path = docs_dir / filename
        file_path.parent.mkdir(parents=True, exist_ok=True)
        
        if data["action"] == "append" and file_path.exists():
            existing = file_path.read_text(encoding="utf-8")
            file_path.write_text(existing + "\n\n" + data["content"])
        else:
            file_path.write_text(f"# {filename.stem}\n\n{data['content']}")
        
        # 更新 FTS5 索引
        await self._update_knowledge_index(project_id, filename, file_path.read_text())
        
        return ActionResult(status="success", data={
            "updated": True,
            "filename": filename,
            "action": data["action"],
        })
```

#### C3. 蒸馏频率控制

避免每个工单都触发 LLM（成本控制）：
```python
# 只对以下类型工单触发蒸馏
DISTILL_TRIGGER_TYPES = {
    "bug_fix",          # Bug 修复：有明确的根因和修复方式
    "engine_compile",   # 编译修复：新规则可以预防
    # 不触发：routine development（代码实现），acceptance（只是验证）
}

# 只在 Reflexion 循环发生过时触发（说明有踩坑过程）
if ticket.get("fix_count", 0) > 0:
    await _trigger_knowledge_distill(ticket_id, project_id)
```

### Phase E：上下文压缩（~1.5 天，可选）

**解决的核心问题**：工单 fix_issues 循环多轮后，context 积累大量工具输出，LLM 成本急剧上升。

#### E1. 简化版压缩（不需要实现 Hermes 的完整 5 阶段）

在 `DevAgent.fix_issues` 中，当循环次数 > 3 且 context 超过 50K token 时：

```python
async def _maybe_compress_context(self, messages: list) -> list:
    """简化版：只做工具结果摘要（第1阶段），不做 LLM 全量压缩"""
    compressed = []
    for msg in messages:
        if msg["role"] == "tool" and len(str(msg.get("content", ""))) > 3000:
            # 用简短摘要替换大型工具输出
            compressed.append({
                **msg,
                "content": f"[摘要] {str(msg['content'])[:200]}... ({len(str(msg['content']))} 字符)"
            })
        else:
            compressed.append(msg)
    return compressed
```

| Phase | 功能 | 工作量 | 价值 | 优先级 |
|---|---|---|---|---|
| **A** | 仓库 docs 同步 + 路径配置 | 1.5 天 | 高（用户呼声） | **P0** |

## 五、不做的事（有意为之的边界）

| 能力 | 决定 | 原因 |
|---|---|---|
| **向量化 RAG** | 暂不做 | 5000 字全量注入对中小项目够用；FTS5 覆盖 80% 场景；向量化引入 embedding 依赖，维护成本高 |
| **记忆提供商插件化** | 暂不做 | Hermes 支持多 provider 是因为对外开放；我们是单租户系统，SQLite 足够 |
| **@ 引用语法** | 暂不做 | 当前 Agent 通过 SOP context 拿到文件列表；@ 引用适合 REPL，不适合工单驱动流程 |
| **多轮 LLM 压缩** | Phase E 只做简化版 | 完整 5 阶段压缩逻辑复杂；先做工具输出摘要，够用再上 LLM 压缩 |

## 六、关键设计决策

### 6.1 FTS5 vs 向量检索

Hermes 选择 FTS5 而非向量的原因：
- SQLite 内置，零依赖
- 支持布尔逻辑（AND/OR/NOT/前缀匹配）
- 对代码关键词（类名、函数名、错误代码）精确匹配比语义相似度更有用
- 更新即时，不需要重新 embedding

我们的场景（UE 编译错误 `C3668`、函数名 `OnRep_Score`、模块名 `UCapsuleComponent`）**关键词搜索比语义搜索更准确**，FTS5 完全够用。

### 6.2 知识蒸馏的 class-first 原则

不记录「工单 T-042 里 TestFPS 的 FPSCharacterBase.cpp 第 22 行修复」这种具体实例，而是记录「UE C++ 常用组件类必须在 .cpp 中 include 对应头文件」这种通用规律。

判断标准：**下次遇到类似问题的 DevAgent，能从这条知识里直接找到答案吗？**

### 6.3 安全扫描是必要的

用户上传的文档或 git 仓库中的 `docs/*.md` 可能含有攻击性内容（恶意第三方代码库、生成的文档）。借鉴 Hermes 的注入模式检测，在同步前扫描，异常文件跳过 + 记录警告。
