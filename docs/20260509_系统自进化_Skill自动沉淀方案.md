# 系统自进化：成功经验自动沉淀为 Skill

> 日期：2026-05-09
> 状态：方案设计（待实现）
> 关联 TODO：H. 系统自进化 Skill 自动沉淀（P2）

---

## 一、问题与目标

### 现状

系统已有四层自进化机制：

| 层次 | 机制 | 覆盖范围 |
|------|------|---------|
| L1 | Reflexion 单工单反思 | 当次失败 → 修复策略 |
| L2 | Failure Library 跨工单记忆 | 历史失败 → 下次规避 |
| L3 | Skills 知识注入 | 专家经验 → prompt 增强 |
| L4 | SelfTest 自测卡点 | 产出质量 → 提前拦截 |

**缺口**：L1/L2 只记录"失败"，L3 的 Skills 内容靠**人工维护**。成功案例（工单验收通过时用了什么有效模式）无法自动流入 Skills，形成经验积累。

### 目标

工单验收通过后，系统自动：
1. 分析本次开发轨迹，识别可复用的技术模式
2. 生成 Skill 草案（非自动上线，需人工确认）
3. 人工确认后写入 skills.json，热重载生效

把"人工总结 → AI 执行"升级为 **"AI 草稿 → 人工确认 → 自动生效"**。

## 二、架构设计

### 2.1 整体流程

```
工单验收通过（acceptance_passed）
        ↓
Orchestrator 触发 SkillExtractorAction
        ↓
读取工单全轨迹：
  - ticket: title / description / module / project_type
  - artifacts: PRD + 架构设计 + 代码 + dev-notes
  - ticket_logs: reflection logs（根因 + 策略变化）
  - agent: 哪个 Agent 执行的
        ↓
LLM 分析：是否有可复用的技术模式？
        ↓
    有 → 生成 Skill 草案，写入 pending_skills 表（status=draft）
    无 → 静默跳过
        ↓
ChatAssistant 下次对话时感知到新草案
→ 主动提示：「发现 1 条新 Skill 草案，是否查看？」
        ↓
人工确认 → ConfirmSkillAction → 写入 skills.json + 热重载
人工拒绝 → pending_skills status=rejected
```

### 2.2 新增组件

```
backend/
├── actions/
│   ├── skill_extractor.py          # 新建：提取 Skill 草案
│   └── chat/
│       └── confirm_skill.py        # 新建：确认/拒绝 Skill 草案
├── skills/
│   └── pending_skills.py           # 新建：草案 CRUD
```

数据库新增表：

```sql
CREATE TABLE pending_skills (
    id          TEXT PRIMARY KEY,
    project_id  TEXT,
    ticket_id   TEXT,
    agent_type  TEXT,
    name        TEXT NOT NULL,          -- Skill 名称（slug）
    description TEXT NOT NULL,         -- 一行描述
    inject_to   TEXT NOT NULL,         -- JSON: ["DevAgent"]
    traits_match TEXT,                 -- JSON: {"any_of": ["engine:ue5"]}
    prompt_content TEXT NOT NULL,      -- Skill 正文（Markdown）
    status      TEXT DEFAULT 'draft',  -- draft | confirmed | rejected
    source_summary TEXT,               -- 提取依据摘要（给人看）
    created_at  TEXT,
    confirmed_at TEXT,
    confirmed_by TEXT
);
```

## 三、核心模块设计

### 3.1 SkillExtractorAction

**触发时机**：Orchestrator 检测到 `acceptance_passed` 后异步触发（不阻塞主流程）。

**输入 context**：
```python
{
    "ticket_id": str,
    "project_id": str,
    "agent_type": str,        # 哪个 Agent 完成的
    "project_traits": list,   # 项目特征
    "ticket_description": str,
    "artifacts": dict,        # PRD / 架构 / 代码摘要
    "reflections": list,      # 历次反思
    "retry_count": int,       # 重试了几次才成功
}
```

**LLM Prompt 核心逻辑**：

```
分析这次成功完成的工单，判断是否存在可以提炼为可复用 Skill 的技术模式。

判断标准（满足其一即提取）：
1. 解决了一个需要特定领域知识才能处理的问题（如 UE 的某类 Bug 规律）
2. 经过多次 Reflexion 后发现了一个通用策略变化
3. 代码里出现了一个可以被未来工单直接复用的模式或约定

不提取的情况：
- 通用编程知识（没有项目/引擎特异性）
- 过于具体（只适用于本工单，无法泛化）
- skills.json 里已有覆盖

输出格式（JSON）：
{
  "should_extract": bool,
  "reason": "为什么提取/不提取",
  "skill": {               // should_extract=true 时才有
    "name": "skill-slug",
    "description": "一行描述，说明何时使用",
    "inject_to": ["DevAgent"],
    "traits_match": {"any_of": ["engine:ue5"]},
    "prompt_content": "..Markdown 格式的 Skill 正文.."
  }
}
```

**降级**：LLM 调用失败 / `should_extract=false` → 静默跳过，不影响主流程。

### 3.2 pending_skills.py

```python
class PendingSkillsManager:
    async def add(self, ticket_id, project_id, agent_type, skill_draft) -> str
    async def list_drafts(self, project_id=None) -> List[dict]
    async def confirm(self, skill_id, confirmed_by=None) -> bool
        # → 写入 skills/packs/<name>/prompt.md
        # → 更新 skills.json
        # → skill_loader.reload()
    async def reject(self, skill_id) -> bool
    async def count_pending(self) -> int
```

**写入 skills.json 的逻辑**：

```python
def _write_to_skills_json(self, skill: dict):
    path = SKILLS_BASE / "packs" / skill["name"] / "prompt.md"
    path.parent.mkdir(exist_ok=True)
    path.write_text(skill["prompt_content"])

    skills_json_path = SKILLS_BASE / "skills.json"
    cfg = json.loads(skills_json_path.read_text())
    cfg[skill["name"]] = {
        "name": skill["description"],
        "prompt_file": f"packs/{skill['name']}/prompt.md",
        "inject_to": skill["inject_to"],
        "traits_match": skill.get("traits_match"),
        "priority": "medium",
        "enabled": True,
        "auto_generated": True,   # 标记机器生成，便于审计
        "source_ticket": skill["ticket_id"],
    }
    skills_json_path.write_text(json.dumps(cfg, indent=2, ensure_ascii=False))
    skill_loader.reload()   # 热重载，立即生效
```

### 3.3 ConfirmSkillAction（Chat 工具）

**tool_schema**：

```python
{
    "name": "confirm_skill",
    "description": "确认或拒绝一条 AI 自动提取的 Skill 草案",
    "input_schema": {
        "properties": {
            "skill_id": {"type": "string"},
            "action": {"type": "string", "enum": ["confirm", "reject"]},
            "edit_content": {"type": "string", "description": "可选：修改 Skill 内容后再确认"}
        },
        "required": ["skill_id", "action"]
    }
}
```

**ChatAssistant 主动感知**：在 `_build_system_prompt` 里检查 `pending_skills.count_pending()`，若 > 0 则在 prompt 里追加提示，让 AI 在合适时机主动询问用户。

## 四、示例效果

**场景**：DevAgent 在 UE 项目中经过 3 次 Reflexion，发现"StaticMesh 引用导致的循环依赖要用 FSoftObjectPath 延迟加载"，最终验收通过。

**自动提取的 Skill 草案**：

```markdown
---
name: ue-softobjectpath-lazy-load
description: UE StaticMesh 循环依赖时改用 FSoftObjectPath 延迟加载
inject_to: [DevAgent]
traits_match: {any_of: [engine:ue5, engine:ue4]}
---

## 场景
当 A 模块依赖 B 模块的 StaticMesh，B 模块又直接或间接依赖 A，
导致 UHT 编译期循环依赖报错。

## 解法
将硬引用 `UStaticMesh*` 改为 `TSoftObjectPtr<UStaticMesh>` 或 `FSoftObjectPath`，
在 BeginPlay 或使用时通过 LoadSynchronous() / StreamableManager 延迟加载。

## 注意
- CDO 里不要调 LoadSynchronous（影响启动时间）
- 需要在 Build.cs 加 "Engine" 模块依赖
```

**对话展示**：

```
AI 助手：「发现 1 条新 Skill 草案（来自工单 #ed915d）：
  ue-softobjectpath-lazy-load — UE StaticMesh 循环依赖时改用 FSoftObjectPath 延迟加载
  要查看详情并决定是否加入技能库吗？」

用户：「看看」

AI 助手：（展示 Skill 内容）「这条经验来自上次 UE FPS 工单经历 3 次 Reflexion 后总结的。
  确认加入技能库，还是拒绝？」

用户：「确认」→ 写入 skills.json，DevAgent 后续 UE 工单自动注入此知识
```

## 六、风险与边界

| 风险 | 缓解 |
|------|------|
| LLM 提取质量差，生成无用 Skill | 人工确认门控，草案不自动生效 |
| Skill 内容过于具体，泛化性差 | Prompt 明确排除"只适用本工单"的案例 |
| skills.json 被机器生成内容污染 | `auto_generated: true` 标记，可批量清理 |
| 热重载影响正在运行的 Agent | reload 只重建缓存，不影响进行中的 action |
| 重复提取相同 Skill | name 唯一约束 + 提取前检查 skills.json 已有条目 |
