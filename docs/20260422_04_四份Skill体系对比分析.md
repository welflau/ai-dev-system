## 二、两大设计派别

四份系统其实分两派：

### 派别 A — Prompt 注入派（ai-dev-system + MagicAI）

**理念**：skill = 塞进 Agent system prompt 的**知识块**。

- Agent 对 skill 无感——只是 prompt 里多了一段内容
- 运行时按 `inject_to: [AgentName]` 选择谁拿到哪些 skill
- 优点：跟 Agent 执行链路完全透明，旧 Agent 代码零改动
- 缺点：**skill 不能"被调用"**，只能被动注入；无法支持 "/code-review" 这类用户触发的端到端流程

### 派别 B — Workflow 调用派（CCGS + AgentHub）

**理念**：skill = 用户 `/slash` 触发的**可调用编排脚本**。

- SKILL.md 的 frontmatter 声明 `allowed-tools / user-invocable / argument-hint`
- body 是步骤化指令："When this skill is invoked: 1. 读 X 文件 2. 调 Task(subagent_type=Y) 3. 汇总"
- 跑在 Claude Code CLI 的 Skill 工具上（或 CodeBuddy 等同类运行时）
- 优点：流程可复用、可组合、可被用户显式触发
- 缺点：依赖 CLI 运行时的原生 Skill 机制；搬到自研系统需要造一套 Skill 执行器

**两派不是替代，是互补**。我们当前是 A 派；B 派在用户触发的流程性操作上有明显价值（`/code-review` / `/milestone-review`）。

## 三、AgentHub 的三个独有亮点

### 3.1 🎯 `paths:` frontmatter 自动匹配（**最值得抄的一项**）

```yaml
---
paths:
  - "**/*.cpp"
  - "**/*.h"
  - "**/*.cc"
---
# C++ Coding Rules (Unreal Engine 5)
...
```

**语义**：编辑 `.cpp` 文件时这份 rule 自动激活，编辑 `.ts` 文件时自动换成 `typescript.md`。

跟我们当前机制对照：
- 我们现在按 **Agent 名**注入 skill（`inject_to: [DevAgent]`）—— DevAgent 永远一锅端拿到 react-dev + fastapi-dev + git-workflow 三个 skill
- AgentHub 按**文件 glob** 注入 rule —— 精准一个数量级，写 `.py` 只注入 Python 规范，写 `.html` 才注入 React/HTML 规范

### 3.2 🧰 rules-bridge-mcp

给不支持原生 `paths:` 的运行时做桥。~100 行 Node.js MCP Server，暴露单个工具：

```
get_coding_rules(file_path) → 匹配的 rule md 内容拼接
```

CodeBuddy CLI 原生支持 `paths:`，但 VSCode/Rider 的 IDE 插件不支持，就用这个 MCP 按需拉取。

**对本项目的价值**：我们已经有 MCP 客户端（v0.16.0）。如果搭起 rules/ 系统，可以照搬此 MCP 让 ChatAssistant 也能动态取规则（"用户问的是 index.html 就拉 react-dev rule"）。

### 3.3 📋 `Global/CodingGuide.md` 全局准则

```yaml
---
type: always
alwaysApply: true
description: IMMO 项目代码通用规范
---
# Coding Guide
## Language Requirements
所有代码内容必须用英文书写（团队母语为中文，不加约束 AI 会混用中文）...
## Naming
- C++/UE 布尔变量前缀 b（e.g., bIsVisible, bHasWeapon）
```

框架性规则（语言、命名、文档注释）**跟技术栈正交**，所有 skill 都应该遵守。

我们现在的情况：4 个 skill 各自写了一遍 "Git 规范、提交 message 规范、FastAPI 规范" —— 重复 + 容易不一致。

### 4.4 AgentHub 的 rules/（真正的宝藏）

| 资产 | 能否用 | 说明 |
|---|---|---|
| `Global/CodingGuide.md`（语言要求、命名、文档） | 🟢 **格式直接抄，内容本土化重写** | 照格式 + `alwaysApply: true` 搭一份我们自己的 |

| `Global/IMMO_Context.md`（项目上下文） | 🟢 **格式抄** | 对应我们的"项目背景"常量 |
| `CodingStyle/*.md`（按语言分文件） | 🟢 **格式抄**，逐语言重写 | 搭 `python.md` / `javascript.md` / `css.md` |
| `DomainSpecific/FileTypeExtension/cpp.md`（用 `paths:` 匹配） | 🟢 **机制抄** | 本项目对应 `.py` / `.html` / `.js` 各一份 |
| `Workflow/compile-check.md` | ⚪ | 我们没编译概念，直接跳 |

## 五、对本项目的建议（按 ROI 排序）

### 🟢 立即做（半天到一天，收益高）

#### 5.1 给 SkillLoader 加 `paths:` 匹配机制

**改动**：
- `backend/skills/loader.py` 支持读 `prompt.md` 的 YAML frontmatter（`paths: [...]`）
- `build_prompt_for_agent(agent_type, current_file=None)` 加个 `current_file` 参数
- 有 `paths:` 的 skill 只在当前文件匹配时注入，没 `paths:` 的维持原来按 `inject_to` 注入

**收益**：DevAgent 写 `.py` 时不再被灌 React 规范、写 `.html` 时不再被灌 FastAPI 规范，精准度上台阶。

**工作量**：~半天（加载器改动 + 现有 4 个 skill 补 `paths:` frontmatter）

#### 5.2 引入 `global.md` 全局准则

**改动**：
- 新建 `backend/skills/rules/global.md`（带 `alwaysApply: true`）
- 抽出 4 个 skill 里重复的"语言要求 / 命名规范 / 文档注释 / Git 提交规范"
- SkillLoader 对每个 Agent 都无条件注入这份

**收益**：消除 4 个 skill 的冗余 + 规范一致性保证。

**工作量**：~2 小时

### 🟡 中期（1-2 天，战略）

#### 5.3 仿 AgentHub 的 rules/ 目录结构

按语言/技术栈拆分：

```
backend/skills/rules/
├── global.md                  ← alwaysApply
├── language/
│   ├── python.md              ← paths: **/*.py
│   ├── javascript.md          ← paths: **/*.js, **/*.ts
│   ├── html.md                ← paths: **/*.html
│   └── css.md                 ← paths: **/*.css
├── domain/
│   ├── fastapi.md             ← paths: backend/**
│   ├── react.md               ← paths: frontend/**
│   └── git.md                 ← alwaysApply (工作流)
```

**收益**：跟 AgentHub 生态对齐，后续能借鉴他们的 rules 升级

**工作量**：~1 天（建目录 + 把现有 4 个 skill 拆解重组）

#### 5.4 rules-bridge-mcp 引入

**前提**：先做完 5.3。

把那 100 行 Node 代码拉进来，挂到我们的 mcp_servers.json，让 ChatAssistant 能在对话里动态拉规则。

**工作量**：~半天

### 🔴 长期（不在当前路径上）

#### 5.5 Workflow 调用派 skill（暂不做）

做 `/code-review` `/milestone-review` 这类可触发 skill：
- 要造 Skill 执行器（解析 frontmatter + 步骤化 body）
- 跟前端聊天 slash 命令系统打通
- 跟 TODO 上已有的 "v0.15.3 前端 SOP 拖拽编辑器" 概念重叠

**建议**：先把 SOP 编辑器做出来，两者合并考虑；不单独做 Skill workflow 执行器。

## 六、不做的决策

### 6.1 不直接 drop-in 任何 skill 内容

- MagicAI 5 个 Godot 特化，污染 prompt
- CCGS 37 个游戏开发特化 + 架构不兼容
- AgentHub 30 个 UE5/C++ 特化 + 架构不兼容

**结论**：外部 skill **内容** 对本项目直接价值为 0，**格式 / 机制**才是可借鉴的。

### 6.2 不换 skill 运行模式

我们当前 Prompt 注入派架构已稳定运行，跟 6 个 Role Agent + Action 池 + SOP 驱动体系 **解耦得很干净**。换成 Workflow 调用派意味着要把 skill 执行器和 Agent 编排合二为一，风险大。

**结论**：保持 Prompt 注入派为主，升级它的精准度（加 `paths:`），而非换骨。

## 七、总结

| 来源 | 核心收获 |
|---|---|
| **MagicAI game_agent** | 证实我们 SkillLoader 的 "MagicAI 包可 drop-in" 声明是真的；内容无用 |
| **CCGS** | Workflow 调用派的代表；`rules/` + `@include` 思路启蒙；37 个 skill 大多游戏特化 |
| **AgentHub (IMMO/UE5)** | ⭐ **最有价值的一份**：`paths:` frontmatter 自动匹配机制 + 全局准则分层 + rules-bridge-mcp 工程实现 |

最值得记住的一句话：

> **外部 skill 生态对我们的价值是"机制"而不是"内容"。**
> **AgentHub 的 `paths:` 自动匹配是四份里最具体、最立等可取的一个改进点。**
