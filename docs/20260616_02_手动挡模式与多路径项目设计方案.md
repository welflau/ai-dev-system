# 手动挡模式 + 多路径项目 + 快速打开目录 设计方案

## 版本
> v1.0 2026-06-16

---

## 背景与动机

现有系统是纯「自动挡」：创建需求 → 拆工单 → Agent 自动流转 → 自动 git commit。

这套流程适合全新项目、从零开始的开发任务。但现实中存在大量场景：
- 项目已有代码，想让 AI 直接读写修改，自己决定何时提交
- 项目由多个路径构成，分别由 git 和 P4 管理
- 不想走工单流程，直接在对话中完成任务
- 想快速打开一个现有目录，不想填一堆配置

本方案新增「手动挡」模式，以及与之配套的多路径项目体系和快速打开目录功能。

---

## 一、手动挡 vs 自动挡

### 核心差异

| 维度 | 自动挡（现有） | 手动挡（新增） |
|---|---|---|
| 任务驱动方式 | 需求 → 工单 → Agent 流转 | 纯对话，AI 直接执行 |
| 工单 | 有 | 无 |
| 自动 git commit | 是 | 否 |
| 自动 P4 submit | 否 | 否 |
| 提交时机 | Agent 每阶段自动提交 | 用户在对话中主动要求 |
| 适合场景 | 全新项目、大型功能开发 | 现有项目修改、快速调试、探索性开发 |

### 模式切换

模式在**项目级别**配置，默认根据创建方式决定：
- 「新建项目（完整配置）」→ 默认自动挡
- 「打开目录（快速通道）」→ 默认手动挡

用户可在项目设置中随时切换。

## 二、「是否创建需求」卡片

手动挡对话过程中，AI 持续理解用户意图。当判断满足以下条件时，主动弹出选择卡片：

**触发条件**（AI 综合判断）：
- 任务描述已足够清晰（目标、范围、验收标准基本明确）
- 涉及多个文件或模块的系统性改动
- 对话轮次 ≥ 2 轮且任务仍在推进

**卡片内容**：
```
┌─────────────────────────────────────────┐
│ 💡 任务已明确，是否创建需求正式跟踪？      │
│                                         │
│ 「优化登录模块性能，引入 JWT 缓存」        │
│                                         │
│  [创建需求]        [继续对话处理]         │
└─────────────────────────────────────────┘
```

- **创建需求** → 切换自动挡，进入工单流转
- **继续对话处理** → 保持手动挡，AI 直接执行，对话中展示完整过程

## 三、多路径项目

### 路径体系

一个项目由多个路径构成，分两类：

**用户配置路径**（手动添加，可多个）：
- 每个路径独立配置 VCS 类型（auto-detect / git / p4 / none）
- 默认权限：可读写

**自动识别路径**（系统从用户配置路径中推断补充）：
- UE 引擎路径 ← 从 `.uproject` 的 `EngineAssociation` 字段读取
- git submodule ← 从 `.gitmodules` 读取
- 其他外部依赖（P4 external mapping 等）
- 默认权限：**只读**，用户确认后可改为可写

### 路径权限规则

```
写文件前：
├─ 只读路径 → 弹确认卡片「该文件在只读路径中，是否允许修改？」
│               └─ 用户允许 → 写入（本次会话有效）
│               └─ 用户拒绝 → 中止该文件修改
├─ git 路径  → 直接写，不 git commit
├─ P4 路径   → p4 edit <file> → 写入，不 p4 submit
└─ 无管理    → 直接写
```

### P4 Checkout 流程

```python
# 写入文件前触发
async def ensure_writable(file_path):
    vcs = detect_vcs(file_path)
    if vcs == "p4":
        result = await run("p4 edit " + file_path)
        if result.returncode != 0:
            raise Exception(f"P4 checkout 失败: {result.stderr}")
    elif vcs == "readonly":
        confirmed = await ask_user_confirm(file_path)
        if not confirmed:
            raise Exception("用户拒绝修改只读文件")
```

P4 环境前提：本地已配置好 P4 环境变量（`P4PORT`、`P4USER`、`P4CLIENT`）或 `.p4config` 文件。
识别不到 P4 环境时：提示用户配置，不强制中断。

### VCS 自动检测规则

| 检测方式 | 判断为 |
|---|---|
| 路径下有 `.git/` | git 管理 |
| 路径下有 `.p4config` 或 `p4 where <file>` 返回成功 | P4 管理 |
| UE `.uproject` 中 `EngineAssociation` 指向的路径 | 自动识别路径（只读）|
| 以上都没有 | 无管理（直接读写）|

## 四、快速打开目录

### 交互入口

用户在 AI 助手对话框中直接发送一个路径：

```
用户：F:\ADS_Projects\MyGame
```

### AI 处理流程

```
收到路径
    ↓
判断路径是否已在系统项目中
  ├─ 已有项目 → 切换到该项目 + 告知用户
  └─ 新路径   → 后台扫描识别（1-3秒）
                    ↓
               弹确认卡片：
               ┌──────────────────────────────────┐
               │ 📁 检测到新项目，是否创建？         │
               │                                  │
               │ 项目名：MyGame                    │
               │ 类型：UE5 / C++ / Blueprint       │
               │ git：Source/（已检测到）           │
               │ P4：Content/（已检测到）           │
               │ 引擎：F:\UE5\（自动识别，只读）     │
               │                                  │
               │  [创建项目]    [取消]              │
               └──────────────────────────────────┘
                    ↓ 用户确认
               创建项目（手动挡，无工单）
               进入对话，AI 汇报识别结果
```

### 自动识别规则

| 特征文件 / 检测方式 | 识别结论 |
|---|---|
| `*.uproject` | UE5 项目；读 `EngineAssociation` 补充引擎路径 |
| `package.json` | Node/JavaScript；读 `name` 字段作项目名 |
| `requirements.txt` / `pyproject.toml` | Python 项目 |
| `*.sln` / `*.csproj` | C# / .NET 项目 |
| `Cargo.toml` | Rust 项目 |
| `go.mod` | Go 项目 |
| `.git/` | 检测到 git，记录为 git 路径 |
| `.p4config` / `p4 where` 返回成功 | 检测到 P4，记录为 P4 路径 |
| `Content/` + `Source/` + `.uproject` | UE 标准目录结构 |
| 目录名 | 作为项目名兜底 |

**识别不到时**：卡片中对应字段标注「未识别，建议手动配置」，不阻断创建流程。

## 五、手动挡对话行为

### AI 执行完成后的汇报

```
✅ 已完成修改，共涉及 4 个文件：

修改文件：
  📝 Source/Login/LoginController.cpp  （git）
  📝 Source/Login/LoginController.h    （git）
  🔓 Content/UI/LoginWidget.uasset     （P4，已 checkout）
  📝 Config/DefaultGame.ini            （P4，已 checkout）

未修改：
  🔒 Engine/Source/Runtime/...         （只读路径，已跳过）

请根据需要自行决定提交：
  git 部分：git add . && git commit
  P4 部分：p4 submit
```

### git 操作权限

手动挡下 git 操作遵循「用户主导」原则：
- AI **不会**自动执行：`git add`、`git commit`、`git push`
- AI **可以**执行（用户要求时）：`git diff`、`git log`、`git status`、`git stash`、用户明确要求的 commit/push
- AI **会自动**执行：读文件、写文件（git 工作区直接修改）

## 六、需改动的模块

| 模块 | 改动说明 |
|---|---|
| `backend/database.py` | projects 表新增 `mode`（auto/manual）、`extra_paths`（JSON，多路径）字段 |
| `backend/api/projects.py` | 新增「扫描识别」端点；projects CRUD 支持 extra_paths |
| `backend/git_manager.py` | 新增 `detect_vcs(path)`；写文件前触发 `ensure_writable()` |
| `backend/p4_manager.py` | 新增 P4 工具类：`p4_edit()`、`p4_info()`、`detect_p4_root()` |
| `backend/orchestrator.py` | 手动挡下跳过自动 commit；跳过工单流转 |
| `backend/agents/chat_assistant.py` | 感知多路径上下文；写文件前调用 `ensure_writable()`；弹「创建需求」卡片逻辑 |
| `frontend/app.js` | 「打开目录」识别卡片；只读确认卡片；手动挡标识 |
| `frontend/styles.css` | 新增卡片样式 |

## 七、实现顺序

```
Phase 1：基础设施
  - p4_manager.py（p4 edit / p4 info / detect）
  - detect_vcs(path) 通用方法
  - projects 表 extra_paths / mode 字段

Phase 2：快速打开目录
  - 扫描识别端点
  - AI 助手识别路径 → 弹确认卡片
  - 创建项目（手动挡默认）

Phase 3：手动挡执行
  - 写文件前 ensure_writable()
  - 禁用自动 git commit（手动挡下）
  - 对话完成汇报（改了哪些文件、VCS 状态）

Phase 4：「创建需求」卡片
  - AI 判断触发时机
  - 卡片 UI + 交互
  - 切换自动挡流程

Phase 5：只读路径确认
  - 用户确认卡片
  - 引擎路径权限管理
```

## 八、验证场景

1. 发送 UE 项目路径 → 识别卡片显示正确信息 → 创建项目进入对话
2. 让 AI 修改 P4 管理的文件 → 自动执行 `p4 edit` → 文件可写
3. 让 AI 修改引擎路径文件 → 弹只读确认 → 用户允许后修改
4. 对话完成后检查 git status → 有未提交修改，无自动 commit
5. 用户说「帮我 commit 一下」→ AI 执行 git commit
6. 对话几轮后 AI 弹「是否创建需求」→ 选创建 → 进入工单流程
