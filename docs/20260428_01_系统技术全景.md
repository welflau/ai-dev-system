## 一、系统是什么

### 1.1 一句话定位

**AI 自动开发系统**是一个**工单驱动的多 Agent 自动软件开发平台**。用户提交需求，系统自动完成需求拆解、架构设计、代码编写、测试验收、Git 管理和部署上线的全流程。

### 1.2 解决的核心问题

直接让 LLM 写代码并部署等于把不确定性推到生产环境：

| 问题 | 表现 |
|------|------|
| 幻觉 | 调用不存在的 API，捏造文件路径 |
| 盲写 | 不看已有代码，重写整个文件 |
| 截断 | max_tokens 不够，代码写到一半停了 |
| 失忆 | 多轮对话后忘记之前的架构决策 |
| 无法验证 | 生成的代码从未真正运行过 |

系统的答案是 **Harness 架构**：AI 是引擎，Harness 是方向盘和刹车。

### 1.3 系统全景图

```
  用户
   │
   │ 一句话描述需求（"做个 UE5 FPS 游戏"）
   ▼
```

## 二、整体架构

### 2.1 Harness 四层模型

```
┌─────────────────────────────────────────────────────────────┐
│  Layer 4：工程规范层                                          │
│  Git 分支隔离 | CI/CD 三环境 | 合并保护 | 产物落盘            │
├─────────────────────────────────────────────────────────────┤
│  Layer 3：质量门禁层                                          │
│  SelfTest(L1静态+L2编译) → CodeReview → Acceptance           │
│  Reflexion 反思 → fix_issues 自动修复闭环                    │
├─────────────────────────────────────────────────────────────┤
│  Layer 2：Agent 执行层                                        │
│  ActionNode JSON 约束输出 | BY_ORDER 顺序执行 | REACT 动态决策│
│  Skills 动态注入 | 已有代码上下文注入 | 对话历史压缩          │
├─────────────────────────────────────────────────────────────┤
│  Layer 1：LLM 能力层                                          │
│  Anthropic Claude（tool_use + vision）                       │
│  prompt caching | 16k max_tokens | temperature=0.3~0.7      │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 技术栈

| 层 | 技术 | 关键选择理由 |
|----|------|------------|
| 前端 | 纯 HTML + Vanilla JS（app.js 11k 行） | 零构建依赖，SSE 原生支持 |
| 后端 | Python 3.12 + FastAPI + asyncio | async 便于 LLM 并发调用 |
| 数据库 | SQLite（WAL 模式 + FTS5） | 单机嵌入，FTS5 支持中文 trigram 搜索 |
| AI | Anthropic Claude（claude-sonnet-4-x） | 原生 tool_use，128k 上下文 |
| VCS | Git（本地裸仓库） | 分支隔离，代码落盘可追溯 |
| 实时推送 | SSE（Server-Sent Events） | 单向流，比 WebSocket 轻量 |

### 2.3 一个需求的完整数据流

```
用户输入："给游戏加个武器系统"
        │
        ▼ POST /api/projects/{id}/requirements
   需求落库（REQ-xxx，status: submitted）
        │
        ▼ Orchestrator 触发
   ProductAgent.analyze_and_decompose
        │  SummarizeCode 读懂现有代码
        │  DecomposeAction 拆成 N 个工单
        ▼
   工单 1: "武器基类 AWeaponBase"  → Ticket-001（pending）
   工单 2: "步枪 ARifle 子类"       → Ticket-002（pending，依赖 001）
   工单 3: "武器拾取组件"           → Ticket-003（pending，依赖 001）
        │
        ▼ 依赖就绪后 Orchestrator 调度
   ArchitectAgent.design_architecture
        │  读已有代码 + 分析增量改动点
        │  输出：files_to_create / files_to_modify / 估时
        ▼
   DevAgent.develop
        │  PlanCodeChangeAction：规划哪些文件改什么
        │  WriteCode / ModifyFile：逐文件精准增量修改
        │  SelfTestAction：Layer1 静态规则 → Layer2 UBT 编译
        ▼
   ReviewAgent.code_review
        │  CodeReviewAction：读实际 Git 文件做审查
        │  pass_score: 7，低分 warning 不阻塞
        ▼
   ProductAgent.acceptance_review
        │  读已有代码 + 工单产物做验收
        │  通过 → acceptance_passed
        │  打回 → Reflexion 反思 → DevAgent.fix_issues → 重试
        ▼
   DeployAgent.deploy
        │  CIStrategy 按项目类型选策略（Web/UE/Default）
        │  push 到 test 分支，触发 CI 流水线
        ▼
   工单状态：deployed ✅
   SSE 推送前端，看板实时更新
```

### 3.1 Agent 一览

```
┌──────────────────────────────────────────────────────────────┐
│                    Agent 流水线                               │

│  │拆单+验收  │   │架构设计   │   │代码编写   │                 │
│  └──────────┘   └──────────┘   └────┬─────┘                 │
│                                      │                       │
│                              ┌───────┴───────┐              │
│                              ▼               ▼              │
│                        ┌──────────┐   ┌──────────┐          │
│                        │  Review  │   │   Test   │          │
│                        │  Agent   │   │  Agent   │          │
│                        │代码审查   │   │测试验证   │          │
│                        └──────────┘   └──────────┘          │
│                              │               │               │
│                              └───────┬───────┘              │
│                                      ▼                       │
│                              ┌──────────────┐               │
│                              │ Deploy Agent │               │
│                              │  部署上线     │               │
│                              └──────────────┘               │
│                                                              │
│  独立运行：┌──────────────────┐                              │
│           │ ChatAssistant    │                              │

│           └──────────────────┘                              │
└──────────────────────────────────────────────────────────────┘
```

### 3.2 各 Agent 职责与核心 Action

| Agent | 任务入口 | 核心 Action | 执行模式 |
|-------|---------|------------|---------|
| **ProductAgent** | analyze_and_decompose / acceptance_review | SummarizeCode → Decompose → AcceptanceReview | BY_ORDER |
| **ArchitectAgent** | design_architecture | DesignArchitecture（ActionNode） | SINGLE |
| **DevAgent** | develop / rework / fix_issues / run_engine_compile | PlanCodeChange → WriteCode → SelfTest | SINGLE（路由分发） |
| **ReviewAgent** | code_review | CodeReviewAction（读实际 Git 文件） | SINGLE |
| **TestAgent** | run_tests / run_playtest | SelfConsistency 投票 / UEPlaytestAction | SINGLE |
| **DeployAgent** | deploy | CIStrategy → UEPackage / WebDeploy | SINGLE |
| **ChatAssistantAgent** | chat / chat_global | 23 个工具（tool_use） | REACT |

### 3.3 三种执行模式

```python
class ReactMode(Enum):
    SINGLE   = "single"    # orchestrator 指定 action，执行一次
    BY_ORDER = "by_order"  # 按 action_classes 顺序依次执行，前一步输出注入后一步
    REACT    = "react"     # LLM 动态决策下一步，最多 max_react_loop=5 轮
```

**BY_ORDER 的关键设计**：后一步的 context 会合并前一步的 files 输出（`dict.update` 语义，key 冲突取后者），保证代码产物在整个链路中累积传递，不会丢失。

### 3.4 工单状态机

```
            需求提交
               │
               ▼
          [ submitted ]
               │  ProductAgent
               ▼
          [ decomposed ]──→ 拆出 N 个工单
               │
    ┌──────────┼──────────┐
    ▼          ▼          ▼   （工单，按依赖顺序调度）
 [ pending ] [ pending ] ...
    │
    │ ArchitectAgent
    ▼
[ architecture_done ]
    │
    │ DevAgent.develop
    ▼
[ development_done ]──→ SelfTest 失败 ──→ [ self_test_failed ]
    │                                            │
    │ ReviewAgent                                │ DevAgent.fix_issues
    ▼                                            ▼
[ review_passed ]                         [ development_done ]
    │
    │  （UE 项目额外经过）
    │  ├─ engine_compile_passed / engine_compile_failed
    │  └─ play_test_passed / play_test_failed
    │
    │ ProductAgent.acceptance_review
    ├──────────────────────────────→ [ acceptance_rejected ]
    │                                         │
    │                               Reflexion │ DevAgent.rework
    ▼                                         ▼
[ acceptance_passed ]                  [ development_done ]
    │                                  （最多重试 5 次，超限强制通过）
    │ DeployAgent
    ▼
 [ deployed ] ✅
```

### 3.5 Orchestrator 调度机制

Orchestrator 是整个系统的"心脏"，负责把对的工单交给对的 Agent。

**两种触发方式**（互为补充）：

```
事件总线（主）：工单状态变更 → publish("ticket_ready") → 立即调度
                                                              ↓
轮询兜底（辅）：每 30s 扫描一次可流转工单 ─────────────────────┘
```

**僵尸检测**：工单处于 `*_in_progress` 状态超过 60s 无心跳，自动重置到对应的完成状态，避免卡死。

**依赖检查**：工单有 `dependencies: ["Ticket-001"]` 时，等所有前置工单到达 `testing_done / deployed` 才开始调度。

## 四、SOP 驱动的流程引擎

### 4.1 为什么要 SOP

早期版本流程硬编码在 orchestrator.py 里，改一个阶段要动几百行代码。引入 SOP（Standard Operating Procedure）后，流程变成配置，逻辑与策略分离。

### 4.2 核心结构：base + fragments

```
backend/sop/
├── _core.yaml          ← 基础流程（所有项目都有）
├── fragments/          ← 按 trait 动态注入的流程片段
│   ├── engine_compile.yaml   （engine:ue* / godot* / unity 项目）
│   ├── play_test.yaml        （engine:ue* 项目）
│   ├── deploy_web.yaml       （platform:web 项目）
│   ├── deploy_ue.yaml        （engine:ue* 项目）
│   ├── smoke_test.yaml
│   ├── i18n_check.yaml
│   ├── multiplayer_stress.yaml
│   └── asset_gen.yaml
└── loader.py           ← compose_sop()：组合最终流程
```

### 4.3 SOP 组合示意

以一个 UE5 游戏项目为例：

```
_core.yaml（基础）          fragments（按 trait 注入）
─────────────────           ─────────────────────────
architecture                engine_compile（priority=95，insert_after: development）
development             ┐
                        ├─  engine_compile  ← 插入（UE 编译验证）
                        ├─  play_test       ← 插入（Automation 测试）
code_review             ┘
acceptance
rework
deploy                  ┐
                        └─  deploy_ue       ← 覆盖（UAT BuildCookRun）

最终组合结果：
architecture → development → engine_compile → play_test → code_review
→ acceptance → rework → deploy_ue
```

### 4.4 stage_overrides：patch 已有 stage 的 config

fragments 除了插入新 stage，还可以通过 `stage_overrides` 修改已有 stage 的配置：

```yaml
# engine_compile.yaml
stage_overrides:
  - id: development          # 要 patch 的 stage
    config:
      ue_precompile: true    # 开启 Layer 2 单文件预编译
      ue_precompile_timeout: 120
```

DevAgent 的 `development` stage 收到 `sop_config.ue_precompile=true` 后，SelfTestAction 会自动触发 Layer 2 UBT 编译检查（详见第六章）。

### 4.5 trait-first 动态组装

项目创建时打上 traits 标签，SOP loader 根据 traits 过滤可用 fragments：

```
项目 traits: [platform:desktop, category:game, engine:ue5, lang:cpp]
                                         │
                          ┌──────────────┼──────────────┐
                          ▼              ▼              ▼
                  engine_compile    play_test      deploy_ue
                  （engine:ue* ✓）  （engine:ue*✓） （engine:ue*✓）
```

```
项目 traits: [platform:web, category:app, framework:react]
                     │
                     ▼
               deploy_web（platform:web ✓）
```

同一套引擎，不同项目类型自动走不同流程，零代码改动。

## 五、ActionNode 与 Action 池

### 5.1 ActionNode：用 JSON Schema 约束 LLM 输出

LLM 输出是自然语言，不可直接执行。ActionNode 把 LLM 的输出约束成严格的结构化 JSON：

```
Prompt（含 JSON Schema 定义）
         │
         ▼ LLM
  {"files": {"src/main.py": "..."}, "summary": "..."}
         │
         ▼ ActionNode.fill()
  校验字段完整性 + 类型 + 必填项
         │
         ▼ 成功
  强类型 Python 对象，可直接使用
```

**核心优点**：
- 幻觉产生的随机文本被强制解析，格式错误立即报错重试
- Schema 注入 prompt 即是文档，LLM "看着 Schema 填表"
- 支持 `BY_ORDER` 模式：前一步的输出字段自动注入后一步的 prompt

### 5.2 关键 Action 一览

| Action | 职责 | 关键设计 |
|--------|------|---------|
| **PlanCodeChangeAction** | 先规划再改代码 | 两阶段：Phase1 产 JSON plan（改哪些文件/函数），Phase2 按 plan 逐文件精准 patch |
| **WriteCodeAction** | 全新生成代码 | 用于空项目首次生成，max_tokens=16000 |
| **CodeReviewAction** | 读实际 Git 文件审查 | 非盲审，pass_score=7，低分 warning 不阻塞流程 |
| **DecomposeAction** | 需求拆工单 | 注入已有文件列表 + 代码摘要，避免拆出重复功能 |
| **SelfTestAction** | 开发后自测 | Web：4 项通用检查；UE：Layer1 静态规则 + Layer2 UBT 编译 |
| **ReflectionAction** | 失败反思 | 结构化输出根因 + 策略变更，注入下次开发 prompt |
| **SummarizeCodeAction** | 代码摘要 | 读仓库文件树 + 关键文件，提炼 project_summary 给后续 Action 使用 |
| **UECompileCheckAction** | UBT 编译验证 | subprocess.Popen + threading，兼容 Python 3.14 + UE 5.7 |
| **UEPlaytestAction** | 自动化游戏测试 | Automation Framework headless，capture_screenshots |

### 5.3 Reflexion 反思闭环

失败不等于结束，Reflexion 让系统从错误中学习再重试：

```
DevAgent 开发完成
        │
        ▼ SelfTestAction
   blocking_issues > 0？
        │ Yes
        ▼
  [ self_test_failed ]
        │
        ▼ Orchestrator → DevAgent.fix_issues
  ReflectionAction 分析失败原因
        │
        ▼ 输出结构化反思
  {
    "root_cause": "AWeaponBase 缺少 UFUNCTION() 宏",
    "missed_requirements": ["OnRep 回调不能加 UFUNCTION"],
    "strategy_change": "先过 R2 规则再写 OnRep",
    "specific_changes": ["移除 OnRep_Health 上的 UFUNCTION()"],
    "confidence": 0.85
  }
        │
        ▼ 注入下次开发 prompt 的 reflection_block
  PlanCodeChangeAction（携带反思上下文）
        │
        ▼
   重新开发 → SelfTest → ...
        │
   最多重试 5 次（force_pass_after_max_rework: true）
```

**特化处理**：
- `engine_compile_failed`：解析结构化 UBT errors（file/line/category），内置 UE 常见坑提示
- `play_test_failed`：注入 failed_tests + playtest_summary
- `acceptance_rejected`：注入 rejection_reason + previous_reflections（最多 3 次历史）

## 六、UE 游戏开发专项支持

### 6.1 UE 四件套

UE 项目开发有其特殊性（巨型引擎、C++ 宏体系、UBT 构建系统），系统为此专门打造了四件套：

```
① 引擎检测          ② 模板实例化        ③ UBT 编译验证      ④ Playtest 测试
─────────────────   ─────────────────   ─────────────────   ─────────────────
启动时扫描本地        从内置模板生成        编译 C++ 代码        自动化运行游戏
UE 安装路径          UE 工程骨架          验证代码正确性        验证功能正确性
 ↓                    ↓                    ↓                    ↓
engine_path →        实例化 .uproject     UBT 编译报告         Automation
ue_version →         .Build.cs           errors/warnings      Framework
ue_api_version       Source/*.cpp/h      结构化解析            headless 运行
```

### 6.2 UE 自测两层架构

单次开发后的自测分两层，快慢结合：

```
DevAgent 写完代码，调用 SelfTestAction
                │
                ▼ 检测到 engine:ue* trait
        ┌───────────────────────────────┐
        │  Layer 1：静态规则检查（<1s）   │
        │                               │
        │  R1  GENERATED_BODY 宏缺失     │
        │  R2  OnRep 禁止 UFUNCTION()   │
        │  R3  #include 路径合法性       │
        │  R4  Build.cs 模块白名单       │
        │  R5  .uproject Modules 同步   │
        │  R6  Target.cs IncludeOrder   │
        │  R7  常用类型必需 header        │
        │                               │
        │  blocking_issues > 0？         │
        └───────┬───────────────┬───────┘
                │ Yes           │ No
                ▼               ▼
         self_test_failed  Layer 2：UBT -SingleFile（30-90s）
         → fix_issues      │
                           │  sop_config.ue_precompile == true
                           │  （engine_compile fragment 自动开启）
                           ▼
                   UECompileCheckAction
                   UBT -SingleFile [first.cpp] -Target=Game
                           │
                    成功 ── ┤── 失败
                    │       │
                    ▼       ▼
               pass    self_test_failed → fix_issues
                       （省掉全项目编译 3-5 分钟）
```

### 6.3 UE Playtest：让 AI 真正"玩"游戏

传统方法只验证编译通过，Playtest 验证游戏逻辑是否真的能跑：

```
TestAgent.run_playtest
        │
        ▼ UEPlaytestAction
  启动 UE Editor（Headless 模式）
  -ExecCmds "Automation RunTests Project."
        │
        ▼ 日志解析
  ┌─────────────────────────────────────────┐
  │  测试能力范围                             │
  │  ✅ 基础启动：游戏能否正常启动到主菜单      │
  │  ✅ 关卡加载：地图/场景是否能正确加载       │
  │  ✅ 角色控制：玩家输入响应是否正常          │
  │  ✅ 游戏逻辑：核心玩法机制验证             │
  │  ✅ 性能检测：帧率、内存占用等基础指标      │
  └─────────────────────────────────────────┘
        │
  failed_tests > 0？
        │ Yes
        ▼
  play_test_failed → Reflexion → DevAgent.fix_issues
  （注入 failed_tests + playtest_summary + UE 常见失败原因）
```

**技术实现**：
- `subprocess.Popen + threading.Thread` 方案（解决 Python 3.14 + Windows + UE 5.7 asyncio 子进程兼容问题）
- `timeout_seconds: 600`，超时强制 kill
- `capture_screenshots: true`，失败截图保存到 `screenshots/` 供 AI 诊断

## 八、CI/CD 与部署

### 8.1 CIStrategy 抽象

```python
class CIStrategy(ABC):
    priority: int = 0          # 数字越大越优先匹配

    def matches(self, project: dict) -> bool: ...
    def get_pipeline(self, project: dict) -> List[Stage]: ...

# 三个策略，按 priority 匹配第一个
UECIStrategy     priority=100  # engine:ue* 项目
WebCIStrategy    priority=50   # platform:web 项目
DefaultStrategy  priority=0    # 兜底
```

### 8.2 CI/CD 流水线对比

| 阶段 | Web 项目 | UE 项目 |
|------|---------|---------|
| 构建 | npm build / python setup | UAT BuildCookRun |
| 测试 | jest / pytest | Automation Framework |
| 打包 | docker build | UE Package（Win/Linux/Mac） |
| 部署 | nginx / k8s | 上传到发布服务器 |
| 环境 | dev / test / prod | dev / test / staging |

### 8.3 三环境隔离

```
dev 环境：开发分支（feature/*）自动部署，快速迭代
    │
    │ CI 通过
    ▼
test 环境：develop 分支，手动 / 自动触发，QA 测试
    │
    │ 验收通过
    ▼
prod 环境：main 分支，发布 tag 触发，生产上线
```

### 8.4 SOP 驱动的部署 fragments

```yaml
# deploy_ue.yaml
stage:
  agent: DeployAgent
  action: deploy
  config:
    strategy: ue
    target_env: test
    build_config: Development
    target_platform: Win64
    run_automation: true
    uat_extra_flags: "-pak -compressed"
```

DeployAgent 读取 `sop_config.strategy`，动态选择对应的 CIStrategy，无需硬编码项目类型判断。

## 九、未来规划

当前系统（v0.19.x）已实现从需求到部署的全链路自动化，但在**智能深度**、**工程可靠性**和**平台化**三个维度上仍有较大提升空间。

```
现在（v0.19）           近期（v0.20）            中期               长期
──────────────        ──────────────────       ──────────────     ──────────────
单机离线开发   ──▶    UE 编辑态感知      ──▶   多 LLM / 并发  ──▶  平台化 / 生态
串行工单调度          错误精准定位              自适应流程          多项目协作
纯 C++ / 代码         Blueprint / 资产          跨平台扩展          插件市场
```

### 9.1 提升 AI 决策质量

**当前短板**：Reflexion 修 bug 时是"离线盲猜"，AI 看不到运行时状态，导致多轮重试浪费在错误方向。

- **错误根因分析**：在 Reflexion 之前增加专门的 `DebugErrorAction`，先结构化解析 stack trace、精准定位 file/line，再交给 Reflexion 制定修复策略，从"猜测原因"变为"知道原因"。
- **UE 编辑态感知（MCP 集成）**：接入 UE Editor 的实时状态查询能力，让 AI 在修复 BP / 资产问题时能直接看到 Editor 现状，而不是依赖静态文件推断。这是 UE 开发质量从"能编译"到"能正确运行"的关键跨越。
- **联网调研能力（ResearchAgent）**：在架构设计阶段接入实时搜索，AI 可主动调研竞品方案、查阅最新文档，减少因知识过期导致的错误决策。

### 9.2 提升工程可靠性

**当前短板**：系统强依赖单一 LLM 提供商，工单串行执行效率低，部分核心模块仍有技术债。

- **多 LLM 支持与降级链**：抽象 LLMClient 层，支持 Anthropic（主）→ Ollama 本地模型（降级）的自动切换，消除对单一 API 的依赖，同时让本地敏感项目可以完全离线运行。
- **并发调度**：Orchestrator 按依赖图并行执行无依赖工单，项目整体交付速度可成倍提升。
- **技术债清理**：Orchestrator（~2000 行单文件）拆分为调度 / 上下文构建 / 结果处理三个模块；前端 `app.js`（11k 行）渐进式模块化；SQLite 在高并发场景下迁 PostgreSQL。

### 9.3 提升使用体验

**当前短板**：修改工作流需要手改 YAML，Agent 行为无可视化配置入口，仓库视图功能初级。

- **SOP 可视化编辑器**：将 YAML 流程转化为可拖拽的 stage 节点图，业务人员无需了解 YAML 即可调整开发流程。
- **Agent 配置面板**：在线切换 ReactMode、启用/禁用指定 Action、调整 LLM 参数，支持针对不同项目类型定制 Agent 行为。
- **仓库视图增强**：Commit Diff 真 side-by-side 分栏、分支关系树形展示（feat/* → develop → main）、大文件截断与语法高亮对齐。

### 9.4 平台化与生态扩展

**方向**：从单机工具演进为可扩展的开发平台。

- **插件市场**：Action / SOP Fragment / Skill Pack 可打包发布和一键安装，社区可贡献垂直领域的专属能力（Unity 游戏、微信小程序、区块链合约等）。
- **多项目协作**：跨项目共享公共库工单，多个项目的 Agent 可以协作开发同一个基础库，避免重复劳动。
- **专项 Agent 扩展**：ReportAgent（每日进展日报）、SecurityAgent（OWASP 扫描）、PerformanceAgent（API 响应回归）等垂直能力 Agent，按需接入流水线。
- **跨平台扩展**：UE 引擎检测从 Windows-only 扩展到 Mac / Linux，支持在 CI 服务器上运行全量 UE 构建验证。
