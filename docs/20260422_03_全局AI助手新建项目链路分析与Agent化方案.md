> 日期: 2026-04-22 | 关联: `docs/20260417_01_ChatAssistant_Agent化迁移方案.md`（项目内聊天已迁移，本文档处理最后一条漏网之鱼）

## 一、现状：全局 AI 助手新建项目的完整链路

### 1.1 端到端路径

```
[项目列表页，右侧「AI 助手」抽屉]
用户输入："帮我新建一个项目"
  │
  ▼
frontend/app.js:6302    POST /api/chat   (无 project_id → 全局聊天)
  │
  ▼
backend/api/chat.py:1401   global_chat_with_ai()
  │ ① 拉 projects 列表当上下文
  │ ② _build_global_system_prompt()  (line 1444)
  │      ↑ system prompt 里写：必填 name + git_remote_url；齐了才输出 [ACTION:CREATE_PROJECT] JSON
  │ ③ llm_client.chat(messages, temperature=0.7, max_tokens=16000)   ← 纯 raw LLM 调用
  │ ④ LLM 回复里带：
  │      [ACTION:CREATE_PROJECT]
  │      {"name": "...", "git_remote_url": "...", ...}
  │      [/ACTION]
  ▼
_parse_global_action()   (line 1498)
  │ 正则 \[ACTION:(\w+)\]\s*(\{.*?\})\s*\[/ACTION\] 抠 JSON → dispatch
  ▼

  │ ⑤ 异步跑 roadmap 生成

返回 {action: {type: "project_created", project_id, name}}
  │
  ▼

  showToast() → 1 秒后 showProjectDetail(project_id) 跳转项目详情页
```

## 二、问题盘点

### 2.1 这条链路是 v0.15.x 残存的 legacy 路径

项目内聊天 (`/api/projects/{id}/chat`) 在 v0.15.2 已经迁到 `ChatAssistantAgent + tool_use`（由 `CHAT_USE_AGENT=true` 控制，P3 起默认开），详见 `docs/20260417_01_ChatAssistant_Agent化迁移方案.md`。

**但全局聊天 (`global_chat_with_ai`) 没迁**。`api/chat.py:282` 那个 `if settings.CHAT_USE_AGENT:` 分支只在**项目内**聊天里生效；全局聊天函数里根本没这个判断——直接 `llm_client.chat()` + 正则抠 `[ACTION:XXX]`。这是唯一还在 prod 跑的 legacy 文本协议路径。

### 2.2 具体问题

| # | 问题 | 位置 | 影响 |
|---|---|---|---|
| 1 | **"确认"只靠 prompt 诱导** | `chat.py:1472-1478` system prompt "等用户确认后再输出 ACTION" | LLM 不听话就直接建项目，无后端拦截。对比：需求创建有 `/confirm-create-requirement` + 前端确认卡片，两步式 |
| 2 | **JSON 解析靠正则** | `chat.py:1502` `\{.*?\}` | LLM 返回格式稍微不标准 → `logger.warning("无法解析")` → 静默吞掉，用户看不到错误，只看到 AI 说"好的"但项目没建 |
| 3 | **Tool use 完全没接上** | 全局聊天入口 | MCP 工具、Skills 注入、`llm_client.chat_with_tools` 都用不上——`CreateProjectAction` 都没有，能力没法被 LLM 当工具调用 |
| 4 | **Skills 不生效** | 同上 | 虽然 v0.16.0 接了 4 个 Skill 包，但因为走的不是 Agent 路径（ContextVar 不被设置），新建项目这条流程拿不到 `git-workflow` Skill（其实正是这条流程需要的）|
| 5 | **没有 Action 抽象** | `_execute_create_project` 是函数 | 不能单测、不能复用、不能被群聊/其它 Agent 引用 |
| 6 | **错误反馈弱** | Git clone 失败、写库失败都只返回 error dict | 没有 Reflexion、没有 Failure Library 记录（v0.16.0 已有设施但这条路径绕过了）|

### 2.3 对比：需求创建已经对了

需求创建在项目内聊天里是这样的（可直接照抄）：

```
LLM tool call: confirm_requirement(title, description, priority)
  ↓
ConfirmRequirementAction（backend/actions/chat/confirm_requirement.py）
  ↓ 返回 action_data = {type: "confirm_requirement", …}（草稿，不落库）
  ↓
前端 chat.py 渲染确认卡片 + 「确认创建」按钮
  ↓
用户点击 → POST /api/projects/{id}/chat/confirm-create-requirement
  ↓
CreateRequirementAction（backend/actions/chat/create_requirement.py） → 落库
```

**两个 Action、两步、用户必须点确认才落库。** 这正是全局聊天新建项目应该抄的形状。

## 三、目标方案

### 3.1 核心原则

1. **对齐项目内聊天架构**：全局聊天也用 `ChatAssistantAgent + tool_use`，让 `CHAT_USE_AGENT` 标志对它同样生效
2. **两步式创建**：LLM 调 `confirm_project`（只生成草稿）→ 用户点确认 → 后端调 `create_project`（落库）
3. **Action 一等公民**：`ConfirmProjectAction` + `CreateProjectAction` 独立到 `backend/actions/chat/`
4. **兜底保留**：保留 `_execute_create_project`（旧路径）+ 降级链，和项目内聊天同款 fallback（`CHAT_USE_AGENT` 路径失败自动回旧路）

### 3.2 目标文件结构

```
backend/actions/chat/
├── confirm_project.py       🆕 生成草稿卡片（不落库）
├── create_project.py        🆕 真正落库（原 _execute_create_project 抽出来）
├── confirm_requirement.py   ✅ 已存在，照着抄
└── create_requirement.py    ✅ 已存在，照着抄

backend/agents/chat_assistant.py   ← 扩展：注册两个 project Action + 区分 global / project 语境
backend/api/chat.py                ← 瘦身：global_chat_with_ai 分支处理 CHAT_USE_AGENT
backend/api/projects.py            ← 复用 /confirm-create-project 端点（或挂到 chat 全局路由）
frontend/app.js                    ← 新增项目确认卡片渲染 + 确认按钮
```

## 四、实施方案（分 3 个 Phase）

### Phase 1：抽 Action（不动路径，只做重构）

**目标**：`_execute_create_project` 从 `chat.py` 抽到独立 Action，跟 `CreateRequirementAction` 对齐形状。不影响现有行为。

#### 1.1 新建 `backend/actions/chat/create_project.py`

照抄 `create_requirement.py` 模板：

```python
class CreateProjectAction(ActionBase):
    @property
    def name(self) -> str: return "create_project"

    @property
    def description(self) -> str:
        return "真正创建项目：clone / init Git 仓库 + INSERT projects + 异步跑 roadmap 生成"

    @property
    def tool_schema(self) -> Dict[str, Any]:
        # ⚠️ 内部 Action，不对 LLM 暴露；LLM 只调 confirm_project
        return {
            "name": self.name,
            "description": "（内部用）实际落库——LLM 不应直接调用，应通过 confirm_project 让用户确认。",
            "input_schema": { … name / description / tech_stack / git_remote_url / local_repo_path … },
        }

    async def run(self, context: Dict[str, Any]) -> ActionResult:
        # ① 字段校验（name + git_remote_url 必填）
        # ② generate_id("PRJ")
        # ③ Git 处理（clone 远程 / init / fetch+reset）
        # ④ INSERT projects
        # ⑤ 异步调度 roadmap 生成
        # ⑥ return ActionResult.ok({"type": "project_created", "project_id": …, "name": …})
```

#### 1.2 新建 `backend/actions/chat/confirm_project.py`

```python
class ConfirmProjectAction(ActionBase):
    @property
    def name(self) -> str: return "confirm_project"

    @property
    def description(self) -> str:
        return "生成项目创建草稿卡片供用户审阅；不落库。用户点确认后才调 create_project。"

    @property
    def tool_schema(self) -> Dict[str, Any]:
        # ✅ 对 LLM 暴露为 tool
        return {
            "name": self.name,
            "description": "草拟一个待创建的项目。用户尚未确认，你必须先列出所有字段让用户审核。",
            "input_schema": {
                "type": "object",
                "properties": {
                    "name":           {"type": "string", "description": "项目名"},
                    "git_remote_url": {"type": "string", "description": "Git 远程仓库 URL"},
                    "description":    {"type": "string", "description": "项目描述，可选"},
                    "tech_stack":     {"type": "string", "description": "技术栈，可选"},
                },
                "required": ["name", "git_remote_url"],
            },
        }

    async def run(self, context) -> ActionResult:
        # 不碰数据库，只把参数打包成 confirm_project 卡片
        return ActionResult.ok({
            "type": "confirm_project",
            "name": context["name"],
            "description": context.get("description", ""),
            "tech_stack": context.get("tech_stack", ""),
            "git_remote_url": context["git_remote_url"],
        })
```

#### 1.3 现有 `_execute_create_project` 改成薄 wrapper

```python
async def _execute_create_project(data: dict) -> Dict:
    from actions.chat.create_project import CreateProjectAction
    result = await CreateProjectAction().run(data)
    return result.data
```

**验收**：legacy `[ACTION:CREATE_PROJECT]` 路径行为**完全不变**。跑一次现有 E2E。

### Phase 2：后端接入 Agent + tool_use

**目标**：让 `CHAT_USE_AGENT=true` 时，全局聊天走 `ChatAssistantAgent`，LLM 通过 tool_use 调 `confirm_project`。

#### 2.1 新增 `/api/chat/confirm-create-project` 端点

挂到 `global_chat_router`（对比需求的 `/api/projects/{id}/chat/confirm-create-requirement` 是项目级）：

```python
class ConfirmCreateProjectRequest(BaseModel):
    name: str
    description: str = ""
    tech_stack: str = ""
    git_remote_url: str
    local_repo_path: str = ""

@global_chat_router.post("/confirm-create-project")
async def confirm_create_project(req: ConfirmCreateProjectRequest):
    """用户点击「确认创建」后真正执行 CreateProjectAction"""
    from actions.chat.create_project import CreateProjectAction
    result = await CreateProjectAction().run(req.model_dump())
    if result.data.get("type") == "error":
        raise HTTPException(400, result.data["message"])
    return result.data
```

#### 2.2 扩展 `ChatAssistantAgent` 暴露两个 project 工具

在 `backend/agents/chat_assistant.py` 的 Action 池里加上 `ConfirmProjectAction`（不加 `CreateProjectAction`，内部用）。Agent 本身已经支持 REACT + tool_use，加 Action 到注册列表即可。

#### 2.3 `global_chat_with_ai` 分支处理

照抄 `chat_with_ai` 的 CHAT_USE_AGENT 分支（`api/chat.py:282-305`）：

```python
@global_chat_router.post("", …)
async def global_chat_with_ai(req: GlobalChatRequest):
    if settings.CHAT_USE_AGENT:
        try:
            return await _global_chat_via_agent(req)
        except HTTPException:
            raise
        except Exception as e:
            logger.warning("全局新路径失败，降级：%s: %s", type(e).__name__, e)
            # 记录 fallback 事件（跟项目内聊天同款埋点）
    return await _global_chat_legacy(req)  # 原函数改名保留
```

`_global_chat_via_agent`：构造 `ChatAssistantAgent` 实例 → 跑 REACT 循环 → 收到 `confirm_project` 工具调用 → 返回 `action = {type: "confirm_project", ...}`。

#### 2.4 Skills / Reflexion / Failure Library 自动生效

因为走了 `ChatAssistantAgent` → `BaseAgent.run_action()` → ContextVar 设置 → ActionNode 注入。`git-workflow` Skill 里的 commit message 规范、PR 流程知识会自动注入到 `confirm_project` 的 LLM 调用。**零改动**。

### Phase 3：前端确认卡片

**目标**：用户看到的不再是"AI 说创建就直接创建"，而是"AI 列出字段 → 用户点确认 → 创建"。

#### 3.1 前端 action 类型分流

`frontend/app.js:6329` 目前只认 `project_created`（已落库）。增加 `confirm_project` 分支：

```javascript
if (action && action.type === 'confirm_project') {
    // 渲染一张确认卡片，带「✅ 确认创建」「❌ 取消」按钮
    renderProjectConfirmCard(action);
}
else if (action && action.type === 'project_created') {
    // 保留现有的落库成功后跳转逻辑
    showToast(…); setTimeout(() => showProjectDetail(action.project_id), 1000);
}
```

#### 3.2 确认按钮调用新端点

```javascript
async function confirmCreateProject(draftData) {
    const resp = await api('/chat/confirm-create-project', {
        method: 'POST',
        body: JSON.stringify(draftData),
    });
    showToast(`项目「${resp.name}」已创建`);
    showProjectDetail(resp.project_id);
}
```

样式照抄现有 `confirm_requirement` 卡片（`app.js` 里已经有）。

## 五、验收标准

| # | 场景 | 期望行为 |
|---|---|---|
| 1 | `CHAT_USE_AGENT=true`，用户说"建一个 FastAPI 脚手架项目 git@xxx" | AI 调 `confirm_project` tool → 前端弹卡片 → 用户点确认 → 创建成功 → 跳转详情页 |
| 2 | `CHAT_USE_AGENT=true`，用户只说"建个项目"（信息不全） | AI 不调 tool，只反问"请提供 Git URL 和项目名"；不出现确认卡片 |
| 3 | `CHAT_USE_AGENT=true`，Agent 路径抛异常 | 日志出现 `"全局新路径失败，降级"`，自动回到 legacy `[ACTION:CREATE_PROJECT]` 路径；用户无感知 |
| 4 | `CHAT_USE_AGENT=false` | 保持 legacy 行为不变（回归测试） |
| 5 | Skills 注入 | 查 ActionNode 调用日志，`git-workflow` Skill 内容出现在 `confirm_project` 的 system prompt |
| 6 | Git URL 无效（clone 失败） | `CreateProjectAction` 返回 error，前端弹 toast，**不跳转**，不破坏 chat 历史 |

## 六、风险 & 缓解

| 风险 | 缓解 |
|---|---|
| LLM 绕过确认直接调 `create_project` | 不把 `CreateProjectAction` 注册到 Agent 工具池（仅 `/confirm-create-project` 端点可达，HTTP 层强制） |
| Agent 路径退化（tool 不调用） | 和项目内聊天同款 fallback + 埋点，观察失败率 |
| 老用户已习惯"说完就建"的速度 | 两步式耗时多 1 个用户点击，但换来不可逆操作的人工确认；对照需求创建已验证这个 trade-off 用户可接受 |
| roadmap 生成失败导致项目半成品 | 保持异步 fire-and-forget，失败记 log 不回滚（当前行为） |

## 九、Self-Consistency / 备选方案

- **不做**：继续用 legacy 路径。代价：正则脆弱 + 无确认 + Skill/MCP 无法复用 + 整条 chat 栈架构不一致。
- **只做 Phase 1**：只抽 Action 不接 Agent。代价：前端体验不变，但后续能力复用仍断层。
- **做到 Phase 2 就停**：接 Agent 但不加确认卡片。代价：LLM 如果 hallucinate 仍可能直接建错项目。
- **全做（推荐）**：~2.5 天，彻底收口，把"AI 助手是真正的 Agent"这个叙事做实。
