## 一、问题背景

### 1.1 现状

当前系统默认**所有项目都是 web + Python/React 栈**：
- 4 个 skill 全是 web 方向（react-dev / fastapi-dev / playwright-e2e / git-workflow）
- SOP `default_sop.yaml` 硬编码 7 阶段且 smoke_test 假设 HTTP 服务
- 6 个 Role Agent（Product/Architect/Dev/Test/Review/Deploy）角色通用但其 prompt 暗含 web 语境
- 项目表 `tech_stack` 只是自由文本字符串，无结构化语义

### 1.2 需求

规划要支持的项目类型：**网页型、客户端型、游戏型、网页游戏、客户端游戏、UE 游戏、Godot 游戏、Unity 游戏、微信小程序 ...**

这些类型**存在多继承关系**：
- 网页游戏 ≡ 网页 ∩ 游戏
- UE 游戏 ≡ 客户端 ∩ 游戏 ∩ UE 引擎
- 微信小程序 ≡ 网页 ∩ 微信特定环境

用**树**（单继承）表达不了；用**扁平枚举**会组合爆炸。

### 1.3 复合问题

还有两条 UX 路径要同时考虑：

| 链路 | 关键问题 |
|---|---|
| 全局 AI 助手**新建项目** | 用户不想填表；LLM 要从对话里**逐步推断**出项目类型，不清楚时要**反问关键维度** |
| 用户**导入已有项目** | 要能**自动探测**仓库类型（扫文件特征），给出候选 + 证据，让用户**确认**而不是从零填 |

## 二、分类体系架构：为什么选 Trait + Preset 混合

### 2.1 四种候选架构对比

| 方案 | 机制 | 表达力 | 工程复杂度 | 适合 |
|---|---|---|---|---|
| **P1 Flat enum** | `type: 'ue-game' \| 'godot-game' \| ...` | 🔴 低（组合爆炸） | 🟢 最简单 | 类型 ≤ 5 且互斥 |
| **P2 Tree 单继承** | `type.parent = ?` 链式查祖先配置 | 🟡 中 | 🟡 中 | 纯父子层级、无横切 |
| **P3 Tags 多标签** | `traits: [web, game, engine:ue5]`；skill 按 `all_of/any_of` 匹配 | 🟢 高 | 🟡 中 | **多维交叉** |
| **P4 Yaml extends 多继承** | `preset.yaml: extends: [web, game]` 拓扑合并 | 🟢 高 | 🔴 高（merge 语义难） | 配置重度 |

### 2.2 推荐方案：P3 为主 + P4 做 UX 糖

**核心**：
- **Skill 匹配、SOP 匹配、MCP 路由**全部走 P3 的 trait 匹配（精准、可组合）
- **用户交互层**用 P4 的 preset 包装成"项目类型"单选（减决策负担）

### 2.3 为什么不纯 P4（Yaml extends）

- stages 列表合并怎么排序？冲突怎么解？语义难定义
- 实际运行配置是多份 yaml 合出来的，debug 困难
- 跟 trait 有职责重复 —— preset 其实就是"命名的 trait 集合"

**结论**：trait 做引擎，preset 做门面。

## 三、数据模型

### 3.1 Traits 分类法（Canonical Taxonomy）

固定枚举表放在 `backend/skills/rules/trait_taxonomy.yaml`，LLM/detector 只能从这里选，不能自创：

```yaml
# 平台维度
platform:
  - web              # 浏览器
  - wechat           # 微信小程序/生态
  - desktop          # 桌面客户端
  - mobile           # 手机 App（iOS/Android）
  - server           # 服务端（无 UI）
  - cli              # 命令行工具

# 类别维度
category:
  - app              # 通用应用（增删改查、工具类）
  - game             # 游戏
  - service          # 后端服务（API、微服务）
  - library          # 第三方库
  - doc              # 纯文档站点

# 引擎维度（category=game 必填）
engine:
  - ue5
  - ue4
  - godot4
  - godot3
  - unity
  - cocos
  - none             # 手写 canvas / WebGL / 自研

# 语言维度（可选，多数能从 engine 推出）
lang:
  - python
  - javascript
  - typescript
  - cpp
  - csharp
  - gdscript
  - rust
  - go

# 框架维度（可选）
framework:
  - react / vue / svelte / fastapi / django / flask / express ...

# 特殊约束维度（可选，未来扩展）
compliance:
  - enterprise       # 企业合规要求
  - i18n             # 国际化必要
  - offline          # 离线运行
  - wechat-review    # 微信审核要求
```

**约束规则**：
- 必填：`platform:*` + `category:*`
- 若 `category:game`，必填 `engine:*`（`none` 是有效值）
- 其余可选，但推荐至少有 `lang:*`

### 3.2 Preset（命名的 trait 组合）

```yaml
# backend/skills/rules/presets.yaml
presets:
  webapp:
    label: "网页应用"
    traits: [platform:web, category:app]
    default_sop: web.yaml

  web-game:
    label: "网页游戏（HTML5）"
    traits: [platform:web, category:game, engine:none]
    default_sop: web-game.yaml

  wechat-miniapp:
    label: "微信小程序"
    traits: [platform:wechat, platform:web, category:app]
    default_sop: wechat-miniapp.yaml

  ue5-game:
    label: "UE5 客户端游戏"
    traits: [platform:desktop, category:game, engine:ue5, lang:cpp]
    default_sop: ue5-game.yaml

  godot-game:
    label: "Godot 客户端游戏"
    traits: [platform:desktop, category:game, engine:godot4, lang:gdscript]
    default_sop: godot-game.yaml

  unity-game:
    label: "Unity 客户端游戏"
    traits: [platform:desktop, category:game, engine:unity, lang:csharp]
    default_sop: unity-game.yaml

  api-service:
    label: "后端 API 服务"
    traits: [platform:server, category:service, lang:python, framework:fastapi]
    default_sop: api-service.yaml

  custom:
    label: "自定义（手动填 traits）"
    traits: []
    default_sop: generic.yaml
```

Preset 是**出厂参数模板**，项目建出来后 `projects.traits` 是**项目自己的状态**（可编辑），两者解耦。

### 3.3 数据库迁移

```sql
ALTER TABLE projects ADD COLUMN traits TEXT DEFAULT '[]';
  -- JSON array: ["platform:web", "category:game", "engine:none"]

ALTER TABLE projects ADD COLUMN traits_confidence TEXT DEFAULT '{}';
  -- JSON object:
  -- {"engine:ue5": {"score": 0.98, "source": "file_detected",
  --                 "evidence": "found MyGame.uproject"}}

ALTER TABLE projects ADD COLUMN sop_template TEXT DEFAULT 'web';
  -- preset 名字（或直接 SOP yaml 名），决定加载哪份 SOP

ALTER TABLE projects ADD COLUMN preset_id TEXT DEFAULT NULL;
  -- 用户建项目时选的 preset；建好后可以改 traits 脱离 preset
```

`traits_confidence.*.source` 枚举：
- `user_declared` — 用户对话里明说（最高可信）
- `file_detected` — 自动扫描识别
- `ai_inferred` — LLM 从上下文推测
- `preset` — 选 preset 带入
- `manual_edit` — 建好后手改

## 四、UX 链路 1：新建项目（对话式渐进识别）

### 4.1 核心原则

**不让用户填表 traits**，但 `confirm_project` 必须带 traits。靠 LLM 从对话逐步提炼，信息不够就**反问关键维度**。

### 4.2 `confirm_project` tool_schema 升级

```python
{
    "name": "confirm_project",
    "input_schema": {
        "properties": {
            "name": ...,
            "git_remote_url": ...,
            "description": ...,
            "preset_id": {
                "type": "string",
                "description": "可选 preset，如 ue5-game / webapp。若填了自动带入 traits"
            },
            "traits": {
                "type": "array",
                "items": {"type": "string"},
                "description": (
                    "项目特征标签。必须包含 platform:* 和 category:*，"
                    "category=game 时必须有 engine:*。"
                    "从固定分类法里选：见 trait_taxonomy.yaml"
                )
            },
            "traits_confidence": {
                "type": "object",
                "description": "每个 trait 的置信度 0-1 + 证据"
            }
        },
        "required": ["name", "git_remote_url", "traits"],
    }
}
```

### 4.3 system prompt 新增规则

```
## 识别项目类型的必填维度
调用 confirm_project 前，必须从对话推断出：
1. platform:* (web/wechat/desktop/mobile/server/cli)
2. category:* (app/game/service/library/doc)
3. 如果 category=game，还要 engine:* (ue5/godot4/unity/none/...)

## 判断准则
- "网页计算器" → platform:web, category:app 齐了 ✅
- "贪吃蛇" → category:game 清楚，platform/engine 未定 ❌ 必须反问
- "UE5 射击游戏" → 全齐 ✅
- "做个网站" → category 不明（app 还是 service？）❌ 反问

## 反问的正确打法
给 3-4 个预设选项（不要让用户自由描述）：

"这个贪吃蛇你打算做成哪种？
  1️⃣ 网页版（HTML5 canvas，浏览器里玩）
  2️⃣ 微信小程序
  3️⃣ Godot/Unity/UE 客户端游戏
  4️⃣ 其他（请描述）"

## traits 齐了才 confirm_project
维度齐 + 用户同意 → 调工具，traits_confidence.source 填 'user_declared'
任一维度不明 → 反问，不调工具
```

### 4.4 对话示例

```
User: 帮我建个贪吃蛇项目
AI  : 好，做贪吃蛇先选个平台：
      1️⃣ 网页版（HTML5 canvas）
      2️⃣ 微信小程序
      3️⃣ Godot 桌面游戏
      4️⃣ 其他
User: 网页版
AI  : 明白。Git 仓库地址给一下？
User: https://github.com/me/snake.git
AI  : [📦 确认创建项目]
      项目：snake
      类型：网页游戏（preset: web-game）
      特征：
        • platform:web ✅ 用户确认
        • category:game ✅ 用户确认
        • engine:none ✅ preset 默认
        • lang:javascript 🟡 推断
      Git: https://github.com/me/snake.git
      [✅ 确认创建] [✗ 取消]
```

## 五、UX 链路 2：导入项目（自动探测 + 证据链）

### 5.1 新 Action：`ProjectTypeDetectorAction`

扫 repo 目录的特征文件，产出候选 traits + 置信度 + 证据。

```python
class ProjectTypeDetectorAction(ActionBase):
    """扫描 repo，从文件特征推断 traits 候选"""

    DETECTION_RULES = [
        # (匹配条件, 产出 traits, 置信度, 证据模板)
        ("*.uproject", ["engine:ue5", "category:game", "platform:desktop"], 0.98, "找到 {file}"),
        ("project.godot", ["engine:godot4", "category:game", "platform:desktop"], 0.98, "找到 {file}"),
        ("ProjectSettings/ProjectVersion.txt", ["engine:unity", "category:game", "platform:desktop"], 0.95, "Unity 项目"),
        ("*.uasset", ["engine:ue5"], 0.80, "UE asset 文件"),
        ("miniprogram/app.json + app.wxss", ["platform:wechat", "platform:web"], 0.95, "微信小程序配置"),
        ("package.json[react in deps]", ["framework:react", "platform:web"], 0.90, "依赖 react"),
        ("package.json[vue in deps]", ["framework:vue", "platform:web"], 0.90, "依赖 vue"),
        ("requirements.txt[fastapi]", ["framework:fastapi", "category:service", "lang:python"], 0.90, "依赖 fastapi"),
        ("Cargo.toml", ["lang:rust"], 0.95, "Rust 项目"),
        ("go.mod", ["lang:go"], 0.95, "Go 项目"),
        ("*.html + <canvas>", ["category:game", "platform:web"], 0.60, "可能的 HTML5 游戏"),
        # ... ~20-30 条规则
    ]

    async def run(self, context):
        repo_path = context["repo_path"]
        candidates = self._scan(repo_path)   # 合并规则输出，去重 trait
        suggested_preset = self._match_preset(candidates)
        return ActionResult(success=True, data={
            "type": "project_type_detected",
            "candidates": [
                {"trait": "engine:ue5", "confidence": 0.98, "evidence": "..."},
                ...
            ],
            "suggested_preset": "ue5-game" or None,
        })
```

### 5.2 导入流程

```
User: 帮我把 https://github.com/me/zombie-survivor.git 导进来
AI  : 好，先探测一下仓库类型。
      [内部] git clone --depth 1 → 临时目录
             → ProjectTypeDetectorAction
             → candidates + suggested_preset
      
AI  : 🔍 探测结果：
      
      推荐类型：**UE5 客户端游戏**（preset: ue5-game）
      
      依据：
        ✅ engine:ue5       (0.98) 找到 ZombieSurvivor.uproject
        ✅ category:game    (0.98) uproject 推断
        ✅ platform:desktop (0.98) uproject 推断
        🟡 lang:cpp        (0.70) Source/ 目录下有 .cpp
        🟡 framework:none  (0.70) 未发现其他框架
      
      [✅ 确认导入] [✏️ 改一下 traits] [✗ 取消]
```

### 5.3 低置信兜底

如果**所有 trait 置信度 < 0.5**（空 repo / detector 没命中 / 新技术栈）→ 落到"新建对话式"那条路径反问用户。

## 七、与 Skills / SOP / MCP 的联动

### 7.1 Skill 按 traits 过滤（核心）

```yaml
# backend/skills/packs/ue_cpp_dev/skill.yaml
name: "UE5 C++ 开发"
description: "UE5 Gameplay Framework, UPROPERTY, UFUNCTION, Blueprint bridge..."
prompt_file: prompt.md
inject_to: [DevAgent, ReviewAgent]
priority: high
traits_match:
  all_of:
    - "engine:ue5"
---

# backend/skills/packs/game_design/skill.yaml
name: "游戏设计"
inject_to: [ProductAgent, ArchitectAgent]
traits_match:
  any_of:
    - "category:game"
---

# backend/skills/packs/fastapi_dev/skill.yaml  （现有 skill 补上 matcher）
inject_to: [DevAgent]
traits_match:
  all_of:
    - "framework:fastapi"
  # 或者更宽：
  # any_of: [framework:fastapi, category:service]
```

SkillLoader 核心变动：

```python
def build_prompt_for_agent(
    self,
    agent_type: str,
    project_traits: List[str] = None,   # ← 新增
) -> str:
    applicable = []
    for skill_id, cfg in self.skills.items():
        if not cfg.get("enabled"):
            continue
        if agent_type not in cfg.get("inject_to", []):
            continue
        if not self._traits_match(cfg.get("traits_match", {}), project_traits):
            continue
        applicable.append(skill_id)
    # ... 拼 prompt
```

`_traits_match()` 实现三元组：`all_of` (AND) / `any_of` (OR) / `none_of` (NOT)，都空则永远匹配（兼容老 skill）。

### 7.2 SOP 按 preset 加载

```
backend/sop/
├── generic.yaml        ← 兜底
├── web.yaml
├── web-game.yaml
├── wechat-miniapp.yaml
├── ue5-game.yaml
├── godot-game.yaml
├── unity-game.yaml
└── api-service.yaml
```

`sop/loader.py` 按 `project.sop_template` 取对应 yaml（不再 hardcode `default_sop.yaml`）。

### 7.3 MCP 按 traits 启用

`mcp_servers.json` 条目加 `enabled_for_traits` 过滤：

```json
{
  "ue-python-bridge": {
    "command": "...",
    "enabled_for_traits": {
      "all_of": ["engine:ue5"]
    }
  },
  "godot-mcp": {
    "enabled_for_traits": {
      "all_of": ["engine:godot4"]
    }
  },
  "filesystem": {
    // 无 enabled_for_traits = 全项目可用
  }
}
```

ChatAssistant 在加载 MCP 工具清单时按当前 `project.traits` 过滤。

### 7.4 `paths:` 文件级过滤（引用之前分析）

跟 AgentHub 的做法一致，skill 内部还可以进一步加 `paths:` 按文件 glob 过滤（项目层 traits 通过 + 文件层 paths 通过，才注入）：

```yaml
# backend/skills/packs/react_dev/skill.yaml
name: "React/HTML 开发"
inject_to: [DevAgent]
traits_match:
  any_of: [framework:react]
paths:           # ← 文件层进一步细分
  - "**/*.html"
  - "**/*.jsx"
  - "**/*.tsx"
```

## 八、分阶段实施计划

| Phase | 内容 | 依赖 | 估时 |
|---|---|---|---|
| **Phase 1** | DB 迁移（traits/preset_id/sop_template/traits_confidence）+ `trait_taxonomy.yaml` + `presets.yaml` | — | 1 天 |
| **Phase 2** | `confirm_project` schema 加 traits/preset_id 必填 + ChatAssistant system prompt 加"识别维度 + 反问规则" | P1 | 1 天 |
| **Phase 3** | 前端 confirm_project 确认卡片显示 traits + 来源徽章 | P2 | 半天 |
| **Phase 4** | `ProjectTypeDetectorAction` + 导入链路（clone → detect → 确认卡片展示 candidates 和 evidence） | P1 | 1.5 天 |
| **Phase 5** | 「项目特征」子 Tab（查看 + 编辑 traits，查看生效 skill/SOP/MCP） | P1+P2+P4 | 半天 |
| **Phase 6** | `SkillLoader.build_prompt_for_agent(project_traits=...)` 按 traits 过滤；现有 4 个 skill 补 `traits_match` frontmatter | P1 | 1 天 |
| **Phase 7** | SOP 按 `sop_template` 加载；拆出 `web.yaml`（现 default） + 新建 2-3 份游戏 SOP 草稿 | P1 | 1-2 天 |
| **Phase 8** | MCP 按 `enabled_for_traits` 启用；mcp_servers.json 升级 schema | P1 | 半天 |
|  | **合计** | | **~7-8 天** |

Phase 1-3 是**对话式新建**的最小可用组，做完能立刻跑通"用户说建个 UE5 游戏 → LLM 对话识别 → 前端卡片显示 traits → 建项目"。

Phase 4-5 补全**导入**和**编辑**两条链路。

Phase 6-8 是让 traits 真正**发挥作用**（skill/SOP/MCP 按 traits 路由），做完一个游戏项目跑通的 value 才能体现。

**推荐节奏**：Phase 1+2+3+6 一批做（4 天）跑通"对话式新建 + skill 按 traits 过滤"的垂直切片，能立刻用；Phase 4+5+7+8 作为第二批次补全（3-4 天）。

## 九、关键决策点（未决 / 待讨论）

### 9.1 traits 的 "canonical" 谁来守护

建议定在 `backend/skills/rules/trait_taxonomy.yaml`（单一权威）。LLM/detector/UI 都只能从这里选，不能自创 `platform:ios-app-store` 这种。

**谁有权扩？** 开发加一条 trait → 改 yaml → commit review → 部署。不让用户在 UI 里自创自由 trait（否则两个项目的 trait 语义会漂移，后续按 trait 匹配 skill 时会乱）。

### 9.2 Detector 跑在哪、什么时候跑

**推荐**：跟现在 `_execute_create_project` 的 clone 流程串起来。用户给 URL → clone 到项目仓库目录 → 立刻跑 detector → 拿到 candidates → 传给前端确认卡片。用户等待约 10-30s（git clone 时间），但得到立即可见的 traits + 证据，不用额外交互。

### 9.3 现有 6 个项目要不要回填 traits

**推荐**：跑一次 batch script → 对每个项目调 `ProjectTypeDetectorAction` → 填 `source: file_detected` → 用户进"项目特征"页点一次"确认"升级为 `user_declared`。

跟之前"读老 bug"一个套路，不 block 当前使用。

### 9.4 preset 改动后，旧项目要不要追溯更新

**推荐不追溯**：preset 只在建项目时作为"出厂参数"使用。用户建好项目后改 preset 定义不影响已有项目（避免已跑一半的工单忽然换 SOP 的惊吓）。

若想手工同步：用户在项目特征页点"更换 preset"按钮，明确触发。

### 9.5 `paths:` 文件级 vs `traits_match` 项目级的层叠

两个是**AND 关系**（都通过才注入）。traits_match 先过滤掉"这个 skill 跟项目类型没关系"，`paths:` 再过滤"跟当前编辑文件没关系"。

对聊天场景（没有"当前编辑文件"的概念）：`paths:` 忽略，只看 traits_match。
对工单/Action 场景（处理具体文件）：两个都生效。

## 十、风险和替代路径

### 10.1 风险

1. **Trait 分类法设计偏差**：一开始选的维度不对（比如没想到要加 `compliance:*`），后续补维度相当于给所有项目回填 —— **缓解**：先定 4-5 个必要维度（platform/category/engine/lang/framework），其他维度 lazy add
2. **LLM 反问环节"太话痨"**：用户只想快速建项目，被连问 3 轮会烦 —— **缓解**：prompt 里强调"用预设选项反问 + 一次问清所有缺的维度"；建 preset 是为了减决策数量
3. **Detector 覆盖率不够**：遇到新技术栈（Rust game、SvelteKit、Bun）识别不出 → 落到兜底反问路径就行
4. **SOP 配置爆炸**：每个 preset 一份 SOP yaml，后续类型多了 yaml 难维护 —— **缓解**：把 SOP 共用部分抽到 `_base.yaml`，新 SOP 用 `include: _base` 引用（不做复杂 extends 多继承）

### 10.2 替代路径：如果做 Phase 1-3 发现 UX 不理想

回退到**简单枚举**（P1 flat enum）加一个 `project_type` 字段，跑几个月积累类型后再切换到 trait。

成本：再做一次 DB migration 把 `project_type` 转成 traits 集合（reversible）。

## 十二、TL;DR

**问题**：支持多种项目类型（网页/客户端/游戏 × UE/Godot/Unity × 微信...），继承关系非树而是 DAG。

**方案**：**Trait (P3) + Preset (P4)** 混合
- Trait = 多维标签（`platform:* / category:* / engine:*` ...）
- Preset = 命名的常用 trait 组合，UI 上作为"项目类型"单选
- Skill / SOP / MCP 全部按 trait 匹配

**UX 三链路**：
1. 新建：LLM 对话识别 + 反问关键维度 + `confirm_project` 带 traits 必填
2. 导入：`ProjectTypeDetectorAction` 扫 repo 文件 → candidates + 证据 → 确认卡片
3. 编辑：项目特征子 Tab，实时显示生效 skill/SOP/MCP

**落地**：7-8 天分 8 个 Phase。Phase 1-3+6 (~4 天) 能跑通对话式新建 + skill 按 trait 过滤的垂直切片。

**下一步决策**：
1. 确认走这个方案，定 Phase 1 动工（DB migration + trait_taxonomy.yaml）
2. 或先锁死一个具体项目类型（Godot 最轻 or UE 生态最全）试点，验证 traits 机制再铺开
3. 或把这个方案先挂着，等现有 web 项目跑稳当再回头

# 附录 A：对既有 Agent / Skill 方案的调整

> 补充于 2026-04-23。
> 背景：本文前 12 章定了 traits/preset 架构后，回头看 `docs/20260422_04_四份Skill体系对比分析.md` 里已经排的两件事（5.1 加 `paths:` + 5.2 抽 `global.md`），会发现它们跟 traits 是**同构的过滤问题**，不能各做各的。Agent / Action / SOP 也有连锁反应。本附录把这些调整并入正文的分阶段计划。

## A.1 Skill 层面：三层过滤统一设计

引入 traits 之前，`docs/20260422_04` 计划在 SkillLoader 上分别加两个独立的匹配字段（`paths:` 文件级、`global.md` alwaysApply 层）。Traits 之后，**三个维度必须合并成一套统一的过滤引擎**：

```
┌────────────────────────────────────────────┐
│ Layer 1: inject_to: [Agent]   ← Agent 级    │  哪些 Agent 需要这块知识
│ Layer 2: traits_match: {...}  ← 项目级      │  哪种项目类型适用
│ Layer 3: paths: [...]         ← 文件级(可选)│  处理什么文件时注入
└────────────────────────────────────────────┘
              三层 AND，最终决定是否注入
```

`rules/` 目录里的规则（比如 `global.md`）则是**没有任何 matcher、永远注入**的最外层兜底。

### A.1.1 现有 4 个 skill 重构归位

在 traits 视角下，不是"加个字段"那么简单，4 个 skill 全部要重构：

| skill | 现状 | 正确归位 |
|---|---|---|
| `git-workflow` | `inject_to: [DevAgent, DeployAgent]` 一锅端 | 🔄 **提升到 `rules/git.md`（alwaysApply）** —— 不是 skill，是全项目通用规范 |
| `fastapi-dev` | `inject_to: [DevAgent]` | 🔄 加 `traits_match: any_of: [framework:fastapi]` —— 非 FastAPI 项目别注入 |
| `react-dev` | `inject_to: [DevAgent]` | 🔄 加 `traits_match: any_of: [framework:react]` + `paths: ["*.html","*.jsx","*.tsx"]` |
| `playwright-e2e` | `inject_to: [TestAgent, ProductAgent]` | 🔄 加 `traits_match: any_of: [platform:web]` —— 客户端游戏要用别的 e2e 工具 |

**实际意义**：之前 `docs/20260422_04` 计划 3 个分散的小改（5.1 paths / 5.2 global / 5.3 rules 目录 = 共 ~5 天），合并到 traits 工程后变成**一次 SkillLoader 大改 + 4 个 skill 重构，合计 ~3 天**，省出 2 天。

## A.2 Agent 层面：3 个必须定的问题

### A.2.1 要不要新增 ArtistAgent

游戏项目需要美术资产生成（UI / 音效 / 模型），6 个现有 Role 里没对应角色。

| 方案 | 说明 | 成本 | 问题 |
|---|---|---|---|
| A. 新增 `ArtistAgent` | 第 7 号 Role | 🟡 中 | 要配 `available_for_traits = [category:game]` 让它只在游戏项目激活 |
| B. DevAgent 装 `asset-gen` skill | 不加 Agent，靠 skill | 🟢 低 | DevAgent prompt 心智混乱（写代码 vs 出图是两种活） |
| C. 完全跳过 | 手工放 assets/，AI 只管代码 | 🟢 零 | 游戏项目一半工作 AI 碰不到 |

**决策**：**方案 A，但接口先建、Agent 类推迟**。即：本次工程加 `BaseAgent.available_for_traits` 字段 + orchestrator 按 traits 过滤的逻辑，但不急着真写 `ArtistAgent` 类，等真做第一个游戏项目时再加。

### A.2.2 Agent prompt 要不要 trait-aware

DevAgent 写 Python 和写 GDScript 心智完全不同，两种实现路径：

- **选项 X（推荐）**：Agent prompt 保持通用，所有技术栈细节靠 skill 注入
- **选项 Y（不推荐）**：Agent prompt 里加 `{{#if traits.engine:ue5}}...{{/if}}` 条件块

**决策**：选项 X。理由：
- Y 把 Agent prompt 变成模板引擎，调试噩梦
- X 下 Agent 类 = 纯角色模板（开发者/产品经理/测试），traits 驱动的知识全在 skill 层
- 边界干净 → Agent **类定义本次不用动**

### A.2.3 Agent 可用性按 traits 过滤

即使不加 ArtistAgent，过滤机制本次也要铺好：

```python
# BaseAgent 加属性（None = 全项目可用，兼容现有 6 个 Role）
class BaseAgent:
    available_for_traits: Optional[dict] = None

# orchestrator 从全局池按项目 traits 过滤
def get_agents_for_project(project_id):
    traits = get_project_traits(project_id)
    return [a for a in self.agents.values()
            if _traits_match(a.available_for_traits, traits)]
```

Future-proof：ArtistAgent / SecurityAgent / I18nAgent 等专业角色后续加入时只声明 `available_for_traits` 即可。

## A.3 Action / SOP 层面

### A.3.1 SOP 多份 vs 单份 + gating

正文 §7.2 已采纳「按 preset 拆多份」。此处补充**消冗余策略**：

| 候选 | 成本 | 问题 |
|---|---|---|
| 复杂 extends 多继承（P4） | 高 | merge 语义难定义，debug 噩梦（见本文 §2.3） |
| 简单 `include: _base.yaml`（推荐） | 低 | 只合并"共用阶段"那一块，不试图做全量合并 |

示例：

```yaml
# backend/sop/_base_common.yaml
common_stages:
  design:
    agent: ProductAgent
    action: DesignAction
  architecture:
    agent: ArchitectAgent
    action: DesignArchitectureAction

# backend/sop/ue5-game.yaml
include: _base_common
stages:
  - $common.design
  - $common.architecture
  - name: engine_compile
    agent: DevAgent
    action: UECompileCheckAction
  - name: play_test
    agent: TestAgent
    action: PlaytestAction
  - ...
```

`include` 只做字符串级模板插值，不做冲突合并，可读性和可调试性都保住。

### A.3.2 Action 池也要按 traits 过滤

ACTION_REGISTRY 当前全局注册。加 trait 过滤机制：

```python
class ActionBase:
    available_for_traits: Optional[dict] = None  # None = 全项目可用

class UECompileCheckAction(ActionBase):
    available_for_traits = {"all_of": ["engine:ue5"]}

class PlaywrightE2EAction(ActionBase):
    available_for_traits = {"any_of": ["platform:web"]}
```

**校验时机**：SOP loader 加载 yaml 时如果引用了跟项目 traits 不匹配的 Action → **启动期报错** 而不是运行期才炸。
