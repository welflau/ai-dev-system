# MCP 客户端实现方案（P3）

> 日期: 2026-04-21 | 对标: MagicAI `src/agents/game_agent/utils/mcp_manager.py` | SDK: `mcp>=1.0.0,<2.0.0`

---

## 一、背景

### 1.1 MCP 是什么

Model Context Protocol — Anthropic 推出的开放协议，标准化 LLM 工具调用：
- 工具服务器（MCP server）作为独立子进程运行（stdio）或 HTTP SSE 端点
- LLM 客户端（Claude Desktop / Claude Code / 我们的 ChatAssistant）连接 server，拉取工具元数据（JSON Schema），调用工具
- 生态：官方 `@modelcontextprotocol/servers` 仓库提供 filesystem / git / fetch / sqlite / postgres / slack / github 等几十个现成 server

### 1.2 本项目现状

- ChatAssistant 有 16 个硬编码 Action（`backend/actions/chat/*.py`），每个手写 `tool_schema`
- 调用走 Anthropic `tool_use` 原生协议
- 非 Chat Agent（Dev/Test/Product/…）完全不用 tool_use —— 走 ActionNode JSON 解析

缺点：
- 每加一个工具（git / 文件 / 网页抓取）都要在本仓库写一份实现
- 无法受益于 MCP 生态的成熟 server
- 别人的 MCP 工具（比如 Notion / Linear / Slack）想接进来只能手工重写

### 1.3 P3 M1 目标

**ChatAssistant 支持外部 MCP server**。现有 16 个内部 Action 保持不变；MCP 工具作为**额外能力**挂进 ChatAssistant 的 tool_use 列表。

用户只需：改 `mcp_servers.json` 把 `enabled` 置 `true` + 装好对应命令行工具 → 重启服务 → MCP 工具立刻出现在 ChatAssistant 的可用工具里。

**明确不做**：
- 内部 Action 迁 MCP（保留现状，工作量大收益低）
- 非 Chat Agent 支持 MCP（Agent 用的是 ActionNode 模式，不是 tool_use）
- 暴露本系统为 MCP server（反向，另外再议）

## 二、核心设计

### 2.1 数据流

```
启动（FastAPI lifespan）
  └─ mcp_client.start_enabled_servers()
       └─ 遍历 mcp_servers.json 里 enabled=true 条目
       └─ 每个 server 起一个 asyncio.Task：
            stdio_client(...) → ClientSession → initialize() → list_tools()
            → 缓存 tool 元数据（Anthropic schema 格式 + mcp__<server>__ 前缀）
       └─ 启动失败 / 超时 → 记 warning，跳过该 server，不阻塞其他

ChatAssistant.chat()
  └─ _exposed_tool_schemas() 返回 内部 16 + MCP N 个
  └─ 请求 Anthropic API，LLM 返回 tool_use block
  └─ _ChatToolExecutor.execute(tool_name, input):
       if tool_name.startswith("mcp__"):
         mcp_client.call_tool(tool_name, input)  # 路由到 server 的调用队列
       else:
         原有内部 Action 路径（不动）
  └─ 结果字符串回给 LLM 继续下一轮

关停（FastAPI lifespan）
  └─ mcp_client.stop_all_servers() → 取消各后台 task，subprocess 自动清理
```

### 2.2 配置文件 `backend/mcp_servers.json`

```json
{
  "filesystem": {
    "type": "stdio",
    "command": "npx",
    "args": ["-y", "@modelcontextprotocol/server-filesystem", "."],
    "env": {},
    "enabled": false,
    "description": "官方文件系统 MCP"
  },
  "fetch": {
    "type": "stdio",
    "command": "uvx",
    "args": ["mcp-server-fetch"],
    "env": {},
    "enabled": false,
    "description": "官方网页抓取 MCP"
  },
  "git": {
    "type": "stdio",
    "command": "uvx",
    "args": ["mcp-server-git"],
    "env": {},
    "enabled": false,
    "description": "官方 Git MCP"
  }
}
```

**默认全部 `enabled: false`**（opt-in）。与 MagicAI 配置格式兼容。

### 2.3 工具命名空间

MCP 工具名在传给 Anthropic API 前加前缀：

```
<server_name>.<tool_name>  →  mcp__<server_name>__<tool_name>
```

| 示例 | 改造后 |
|---|---|
| filesystem.read_file | `mcp__filesystem__read_file` |
| git.status | `mcp__git__status` |
| fetch.fetch | `mcp__fetch__fetch` |

这与 Claude Code / MagicAI 使用的规则一致。`_ChatToolExecutor.execute()` 通过前缀判断路由方向。

### 2.4 核心组件 `backend/mcp_client.py`

```python
class MCPClient:
    async def start_enabled_servers() -> None
    async def stop_all_servers() -> None
    def list_all_tool_schemas() -> List[Dict]  # 已加 mcp__ 前缀
    async def call_tool(tool_name, tool_input) -> str
    def is_mcp_tool(tool_name) -> bool
    def get_status() -> Dict
```

**内部结构**：
- 每个 server 一个 `_ServerConnection` 对象：
  - 后台 `asyncio.Task` 持有 `ClientSession` 整个生命周期
  - 对外通过 `asyncio.Queue` 接受调用请求（串行消费，避免抢占 session）
  - `status: pending | running | error | stopped`
- 启动超时 20s、调用超时 30s（硬编码，后续可配）
- 响应 > 10000 字符自动截断 + 提示

### 2.5 Schema 转换

MCP `Tool` 对象（来自 `mcp.types.Tool`）包含 `name` / `description` / `inputSchema`（JSON Schema）。Anthropic `tool_schema` 字段完全一致（仅 `input_schema` 键名下划线），所以转换只改两件事：

```python
{
    "name": f"mcp__{server_name}__{tool.name}",           # 加前缀
    "description": (tool.description or "")[:1000],        # 描述截断
    "input_schema": tool.inputSchema or {"type": "object", "properties": {}},
}
```

### 2.6 ChatAssistant 集成

**集成点 1**：`_exposed_tool_schemas()` 追加 MCP 工具

```python
# backend/agents/chat_assistant.py
def _exposed_tool_schemas(self):
    schemas = [internal action schemas ...]
    try:
        from mcp_client import mcp_client
        schemas.extend(mcp_client.list_all_tool_schemas())
    except Exception as e:
        logger.warning("合并 MCP 工具列表失败: %s", e)
    return schemas
```

**集成点 2**：`_ChatToolExecutor.execute()` 前缀路由

```python
async def execute(self, tool_name, tool_input):
    if tool_name.startswith("mcp__"):
        try:
            from mcp_client import mcp_client
            return await mcp_client.call_tool(tool_name, tool_input if isinstance(tool_input, dict) else {})
        except Exception as e:
            return json.dumps({"error": f"MCP 调用失败: {e}"}, ensure_ascii=False)
    # ... 原有内部 Action 分发不动 ...
```

### 2.7 生命周期集成（`backend/main.py::lifespan`）

```python
# 启动段
try:
    from mcp_client import mcp_client
    await mcp_client.start_enabled_servers()
    active = [n for n, s in mcp_client.get_status()["servers"].items()
              if s.get("status") == "running"]
    if active:
        logger.info("MCP 客户端就绪：%d 个 server 运行中 (%s)",
                    len(active), ", ".join(active))
except Exception as e:
    logger.warning("MCP 客户端启动失败: %s", e)

# 关停段
try:
    from mcp_client import mcp_client
    await mcp_client.stop_all_servers()
except Exception as e:
    logger.warning("MCP 客户端关闭异常: %s", e)
```

### 2.8 状态查询 API

新增路由 `backend/api/mcp_status.py`：

```
GET /api/mcp/status
→ {

    "filesystem": {"enabled": true, "status": "running",

    "fetch": {"enabled": false, "status": "disabled", ...}
  }
}
```

供：
- 未来前端管理页
- ChatAssistant 自问自答"我接了哪些 MCP"

## 六、未来扩展

### 6.1 前端 MCP 管理页

类似 SOP / Skills 管理页，做一个 `/mcp` 页面：
- 列出所有 server 卡片（名字、命令、状态、工具列表）
- 启用 / 禁用 toggle（更新 mcp_servers.json + `POST /api/mcp/reload`）
- 实时状态刷新（SSE）

工作量 1-2 天。

### 6.2 HTTP SSE 类型 MCP server

当前只支持 stdio。未来支持远程 server（HTTP SSE）：`type: "sse"` + `url: "https://..."` → `sse_client()` 连接。

### 6.3 内部 Action 反向暴露为 MCP server

把本项目的 `GetRequirementPipelineAction` / `GetTicketStatusAction` / `GenerateDocumentAction` 等 Action 封装成一个 MCP server，让 Claude Desktop 等外部客户端也能调。

工作量 2-3 天。

### 6.4 非 Chat Agent 支持 MCP

当前只有 ChatAssistant 用 tool_use。DevAgent / TestAgent 等走 ActionNode JSON。要让它们用 MCP，需要：
- 所有 Agent 切到 `chat_with_tools` 调用方式
- 或者写一个 ActionNode ↔ MCP 的桥接层

这是 ⭐ 架构级改动，建议生产流量验证后再评估。

## 七、风险与降级

| 风险 | 现状 / 缓解 |
|---|---|
| `mcp` SDK 接口不稳（v1.x 新库） | requirements 锁 `mcp<2.0.0`；MCPClient 包一层抽象 |
| Windows subprocess spawn 失败 | 单 server 启动失败只 warning 跳过，不阻塞 |
| MCP 工具调用超时 | 硬编码 30s 超时，返回 error JSON |
| 工具名冲突 | `mcp__` 前缀强制隔离 |
| MCP 响应过大 | 超 10000 字符截断 + 提示 |
| pywin32 postinstall 未跑 | README / 安装文档需注明；首次装 mcp 时要提示 |

## 八、与 MagicAI 的差异

| 特性 | MagicAI | 本方案 |
|---|---|---|
| 集成层 | `mcp_manager.py` 只转换配置 → SDK 原生 | 自建完整客户端（session 生命周期 + 队列） |
| 启动时机 | Agent 启动时传给 CodeBuddySDK | FastAPI lifespan |
| 工具命名 | `mcp__<server>__<tool>` | 同 |
| 调用链 | SDK 自动 | 手动调 `session.call_tool` + 队列串行化 |
| 配置格式 | `mcp_servers.json` | 完全兼容 |

本方案比 MagicAI **多做的**：subprocess 生命周期管理（MagicAI 委托给 SDK），适应本项目没有 SDK 的现状。

## 九、一句话总结

> 给 ChatAssistant 装一个 MCP 客户端适配层，让它能在保留 16 个内部工具的同时**额外**接入任何官方 / 第三方 MCP server。用户只需改 `mcp_servers.json` 的 `enabled` 开关，不用改一行代码就能给 AI 助手扩能（文件系统、网页抓取、Git、Slack、Notion……）。
