# Reflexion 反思框架 — 实现方案

> 日期: 2026-04-19 | 对标: MagicAI 的 Reflexion + Failure Library | 对标论文: Shinn et al. 2023 *Reflexion: Language Agents with Verbal Reinforcement Learning*

---

## 一、背景与问题

### 1.1 现有返工机制

本系统 SOP 有两类失败路径：

```
development_done → (ProductAgent 验收) → acceptance_rejected
                                          → goto rework
                                          → DevAgent 再跑一遍 _do_develop
                                          
testing_done     → (TestAgent 测试) → testing_failed
                                       → goto fix_issues
                                       → DevAgent 再跑一遍 _do_develop
```

触发后的重写流程在 `backend/agents/dev.py` 的 `_do_rework` / `_do_fix_issues`：

```python
async def _do_rework(self, context):
    rejection_reason = context.get("rejection_reason", "")
    context["ticket_description"] = f"{context['ticket_description']} [返工原因] {rejection_reason}"
    return await self._do_develop(context)   # 只是把失败原因拼进描述，重新发 prompt
```

### 1.2 现有机制的缺陷

1. **纯"拼串"无结构化反思**：把 rejection_reason 文本直接拼到 ticket_description 后面。LLM 看到的是"原需求 + [返工原因] xxx"这种朴素拼接，**没有引导 LLM 系统性分析失败根因**。

2. **重复踩坑**：如果 LLM 一开始就没理解需求，靠同样的 prompt 再走一遍，大概率犯同样错误。实测数据显示：当前"添加访问计数功能"需求被**打回 5 次 → 强制通过**，其实 5 次都是"DevAgent 声称自测通过但 ProductAgent 发现完全没实现"。

3. **max_retries=5 的硬保底**：`orchestrator.py:850` 的 `MAX_RETRIES=5`，打回超 5 次直接 `force_pass`（标记完成），**工单"完成"了但功能没做出来**。这是典型的"掩盖失败"——系统指标看起来 OK，实际产出废品。

4. **无跨工单学习**：一个工单踩过的坑，下一个类似工单还会踩。没有失败知识沉淀。

### 1.3 Reflexion 思路（MagicAI 的做法）

```
失败 → LLM 反思（分析根因 / 提炼策略 / 指出具体修改点）
     → 把反思作为 prompt context 注入下一次尝试
     → 重试次数多时：N 次反思累积在 prompt 里
     → 成功则沉淀到 Failure Library，后续类似失败可直接检索
```

论文实证：在代码生成/游戏任务上，Reflexion 比朴素重试**相对提升 20-30% 成功率**。

## 二、核心设计思路

### 2.1 三件事

| # | 动作 | 落在哪 |
|---|---|---|
| 1 | **失败反思**：让 LLM 结构化分析失败原因 | 新增 `ReflectionAction` |
| 2 | **策略注入**：把反思结构化地注入重试 prompt | 改 `DevAgent._do_rework` / `_do_fix_issues` |
| 3 | **失败知识沉淀**（P2）：建 failure_cases 表，相似失败可检索历史 | 新增表 + 语义检索 |

本文档**只详述 #1 + #2**（P1 范围）。#3（Failure Library）放到 **P2 阶段**，作为独立文档再写。

### 2.2 反思输出的结构化 Schema

不让 LLM 自由输出"你觉得哪错了"，而是强制产出固定 schema：

```json
{
  "root_cause": "上一次开发没有在 index.html 里插入计数器的 DOM 元素，导致 ProductAgent 验收时在页面上看不到",
  "missed_requirements": [
    "右下角固定位置的 DOM 元素",
    "每次页面加载时计数+1 的 JS 逻辑",
    "localStorage 持久化"
  ],
  "previous_attempt_issue": "DevAgent 自测时只跑了 pytest（后端空），没有检查页面是否真的有新增 DOM",
  "strategy_change": "这次必须直接修改 index.html 的 <body> 内部，新增 <div id=\"counter\"> 元素 + <script> 块处理 localStorage 计数",
  "specific_changes": [
    "在 index.html 的 </body> 前加 <div id=\"visit-counter\" style=\"position:fixed;bottom:10px;right:10px;\"></div>",
    "加 <script> 读/写 localStorage.visits，+1 后渲染到 #visit-counter.textContent",
    "自测不再跑 pytest，改为 Playwright 访问页面截图确认右下角有数字"
  ],
  "confidence": 0.85
}
```

**设计要点**：
- `root_cause`：一句话根因（不是"我没写好"这种废话）
- `missed_requirements`：具体漏掉了哪几条需求点
- `previous_attempt_issue`：上一次**自测环节**为啥没挡住（关键——暴露"自测 pass 但验收 fail"的 gap）
- `strategy_change`：这次具体要改哪里的实现方式
- `specific_changes`：可执行的具体修改指令
- `confidence`：LLM 对这次反思的自评信心（辅助打回阈值）

### 2.3 注入重试的 Prompt 结构

旧 prompt（简化）：
```
你是 DevAgent，现在要开发这个工单：
【工单描述】在右下角添加访问计数功能 [返工原因] 缺少计数器 DOM 元素

【需求】...
【架构】...
【现有代码】...

请生成代码。
```

新 prompt：
```
你是 DevAgent，现在是第 {retry_count} 次尝试开发这个工单。

【原需求】...
【原架构】...

【上次失败的反思】
- 根本原因：上一次开发没有在 index.html 里插入计数器的 DOM 元素
- 漏掉的需求点：
  1. 右下角固定位置的 DOM 元素
  2. 每次页面加载时计数+1 的 JS 逻辑
  3. localStorage 持久化
- 自测环节为何没拦住：DevAgent 自测只跑了 pytest，没有检查页面 DOM
- 本次应该调整的策略：直接修改 index.html 的 <body> 内部
- 具体修改指令：
  1. 在 index.html 的 </body> 前加 <div id="visit-counter">
  2. 加 <script> 读写 localStorage.visits
  3. 自测改为 Playwright 截图

【现有代码】...

请基于以上反思，重写代码。这次**必须**确保 index.html 里新增了可见的计数器 DOM 元素。
```

**关键改进**：把模糊的"返工原因"变成**结构化诊断 + 具体指令**，LLM 压力更大、不容易敷衍。

### 2.4 多次重试的反思累积

第 1 次失败 → 反思 R1 → 第 2 次尝试（prompt 含 R1）
第 2 次失败 → 反思 R2（prompt 含 R1 + 前一次产物）→ 第 3 次尝试（prompt 含 R1 + R2）
...

设计：**最多保留最近 3 次反思**在 prompt 里（避免 context 爆炸）。每次反思要明确引用"上一次反思里我说 X，但这次仍然失败，说明 X 的策略不对，需要 Y"。

## 三、详细改动清单

### 3.1 新增文件：`backend/actions/reflection.py`

```python
"""
ReflectionAction — 失败反思 Action

输入 context：
  ticket_description    : 原工单描述
  rejection_reason      : ProductAgent 验收的打回理由（acceptance_rejected 路径）
  test_issues           : TestAgent 的测试问题（testing_failed 路径）
  previous_code         : 上次写的代码（dict[path -> content]，来自 git 仓库）
  retry_count           : 当前第几次重试
  previous_reflections  : 历次反思列表（最多 3 条）
  failure_type          : "acceptance_rejected" / "testing_failed"

输出：
  ActionResult.data = {
    "reflection": {
      "root_cause": str,
      "missed_requirements": list[str],
      "previous_attempt_issue": str,
      "strategy_change": str,
      "specific_changes": list[str],
      "confidence": float,
    }
  }
"""
import json
import logging
from typing import Any, Dict
from actions.base import ActionBase, ActionResult
from llm_client import llm_client

logger = logging.getLogger("actions.reflection")


_REFLECTION_SYSTEM_PROMPT = """你是一位资深技术主管，正在复盘一次失败的开发工单。
你的任务：诊断失败根因，提出精确的下一次尝试策略。

输出要求：严格的 JSON，不要 markdown 包裹，字段见用户消息。"""


class ReflectionAction(ActionBase):

    @property
    def name(self) -> str:
        return "reflect"

    @property
    def description(self) -> str:
        return "对失败的工单做结构化反思，输出根因 + 策略调整 + 具体修改指令"

    async def run(self, context: Dict[str, Any]) -> ActionResult:
        ticket_desc = context.get("ticket_description", "")
        failure_type = context.get("failure_type", "unknown")
        rejection_reason = context.get("rejection_reason", "")
        test_issues = context.get("test_issues", [])
        previous_code = context.get("previous_code", {})
        retry_count = context.get("retry_count", 1)
        previous_reflections = context.get("previous_reflections", [])

        # 构建失败信号描述
        if failure_type == "acceptance_rejected":
            failure_signal = f"ProductAgent 验收不通过。理由：{rejection_reason}"
        elif failure_type == "testing_failed":
            issues_text = "\n".join(f"  - {i}" for i in test_issues) if test_issues else "无具体 issue"
            failure_signal = f"TestAgent 测试不通过。问题列表：\n{issues_text}"
        else:
            failure_signal = f"失败类型: {failure_type}"

        # 历次反思摘要（最多 3 条）
        prev_reflection_text = ""
        if previous_reflections:
            keep = previous_reflections[-3:]
            lines = []
            for i, r in enumerate(keep, 1):
                lines.append(f"[第 {i} 次反思] 根因: {r.get('root_cause', '?')[:150]}")
                lines.append(f"  策略: {r.get('strategy_change', '?')[:150]}")
            prev_reflection_text = "\n## 历次反思（仍然失败）\n" + "\n".join(lines) + "\n请注意：上面的策略都没生效，这次要换一种思路。"

        # 现有代码摘要（前 800 字）
        code_summary = ""
        if previous_code:
            lines = []
            for path, content in list(previous_code.items())[:5]:
                snippet = content[:200].replace("\n", " ")
                lines.append(f"- {path}: {snippet}...")
            code_summary = "\n## 上次产出的代码摘要\n" + "\n".join(lines)

        user_prompt = f"""## 任务
复盘开发失败并产出结构化反思。

## 原工单描述
{ticket_desc}

## 本次失败信号（第 {retry_count} 次重试）
{failure_signal}
{code_summary}
{prev_reflection_text}

## 要求：输出严格 JSON
{{
  "root_cause": "一句话说清楚失败的根本原因（不要笼统，要具体到文件/函数/逻辑）",
  "missed_requirements": ["原需求里 DevAgent 漏掉/误解的具体条目 1", "条目 2"],
  "previous_attempt_issue": "上一次的自测环节为什么没拦住这个问题（是自测范围不够？验收标准不一致？）",
  "strategy_change": "这一次的策略与上一次的本质区别是什么",
  "specific_changes": ["具体修改 1（到文件/行）", "修改 2"],
  "confidence": 0.0 到 1.0
}}

输出纯 JSON，不要包 markdown。"""

        messages = [
            {"role": "system", "content": _REFLECTION_SYSTEM_PROMPT},
            {"role": "user", "content": user_prompt},
        ]

        try:
            raw = await llm_client.chat(messages, temperature=0.3, max_tokens=2000)
            # 清理 ```json``` 包裹
            cleaned = raw.strip()
            if cleaned.startswith("```"):
                cleaned = "\n".join(l for l in cleaned.split("\n")
                                    if not l.strip().startswith("```"))
            reflection = json.loads(cleaned)
        except Exception as e:
            logger.warning("Reflection LLM 失败或 JSON 解析失败: %s", e)
            # 降级：返回最小可用反思
            reflection = {
                "root_cause": f"LLM 反思失败（{e}），降级仅透传失败信号",
                "missed_requirements": [],
                "previous_attempt_issue": "",
                "strategy_change": rejection_reason or str(test_issues),
                "specific_changes": [],
                "confidence": 0.3,
            }

        logger.info("🔍 反思完成：根因=%s...", reflection.get("root_cause", "")[:80])

        return ActionResult(
            success=True,
            data={"reflection": reflection, "retry_count": retry_count},
        )
```

### 3.2 改动：`backend/agents/dev.py`

注入 ReflectionAction + 在 rework/fix_issues 里先反思再开发。

```python
# ---------- 新 import ----------
from actions.reflection import ReflectionAction

# ---------- 改 action_classes ----------
class DevAgent(BaseAgent):
    action_classes = [
        WriteCodeAction,
        PlanCodeChangeAction,
        SelfTestAction,
        ReflectionAction,   # 新增
    ]
    # ...

    # ---------- 改 _do_rework ----------
    async def _do_rework(self, context: Dict[str, Any]) -> Dict[str, Any]:
        """返工（带反思）"""
        context["failure_type"] = "acceptance_rejected"

        # 拉上次代码 + 历次反思
        await self._enrich_retry_context(context)

        # 反思
        refl_result = await self.run_action("reflect", context)
        reflection = refl_result.get("reflection", {})

        # 把反思结构化注入开发 context（不再粗暴拼字符串）
        context["reflection"] = reflection
        context["retry_count"] = context.get("retry_count", 1)

        # 依旧调原开发流程（改造的是 prompt 层，见 3.3）
        result = await self._do_develop(context)
        # 把本次反思带出，供 orchestrator 记到 ticket_logs
        result["last_reflection"] = reflection
        return result

    async def _do_fix_issues(self, context: Dict[str, Any]) -> Dict[str, Any]:
        """测试失败修复（带反思）"""
        context["failure_type"] = "testing_failed"
        await self._enrich_retry_context(context)

        refl_result = await self.run_action("reflect", context)
        reflection = refl_result.get("reflection", {})

        context["reflection"] = reflection
        context["retry_count"] = context.get("retry_count", 1)

        result = await self._do_develop(context)
        result["last_reflection"] = reflection
        return result

    async def _enrich_retry_context(self, context):
        """从 DB/Git 拉取重试所需的上下文：previous_code + previous_reflections + retry_count"""
        from database import db

        ticket_id = context.get("ticket_id")
        if not ticket_id:
            return

        # 重试次数（排除自己本次）
        r = await db.fetch_one(
            "SELECT COUNT(*) AS c FROM ticket_logs WHERE ticket_id = ? AND action = 'reject'",
            (ticket_id,),
        )
        context["retry_count"] = (r["c"] if r else 0) + 1

        # 历次反思
        refl_logs = await db.fetch_all(
            """SELECT detail FROM ticket_logs
               WHERE ticket_id = ? AND action = 'reflection'
               ORDER BY created_at DESC LIMIT 3""",
            (ticket_id,),
        )
        prevs = []
        for log in reversed(refl_logs):   # 时间正序
            try:
                prevs.append(json.loads(log["detail"]))
            except Exception:
                pass
        context["previous_reflections"] = prevs

        # 上次代码（从 feat 分支读）
        # 简化：用 existing_code 占位；完整实现应读 Git
        context["previous_code"] = context.get("existing_code", {})
```

### 3.3 改动：`backend/actions/write_code.py` 和 `actions/plan_code_change.py`

在这些 Action 的 prompt 构造里，如果 `context["reflection"]` 存在则插入反思段落。

```python
def _build_prompt(self, context):
    ticket_desc = context["ticket_description"]
    existing = context.get("existing_code", {})
    prd = context.get("prd_content", "")

    # --- 新增 ---
    reflection_block = ""
    reflection = context.get("reflection")
    retry_count = context.get("retry_count", 1)
    if reflection and retry_count > 1:
        reflection_block = self._format_reflection_block(reflection, retry_count)
    # --- 新增结束 ---

    return f"""你是 DevAgent，现在是第 {retry_count} 次尝试这个工单。

## 原需求
{ticket_desc}

## PRD
{prd}

{reflection_block}

## 现有代码
{self._format_existing(existing)}

请基于以上所有信息，生成代码修改计划。
"""

def _format_reflection_block(self, reflection, retry_count):
    return f"""## 上一次失败的反思（第 {retry_count - 1} 次失败后）

**根因**：{reflection.get('root_cause', '')}

**上一次漏掉 / 误解的需求点**：
{chr(10).join(f'- {x}' for x in reflection.get('missed_requirements', []))}

**上一次自测环节为何没拦住**：{reflection.get('previous_attempt_issue', '')}

**本次策略调整**：{reflection.get('strategy_change', '')}

**具体修改指令**：
{chr(10).join(f'- {x}' for x in reflection.get('specific_changes', []))}

⚠️ 本次**必须**按上述具体修改指令执行，不能再犯同样的错误。
"""
```

### 4.1 改前

    │
    ▼

    │
    ▼

    ▼
生成代码 → ProductAgent 验收 → 大概率同样的失败

### 4.2 改后

```
acceptance_rejected
    │
    ▼
Orchestrator 发 rework
    │
    ▼
DevAgent._do_rework
    │  1. _enrich_retry_context：
    │     - 拉 retry_count（第几次）

    │
    ├─ 通过 → 继续流程
    └─ 再次失败 → 回到最上面，这次 prompt 里会有 2 条历次反思

    with patch("llm_client.llm_client.chat", AsyncMock(return_value=fake_response)):

## 八、风险与权衡

| 风险 | 缓解 |
|---|---|
| 反思本身也会失败（LLM 超时/返回无结构数据） | 降级路径：反思失败时退回旧逻辑（只拼 rejection_reason），不阻塞主流程 |
| 反思 prompt 膨胀 context | 限制历次反思数 ≤ 3；每条反思文本截断到 300 字 |
| LLM 反思"敷衍"，输出模板化废话 | system prompt 里强调"必须具体到文件/函数/行"；若 `confidence < 0.5` 记 warning |
| 反思 token 成本翻倍 | 每次重试增加 1 次 LLM 调用。测算：假设原来 1 次开发 ~5k token，反思 ~1k token。成本增加 20%，换取 25%+ 成功率提升，ROI 为正 |
| 重试次数过多时反思 context 堆积 | 只保留最近 3 条；>5 次强制走 force_pass（保留现有兜底） |
| 反思结果可能误导（错判根因） | 累积反思机制：下一次反思会看到"上一次说是 X，但还是失败"，LLM 被迫换思路 |

## 九、与 MagicAI 的差异

MagicAI 的 Reflexion 更完整，本方案是**精简 + 贴合本项目架构**的版本：

| 特性 | MagicAI | 本方案（M1） |
|---|---|---|
| 反思触发点 | 任何 Agent 失败 | 仅 DevAgent 的 rework/fix_issues |
| 反思粒度 | LangGraph 节点级 | SOP stage 级 |
| 多候选生成（Self-Consistency） | ✅ N=5 投票 | ❌（留给 M3） |
| Failure Library | ✅ 向量检索历史 | ❌（留给 M3） |
| 模型 | claude-opus-4.6 / claude-sonnet-4.6 | claude-sonnet-4.6（本项目现有配置） |

**可以接受**：本方案只要做到"比现在强"就有价值，不必一步到位 MagicAI 级别。
