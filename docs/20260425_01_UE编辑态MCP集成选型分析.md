# UE 编辑态 MCP 集成选型分析

> 日期：2026-04-25
> 起因：用户要求对比 `flopperam/unreal-engine-mcp` 与 `Italink/UnrealClientProtocol` 两个 UE 相关开源项目
> 版本上下文：v0.19（已交付 UECompileCheckAction / UEPlaytestAction / Reflexion UE 支路）
> 决策：**v0.19.x 不接入**，作为 v0.20+ 候选能力纳入 TODO；详见第 7 节

---

## 1. 为什么会问这个问题

v0.18/v0.19 把 UE 的「**离线**」能力填满了：

- UBT 编译 + 结构化错误解析（`UECompileCheckAction`）
- 模板实例化 + Git commit（`InstantiateUETemplateAction`）
- Automation Framework headless 测试（`UEPlaytestAction`）
- 编译 / 测试失败走 Reflexion 自动修复

但**编辑态**（editor 开着的时候直接操纵 Blueprint / 资产 / 关卡）一直没做。这带来几个痛点：

| 痛点 | 当前处理方式 | 代价 |
|---|---|---|
| AI 要改一个 Blueprint 的变量值 | 只能让 DevAgent 改 C++ 源码 + UHT 重生成 + UBT 重编 | 每次 ~2-5 min 编译 |
| AI 要往测试地图加一个 actor 验证 | 只能让 DevAgent 写 Functional Test C++ 类 | 需要写完整 C++ + 编译 |
| Reflexion 反思时想知道「当前 editor 里 XX 资产长啥样」 | 无法看 editor 状态，只能靠 C++ 源码 + 日志猜 | 反思质量受限 |
| AI 要批量改一批 material 参数 | 逐个写 C++ 资产迁移脚本 | 低效、高风险 |

两个项目都声称能解决这类问题——让 AI 通过 IPC 协议**远程操纵正在运行的 Unreal Editor**。本文逐项拆解它们到底能干什么、跟我们系统怎么对接、要付什么代价。

## 2. flopperam/unreal-engine-mcp 详解

### 2.1 基本信息

| 项 | 值 |
|---|---|
| 仓库 | https://github.com/flopperam/engine-mcp |
| Stars / Forks | 851 / 166 |
| 最近活跃 | 持续更新（截至 2026-04） |
| License | MIT |
| 语言分布 | C++ 59% · Python 40% |
| UE 版本 | 5.5 / 5.6 / 5.7 |
| 提交数 | 92 |
| Open issues | 3 |

### 2.2 架构

```
┌─────────────────┐      MCP         ┌────────────────────┐      TCP       ┌──────────────────┐
│  AI Client      │  ───────────────▶│ Python MCP Server  │───────────────▶│ C++ UnrealMCP    │
│ (Cursor / CC /  │   stdio or SSE   │ (unreal_mcp_server │   socket       │ Plugin (editor)  │
│  Windsurf ...)  │                  │  _advanced.py)     │                │                  │
└─────────────────┘                  └────────────────────┘                └──────────────────┘
                                                                                    │
                                                                            runs inside UE
                                                                            Editor 5.5+
```

- 前端是**标准 MCP server**，任意支持 MCP 的 AI 客户端都能连
- 后端是**必须装到 UE 项目里的 C++ 插件**（Plugins/UnrealMCP）
- 还提供 **hosted remote MCP**（flopperam.com/mcp），免部署但要把 editor 暴露给第三方服务（一般不会用）

### 2.3 暴露的工具集（MCP tool surface）

**Blueprint 图编程（23+ 工具）**：
- `add_node`, `connect_nodes`, `delete_node`, `set_node_property`
- `create_variable`, `create_function`, `add_function_input`, `add_function_output`
- `read_blueprint_content`, `analyze_blueprint_graph`
- `compile_blueprint`

**关卡/世界构建（高层语义）**：
- `create_town`, `construct_house`, `construct_mansion`
- `create_tower`, `create_castle_fortress`, `create_suspension_bridge`
- `create_maze`, `create_aqueduct`

**Actor / 物理**：
- `get_actors_in_level`, `spawn_physics_blueprint_actor`
- `set_actor_transform`, `apply_material_to_actor`

### 2.4 评价

**优点**：
- 高层语义工具对 demo / 快速原型很炫（一句话建城堡）
- 成熟度高（851★），bug 踩得差不多了
- Blueprint 节点级的细粒度操作完整

**缺点**：
- 高层工具太「开箱即用」但**反而束手束脚**——我们的项目不一定要建城堡，这些 API 用不上
- `create_town` 这类 DSL 把"业务逻辑"硬编码到 MCP 里，拓展到新项目形态要 fork 插件源码
- Python MCP server 是独立进程，额外一份运行时要管

## 3. Italink/UnrealClientProtocol (UCP) 详解

### 3.1 基本信息

| 项 | 值 |
|---|---|
| 仓库 | https://github.com/Italink/UnrealClientProtocol |
| Stars / Forks | 105 / 13 |
| 最近活跃 | 活跃开发 |
| License | MIT |
| 语言分布 | C++ 92% · Python 7.5% |
| UE 版本 | 5.x |
| Open issues | 1 |

### 3.2 架构

```
┌─────────────────┐     TCP + JSON     ┌─────────────────────┐
│  AI Agent       │  ─────────────────▶│ UCP Plugin          │
│ (任何支持 TCP   │   port 9876        │ (内嵌在 UE Editor)  │
│  客户端的系统)  │                    │                     │
└─────────────────┘                    └─────────────────────┘
                                               │
                                       reflection-driven
                                               │
                                    ┌──────────┴──────────┐
                                    │  UE Reflection API  │
                                    │ (UFunction/UProperty│
                                    │  / UClass metadata) │
                                    └─────────────────────┘
```

- **零 engine 修改**——纯插件
- Editor 启动时自动拉起一个 TCP 监听（loopback-only，默认 9876）
- 协议：**JSON-over-TCP**，自定义 schema

### 3.3 分层技能设计

| 层 | 作用 |
|---|---|
| Transport | TCP/JSON 序列化反序列化 |
| Foundation | `UObject.GetProperty` / `SetProperty` / `CallFunction` ——基于 reflection 的万能原子 |
| Domain Skills | Actor / Asset / Material / Blueprint 五个域，每域若干操作 |

### 3.4 暴露能力

- **Object**：任意 `UObject` 的 property 读写、metadata 自省
- **Actor**：spawn / delete / duplicate / transform / selection
- **Asset**：搜索、依赖分析、CRUD
- **Material**：节点图编辑、HLSL、MaterialInstance、compile
- **Blueprint**：节点图、函数、事件、变量
- **通用**：任意 `UFunction` 远程调（通过 reflection）

### 3.5 NodeCode——它的独门设计

**核心创新**：把 Blueprint / Material 的可视化图转换成**文本 IR**，让 AI 当"代码"读写。

意义：
- AI 原生是处理文本的，直接操纵图结构（`add_node + connect_nodes`）要调很多次工具、容易出错
- NodeCode 类似 "把 BP 序列化成一个人类可读的类 YAML 结构" → AI 一次性读写完整 BP

对我们系统的意义很大——DevAgent 的 Reflexion 已经是文本驱动的反思，NodeCode 让它能"写 BP 就像写 C++"。

### 3.6 安全控制

- Loopback-only 绑定（默认只允许 127.0.0.1）
- Allowlist / blocklist 机制
- 集成 UE 的 Undo/Redo 系统——改错能撤销

### 3.7 评价

**优点**：
- 反射驱动 = 覆盖面最广，没有"哪个 API 不支持"的死角
- 纯插件、零独立进程、零 engine 修改
- NodeCode 抽象对 AI 友好
- 安全模型考虑周到

**缺点**：
- 成熟度相对低（105★）——边缘情况要自己踩
- Latent functions（延迟执行的 UE 节点）不支持
- Delegate 类型的参数不支持
- 中文文档为主，国际化不足

## 4. 两者正面对比

| 维度 | flopperam/unreal-engine-mcp | Italink/UnrealClientProtocol |
|---|---|---|
| **抽象层次** | 高层语义 + 低层原子混合 | 纯低层原子 + reflection |
| **接入协议** | MCP（stdio/SSE）→ TCP | 直接 TCP JSON |
| **独立进程** | 需要 Python MCP server | 仅 editor 内插件 |
| **跟我们 Agent 架构契合** | 中（我们已有 tool_use，不太需要外部 MCP server） | 高（直接 TCP 客户端接进 `ActionBase`） |
| **BP 编辑** | 节点级 API | 节点级 + NodeCode 文本 IR |
| **材质编辑** | actor 上 apply material | 完整 material 图 + HLSL + MaterialInstance |
| **资产管理** | 弱 | 强（依赖分析、搜索、CRUD） |
| **域覆盖** | BP 强、世界构建强、其他一般 | 全面：Object / Actor / Asset / Material / BP |
| **"一键建城堡"类工具** | 有 | 无 |
| **成熟度** | 高（851★） | 中（105★） |
| **改插件成本** | 高（内置 DSL 多，改一处影响多处） | 中（reflection 驱动，加 skill 相对干净） |
| **我们系统接起来的代码量** | 中（还得包一层 MCP 客户端） | 小（直接 TCP JSON，~200 行） |

**一句话总结**：
- flopperam 像 "**带示例的应用**"——适合快速演示 + BP 深度编辑场景
- UCP 像 "**SDK**"——适合构建长期的 AI-Editor 联动能力

## 5. 与我们系统的契合度分析

### 5.1 当前系统 UE 能力快照（v0.19）

```
离线链路（✅ 已实现）：
  建模板 → UBT 编译 → Automation 测试 → 失败 → Reflexion → DevAgent 改 C++ → UBT 重编 → ...

编辑态链路（❌ 完全缺失）：
  AI 改 BP / 资产 / 材质 / 关卡 actor 的能力为 0
```

### 5.2 接入 UCP 能补的能力（按价值排序）

| # | 能力 | 当前实现方式 | 接 UCP 后 | 价值 |
|---|---|---|---|---|
| 1 | **Reflexion 时查 editor 现状** | 盲猜 | `get_actors_in_level` / `read_blueprint` | ★★★★★ |
| 2 | **改 BP 变量默认值** | 改 C++ `UPROPERTY(EditDefaultsOnly)` 默认值 + 重编 | `set_property` 一行 | ★★★★ |
| 3 | **批量改 material 参数** | 写 C++ commandlet | `set_material_parameter` | ★★★★ |
| 4 | **生成 Functional Test 的测试场景** | 写 C++ 测试类手动 spawn | `spawn_actor` + `set_transform` | ★★★★ |
| 5 | **资产依赖分析** | 无 | `analyze_dependencies` | ★★★ |
| 6 | **BP 图编辑（加节点连线）** | 改 C++ 重编 | NodeCode 文本 IR | ★★★ |
| 7 | **关卡动态构建（建城堡那种）** | 不做 | flopperam 有，UCP 要自己写 | ★★（非核心） |

### 5.3 对接点设计（UCP 方案）

**新 Action**：`UEEditorControlAction`

```python
# backend/actions/ue_editor_control.py
class UEEditorControlAction(ActionBase):
    available_for_traits = {"any_of": ["engine:ue5", "engine:ue4"]}

    async def run(self, context):
        op = context["op"]              # "get_actors" / "set_property" / "spawn_actor" / ...
        args = context.get("args", {})
        # TCP 连接 127.0.0.1:9876，JSON 请求响应
        resp = await self._ucp_call(op, args)
        return ActionResult(success=True, data=resp)
```

**新 skill pack**：`backend/skills/packs/unreal_editor_control/prompt.md`
- 列出所有 UCP 原子操作 + 使用场景
- 注入条件：`inject_to=[DevAgent, TestAgent]` + `traits_match: any_of=engine:ue*`

**Reflexion 增强**：
- `DevAgent._do_fix_issues` 在 `play_test_failed` 分支里，反思前先调 UCP 取测试地图现状 → 注入 context → ReflectionAction 出更准确的策略

**模板注入**：
- `InstantiateUETemplateAction` 选项加 `install_ucp: true`，instantiate 时把 UCP 插件拷到 `Plugins/UCP/`
- 前端方案卡加个复选框 "启用编辑态 MCP（自动装 UCP 插件）"

**启动约束**：
- UE Editor 必须开着且 UCP 插件已启用
- 连不上 9876 → Action 返回 `status=error` → orchestrator 走 BLOCKED + 提示用户"开 editor 后重试"

### 5.4 为什么不选 flopperam

1. **架构重复**：我们已有 tool_use + Action + Skill 体系，再套一层外部 Python MCP server 是冗余
2. **DSL 绑定**：`create_town` / `create_castle` 这类工具对一般项目是噪音
3. **Python 进程管理**：多一个独立 server 要管启停，增加运维复杂度
4. **BP 操作接口与 UCP 重叠**，UCP 的 NodeCode 更优

## 6. 风险与阻碍

### 6.1 必须开 Editor 的悖论

当前 `UEPlaytestAction` 是 `-unattended` headless 跑完就退，不保持 editor 开。接入 UCP 意味着：
- 要么**常驻一个 editor 进程**（占用 ~4GB 内存 + 显存，启动 30s+）
- 要么**每次 action 前启 editor**（延迟叠加到每次 AI 反思）

**候选方案**：
- 在项目设置里加「**持久化 editor 进程**」开关，默认关
- 开启后后端起一个守护进程 pinger，每次 AI 决定要用 UCP 时 lazy-start editor
- Editor 空闲 N 分钟无操作自动 kill

### 6.2 资产破坏风险

AI 随手改 BP 可能把用户项目搞坏：
- 改错变量默认值 → 用户之前调好的 gameplay feel 丢了
- 删 actor / asset → 引用链断裂
- 改材质参数 → 一大片视觉失真

**兜底机制**：
- 所有 UCP 操作**前置 git stash + 结束后 diff 高亮**，异常时一键 revert
- UCP 本身集成 Undo/Redo，但跨 editor 重启无效
- 每次 UCP 操作前自动 `git commit -m "ai-ucp: before <op>"` 做回滚点

### 6.3 成熟度 vs 价值

UCP 只有 105★，远低于 flopperam 的 851★。意味着：
- 边缘情况要自己踩坑（latent function、delegate、PIE 模式、blueprint 继承链等）
- 可能需要 fork 自己维护

折中：**只用 UCP 的 Foundation + Actor/Asset 层**（这两层最通用），Material / BP 的高级操作按需逐步验证。

### 6.4 MCP 生态竞品可能出现

MCP 标准刚出一年多，UE 这方向还会有更多项目。今天接 UCP，明年可能出个 Epic 官方的、或者 1k+★ 的替代品。

缓解：**封装一个 `UEEditorControlAction` 接口层**，上层 Agent 代码只依赖接口；底层实现 UCP / flopperam / 官方方案可以后续替换。

## 7. 决策：v0.19.x 不接，v0.20 候选

### 7.1 理由

1. **v0.19 UE playtest 还没真机实操过**——先把已有链路验收完
2. **常驻 editor 进程**是个大子系统，不适合塞在 v0.19 零碎做
3. **TestFPS 当前项目是纯 C++**，没有 BP 重度编辑需求，UCP 价值还没被验证
4. **MCP 生态未稳**，早接可能两个月后推倒重来

### 7.2 v0.20 接入前提

需要同时满足：
- 用户有一个 BP / Asset 重度项目（TestFPS 不算）
- v0.19 的离线链路在真项目上稳定跑通 ≥ 3 轮
- UCP 没有重大 API break（观察 3-6 个月）

### 7.3 若决定做，建议的 Phase 拆分

| Phase | 内容 | 估时 |
|---|---|---|
| ①  | `UEEditorControlAction` 底层：TCP 连 UCP + 3 个原子（get_actors / read_property / call_function） | 1 天 |
| ②  | 常驻 editor 守护（启停管理 + 健康检查 + SSE 日志）| 1.5 天 |
| ③  | Skill Pack `unreal_editor_control` | 0.5 天 |
| ④  | 模板实例化时选配 UCP 插件 copy + 前端卡片选项 | 0.5 天 |
| ⑤  | Reflexion 支路调 UCP 取 editor 现状 | 1 天 |
| ⑥  | BP 编辑场景（NodeCode 读写 BP 文件） | 2 天 |
| ⑦  | 资产破坏兜底（git stash + 异常 revert） | 1 天 |
| ⑧  | smoke test + 真项目实操 | 2 天 |

**合计 ~9.5 天**，跟 v0.19 ② UE run_playtest 估时相当。按 v0.18/v0.19 的经验实际可能 3-5 天。

---

## 8. 一句话结论

**UCP 是正确的长期方向**（reflection-based / 原子 / NodeCode / 契合 Agent 架构），**flopperam 是正确的短期 demo 参考**（高层 DSL / 成熟），但**两者都不是 v0.19 的当下需求**——先把已有 UE 离线链路在真项目跑稳再说。
