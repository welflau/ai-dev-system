## 三、已交付能力详解

### 3.1 Reflexion 反思框架

**问题**：失败→打回→硬重试，实测"访问计数"需求被打回 5 次 force_pass。

**解法**：DevAgent 失败后调 `ReflectionAction` 做**结构化反思**（root_cause / missed_requirements / previous_attempt_issue / strategy_change / specific_changes / confidence），把结果以「## 上一次失败的反思」段注入重试的 prompt。最多累积 3 条历次反思在 context 里。

**关键改动**：
- 新增 `backend/actions/reflection.py`
- `backend/agents/dev.py::_do_rework` / `_do_fix_issues` → 统一走 `_do_retry_with_reflection`
- `backend/actions/plan_code_change.py` / `write_code.py` prompt 模板加反思段
- SOP yaml 加 `enable_reflection` flag（A/B 可关）
- 单测 `_test_reflection.py` 5 用例

### 3.2 Skills 注入系统（P0）

**问题**：各 Agent 的领域知识（React 模式、FastAPI 规范、Playwright 用法、Git 工作流）硬编码在 Action prompt 里，调整需改源码。

**解法**：模仿 MagicAI 的 Skills 机制——一个 `.md` 文件 = 一个 Skill + `skills.json` 决定挂给哪些 Agent。用 `contextvars.ContextVar` 透明注入，**现有 Action 代码零改动**。

**MagicAI Skill 包可直接 drop-in**（目录结构对齐）。

**关键改动**：
- 新增 `backend/skills/{loader.py, skills.json, packs/<id>/prompt.md}`
- 4 个初始 Skill：react_dev / fastapi_dev / playwright_e2e / git_workflow
- `BaseAgent.__init__` 加载 `self._skills_prompt`，`run_action` 用 ContextVar 注入
- `ActionNode._compile` 从 ContextVar 读 skills 并 prepend
- `ChatAssistant._build_system_prompt` 单独注入（不走 ActionNode）
- 单测 `_test_skills.py` 8 用例

**效果**：启动日志实测 `🎓 DevAgent 加载 3 个 Skill: react-dev, fastapi-dev, git-workflow`。

### 3.3 Failure Library（P1）

**问题**：Reflexion 反思只在**同一工单内**共享，工单 A 踩过的坑工单 B 会原样再踩。

**解法**：新增 `failure_cases` 表 + `FailureLibrary` 服务。DevAgent 失败时 record，ticket 转 acceptance_passed / testing_done 时 mark_resolved。未来工单 rework 时按 `agent_type + module + failure_type + 关键词 LIKE` 检索历史 top 3，注入 ReflectionAction 的 prompt。

无向量检索依赖——纯 Python 关键词提取（中文 2+ 字 / 英文 3+ 字 / 停用词过滤）+ SQLite LIKE。未来流量大了可升级 FTS5 或 embeddings。

**关键改动**：
- 新增 `backend/failure_library.py` + `failure_cases` 表（3 索引）
- `backend/_migrate_failure_library.py` 一次性回灌脚本（幂等，按 ticket_id + root_cause 去重）
- `orchestrator` 反思日志处 → `record()`；`_handle_agent_result` 成功分支 → `mark_resolved()`
- `DevAgent._enrich_retry_context` → `search_similar()`
- `ReflectionAction` 渲染「## 历史相似失败（跨工单）」段
- 单测 `_test_failure_library.py` 6 用例

**效果**：回灌实测把本地 DB 中 4 条"HelloWorld 时钟"历史反思入库，后续相似 frontend 工单 rework 时自动拉到这些前车之鉴。

### 3.4 Session Transcript（P2）

**问题**：调试要去 `server.log` + `ticket_logs` 表 + `llm_conversations` 表三处拼凑。

**解法**：利用 `orchestrator._log()` + `llm_client._save_conversation()` 两个天然 chokepoint，镜像写入 `backend/logs/session_<req_id>/` 目录。

```
backend/logs/session_REQ-xxx/
├── transcript.txt       # 人读：时间戳 + emoji + 事件一行一条
└── tool_calls.jsonl     # 结构化：每行一个 JSON（含 detail 嵌套）
```

LLM 调用只记 tokens / duration / status，不记完整 prompt（已在 DB 表里）。

**关键改动**：
- 新增 `backend/session_logger.py`（SessionLogger 单例，per-req asyncio.Lock 防并发乱序）
- `orchestrator._log` + `llm_client._save_conversation` 各 +15 行镜像调用（try/except 降级）
- `.gitignore` 的 `logs/` 规则已覆盖，无需改
- 单测 `_test_session_logger.py` 5 用例（含 100 并发）

**效果**：端到端模拟输出示例：
```
[01:16:41] 🤖 Orchestrator.start · ticket T-e2e001 · pending → development
[01:16:41] 🧠 DevAgent.write_code → LLM call (1234/567 tokens, 2.0s)
[01:16:41] ❌ ProductAgent.reject · acceptance_pending → acceptance_rejected
[01:16:41] 🔍 DevAgent.reflection · [反思 #2] 根因: 没在 index.html 加 DOM
[01:16:41] ✅ ProductAgent.accept · acceptance_pending → acceptance_passed
```

## 六、下一步选项

### 选项 A：继续做 P3 MCP 协议统一

**价值**：把 ChatAssistant 的 Anthropic tool_use 硬编码工具 + 其他 Agent 的 JSON parsing 全部迁到 MCP 协议。能接入第三方 MCP 服务器生态（官方 git-mcp / filesystem-mcp / fetch-mcp）。

**工作量**：4-5 天（改 LLM 调用层 + Agent 通信层）。

**风险**：较大重构，短期收益不如 P0/P1/P2 明显。

### 选项 B：观察期 + 跑生产流量验证 P0/P1/P2

**价值**：拿到实测数据：
- Failure Library 命中率有多高？检索出来的案例真的有帮助吗？
- Skills 注入是否让 DevAgent 产出质量提升（减少重试次数）？
- Session Transcript 是否真的加速了调试？

**工作量**：0（跑几天即可）。

### 选项 C：补齐 Self-Consistency（⭐⭐⭐ 剩下的一项）

**价值**：对架构设计、代码生成这类发散任务，生成 N=3 个候选 → Critic 打分 → 选 best。论文实测质量 +25%。

**工作量**：2 天（主要是改 ArchitectAgent 和 DecomposeAction 的 run 接口）。

**代价**：LLM token ×3。

## 七、结语

3 天（4-18 ~ 4-20）内把 MagicAI 对比分析里的 3 项 ⭐⭐⭐ 中的 2 项（Reflexion + Failure Library）+ 2 项 ⭐⭐（Skills + Session Transcript）全部落地，总计新增 1500+ 行生产代码 + 500+ 行测试 + 2000+ 行文档。剩余未完成事项以 **P3 MCP 协议统一** 工作量最大但最不紧迫。建议先进观察期，用实际需求检验 P0/P1/P2 的效果数据，再决定 P3 的投入节奏。
