# v0.18 UE 深耕 工作安排

> 日期：2026-04-24  
> 接续：`2026-04-24_v0.17-diag_TestFPS死循环与AI诊断.md`（v0.17 完结后的下一个主线）

## 0. TL;DR

v0.17 把 UE 项目的所有"架构钩子"（trait / preset / SOP fragment / pipeline_view / MCP 过滤）都挂好了，但**执行层全是空的**。TestFPS 实战暴露：6 个工单全部卡在 `engine_compile` 阶段，因为 `DevAgent.run_engine_compile` 等 5 个 UE 专属 action 都只是声明未实现。

v0.18 主题：**把 v0.17 挂好的 UE 钩子真正填上内容**，让 TestFPS 能端到端跑通 `architecture → template 实例化 → development → engine_compile`。

**不做**：`run_playtest` / `run_network_stress` / `generate_assets`（留 v0.19+），Blueprint 直接编辑（架构上永远不做）。

**工期预估**：14 天（7 个 Phase）。

## 1. 背景

### 1.1 当前状态（v0.17 交付后）

TestFPS 项目 `traits = [engine:ue5, category:game, genre:fps, multiplayer, ...]`，v0.17 正确派生出 10 阶段 SOP：

```
architecture → asset_gen → development → engine_compile → play_test 
            → multiplayer_stress → code_review → acceptance → deploy → rework
```

但实际跑时：

| 阶段 | agent.action | 实现状态 | 后果 |
|---|---|---|---|
| engine_compile | DevAgent.run_engine_compile | ❌ 未实现 | BLOCKED |
| play_test | TestAgent.run_playtest | ❌ 未实现 | BLOCKED |
| multiplayer_stress | TestAgent.run_network_stress | ❌ 未实现 | BLOCKED |
| asset_gen | ArtistAgent.generate_assets | ❌ Agent 类都没有 | BLOCKED |
| i18n_check | DevAgent.check_i18n_completeness | ❌ 未实现 | BLOCKED |

同时 **Skills** 里 0 个 UE 相关（只有通用 react/fastapi/playwright/git）— DevAgent 写 UE C++ 时 LLM 没有领域规范注入，**从零盲写**的代码几乎不可能通过 UE 编译。

### 1.2 TestFPS 实战驱动出的 3 个深层需求

用户在讨论里依次提出：

1. **从零写 UE C++ 不可靠** → 应该用 UE 自带的**项目模板**机制（`Templates/TP_FirstPerson` 等），让 LLM 在可编译骨架上改而非从零写
2. **生成框架需要对话式反馈** → 不能一键黑盒执行，关键节点要弹卡片让用户确认（模板选择、是否跑基线编译等）
3. **引擎本身要自动检测** → UE 引擎可能是 Launcher 装的也可能是自编译的，系统要能识别多个版本并让用户选

这 3 个需求合起来构成 v0.18 的**核心四件套**：
- **引擎检测** —— 先知道有没有引擎、是哪个版本、什么类型
- **模板实例化** —— 生成可编译骨架（降低 LLM 任务难度）
- **对话式交互** —— 关键节点用卡片反馈 + 确认
- **编译验证** —— UECompileCheckAction + 失败反馈链

## 3. 核心方案

### 3.1 Part 1：UE 引擎检测（UEEngineResolver）

#### 识别机制

UE 引擎在系统里留痕的 4 个权威源：

| 源 | 位置 | 用途 |
|---|---|---|
| **HKLM 注册表** | `HKLM\SOFTWARE\EpicGames\Unreal Engine\{5.0..5.4}\InstalledDirectory` | 官方 Launcher 版本路径 |
| **HKCU 注册表** | `HKCU\SOFTWARE\Epic Games\Unreal Engine\Builds\{GUID}` | 自编译 build 路径（GenerateProjectFiles 时登记） |
| **Build.version** | `<Engine>/Engine/Build/Build.version`（JSON） | 权威版本号：`{MajorVersion, MinorVersion, PatchVersion, Changelist, ...}` |
| **类型标志文件** | `<Engine>/Engine/Build/InstalledBuild.txt` / `SourceDistribution.txt` | 前者=官方；后者=GitHub 源码 |

#### 解析流程（给定 .uproject 路径）

```
Step 1. 读 .uproject 的 EngineAssociation 字段
        ├─ "5.3"       → 查 HKLM → 官方 UE_5.3 路径
        ├─ "{GUID}"    → 查 HKCU → 自编译路径
        └─ ""          → 扫相对路径（.uproject 所在目录及上级找 Engine/）

Step 2. 读 Engine/Build/Build.version → 权威版本号

Step 3. 判定类型
        • InstalledBuild.txt 存在 → type=launcher
        • SourceDistribution.txt 存在 → type=source_build
        • 都没 → type=unknown（警告）

Step 4. 存在性验证
        • Engine/Binaries/DotNET/UnrealBuildTool/UnrealBuildTool.exe → 能编
        • 缺 → 提示"需先 GenerateProjectFiles.bat + Build.bat 编 Engine"

Step 5. 列举所有候选
        扫 HKLM 所有版本 + HKCU 所有 GUID → 候选列表供用户挑
```

#### 数据结构

```python
@dataclass
class UEEngineInfo:
    path: str                    # "C:/Program Files/Epic Games/UE_5.3"
    version: str                 # "5.3.2"
    version_full: dict           # Build.version 原文 JSON
    type: Literal["launcher", "source_build", "unknown"]
    engine_association: str      # 来自 .uproject（或空）
    has_ubt: bool
    has_editor: bool
    warnings: list[str]
    source_guid: str | None      # 自编译时的 HKCU GUID
```

#### 新模块 API：`backend/engines/ue_resolver.py`

```python
def detect_installed_engines() -> list[UEEngineInfo]:
    """扫 HKLM + HKCU，返回所有已知引擎"""

def resolve_project_engine(uproject_path: str) -> UEEngineInfo | None:
    """为某 .uproject 找对应引擎（按 EngineAssociation）"""

def verify_engine(path: str) -> UEEngineInfo:
    """给定路径做完整验证"""
```

#### 边界情况

| 场景 | 处理 |
|---|---|
| 找到多个同版本 | 列出让用户选，默认取 HKLM 排序第一 |
| .uproject 版本与已装版本不一致 | 警告但允许继续（UBT 多数情况能跨版本编） |
| 自编译但没跑 GenerateProjectFiles | 识别出来并提示设置步骤 |
| EngineAssociation 为空字符串 | 特殊：可能是 BP-only 或 relative engine，扫相对路径 |
| Mac/Linux | v1 不支持，直接提示"暂只支持 Windows 自动检测，请手填路径" |

### 3.2 Part 2：UE 模板实例化（InstantiateUETemplateAction）

#### 核心思想

UE 本身自带完整模板系统（`<Engine>/Templates/TP_FirstPerson` 等），这些是 Epic 官方验证必过编译的 C++ 骨架。让 LLM 在这个基础上改，比从零写 UE C++ 可靠 10 倍。

#### Trait → Template 映射（`backend/templates/ue_templates.yaml`）

```yaml
mappings:
  - template: TP_FirstPerson   # C++ 变体
    match: {all_of: [engine:ue5, category:game], any_of: [genre:fps, genre:first_person]}
  - template: TP_ThirdPerson
    match: {all_of: [engine:ue5, category:game], any_of: [genre:third_person, genre:action]}
  - template: TP_TopDown
    match: {all_of: [engine:ue5, category:game], any_of: [genre:topdown, genre:rpg, genre:rts]}
  - template: TP_Vehicle
    match: {all_of: [engine:ue5, category:game], any_of: [genre:racing, genre:vehicle]}
  - template: TP_VR
    match: {all_of: [engine:ue5, category:game, platform:vr]}
  - template: TP_Blank         # 兜底
    match: {all_of: [engine:ue5]}
    priority: 0
```

TestFPS 的 `[engine:ue5, genre:fps, multiplayer]` 直接命中 `TP_FirstPerson`。

#### 实例化步骤

```
1. 拷贝 <Engine>/Templates/TP_FirstPerson/ → <project_repo>/
2. 文件名替换 TP_FirstPerson* → <ProjectName>*（含 .cpp/.h/.uproject/.Build.cs）
3. 文件内容替换：
   • class 前缀 ATP_FirstPersonCharacter → A<ProjectName>Character
   • include 路径
   • GENERATED_BODY 相关宏
4. .uproject：更新 Modules[].Name
5. .Build.cs：更新模块名
6. 清除模板元数据（Templates/Config/TemplateDefs.ini 不拷贝）
7. Git commit 到 feat/ue-framework 分支
```

#### 参考实现来源

Epic 的模板已经做过参数化（`Templates/Config/TemplateDefs.ini` + token 替换机制），我们可以：
- **A**：直接调 UE 自带的模板实例化工具（如果有 CLI）
- **B**：自己实现 rename 逻辑（参考 Epic 的做法）—— 推荐，控制力强

#### 可选：模板自定义 UI

v0.18 先不做，卡片里只给"选一个模板"，不做"定制模板参数"。

### 3.4 Part 4：UECompileCheckAction + Skill Pack

#### UECompileCheckAction

**命令**：
```
<UE>/Engine/Binaries/DotNET/UnrealBuildTool/UnrealBuildTool.exe 
    <TargetName> Win64 Development <.uproject>
```

默认超时 600s（SOP fragment 里已声明），可配。

**错误解析**：
- 正则抓 `<file>(<line>): error <code>: <msg>`
- 链接错误 `LNK2019 / LNK2001` 单独分类
- 警告 `warning C<code>` 不阻塞，但记录
- 解析失败降级：返回全 stdout + stderr

**结构化输出**：
```python
{
  "status": "success" | "error",
  "exit_code": int,
  "duration_ms": int,
  "errors": [
    {"file": "FPSCharacter.h", "line": 42, "code": "C2065", "msg": "未定义标识符"},
    ...
  ],
  "warnings": [...],
  "raw_output": "...",  # 原始完整输出（截断前 N KB）
  "products": ["Binaries/Win64/UnrealEditor-TestFPS.dll"],
}
```

#### Skill Pack 1: `unreal_cpp_dev`

`backend/skills/packs/unreal_cpp_dev/prompt.md` 覆盖：
- UCLASS/UFUNCTION/UPROPERTY 宏规则 + 修饰符（BlueprintCallable / EditAnywhere / ...）
- FString/FText/FName 的正确用法（别混淆）
- TSubclassOf / TSoftObjectPtr / TSharedPtr 引用规则
- Delegate 模式（DECLARE_DYNAMIC_MULTICAST_DELEGATE_*）
- Build.cs 的 PublicDependencyModuleNames / PrivateDependencyModuleNames
- GC 规则（UObject 派生才能被 GC 管）
- 命名约定（A 前缀 Actor / U 前缀 UObject / F 前缀 struct / E 前缀 enum）
- 常见坑：const correctness / forward declaration / PIMPL

`skills.json` 声明：
```json
"unreal-cpp-dev": {
  "inject_to": ["DevAgent"],
  "traits_match": {"any_of": ["engine:ue5", "engine:ue4"]},
  "paths": ["Source/**/*.h", "Source/**/*.cpp", "**/*.Build.cs"]
}
```

#### Skill Pack 2: `unreal_project`

覆盖：
- .uproject 结构（Modules / Plugins / EngineAssociation）
- Source/ 目录布局（一个模块一个子目录）
- Config/ 常用 .ini 文件（DefaultEngine.ini / DefaultGame.ini / DefaultInput.ini）
- Plugins/ 组织方式
- 模板起点："当前项目基于 TP_FirstPerson 模板，已有 XX，不要重复创建"

`inject_to: [ArchitectAgent, DevAgent]`，同 UE traits 匹配。

#### 编译失败反馈链（Reflexion 复用）

```
engine_compile 阶段 DevAgent.run_engine_compile → 返回 status=error + errors[]
    ↓
_handle_agent_result 走现有 reject_goto: development
    ↓
orchestrator 下次 poll 触发 DevAgent.fix_issues
    ↓
ReflectionAction 看 errors[] + 源文件 → 结构化反思 → 精准修改
    ↓
重新触发 engine_compile
```

**没有新机制**，完全复用 v0.17 已有的 reject_goto + Reflection。

## 5. 技术关键点

### 5.1 UBT 命令组装

```python
def build_ubt_command(engine_path, target_name, uproject_path, 
                     platform="Win64", config="Development"):
    ubt = Path(engine_path) / "Engine/Binaries/DotNET/UnrealBuildTool/UnrealBuildTool.exe"
    return [
        str(ubt),
        target_name,
        platform,
        config,
        str(uproject_path),
        "-waitmutex",           # 避免并发冲突
        "-NoHotReload",          # 避免误触发热重载
        "-progress",             # 输出进度行便于解析
    ]
```

### 5.2 模板实例化 rename 规则

| 替换范围 | 原 | 新（示例 project=TestFPS） |
|---|---|---|
| 文件路径 | `Source/TP_FirstPerson/` | `Source/TestFPS/` |
| 文件名 | `TP_FirstPersonCharacter.cpp` | `TestFPSCharacter.cpp` |
| class 名 | `ATP_FirstPersonCharacter` | `ATestFPSCharacter` |
| module 名 | `TP_FirstPerson_API` | `TESTFPS_API`（大写） |
| include 路径 | `#include "TP_FirstPersonCharacter.h"` | `#include "TestFPSCharacter.h"` |
| .uproject Modules | `{"Name": "TP_FirstPerson"}` | `{"Name": "TestFPS"}` |
| Build.cs | `public class TP_FirstPerson : ModuleRules` | `public class TestFPS : ModuleRules` |

**坑**：
- Module API 宏是全大写（`<MODULE>_API`）
- 某些地方 `FTP_FirstPerson*` 混杂，要按正则严格匹配"标识符边界"
- `.Build.cs` 里的 `PublicIncludePaths.AddRange(new string[] {...})` 路径串可能嵌模块名

### 5.3 SSE 进度事件

```json
// 事件类型: ue_template_progress
{
  "ticket_id": null,
  "project_id": "PRJ-xxx",
  "card_id": "ue-fw-abc123",    // 前端用这个 key 更新同一张卡
  "step": "copy_files",         // detect_engine | copy_files | rename | update_uproject | git_commit | done
  "status": "in_progress",      // pending | in_progress | done | error
  "progress": {"current": 12, "total": 23},
  "message": "拷贝 Source/ 目录..."
}
```

前端收到相同 `card_id` 直接替换卡片 DOM，而不是堆新消息。

## 6. 风险与缓解

| 风险 | 可能性 | 影响 | 缓解 |
|---|---|---|---|
| 开发机无 UE 环境 → E 阶段无法实跑 | 中 | 高 | A/A.3/A.5/C 都能 mock 测试；E 阶段明确要 UE 机器 |
| UBT 不同版本输出格式差异 | 中 | 中 | 第一版锁 UE5.3+；错误解析宽松正则 + 降级传全 stdout |
| .uproject 路径在 P4/Git 混用 | 低 | 中 | 配置里让用户填绝对路径或仓库相对路径，系统规范化 |
| 模板 rename 漏掉某些文件 | 中 | 高 | 编译时会暴露出来 + 自动 fix_issues 链；加完整单测 |
| 自编译 UE 用户路径奇怪 | 中 | 低 | 手填路径兜底 |
| 用户选了 UE5.4 但项目 .uproject 关联 5.3 | 中 | 低 | 警告但允许（多数情况 UBT 能跨编） |
| 首次编译慢（2-5 分钟）用户等不及 | 高 | 低 | 允许跳过 + SSE 进度可见 + 后台跑 |
| LLM 不理解模板边界，重复创建 TP 里已有的类 | 中 | 中 | `unreal_project` skill 里明确说"模板已有 XX，不要重复创建" |

## 9. 成功指标

### 9.1 功能指标

- 在装有 UE5.3+ 的 Windows 机器上，从"创建空 UE5 项目 + 对话'做个 FPS 游戏'"到"engine_compile 阶段通过"全流程不需要人工干预代码
- TestFPS 至少 1 个工单能走完 engine_compile 阶段（首次编译通过或经过 1-2 轮 fix_issues 后通过）
- 引擎检测能识别开发机上 ≥1 个 UE 版本

### 9.2 质量指标

- UECompileCheckAction 错误解析率 ≥ 90%（能从 UBT 输出里正确抽 90% 以上的错误到结构化列表）
- Reflection 首次修复通过率 ≥ 30%（v0.17 Web 项目约 40-60%，UE 第一版期望 30%）
- 模板实例化成功率 = 100%（rename 100% 可靠，编译不一定过但文件要 OK）

### 9.3 用户体验指标

- 从 [✓ 确认生成] 到"文件落地"的平均耗时 ≤ 10s
- 基线编译首次（首次启 UE 编译）≤ 5 分钟
- 增量编译（改几个文件后再编）≤ 1 分钟
