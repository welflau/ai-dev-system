## Phase 1: v0.13 — SOP 配置化 + 事件驱动

> 目标：Agent 流程从硬编码变为可配置，执行从轮询变为事件触发
> 预计：5 个工作日

### 节点 1.1: SOP 配置文件

**开发内容**：
- 新增 `backend/sop/` 目录
- 创建 `default_sop.yaml` 定义默认工作流
- 解析引擎：读取 YAML → 生成 transition_rules
- orchestrator 启动时加载 SOP 配置而非硬编码

**配置文件格式**：
```yaml
# sop/default_sop.yaml
name: "标准软件开发流程"
version: "1.0"

stages:
  - id: architecture
    agent: ArchitectAgent
    action: design_architecture
    trigger_on: pending           # 什么状态触发
    success_status: architecture_done
    
  - id: development
    agent: DevAgent
    action: develop
    trigger_on: architecture_done
    success_status: development_done
    
  - id: acceptance
    agent: ProductAgent
    action: acceptance_review
    trigger_on: development_done
    success_status: acceptance_passed
    reject_status: acceptance_rejected
    reject_goto: development      # 打回到哪个阶段
    
  - id: testing
    agent: TestAgent
    action: run_tests
    trigger_on: acceptance_passed
    success_status: testing_done
    reject_status: testing_failed
    reject_goto: development
    pass_threshold: 60            # 通过率阈值

  - id: rework
    agent: DevAgent
    action: rework
    trigger_on: acceptance_rejected
    success_status: development_done

  - id: fix_issues
    agent: DevAgent
    action: fix_issues
    trigger_on: testing_failed
    success_status: development_done
```

**预期效果**：
- ✅ 修改 YAML 即可调整开发流程，无需改代码
- ✅ 可以跳过某个阶段（如删除 architecture 阶段）
- ✅ 可以调整打回目标（如测试不通过打回架构而非开发）
- ✅ 每个项目可以有不同的 SOP

**验证标准**：
1. 删除 YAML 中的 architecture 阶段 → 工单从 pending 直接进入 DevAgent
2. 修改 pass_threshold 为 80 → 通过率 70% 的工单被打回
3. 新增一个自定义阶段 → 系统正常执行

### 节点 1.2: 内部事件总线

**开发内容**：
- 新增 `backend/event_bus.py`，基于 asyncio.Queue 的内部事件总线
- 工单状态变更时发布内部事件（区别于 SSE 前端事件）
- Agent 执行完成后直接触发下一个 Agent，不等轮询
- 保留轮询作为兜底（处理遗漏事件和僵尸工单）

**架构**：
```
当前（轮询驱动）：
  工单状态变更 → 写入数据库 → 等 10 秒 → 轮询扫描 → 触发 Agent

改进后（事件驱动 + 轮询兜底）：
  工单状态变更 → 写入数据库 → 发布内部事件 → 立即触发 Agent
                                              ↑
                              轮询每 30 秒兜底扫描（处理遗漏）
```

**预期效果**：
- ✅ 工单流转延迟从 10 秒降到 < 1 秒
- ✅ 一个需求从提交到全部工单完成的总时间缩短 30%+
- ✅ 轮询间隔可以放宽到 30 秒（降低 CPU 开销）
- ✅ 实时日志中看到工单状态连续变更，无等待间隔

**验证标准**：
1. 提交需求 → 拆单后工单立即开始架构设计（无 10 秒等待）
2. DevAgent 完成 → ProductAgent 立即接手验收（< 1 秒）
3. 杀掉事件总线进程 → 轮询兜底仍能正常调度

## Phase 2: v0.14 — Agent 能力层重构

> 目标：Agent 的 Role 和 Action 分离，能力可组合、可扩展
> 预计：5 个工作日

### 节点 2.1: Action 层抽离

**开发内容**：
- 新增 `backend/actions/` 目录
- 每个 Action 独立一个类：`DesignArchitecture`, `WriteCode`, `RunTests`, `AcceptanceReview` 等
- Agent（Role）只持有 Action 列表，不包含业务逻辑
- Action 接口统一：`async def run(context: Dict) -> ActionResult`

**目录结构**：
```
backend/actions/
  ├── base.py              # ActionBase 抽象类
  ├── design_architecture.py
  ├── write_code.py
  ├── self_test.py
  ├── acceptance_review.py
  ├── run_tests.py
  ├── code_review.py
  └── deploy.py
```

**Agent 定义变为**：
```python
class DevAgent(Role):
    actions = [WriteCode, SelfTest, GenerateDevNotes]
    
class FullStackAgent(Role):   # 新：全栈 Agent
    actions = [DesignArchitecture, WriteCode, SelfTest, RunTests]
```

**预期效果**：
- ✅ 新增 Agent 只需组合已有 Action，无需从零写代码
- ✅ 同一个 Action 可以被多个 Agent 复用
- ✅ 可以动态给 Agent 添加/移除能力
- ✅ Action 可以独立测试和调试

**验证标准**：
1. 创建一个 FullStackAgent（组合架构+开发+测试）→ 能正常执行
2. 给 DevAgent 动态添加 RunTests Action → 开发完自动跑测试
3. 单独测试 WriteCode Action → 不依赖 Agent 上下文

---

### 节点 2.2: Agent Memory 统一接口

**开发内容**：
- 新增 `backend/memory.py`
- AgentMemory 类封装对 artifacts/ticket_logs/llm_conversations 的查询
- 提供 `get_memories(k)`, `search(keyword)`, `get_context()` 等接口
- Agent 通过 Memory 接口获取历史信息，而非直接查数据库

**接口设计**：
```python
class AgentMemory:
    async def get_recent(self, k: int = 5) -> List[Memory]:
        """获取最近 k 条记忆（跨 artifacts + logs + conversations）"""
    
    async def get_by_ticket(self, ticket_id: str) -> List[Memory]:
        """获取某工单的全部记忆"""
    
    async def get_code_context(self, project_id: str) -> Dict:
        """获取项目代码上下文（已有文件 + 入口代码）"""
    
    async def search(self, query: str) -> List[Memory]:
        """关键词搜索历史记忆"""
```

**预期效果**：
- ✅ Agent 不再直接操作数据库，代码更清晰
- ✅ `_build_context` 和 `_collect_existing_code` 统一到 Memory 层
- ✅ 未来可接入向量数据库做语义搜索
- ✅ Memory 可以跨工单共享（同需求下的工单互相感知）

**验证标准**：
1. DevAgent 通过 Memory 获取已有代码（替代 _collect_existing_code）
2. ArchitectAgent 通过 Memory 获取同需求其他工单进展
3. search("登录") → 返回所有包含"登录"的历史产物

---

### 节点 2.3: Agent 注册中心

**开发内容**：
- orchestrator 不再硬编码 Agent 实例化
- Agent 通过装饰器自动注册：
  ```python
  @register_agent("DevAgent")
  class DevAgent(Role):
      actions = [WriteCode, SelfTest]
  ```
- 前端 Agent 配置页可以启用/禁用 Agent
- 支持自定义 Agent（用户 Python 脚本放入 `agents/custom/` 自动加载）

**预期效果**：
- ✅ 新增 Agent 只需写一个 Python 文件 + 装饰器
- ✅ 不需要修改 orchestrator 代码
- ✅ 前端可以开关 Agent（如禁用 ReviewAgent 跳过审查）
- ✅ 为未来插件市场打基础

**验证标准**：
1. 在 `agents/custom/` 放入 `security_agent.py` → 系统自动加载
2. 前端禁用 ReviewAgent → 工单流转跳过审查阶段
3. API `/api/agents` 返回所有注册的 Agent（含自定义）

## Phase 3: v0.15 — 智能增强

> 目标：多模型支持、并发调度、竞品分析
> 预计：6 个工作日

### 节点 3.1: 多 LLM 支持

**开发内容**：
- llm_client 支持多个 LLM 后端配置
- 按 Agent 指定不同模型（如架构用 Claude Opus，开发用 Sonnet，审查用 Haiku）
- 支持 Ollama 本地模型
- 模型降级链：主模型超时 → 备用模型 → 本地模型 → 规则引擎

**配置**：
```yaml
# .env 或前端配置
llm_providers:
  primary:
    type: anthropic
    model: claude-sonnet-4
    for: [DevAgent, ArchitectAgent, ProductAgent]
  secondary:
    type: ollama
    model: codellama:13b
    for: [ReviewAgent, TestAgent]
  fallback:
    type: ollama
    model: llama3:8b
```

**预期效果**：
- ✅ API 成本降低 50%+（低优先级任务用本地模型）
- ✅ 即使云端 API 挂了，本地模型仍能运行基础流程
- ✅ 不同 Agent 用最适合的模型（代码生成用大模型，审查用小模型）

**验证标准**：
1. DevAgent 使用 Claude Sonnet → ReviewAgent 使用 Ollama → 各自正常工作
2. 主模型超时 → 自动切换备用模型 → 日志中可见切换记录
3. 前端 Agent 配置页可以调整每个 Agent 的模型

---

### 节点 3.2: 并发调度优化

**开发内容**：
- 无依赖的工单可以并行执行（同一需求下多个独立工单）
- Agent 并发池：同一个 Agent 类型可以同时处理多个工单
- 并发控制：限制最大并发数（如最多 3 个 DevAgent 同时工作）
- 前端 Agent 监控面板展示并发状态

**预期效果**：
- ✅ 一个需求拆出 6 个工单，3 个无依赖的工单同时开发
- ✅ 总完成时间缩短 40~60%
- ✅ Agent 监控面板显示"DevAgent: 3/3 busy"
- ✅ 不会因为并发导致 Git 冲突（文件写入加锁）

**验证标准**：
1. 提交需求拆出 5 个工单，其中 3 个无依赖 → 3 个同时开始架构设计
2. 设置 max_concurrency=2 → 最多 2 个工单并行，第 3 个排队
3. 并行工单写入同一个 Git 仓库 → 无冲突（串行 commit）

---

### 节点 3.3: 竞品分析 Agent

**开发内容**：
- 新增 ResearchAgent（竞品分析 + 技术调研）
- 在需求分析阶段，先做市场调研再拆单
- 支持联网搜索（可选）或基于知识库分析
- 输出竞品分析报告作为 artifacts

**SOP 配置**：
```yaml
stages:
  - id: research
    agent: ResearchAgent
    action: competitive_analysis
    trigger_on: submitted
    success_status: researched
    optional: true              # 可跳过
    
  - id: decompose
    agent: ProductAgent
    action: analyze_and_decompose
    trigger_on: researched
    # ...
```

**预期效果**：
- ✅ 需求分析前先了解同类产品怎么做的
- ✅ 拆单更合理（参考竞品功能点）
- ✅ 生成竞品分析报告存入仓库
- ✅ 可选阶段，不需要时可跳过

**验证标准**：
1. 提交"开发一个 Todo 应用" → ResearchAgent 分析 Todoist/TickTick 等
2. 竞品报告出现在 artifacts 中
3. 在 SOP 中设置 `optional: true` + 前端跳过 → 直接进入拆单

## Phase 4: v0.16 — 平台化

> 目标：插件市场、多项目协作、数据分析
> 预计：10+ 工作日

### 节点 4.1: 插件市场

**开发内容**：
- Agent 插件打包格式（含 metadata.yaml + Python 代码 + 依赖声明）
- 插件仓库（本地目录 `plugins/` 或远程 Git URL）
- 前端插件管理页：安装/卸载/启用/禁用
- 安全沙箱：插件在受限环境中运行

**插件结构**：
```
plugins/
  security-scanner/
    ├── metadata.yaml    # 名称、版本、依赖
    ├── agent.py         # Agent 定义
    ├── actions/         # Action 实现
    └── requirements.txt # Python 依赖
```

**预期效果**：
- ✅ 第三方开发者可以编写自定义 Agent 插件
- ✅ 安装插件如同安装 VS Code 扩展一样简单
- ✅ 社区可以贡献"安全扫描 Agent"、"性能测试 Agent"等
- ✅ 插件间通过 SOP 配置串联

---

### 节点 4.2: 多项目协作

**开发内容**：
- 跨项目依赖声明（项目 A 的 API 被项目 B 依赖）
- 共享组件库（多个项目复用同一组 UI 组件）
- 全局 Dashboard（一个页面看所有项目进度）
- 跨项目搜索（搜索所有项目的代码和文档）

**预期效果**：
- ✅ 微服务架构下，前端项目依赖后端项目 API → 自动检测接口变更
- ✅ 共享组件更新后 → 依赖项目自动收到通知
- ✅ 管理层一个 Dashboard 看所有项目状态

### 节点 4.3: Data Interpreter

**开发内容**：
- 参考 MetaGPT 的 Data Interpreter 能力
- 新增 DataAgent — 数据分析、可视化、报告生成
- 支持 CSV/Excel 导入 → 自动生成分析代码 → 执行 → 输出图表
- 集成到项目统计面板

**预期效果**：
- ✅ 上传数据文件 → AI 自动分析趋势、生成图表
- ✅ 项目数据分析：工单完成趋势、Agent 效率分析、代码量增长
- ✅ 自然语言查询："最近一周哪个 Agent 效率最低？"
