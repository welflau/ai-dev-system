## 一、背景与定位

### 为什么需要策划知识库

策划 Agent 生成 PRD 的质量，取决于它能调取多少相关领域知识。当前系统的知识库（`knowledge_index`）只有：
- 项目自身的历史文档
- 系统 dev-notes
- 已完成工单的经验总结

**完全缺失**的是：某类游戏/产品的通用设计知识——比如 RPG 战斗系统有哪些设计模式、Roguelike 的 Meta 成长怎么设计、小程序的购物流程行业标准是什么。

没有领域知识，策划 Agent 只能"靠直觉"写 PRD，质量不稳定。

### 两类知识库的定位区分

| | 项目知识库（现有）| 策划知识库（新建）|
|---|---|---|
| 内容 | 本项目踩过的坑、历史 PRD | 行业通用设计知识、设计模式 |
| 范围 | 项目私有 | **全局共享，跨所有项目** |
| 来源 | Agent 产出 + 用户上传 | 人工整理 + 行业资料 |
| 更新频率 | 随项目进展持续更新 | 相对稳定，定期扩充 |
| 查询时机 | Agent 执行任何阶段 | **策划 Agent 生成 PRD 前重点查询** |

## 二、存储设计

### 物理存储：独立 Git 仓库

与美术资产库（`G_ArtRes`）保持一致的模式——**独立 Git 仓库，存放在主项目外部**。

```
F:/A_Works/
├── ai-dev-system/          ← 主系统代码
├── G_ArtRes/               ← 全局美术资产库
└── G_DesignKnowledge/            ← 全局知识库（新建）
    ├── planning/           ← 策划知识（本文档范围）
    ├── design/             ← UX/交互设计知识（后续扩展）
    └── engineering/        ← 工程经验知识（后续扩展）
```

**独立仓库的好处**：
- 知识内容独立版本管理，不污染主项目 Git 历史
- 多个 ai-dev-system 实例可以共享同一套知识
- 团队成员可以单独维护知识库内容，不需要接触系统代码

### .env 配置

```env
# 全局知识库配置
GLOBAL_KNOWLEDGE_REPO_URL=https://github.com/welflau/G_DesignKnowledge.git
GLOBAL_KNOWLEDGE_LOCAL_PATH=F:/A_Works/G_DesignKnowledge
GLOBAL_KNOWLEDGE_AUTO_PUSH=true
```

## 四、数据库设计

### 新增 planning_knowledge 表

与项目知识库（`knowledge_index`）独立，避免互相污染：

```sql
CREATE TABLE IF NOT EXISTS planning_knowledge (
    id           TEXT PRIMARY KEY,       -- PKB-xxxxxxxx
    filename     TEXT NOT NULL,          -- 相对于 G_DesignKnowledge/ 的路径
    title        TEXT NOT NULL,          -- 文档标题
    category     TEXT NOT NULL,          -- games/apps/universal
    subcategory  TEXT,                   -- rpg/fps/ecommerce 等
    tags         TEXT,                   -- JSON 数组，如 ["战斗","技能","RPG"]
    content      TEXT NOT NULL,          -- 文档全文
    summary      TEXT,                   -- 200字摘要（供快速展示）
    source       TEXT,                   -- manual/gdc/book/research
    updated_at   TEXT NOT NULL
);

-- FTS5 全文检索
CREATE VIRTUAL TABLE IF NOT EXISTS planning_knowledge_fts
USING fts5(
    title,
    content,
    tags UNINDEXED,
    content='planning_knowledge',
    content_rowid='rowid',
    tokenize='trigram'
);
```

### 与现有 knowledge_index 的关系

```
knowledge_index         ← 项目级知识（项目文档/dev-notes/历史工单）
planning_knowledge      ← 领域通用知识（跨项目，本文档范围）

策划 Agent 查询时，同时检索两个来源，综合后注入 prompt
```

## 五、检索与注入机制

### 策划 Agent 查询时机

```
ProductAgent 完成需求分析
    ↓
策划 Agent 开始生成 PRD 前
    ↓
① SearchPlanningKnowledgeAction
   输入：需求标题 + 项目 traits（判断游戏类型/产品类型）
   → 查 planning_knowledge_fts，返回最相关的 2-3 篇文档摘要
    ↓
② SearchKnowledgeAction（现有）
   → 查 knowledge_fts，返回本项目历史相关 PRD
    ↓
③ 综合两个来源 → 注入策划 Agent prompt
   context["domain_knowledge"] = 领域知识摘要
   context["project_knowledge"] = 项目历史经验
    ↓
策划 Agent 生成 PRD（有领域知识支撑）
```

### 查询路由逻辑

系统根据项目 traits 自动判断查哪个分类：

```python
def get_planning_search_path(traits: List[str]) -> str:
    if "category:game" in traits:
        if "engine:ue5" in traits or "genre:fps" in traits:
            return "planning/games/fps"
        elif "genre:rpg" in traits:
            return "planning/games/rpg"
        elif "genre:roguelike" in traits:
            return "planning/games/roguelike"
        else:
            return "planning/games"        # 通用游戏知识
    elif "platform:wechat" in traits:
        return "planning/apps/miniapp"
    elif "platform:web" in traits:
        return "planning/apps"
    else:
        return "planning/universal"        # 兜底：通用知识
```

## 六、内容来源与初始化策略

### 内容质量原则

- **精而不泛**：每篇文档聚焦一个具体系统，不写大而全的概述
- **可操作**：包含具体的设计决策示例和参数参考，不只是原则描述
- **有取舍**：明确说明哪些方案适合什么场景，哪些是反模式

### 初始内容来源

| 来源 | 获取方式 | 质量 | 工作量 |
|---|---|---|---|
| **团队经验沉淀** | 把团队内部已有的设计经验整理成文档 | ⭐⭐⭐⭐⭐ | 中 |
| **GDC 演讲整理** | AI 辅助整理 GDC Vault 中的演讲要点 | ⭐⭐⭐⭐ | 中 |
| **开源 GDD 参考** | GitHub 上的游戏设计文档、设计模式整理 | ⭐⭐⭐ | 低 |
| **项目经验反哺** | 每个项目完成后，把有价值的设计决策沉淀到领域库 | ⭐⭐⭐⭐⭐ | 低（自动积累）|
| **竞品反拆**（KM 第三篇）| AI 浏览器自动分析竞品，产出结构化设计文档 | ⭐⭐⭐⭐ | 自动化 |

### 最小可用集（优先整理）

先整理这 10 篇，策划 Agent 就能受益：

1. `universal/PRD模板.md` — 标准 PRD 结构
2. `universal/验收标准写法规范.md` — 量化验收 50+ 示例
3. `universal/新手引导设计.md` — 通用 Onboarding 设计
4. `games/rpg/战斗系统设计.md` — RPG 最常见系统
5. `games/roguelike/随机生成设计.md` — Roguelike 核心
6. `games/fps/武器系统设计.md` — FPS 武器平衡
7. `games/casual/超休闲设计原则.md` — 超休闲通用
8. `apps/ecommerce/购物流程设计.md` — 电商核心流程
9. `apps/social/Feed设计原则.md` — 社交产品核心
10. `apps/miniapp/小程序设计规范.md` — 微信小程序

## 七、API 端点

```
# 检索策划知识库
GET  /api/planning-knowledge
     ?q=<关键词>
     &category=games|apps|universal
     &subcategory=rpg|fps|ecommerce
     &tags=<逗号分隔>
     &limit=5
     → [{ id, title, category, summary, relevance_score }]

# 获取文档全文
GET  /api/planning-knowledge/{id}
     → { id, title, content, tags, source, updated_at }

# 同步远端知识库
POST /api/planning-knowledge/sync
     → { pulled: N, updated: N, new: N }

# 统计
GET  /api/planning-knowledge/stats
     → { total, by_category, recently_updated }
```

## 八、与 KM Insight 系统的关系

KM 文章中的 Insight 系统（结构化经验 + 置信度 + 自动匹配）和本文档的策划知识库是**互补的**：

| | 策划知识库（本文档）| KM Insight 系统 |
|---|---|---|
| 内容类型 | 行业通用设计知识（外部输入）| 本系统积累的实战经验（内部沉淀）|
| 来源 | 人工整理 + 行业资料 | Agent 执行过程中自动沉淀 |
| 结构 | Markdown 文档，全文检索 | 结构化条目，置信度评分 |
| 查询方式 | 关键词 + 类型过滤 | 自动匹配 + 置信度阈值 |

**长期理想状态**：
```
策划 Agent 查询
    ↓
① 策划知识库（领域通用知识）→ "RPG 战斗系统通常这么设计..."
② 项目知识库（本项目历史）→ "上次做类似系统时遇到了这个问题..."
③ Insight 系统（实战经验）→ "高置信度：验收标准必须包含边界条件"
    ↓
综合三个来源 → 最高质量的 PRD
```

## 九、实施计划

| 阶段 | 内容 | 工期 | 完成后效果 |
|---|---|---|---|
| **Phase 1** | 建立 `G_DesignKnowledge/` 仓库 + `planning_knowledge` 表 + FTS5 索引 | 0.5 天 | 基础设施就绪 |
| **Phase 2** | 编写最小可用集 10 篇文档，导入知识库 | 2 天 | 策划 Agent 有基础领域知识 |
| **Phase 3** | `SearchPlanningKnowledgeAction` + traits 路由 + 注入策划 Agent | 1 天 | 策划 Agent 自动检索领域知识 |
| **Phase 4** | 项目完成后自动反哺机制（好的设计决策沉淀到领域库）| 1 天 | 知识自动积累 |
| **Phase 5** | 竞品反拆管线（AI 浏览器自动分析竞品入库）| 3 天 | 持续更新行业知识 |

**总计约 7.5 天**（Phase 1-3 是核心，4-5 天可独立上线）
