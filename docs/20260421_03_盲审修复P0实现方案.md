# 盲审修复 P0 — 实现方案

> 日期: 2026-04-21 | 对应 TODO.md 两项 P0：ProductAgent 拆单不看代码 + ReviewAgent 完全盲审

---

## 一、背景

### 1.1 TODO.md 里挂了半年的两个 P0

```
| P0 | ProductAgent 拆单 | 不看已有代码就拆单 | 注入 existing_files/code，拆单时参考已有架构 |
| P0 | ReviewAgent       | 完全盲审，不读代码 | 读取实际代码内容 + ActionNode + SOP 配置       |
```

### 1.2 两个问题的真实情况（探索后）

**ProductAgent 拆单**：
- `DecomposeAction.run()` 的 prompt 模板 **早已经有** `existing_files` / `existing_code` / `project_summary` 槽位（`actions/decompose.py:31-65`）
- `ProductAgent._analyze_and_decompose()` 也会在有代码时先跑 `SummarizeCodeAction` 补 `project_summary`（`agents/product.py:33-37`）
- **但是**：`orchestrator.py:530` 触发拆单时只传了 3 个字段：
  ```python
  result = await agent.execute("analyze_and_decompose", {
      "title": requirement["title"],
      "description": requirement["description"],
      "priority": requirement["priority"],
  })
  ```
  没走 `_build_context()`，没注入 `existing_files` / `existing_code`——**所有上游配好的槽位都是空的**。
- 实测表现：已经有现成文件（比如 `index.html`）的项目，拆单时 LLM 会拆出"创建 index.html"这种和已有架构冲突的工单。

**ReviewAgent**：
- `CodeReviewAction` 本身功能**完全正确**：读 `dev_result.files` + `existing_code`，用 ActionNode，读 SOP `max_code_chars` 配置。不是盲审。
- 问题是：`ReviewAgent` 类只是个空壳容器（`agents/review.py:11-19`），orchestrator 从未直接调用它
- `CodeReviewAction` 是**作为 `TestAgent` 的 Phase 2** 在跑（`agents/test.py:72-78`）
- 这让代码审查变成"测试过程的副作用"而非一等 SOP 阶段，也让 `ReviewAgent` 类处于"写了不用"状态

### 1.3 目标

两个问题合并修：
1. **拆单注入代码上下文** — 最小改动，让 DecomposeAction 用上已有槽位
2. **ReviewAgent 抽成独立 SOP 阶段** — `dev → code_review → acceptance → testing`，让代码审查成为可见的一等公民

## 二、P0.1 ProductAgent 拆单注入代码上下文

### 2.1 单点改动

`orchestrator.py:530` 附近，触发 decompose 前加一段读代码上下文的逻辑：

```python
# 读取已有代码上下文（避免盲拆单）
from memory import AgentMemory
try:
    code_ctx = await AgentMemory(project_id).get_code_context()
    existing_files = code_ctx.get("file_list", [])
    existing_code = code_ctx.get("code", {})
    if existing_files:
        logger.info("📂 decompose 读到代码上下文: %d 文件, %d 片段",
                    len(existing_files), len(existing_code))
except Exception as e:
    logger.warning("decompose 读代码上下文失败: %s", e)
    existing_files, existing_code = [], {}

result = await agent.execute("analyze_and_decompose", {
    "title": requirement["title"],
    "description": requirement["description"],
    "priority": requirement["priority"],
    "project_id": project_id,
    "existing_files": existing_files,
    "existing_code": existing_code,
})
```

`AgentMemory.get_code_context()` 已有实现（`memory.py:83-129`）——复用而非新写。

### 2.2 整条链路

```
orchestrator 触发
  └─ AgentMemory.get_code_context() → {file_list, code}
  └─ ProductAgent.execute("analyze_and_decompose", ctx_with_files)
       └─ _analyze_and_decompose: existing_files 非空 → run SummarizeCodeAction
            └─ 产出 project_summary，写回 context
       └─ run_action("decompose", ctx)
            └─ DecomposeAction.run:
                 - 读 project_summary → 填 "项目现有功能" 段
                 - 读 existing_files → 填 "项目文件" 段
                 - LLM 看到完整上下文 → 拆单时避开已有文件
```

## 三、P0.2 ReviewAgent 独立 SOP 阶段

### 3.1 新 SOP 流程

```
旧:  pending → architecture → development → acceptance → testing → done
新:  pending → architecture → development → code_review → acceptance → testing → done
                                                ↑ 新增
```

**review 通过标准（保守、增量安全）**：
- ReviewAgent 总是输出 `review_passed`（不阻塞后续）
- 如果 `score < sop_config.pass_score`（默认 7），写 warning 级别日志 + artifact，但不退回 rework
- 未来可在 SOP config 加 `block_on_low_score: true` 开启硬门控

这样**旧 reject 路径完全不动**（acceptance_rejected / testing_failed），只在 dev 完成后多一个代码审查 artifact 产出 + 可见化审查结论。

### 3.2 具体改动

#### 3.2.1 `backend/models.py`

新增两个 TicketStatus 枚举：

```python
REVIEW_IN_PROGRESS = "review_in_progress"
REVIEW_PASSED = "review_passed"
```

`BOARD_COLUMNS["development"]` 加入这两个状态（审查归在"开发中"看板列，UI 不需改）。
`STATUS_LABELS` 加对应中文名"审查中"/"审查通过"。

#### 3.2.2 `backend/sop/default_sop.yaml`

在 `development` 和 `acceptance` 之间插入新 stage，同时调整 `acceptance.trigger_on` 和 `testing.config.code_review`：

```yaml
- id: code_review
  name: 代码审查
  agent: ReviewAgent
  action: code_review
  trigger_on: development_done
  success_status: review_passed
  description: ReviewAgent 读取实际代码做审查（非盲审）
  config:
    max_code_chars: 3000
    pass_score: 7
    block_on_low_score: false

- id: acceptance
  # trigger_on: development_done  → review_passed
  trigger_on: review_passed
  ...

- id: testing
  config:
    # code_review: true → false（已独立成阶段）
    code_review: false
    ...
```

`pipeline_view.development` 组加入 `code_review` stage 和 `review_in_progress` running status，前端 UI 保持 3 大步不变。

#### 3.2.3 `backend/agents/review.py`

加 `execute()` 路由：

```python
async def execute(self, task_name, context):
    if task_name == "code_review":
        result = await self.run_action("code_review", context)
        result["status"] = "review_passed"  # 不阻塞；分数由 orchestrator 判是否 warning
        return result
    return {"status": "error", "message": f"ReviewAgent 未知任务: {task_name}"}
```

#### 3.2.4 `backend/orchestrator.py::_handle_agent_result`

加 `elif agent_name == "ReviewAgent"` 分支：

```python
elif agent_name == "ReviewAgent":
    new_status = TicketStatus.REVIEW_PASSED.value
    score = result.get("score", 7)
    pass_score = (ticket.get("_sop_config") or {}).get("pass_score", 7)

    await db.update("tickets", {"status": new_status, ...})

    # 低分只记 warning（block_on_low_score=false 默认）
    log_level = "warning" if score < pass_score else "info"
    await self._log(..., "code_review", ..., log_level, detail_data={"score": score, ...})

    # artifact 产出
    await db.insert("artifacts", {"type": "code_review", ..., "content": result_json})
```

#### 3.2.5 `backend/agents/test.py`

`_code_review` 调用包在 `sop_config.code_review` flag 后，默认关闭（SOP 已设 false）：

```python
if sop_cfg.get("code_review", False):
    review = await self._code_review(context)
    ...
else:
    review = {"score": 0, "issues": [], "detail": "(已独立为 ReviewAgent.code_review 阶段)"}
```

#### 3.2.6 `frontend/app.js`

- `allStatuses` 两处加 review_in_progress / review_passed 选项
- `STATUS_LABELS` / `statusLabels` 三处加对应中文
- 看板列表不变（review_* 归并到"开发中"列）

## 五、风险与降级

| 风险 | 缓解 |
|---|---|
| 新 SOP 阶段打破 in-flight 工单 | 只影响 development_done 之后的新流转；已过 acceptance 的工单不受影响 |
| 状态枚举变动影响前端 status filter | 前端 allStatuses + STATUS_LABELS 三处同步更新 |
| 每工单多一次 LLM 调用（成本） | CodeReview prompt 约 3000 字符上限 + 1500 tokens 输出，成本约 +$0.02/工单；换来可见化审查 artifact |
| 低分频繁 warning 噪音 | 默认 block_on_low_score=false，只记日志不阻塞 |
| TestAgent 原 review 分数影响 pass_rate | 移除后 TestAgent 计算改为只含 static + functional + pytest；pass_rate 可能略变 |
