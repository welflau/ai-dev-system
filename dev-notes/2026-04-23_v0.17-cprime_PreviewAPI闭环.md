## 一、UI 重构：系统设置 Modal（commit `210a470`）

### 1.1 问题背景

v0.17 Phase A.6 匆忙把 Traits 查看器 UI 塞在了「项目 → 设置 → Agent / 扩展」底下的新子 Tab。用户提醒：

> Traits 配置界面放到外面来，在项目列表这个页面添加一个系统设置按钮，弹出系统设置窗，将全局知识库设置集成到这里来。

确实错位：Traits (taxonomy + presets) 和全局知识库**都跟具体项目无关**，是**系统级**配置。放在项目级设置下会让用户每次想改都得先进一个项目。

### 1.2 改动

```
项目列表页 page-header-actions:
  之前: [📚 全局知识库]  [📥 导入现有工程]  [+ 新建项目]
  现在: [⚙️ 系统设置]   [📥 导入现有工程]  [+ 新建项目]
```

`showSystemSettingsModal(initialTab='knowledge')` —— 带 2 Tab 的 modal：
- **📚 全局知识库** —— 从原独立 modal 集成进来（`_loadGlobalKnowledgeIntoModal` 复用）
- **🏷 Traits 配置** —— 从项目级 Agent/扩展 子 Tab 移到这里（taxonomy / presets / 匹配测试）

向后兼容：旧 `showGlobalKnowledgeModal()` 降级为 `showSystemSettingsModal('knowledge')` 的薄 alias，保留作未来快捷键可能的入口。

移除：项目级 Agent/扩展 下的「🏷 Traits」子 Tab（~45 行 HTML），`switchAgentConfigTab('traits')` 分支。

### 1.3 预留扩展

`showSystemSettingsModal` 的 Tab 结构可继续加：
- 全局 MCP 启用/禁用开关
- LLM 配置（全局 API key、模型降级链）
- 系统环境变量
- 全局 skill.json 编辑（未来 Phase E 可能用到）

## 二、Phase C' — Preview Assembly API（commit `89ee8c6`）

### 2.1 设计

`POST /api/projects/preview-assembly`

**入参**：
```json
{
  "traits": ["platform:desktop", "category:game", "engine:ue5", "lang:cpp"],
  "ticket_type": "feature",
  "include_rules": true
}
```

**返回**：
```json
{
  "traits": [...],
  "ticket_type": "feature",
  "effective_config": {
    "sop": {
      "stages": [{id, name, agent, action, trigger_on, success_status, description}],
      "fragments_activated": ["asset_gen", "engine_compile", "play_test"]
    },
    "agents": {
      "active": ["ProductAgent", "ArchitectAgent", "DevAgent", ...],
      "excluded": [{agent, reason}]
    },
    "skills_by_agent": {
      "DevAgent": [{id, name, priority, matched}],
      "TestAgent": [...]
    },
    "rules": [{id, description, alwaysApply}],
    "mcps": [{name, enabled, status, applicable_to_traits, tools_count}]
  },
  "warnings": ["缺 platform:* ...", "category:game 但缺 engine:*"],
  "suggestions": [
    {"hint": "若需国际化，可加 i18n，SOP 会自动插入 i18n_check 阶段", "trait": "i18n"}
  ]
}
```

**实现**：底层复用已就位的 `sop_loader.compose()` + `skill_loader.get_skills_for_agent()` + `skill_loader.get_rules_for_context()` + `orchestrator.agents` + `mcp_client.get_status()`，拼装成统一 preview payload。**纯只读，不落库不触发任何副作用**。

### 2.2 Warning 规则（~10 行逻辑）

- 缺 `platform:*` → warning
- 缺 `category:*` → warning
- `category:game` 但缺 `engine:*` → warning（SOP 会没 engine_compile 阶段）
- SOP 阶段数为 0 → warning

### 2.3 Suggestion 规则（鼓励补充 trait）

- `category:game` 且 `has_engine` 但无 `multiplayer:true` → 提示"若多人可加 multiplayer"
- `platform:web` 但无 `i18n` → 提示"若国际化可加 i18n"

Suggestion 不 block 用户，只是 hint。未来可扩展更多规则（如"production 阶段需要 compliance"等）。

### 2.4 前端 confirm_project 卡片集成

**卡片新增**：
- `data-traits` 属性（JSON encoded traits 数组）
- `data-preset-id` 属性
- 正文显示 **Traits chips**（紫色 border，跟 Traits Tab 色调一致）
- 正文显示 **Preset badge**（如 `📦 Preset: ue5-game`）
- **🔮 预计组装** 小框，卡片插 DOM 后 `setTimeout(() => loadProjectAssemblyPreview(cardId, traits), 50)` 异步填充

**预计组装小框内容**：
```
🔮 预计组装
SOP 流程（9 阶段）：架构设计 → 美术资产生成 → 代码开发 →
                   引擎编译验证 → 游戏玩法测试 → 代码审查 →
                   产品验收 → 部署
激活 fragments: asset_gen, engine_compile, play_test
Skill 注入：DevAgent→react-dev, DevAgent→git-workflow, TestAgent→playwright-e2e
MCP: filesystem, fetch, git
⚠️（若有）缺 engine:*
💡 若需国际化，可加 i18n ...
```

**点「确认创建」**时把 `traits` + `preset_id` 透传到后端，`CreateProjectAction` 写入 DB。

### 2.5 容错

- `loadProjectAssemblyPreview` 静默失败（network error 或 API 500）→ 只在预览框显示 "预览不可用"，不阻塞确认按钮
- 保证 preview API 挂掉时用户依然能正常建项目

## 五、垂直切片完整闭环

v0.17 到 Phase C' 为止的完整链路（都已跑通）：

```
用户在 AI 助手说："做个 Godot 多人游戏 git url ..."
    ↓
ChatAssistantAgent.chat_global (Phase A.5 system prompt 含反问规则)
    ↓
LLM 识别 → 调 confirm_project tool
    ↓ traits=[platform:desktop, category:game, engine:godot4, multiplayer:true]
    ↓ preset_id=godot-game
    ↓
前端 appendChatBubble 渲染 chat-confirm-project-card
    ↓
setTimeout → loadProjectAssemblyPreview(cardId, traits)  (Phase C')
    ↓
POST /api/projects/preview-assembly
    ↓
sop_loader.compose(traits, ticket_type)  (Phase C.1)
  + skill_loader.get_skills_for_agent(agent, traits)  (Phase B.1 三层过滤)
  + orchestrator.agents 按 BaseAgent.available_for_traits 过滤  (Phase C.2)
  + mcp_client.get_status()
    ↓
返回 effective_config + warnings + suggestions
    ↓
卡片底部填充"🔮 预计组装"小框：SOP 9 阶段 + Skills 注入表
    ↓
用户点 ✅ 确认创建
    ↓
POST /api/chat/confirm-create-project （含 traits + preset_id）
    ↓
CreateProjectAction.run()  (Phase A.4)
  → Git clone + INSERT projects (含 traits/traits_confidence/preset_id)
    ↓
跳转项目详情页
```

**现在建一个 Godot 多人游戏项目就能立刻跑上面这 9 阶段 SOP**（从 `orchestrator._load_sop` 改用 `compose_sop` 的工作会在 Phase D 或专门的集成 PR 里做，目前 SOP 改动尚未接入 orchestrator 实际调度 —— 是 Phase D 的前置项）。

## 六、遗留 / 下一步

### 6.1 SOP 组合器尚未接入 orchestrator 实际调度

当前 `compose_sop` 只在 Preview API 里被调用展示。**orchestrator 实际跑工单时还是加载 `default_sop.yaml`**。要让游戏项目真跑起来对应的 SOP，需要：

- `orchestrator._load_sop_for_project(project_id)` 按 `projects.traits` 调 `compose_sop` 产出 effective SOP → 转 transition_rules
- 这一步放在 **Phase D 前置** 或 **单独 commit** 做。目前 web 项目继续跑原 default_sop，游戏项目没 SOP 能调度（理论上用不了）

### 6.2 Phase D 导入链路

`ProjectTypeDetectorAction` 扫 repo 文件（`*.uproject` / `project.godot` / `package.json` / `requirements.txt` ...）推断 traits。给 git URL → clone 到项目目录 → detect → 确认卡片展示 candidates + 证据链。

估 1.5 天。

### 6.3 Phase E 项目特征编辑 UI

项目详情页加「项目特征」子 Tab：查看 + 编辑当前项目 traits + 实时 preview 刷新（复用 C' 的 API）。

估 1 天。

### 6.4 Phase F MCP trait 过滤

`mcp_servers.json` 加 `enabled_for_traits`。估 0.5 天。
