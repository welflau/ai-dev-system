# 开发日志 — 2026-04-22（延伸场）

## 版本
- v0.16.5 → **v0.16.6**

## 主题
CI/CD pipeline 自动合并 develop → main 失败时，**按文件类型分派自动解冲突策略**：
- 报告类文件（`docs/**/*.md`, `dev-notes/**`, `screenshots/**`）→ `git checkout --theirs` 吞掉 ours，保留新来的
- 代码类文件（`.py/.js/.html/...`）≤ 3 个 → 调 LLM Action (`ResolveMergeConflictAction`) 尝试语义合并
- 未知类型 / 超限 → 仍旧 abort + 人工介入

## 背景：为什么 CI 日志一直刷冲突

今天晚上排查 v0.16.4 bug 时，发现 server.log 每分钟都刷这条：

```
WARNING  构建失败 ci--2026: 合并到 main 失败: 合并冲突:
```

根因：**HelloWorld 项目**的 `main` 分支上有个 `71260b7 [DevAgent] 返工` commit —— DevAgent 直接 push 到了 main（流程异常，按正常 feature → develop → main 不应该发生），和 develop 并行修改了 `index.html` / `calendar.html`。CI 每 60 秒重试 develop → main 合并，每次都撞冲突 abort。

即时处理：手工 `git merge develop -X theirs --no-ff` 让 develop 覆盖 main（develop 是走了完整 review/accept 循环的产出，比 main 上孤立的 71260b7 更可信），push 后 CI 自动绿回。

但这是**创可贴级处理**。架构上要问：**CI 能不能自己搞定这类冲突？**

---

## 策略：按文件类型分派

冲突文件分三类：

### 1. 报告类（docs/, dev-notes/, screenshots/, *.md 等）
Agent 生成的 review/acceptance/dev-notes 报告冲突，通常只是**路径命名或时间戳导致的假冲突**（两边各生成了一份报告放在不同子目录），语义上不需要人工决断。

**策略**：`git checkout --theirs <file>` + `git add <file>` —— 用 develop 版本吃掉 main 的，自动通过。

### 2. 代码类（.py/.js/.html/.css/...）
这才是真冲突 —— 两边对同一代码文件做了不同修改。

**策略**：
- 文件数 ≤ 3 → 调 `ResolveMergeConflictAction`，让 LLM 读文件（含 `<<<<<<<` 标记）+ 两边分支名，产出合并版本
- 文件数 > 3 → 放弃 AI 解（避免 token 爆炸 + 错误累积），abort 让人工介入

**安全阀**：
- 单文件 > 2000 行 → decline
- 单文件冲突块 > 10 个 → decline
- LLM 产出还有 `<<<<<<<` 标记 → 判失败
- 任一代码文件解失败 → 全部 abort（不允许部分解决部分 fail 的半成品）

### 3. 未知类型（.bin / .lock / 各种配置）
**直接放弃** —— 不确定是代码还是数据，错解风险大。

## 实现

### 新 Action `backend/actions/resolve_merge_conflict.py`（~140 行）

```python
class ResolveMergeConflictAction(ActionBase):
    @property
    def name(self): return "resolve_merge_conflict"

    # 不对 LLM 暴露为 tool；只在 CI 里程序化调用
    @property
    def tool_schema(self): return {"name": self.name, "description": "（内部用）..."}

    async def run(self, context):
        files = context["files"]   # [{path, content, ours_label, theirs_label}]
        resolved, failed = {}, []
        for entry in files:
            # 1. 过安全阈值（行数 / 冲突块数）
            # 2. 读冲突标记正则找 hunks
            # 3. 调 llm_client.chat 低温合并
            # 4. 校验产出不含冲突标记
        return ActionResult(success=bool(resolved), data={"resolved": {...}, "failed": [...]})
```

**Prompt 设计要点**：
- 告诉 LLM 两侧分支名（ours / theirs），冲突块标记语义
- 合并原则明确排序：**保留两边新功能** > **同处冲突优先 theirs** > **保持语法合法** > **不动未冲突部分**
- 要求**只输出文件内容**（不加代码块 ```、不加解释）
- 允许在不确定处加 `TODO: manual review` 注释但**必须给默认值**，不准留 marker
- 剥壳 `_strip_code_fence()` 兜底：LLM 不听话用 ``` 包的情况

### 扩展 `git_manager.merge_branch`

加 `keep_conflict=False` 默认参数：
- `False`（原行为）：冲突时 abort，返回 `{success: False, error: ...}`
- `True`：**不** abort，保留 conflict 标记和 unmerged index，额外返回 `{conflict_files, repo_dir, has_conflict}` 给调用方处理

新增 `abort_merge(project_id)` + `finalize_merge(project_id, message)` 两个工具方法，让调用方能分步控制"试解 → 要么 commit 要么 abort"。

```python
async def merge_branch(self, project_id, source, target, message=None, keep_conflict=False):
    ...
    if rc != 0:
        if keep_conflict:
            return {
                "success": False,
                "error": ...,
                "conflict_files": await self._list_conflict_files(repo_dir),
                "repo_dir": repo_dir,
                "has_conflict": True,
            }
        await self._run_git(repo_dir, "merge", "--abort")
        return {"success": False, "error": ...}
```

### ci_pipeline.py 集成

原 merge 失败就 `_fail_build`，现在多一步：

```python
merge_result = await git_manager.merge_branch(..., keep_conflict=True)

if not merge_result["success"] and merge_result.get("has_conflict"):
    merge_result = await self._auto_resolve_conflict(build_id, project_id, merge_result, logs)

if merge_result["success"]:
    # success path (含自动解决后的 success)
else:
    # fail_build (所有 auto_resolve 失败后会走这里，且 merge 已 abort)
```

`_auto_resolve_conflict()`：
1. 分类冲突文件 → report / code / unknown
2. unknown 存在 OR code 文件数超限 → abort, 返回原 failure
3. report files 一个个 `checkout --theirs` + `git add`
4. code files 读内容 → `ResolveMergeConflictAction.run({files: [...]})` → 写回 + add
5. 全成功 → `finalize_merge` (commit + push)，任何一步失败 → `abort_merge` + 返回原 failure

## 关键设计决策

### 为什么 `-X theirs` 对报告类文件是安全的？
报告文件（`code-review.md`、`acceptance-review.md`、截图）**本质是只读产出**，下一轮 Agent 运行会重新生成一份覆盖。选哪边都不会丢"用户在意的东西"，选 theirs (= develop) 额外好处是保持构建方向一致。

### 为什么 LLM 解冲突先选 theirs 作为默认？
在 develop → main 这个方向上：
- `ours` = main = 当前稳定版
- `theirs` = develop = 要集成进来的新代码

如果 LLM 实在拿不准谁对，**"集成新代码"的语境下偏向 theirs 符合 CI 的语义**。

### 为什么限 ≤ 3 个代码文件？
LLM 解每个文件要读完整文件内容（可能几百行）+ 写回合并版本。3 个文件已经是 token 量的现实上限。更多文件意味着这不是"小冲突"，是"两条分支分叉了"，该人工介入。

### 为什么失败时 abort 而不是保留现场？
保留冲突现场让人手工解，听起来友好，实际上：
- CI 跑在后端服务里，没人盯着
- 下一轮 CI 重试前必须清理现场，否则累积
- 暴露冲突态给用户需要新 UI 入口 + 权限管理

所以保持"要么干净成功，要么彻底 abort 回到干净 failed"这个二值语义。人工解决靠项目仓库的本地 git，不经过 CI。

### 为什么这个 Action 不对 LLM 暴露为 tool？
`ConfirmProjectAction` / `GitMergeAction` 这些是让聊天 LLM 在 tool_use 循环里用的；但 `ResolveMergeConflictAction` 是 CI 基础设施组件，让聊天 LLM 有机会调用它反而危险（可能被恶意 prompt 诱导乱改代码库）。它只从 `_auto_resolve_conflict` 程序化调用。

### 真实 LLM 解冲突

构造一个典型"双方各加功能"的 `index.html` 冲突：
- ours：加了访问计数器（visit-counter div + 计数 JS）
- theirs：加了时钟（clock div + setInterval JS）

LLM 产出：
```html
<!DOCTYPE html>
<html>
<head><title>HelloWorld</title></head>
<body>
<h1>HelloWorld</h1>
<div id="visit-counter">访问次数：0</div>
<div id="clock"></div>
<script>
  let n = 0;
  document.addEventListener("DOMContentLoaded", () => {
    const el = document.getElementById("visit-counter");
    el.textContent = "访问次数：" + (++n);
  });

  function tick() {
    document.getElementById("clock").textContent = new Date().toLocaleTimeString();
  }
  setInterval(tick, 1000);
  tick();
</script>
</body>
</html>
```

两个 `<div>` 并排，两段 JS 合到一个 `<script>` 块里，**无残留冲突标记，两边功能都保留**。✅
