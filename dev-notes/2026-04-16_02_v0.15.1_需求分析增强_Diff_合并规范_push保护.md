## 主要改动

### 一、需求分析增强（解决"改个颜色重写整个页面"问题）

**根因分析**：
1. 过度拆分 — "改颜色"被拆成 6 个工单
2. 过度设计 — 简单修改也走 ArchitectAgent
3. LLM 重写习惯 — 给完整文件就全部重写

**解决方案**：

#### SummarizeCodeAction（新增，Action 池第 8 个）
- 拆单前先读懂项目：现有功能清单、技术栈、文件职责
- 对标 MetaGPT `SummarizeCode`
- 输出 `ProjectSummary` Schema（features/file_roles/tech_stack/complexity）

#### DecomposeAction 增强
- 注入项目理解报告（SummarizeCode 输出）
- 先判断复杂度：
  - `simple`（改颜色/改文案）→ 1 个工单
  - `medium`（小功能）→ 2-3 个工单
  - `complex`（大功能）→ 4-6 个工单
- 已有功能不重复拆单

#### 简单需求跳过架构
- `complexity == "simple"` → 工单初始状态 `architecture_done`
- 直接进 DevAgent，跳过 ArchitectAgent

#### ProductAgent 改造
```
之前: 直接拆单（不看项目）
现在: SummarizeCodeAction（读懂项目）→ DecomposeAction（基于理解拆单）
```

### 二、开发笔记新增 Diff

- `SelfTestAction._generate_diff()` 对比 existing_code 和新产出
- 三种展示：
  - 新建文件：前 20 行 `+` 前缀
  - 修改文件：unified diff 格式（最多 50 行）
  - 无变化：标注
- dev-notes.md 新增"代码变更 (Diff)"章节

### 三、Git 合并规范（两步合并）

**之前**：feat 完成 → 直接 merge feat→develop → 可能冲突
**现在**：
1. Step 1: merge develop→feat（拉最新到 feat，冲突在 feat 解决）
2. Step 2: merge feat→develop（此时无冲突）
- 冲突时停止并记录，不污染 develop

### 四、Agent Push 保护

- `git_manager.push()` 检测调用栈，Agent 不能 push 到 main/master
- `orchestrator.process_ticket()` 切换分支失败且在 main → 阻止执行
- `openArtifactFile` 关闭抽屉 class 名修复（open→active）
- 设置子菜单展开高度修复（240px→400px）

### 五、产品验收改为运行截图

- 不再审查代码片段（之前因截断导致误判"不完整"）
- 启动 HTTP server → 截图 → 截图+需求描述发给 LLM
- 基于运行效果判断是否通过
