## 主题
用户发现「生效配置预览的 SOP」跟「需求详情页 Pipeline 卡片」不一致：
- 配置预览按 traits 组装出 N 条 SOP 规则（v0.17 已就绪）
- 但 Pipeline 视图仍是**全局硬 5 格**（`requirement_analysis → architecture → development → testing → merge_develop`），读的是 `orchestrator._sop_config`（默认 `default_sop.yaml`），没吃 traits

落差：UE5 项目 SOP 里明明有 `asset_gen`、`engine_compile`、`play_test`、`multiplayer_stress`，但 Pipeline 卡片永远只亮 5 格。本轮把这块也改成按项目 traits 动态派生。

---

## 一、方案（Option C：Pipeline 也 trait-aware）

### 1.1 核心思想

`pipeline_view` 不再只放在 `default_sop.yaml`，改成：
- `_core.yaml` 提供**基础 UI 视图**（4 格：需求 → 架构 → 开发 → 合入）
- 每个 fragment 声明 `pipeline_view_contrib`，说明自己归到哪个 UI 组（或新建组）
- `compose_sop` 合并：已有组 append 进去，没有组按 `insert_after/before` 建新格

效果：
- web 项目（空 traits）→ 4 格
- web+service+git → 5 格（加"测试验证"容纳 smoke_test）
- UE5 FPS multiplayer → 6 格（加"资产准备" + "测试验证"含 engine_compile/play_test/mp_stress）

### 1.2 为什么选 C

前面讨论过三个选项：

| 选项 | 描述 | 被否原因 |
|---|---|---|
| A | Pipeline 完全跟着 SOP stages 走，一对一 | UI 会出现 7~9 格，太细；合并小步骤难 |
| B | 继续硬 5 格，fragment 执行时透明穿插 | 用户看不到 engine_compile 是否跑了，黑盒 |
| **C** | **UI 组是声明式的，fragments 声明归组** | ✅ UI 清爽 + 可见性够 + fragment 作者控制 |

### 1.3 拒绝方案：per-project yaml 落盘

**用户追问**："每个项目没有自己的 yaml？那 SOP 存到哪了？"

答：**不落盘**。项目 SOP 是**运行时从 DB traits 现算的**，只临存在 `orchestrator._project_rules_cache`。

```
DB: projects.traits (JSON 列)     ← 唯一持久化
            │
            ▼
    compose_sop(traits)           ← 每次按需组装
            │
            ▼
  内存缓存: _project_rules_cache[(project_id, traits_tuple)]
            │
            ▼
      给 orchestrator / API 用
```

好处：
- 单一真相源，改 fragment 所有匹配项目立即生效
- 不会产生 N 项目 = N yaml 的文件爆炸
- traits 改了自动重组（`invalidate_project_rules()` 清缓存）

代价：
- 项目无法逃出 trait + fragment 的表达力
- 想看某项目当前 SOP 只能调 `compose_sop(traits)` 或 `POST /api/projects/preview-assembly`

如果将来需要"某项目一次性特殊 stage"，两条路（目前都没做）：
1. `projects` 表加 `custom_sop_yaml` 字段，compose 后 merge
2. `extra_fragments` 字段，指定额外启用的 fragment id

## 二、落地：5 个子任务

### C.1 `_core.yaml` 加 `pipeline_view` 基础视图

```yaml
pipeline_view:
  stages:
    - key: requirement_analysis
      synthetic: requirement_status
      name: 需求分析
      icon: 📋
    - key: architecture
      sop_stages: [architecture]
      running_statuses: [architecture_in_progress]
      name: 架构设计
      icon: 🏗️
    - key: development
      sop_stages: [development, code_review, acceptance, rework]
      running_statuses: [development_in_progress, review_in_progress, acceptance_rejected]
      name: 开发实现
      icon: 💻
    - key: merge_develop
      synthetic: merge
      name: 合入Develop
      icon: 🔀
```

**顺带修复**：`_core.yaml` 本来 `acceptance.reject_goto: rework` 但没定义 `rework` stage，导致 `build_pipeline_stages` 报 `pipeline_view 引用了不存在的 SOP stage: rework` warning。补上：

```yaml
- id: rework
  agent: DevAgent
  action: fix_issues
  trigger_on: acceptance_rejected
  success_status: development_done
  config:
    enable_reflection: true
    reflection_max_prev: 3
```

### C.2 fragments 声明 `pipeline_view_contrib`

6 个 fragment 全部加归组声明：

| fragment | group | create_if_missing |
|---|---|---|
| smoke_test | testing | 测试验证 🧪 insert_after development |
| play_test | testing | 同上 |
| engine_compile | testing | 同上 |
| multiplayer_stress | testing | 同上 |
| i18n_check | testing | 同上 |
| asset_gen | **asset_preparation** | 资产准备 🎨 insert_before development |

示例（smoke_test.yaml）：

```yaml
pipeline_view_contrib:
  group: testing
  create_if_missing:
    name: 测试验证
    icon: 🧪
    insert_after: development
```

### C.3 `compose_sop` 合并 pipeline_view

新增 `_merge_pipeline_view(core_view, frag_entries)`：
- 遍历活跃 fragments
- 每个 fragment 找 `contrib.group` 在 core_view.stages 里有没有
  - 有：append fragment 的 `stage.id` 到 `sop_stages`，追加 `trigger_on/reject_status` 到 `running_statuses`
  - 没：按 `create_if_missing` 建新 UI 组，按 `insert_after/before` 定位（兜底插到 merge_develop 之前）
- 返回合并后的 pipeline_view，`compose_sop` 把它塞进 return 的 config

兜底：没声明 `pipeline_view_contrib` 的 fragment 默认挂到 `testing` 组（避免漏）。

### C.4 `api/requirements.py` 按项目 traits 派生

原来：
```python
_pipeline = build_pipeline_stages(orchestrator._sop_config or {})
```

改成：
```python
project_row = await db.fetch_one("SELECT traits FROM projects WHERE id = ?", (project_id,))
project_traits = json.loads(project_row.get("traits") or "[]")
try:
    sop_config = compose_sop(traits=project_traits, ticket_type=None)
except Exception:
    sop_config = orchestrator._sop_config or {}  # 兜底
_pipeline = build_pipeline_stages(sop_config)
```

**不在此处 ticket_type 精细过滤**：Pipeline 是**需求级**视图，要展示所有可能的阶段（该需求下的不同 ticket_type 可能各需要不同 fragment，UI 展示"最大包络"即可）。

### C.5 E2E 验证

```
=== 场景 1: 空 traits（老 web 项目）===
UI order: [requirement_analysis, architecture, development, merge_develop]   ← 4 格

=== 场景 2: category:service + vcs:git ===
UI order: [requirement_analysis, architecture, development, testing, merge_develop]   ← 5 格

=== 场景 3: engine:ue5 + category:game + multiplayer:true ===
UI order: [requirement_analysis, architecture, asset_preparation, development, testing, merge_develop]   ← 6 格
composed_from_fragments: [multiplayer_stress, play_test, engine_compile, asset_gen]
```

零 warning，所有场景正确。

## 四、架构补充：SOP 在哪？

本轮用户的第二个问题触发了一份**运行时架构速查**，补到这里供未来 contributor 查：

### 4.1 SOP 的三种形态

| 形态 | 存在位置 | 生命周期 | 用途 |
|---|---|---|---|
| **模板** | `backend/sop/_core.yaml` + `fragments/*.yaml` | 磁盘，git | 组装材料 |
| **全局默认** | `backend/sop/default_sop.yaml` | 磁盘，git | 空 traits 兜底 |
| **项目实例** | `orchestrator._project_rules_cache[(pid, traits_tuple)]` | **仅内存** | 实际调度使用 |

关键：**项目实例不落盘**。进程重启后首次访问该项目时由 `_get_rules_for_project()` 重算。

### 4.2 各 API 读的是哪份

| API / 触点 | 读什么 SOP |
|---|---|
| `orchestrator._poll_once` 调度 | `_get_rules_for_project(project.id)` → 按 traits compose |
| `/api/projects/preview-assembly` | `compose_sop(traits)` 现场算给前端预览 |
| `/api/requirements/.../pipeline` (v0.17.1 新) | 按项目 traits compose_sop → build_pipeline_stages |
| `/api/sop/config` | `orchestrator._sop_config`（**全局 default**，不知道项目） |
| 前端"流程配置"页 | 同上，`/api/sop/config` |

**待厘清项**：`/api/sop/config` 仍然只展示全局 default SOP。用户在"流程配置"页看到的跟"项目特征"预览出来的不是同一份。要么：
- 改 `/api/sop/config` 接收 `project_id` 参数按项目返回
- 或明确"流程配置"是全局模板的编辑入口，项目页另有入口看项目实际 SOP

（这是 v0.17.x 的 follow-up，本轮未动）

### 4.3 缓存失效链

改 traits 的所有路径都要调 `invalidate_project_rules(project_id)`：

```
PATCH /api/projects/{id}/traits     → 已接（Phase E）
confirm_project Action               → 新建，尚无缓存，无需失效
导入时 create_project + traits       → 新建，同上
（将来）批量 trait 修改               → 需记得接
```
