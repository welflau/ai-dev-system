## Phase 1 — Pre/Post Tool Hooks（P0，2-3 天）

### 目标

在每次 Action 执行前后提供统一的钩子点，解锁审计、限流、错误追踪等横切逻辑。

### 交付物

| 文件 | 类型 | 说明 |
|------|------|------|
| `backend/hooks/__init__.py` | 新建 | 模块入口 |
| `backend/hooks/types.py` | 新建 | HookEvent 枚举 + ToolHookContext 数据类 |
| `backend/hooks/registry.py` | 新建 | HookRegistry 单例（注册 + emit） |
| `backend/hooks/builtin.py` | 新建 | 3 个内置 Hook（见下） |
| `backend/actions/executor.py` | 新建 | run_action() 包装器，含 Pre/Post/Error 触发 |
| `backend/agents/base.py` | 修改 | run_action() 改调 executor.run_action() |
| `backend/agents/chat_assistant.py` | 修改 | execute() 改调 executor.run_action() |
| `backend/main.py` | 修改 | lifespan 加 register_builtin_hooks() |
| `backend/database.py` | 修改 | 新增 tool_audit_log 表 |

### 内置 Hook 清单

| Hook 名 | 触发时机 | 功能 |
|---------|---------|------|
| `audit_log_hook` | POST_TOOL_USE | 每次工具调用写 tool_audit_log 表（tool_name / project_id / ticket_id / duration_ms / success） |
| `shell_rate_limit_hook` | PRE_TOOL_USE | ShellAction 每 Ticket 调用超 50 次时 raise，防滥用 |
| `failure_library_hook` | TOOL_ERROR | 工具报错自动写入 failure_library 表 |

### 验收标准

- [ ] `hook_registry.emit(ToolHookContext(...))` 调用所有已注册 Hook，单个 Hook 失败不中断主流程
- [ ] 执行一次带工具调用的聊天，`tool_audit_log` 表有记录
- [ ] ShellAction 连续调用 51 次抛出限流错误
- [ ] 工具执行报错时，`failure_library` 表有新增记录

### 交付物

| `backend/llm_client.py` | 修改 | chat_with_tools / chat_with_tools_stream 接受 budget 参数 |

### Budget 配置项

```python

AGENT_MAX_TOKENS   = 200_000    # 工单 Agent 单次执行 Token 上限
AGENT_MAX_TURNS    = 50         # 工单 Agent 工具调用轮次上限
AGENT_MAX_SECONDS  = 600        # 工单 Agent 执行时间上限（秒）
CHAT_MAX_TOKENS    = 100_000    # 聊天会话 Token 上限
CHAT_MAX_TURNS     = 30         # 聊天工具调用轮次上限
CHAT_MAX_SECONDS   = 180        # 聊天会话时间上限（秒）
```

### 验收标准

- [ ] 将 CHAT_MAX_TURNS 设为 2，发一条会触发多次工具调用的消息，第 3 次应被中断并返回 BudgetExceededEvent
- [ ] Budget 超限时前端显示友好提示（非 500 错误）
- [ ] .env 配置变更生效，无需修改代码

## Phase 3 — 异步权限审批系统（P2，5-7 天）

### 目标

为高风险操作（删除文件、force-push、DROP TABLE 等）建立人工审批通道，适配 ADS 无人值守的自动化定位。

### 设计思路

不采用 Claude Code 的"实时弹窗"（会挂起后台进程），改为"异步审批"：
1. 检测到高风险操作 → 写入 `permission_requests` 表，协程挂起（最多等待 5 分钟）
2. 前端显示待审批列表 → 用户点击批准/拒绝
3. WebSocket 通知后端 → `future.set_result()` → 协程恢复执行
4. 超时未审批 → 自动拒绝（fail-closed 原则）

### 交付物

| 文件 | 类型 | 说明 |
|------|------|------|
| `backend/permissions/__init__.py` | 新建 | 模块入口 |
| `backend/permissions/gate.py` | 新建 | PermissionGate 类 + HIGH_RISK_TOOLS 配置 |
| `backend/database.py` | 修改 | 新增 permission_requests 表 |
| `backend/actions/executor.py` | 修改 | run_action() 加入 permission_gate.check() |
| `backend/api/permissions.py` | 新建 | GET /pending + POST /{id}/resolve 端点 |
| `frontend/index.html` | 修改 | 新增「待审批」页面或 Tab |
| `frontend/app.js` | 修改 | 审批列表 UI + 批准/拒绝操作 |

### 高风险工具初始清单

| 工具 | 触发条件 | 风险说明 |
|------|---------|---------|
| `ShellAction` | 命令含 `rm -rf`、`git push --force`、`DROP TABLE` | 不可逆的删除/覆盖 |
| `GitMerge` | 目标分支为 `main`/`master` | 影响主干 |
| `shell` (新工具) | 命令含危险关键词 | 同上 |

### 验收标准

- [ ] ShellAction 执行 `rm -rf` 命令时触发挂起，前端出现待审批提示
- [ ] 用户点击「批准」后命令继续执行
- [ ] 用户点击「拒绝」后返回错误，工单状态置为 BLOCKED
- [ ] 5 分钟未操作自动拒绝，工单状态置为 BLOCKED
- [ ] 低风险工具正常执行，无需审批（零性能影响）

## Phase 4 — 子任务派发（P3，3-4 天）

### 目标

给 Agent 增加在执行中动态创建子任务的能力，实现有限度的递归 Agent，而不需要重写 Orchestrator 核心。

### 设计思路

不重构整个 Orchestrator，而是新增 `dispatch_subtask` Action：
1. Agent 调用此工具 → 创建子 Ticket（关联父 Ticket）
2. 子 Ticket 进入 PENDING 状态 → Orchestrator 下一轮 poll 自动拾取
3. 父 Ticket 等待所有子 Ticket 完成后再推进状态机

### 交付物

| 文件 | 类型 | 说明 |
|------|------|------|
| `backend/actions/chat/dispatch_subtask.py` | 新建 | DispatchSubtaskAction（Chat Tool + Orchestrator Tool 复用）|
| `backend/database.py` | 修改 | tickets 表新增 parent_ticket_id 字段 |
| `backend/orchestrator.py` | 修改 | 父子 Ticket 追踪逻辑（等待子任务完成再继续）|
| `frontend/app.js` | 修改 | 工单看板展示父子关系 |

### 验收标准

- [ ] DevAgent 在执行中可以派发子 Ticket 给 TestAgent
- [ ] 父 Ticket 在所有子 Ticket TESTING_DONE 后才推进到下一阶段
- [ ] 工单看板中父子任务有层级展示
- [ ] 循环派发检测（子 Ticket 不能再派发给同级父 Ticket）

## 风险备案

| 风险 | 概率 | 影响 | 缓解方案 |
|------|------|------|---------|
| QueryEngine 重构引发流式输出回归 | 中 | 高 | 新路径加 feature flag，旧路径保留 2 周 |
| Hook 执行性能影响聊天延迟 | 低 | 中 | Hook 内部加超时（2s），fail-open |

| Budget 参数设置过紧导致正常任务截断 | 中 | 中 | 初始值设为当前实际用量的 3 倍，灰度上线 |
| Phase 3 前端工作量超出预期 | 低 | 低 | 先做最小可用 UI（纯列表页），交互后续迭代 |
