## 一、Harness 平台升级（Phase 1-4 全部完成）

- `hooks/types.py`：`HookEvent` 枚举 + `ToolHookContext` 数据类
- `hooks/registry.py`：`HookRegistry` 单例，`blocking` 参数控制限流阻断，内置调用统计

- `query_engine/budget.py`：Token/轮次/时间三重安全阀
- `.env`/`config.py`：6 个预算配置项（CHAT/AGENT × MAX_TOKENS/TURNS/SECONDS）
- `llm_client.chat_with_tools_stream`：接入 Budget，超限 yield `budget_exceeded` 事件

### Phase 2 — QueryEngine 统一 LLM 循环
- `query_engine/events.py`：8 种 QueryEvent 类型
- `query_engine/executor.py`：ToolExecutorProtocol + ChatToolExecutorAdapter + OrchestratorToolExecutorAdapter
- `query_engine/engine.py`：QueryEngine 主类，含 Budget/Hooks/工具循环
- `chat_assistant.py`：`chat_stream` / `chat_global_stream` 改走 QueryEngine（向后兼容）
- `base.py`：`get_tool_schemas()` + `_react_with_think_inner` 接入 QueryEngine

### Phase 3 — 异步权限审批
- `permissions/gate.py`：PermissionGate + 6 条高风险规则（rm -rf/force-push/DROP TABLE 等）
- `asyncio.Future` 挂起协程，300s 超时自动拒绝（fail-closed）
- `api/permissions.py`：3 个端点（pending/resolve/history）
- 前端：顶部红色审批按钮 + 审批弹窗

### Phase 4 — 子任务派发
- `actions/chat/dispatch_subtask.py`：创建子 Ticket + 父 Ticket 置为 waiting_subtasks
- Orchestrator：`_check_subtasks_complete` + `_resume_parent_ticket`

## 四、重大 Bug 修复

### WAL 积压 5.5 GB → DB 不可用
- 根因：Orchestrator 长期高并发写入，WAL 文件从未 checkpoint
- 修复：`DB_PATH` 配置化，切换到干净副本 `ai_dev_system_new.db`
- 预防：`wal_autocheckpoint=500`，降低并发 3→1

### AI 助手无法对话
- 根因 1：`_save_chat_message` 同步等待 DB（被 Orchestrator 锁），stream 在 yield 前崩溃
- 修复 1：改为 `asyncio.create_task`（fire-and-forget）
- 根因 2：`db.fetch_all(projects)` 抛异常，generator 在任何输出前失败
- 修复 2：try/except 捕获，失败时用空列表继续

### Orchestrator `deployed → deploy` SOP 死循环
- 根因：`_relink_stage_transitions` 把 core deploy 阶段链接到 deploy_web fragment 的 success_status=deployed
- 修复：`_TERMINAL_STATUSES = frozenset({'deployed', 'cancelled'})`，硬排除终态

### 聊天输入卡顿
- 根因：`autoResizeChatInput()` 调用 `scrollHeight` 触发同步全页面 Layout 计算，日志面板 DOM 节点多导致耗时数百毫秒
- 修复：`.log-panel { contain: layout style }` + `requestAnimationFrame` debounce

### 日志过滤不生效（新条目绕过）
- 根因：`addLog()` 完全没有过滤逻辑，新进来的 System 条目直接 append
- 修复：`addLog` 和 `appendLogEntry` 追加三项过滤（Agent/Level/关键字）

### ActionNode LLM 无超时
- 根因：Orchestrator Agent 通过 ActionNode 直接调 `llm.chat_json()`，没有超时保护
- 修复：`asyncio.wait_for(timeout=AGENT_MAX_SECONDS)` + 超时 emit `TOOL_ERROR` Hook

### Action 卡片不显示（payload 冲突）
- 根因：`yield {"type": "action", **event.action_data}` 中 event.action_data 的 type 覆盖事件 type
- 修复：改为 `yield {"type": "action", "payload": event.action_data}`

## 五、稳定性措施

| 措施 | 效果 |
|------|------|
| DB 重试（execute/insert 各 3 次）| DB 锁时自动等待，聊天不中断 |
| Orchestrator 并发 3→1 | 减少 DB 写锁竞争 |
| busy_timeout 60s + cache 32MB | 提高 DB 并发容忍度 |
| `_TERMINAL_STATUSES` 硬编码 | 防止 SOP 配置错误死循环 |
| DeployAgent `.pop(key, None)` | 修复并发 KeyError |
| CI lint 排除第三方 skill 目录 | GitHub Actions 全绿 |
