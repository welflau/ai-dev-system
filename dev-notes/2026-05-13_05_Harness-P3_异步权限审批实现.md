## 二、核心流程

```
AI 调用高风险工具（如 shell: rm -rf）
      │
      ▼
permission_gate.check(tool_name, tool_input, context)
      │
      ├─ detect_risk() → None → 低风险，直接放行
      │
      └─ detect_risk() → "删除文件（rm -rf）" → 高风险：
           │
           ├─ 写 permission_requests 表（status=pending）
           ├─ SSE 推送 permission_request 事件
           ├─ asyncio.Future + asyncio.wait_for(timeout=300s)
           │
           ▼ 协程挂起
           │
    用户在前端点击「批准」→ POST /api/permissions/{id}/resolve {approved: true}
           │
           ├─ permission_gate.resolve(id, True) → future.set_result(True)
           ├─ 更新 DB status=approved
           │
           ▼ 协程恢复，工具正常执行
           │
    超时（300s）→ 自动拒绝 → PermissionDeniedError
    用户点击「拒绝」→ future.set_result(False) → PermissionDeniedError
```

## 三、高风险规则清单

| 工具 | 检测模式 | 风险说明 |
|------|---------|---------|
| `shell` | `rm\s+-[rRfF]` | 删除文件（rm -rf）|
| `shell` | `git\s+push\s+.*--force` | 强制推送 |
| `shell` | `DROP\s+TABLE\|DROP\s+DATABASE` | 删除数据库 |
| `shell` | `format\s+[a-zA-Z]:` | 磁盘格式化 |
| `shell` | `shutdown\|reboot\|halt\|poweroff` | 关机/重启 |
| `git_merge` | 目标含 `main\|master` | 合入主干分支 |

规则定义在 `permissions/gate.py`，后续直接追加 `_HIGH_RISK_RULES` 列表即可扩展。

---

## 四、接入点（两条路径双覆盖）

- **`actions/executor.py`**：Orchestrator `run_action()` 调 `run_action_with_hooks()`，PRE Hook 之后执行权限检查  
- **`query_engine/engine.py`**：ChatAssistant 走 QueryEngine，PRE Hook 之后执行权限检查  

两条路径都覆盖，无论 AI 通过哪条路径执行高风险工具，都会触发审批。

---

## 五、前端 UI

- **顶栏红点按钮**：有待审批时显示 `🔒 审批` + 数量角标  
- **审批弹窗**：SSE 收到 `permission_request` 自动弹出  
- **页面加载时**：轮询 `/api/permissions/pending`，恢复刷新前漏掉的 pending  
- **批准/拒绝**：分别调 `POST /api/permissions/{id}/resolve`
