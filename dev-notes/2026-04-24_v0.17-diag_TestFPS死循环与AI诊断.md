## 一、诊断现场

### 1.1 DB 取证

直奔 `ai_dev_system.db`，看 TestFPS 工单和日志：

| ticket | status | logs |
|---|---|---|
| TK-48c27c | development_done | **668** 条 |
| TK-1eb1bb | development_done | 211 条 |
| TK-c6b510 | development_done | 81 条 |
| 另 3 条 | development_done | 79~80 条 |

20 分钟内（10:15→10:36）产生 1200+ 条日志，全是：

```
DevAgent.complete  from=development_done  to=development_done
详情: 开发完成 | 自测: 未自测
```

"未自测" 对上用户报告 —— `test_summary = self_test.get("summary", "未自测")` 的兜底字符串。

### 1.2 追链

1. **TestFPS 的 compose_sop 输出**：
   ```
   architecture → asset_gen → development → engine_compile → play_test →
     multiplayer_stress → code_review → acceptance → deploy → rework
   ```
   `engine_compile.trigger_on=development_done, agent=DevAgent, action=run_engine_compile`

2. **DevAgent 的实现**（`agents/dev.py:34`）：
   ```python
   async def execute(self, task_name, context):
       if task_name == "develop": ...
       elif task_name == "rework": ...
       elif task_name == "fix_issues": ...
       return {"status": "error", "message": f"未知任务: {task_name}"}
   ```
   `run_engine_compile` **未实现**，走 fallthrough → 返回 error。

3. **orchestrator 的 `_handle_agent_result`**（bug 现场）：
   ```python
   # 无视 result["status"] == "error"
   if agent_name == "ArchitectAgent": ...
   elif agent_name == "DevAgent":
       new_status = TicketStatus.DEVELOPMENT_DONE.value  # ← 硬编码
       await db.update(... status=DEVELOPMENT_DONE ...)
       log "开发完成 | 自测: 未自测"
   ```

4. **轮询循环**：ticket 重新落回 `development_done` → `_poll_once` 匹配 `engine_compile` 规则 → 再分派 DevAgent → error → 写回 `development_done` → …

一圈下来：没有任何状态变化，但每圈都跑一次 LLM 调用（`run_action("write_code")` + `run_action("self_test")`）。668 次 ≈ 数百万 tokens 烧掉 + CPU 占满。

### 1.3 顺带挖出的第二个 bug：rework trigger 覆盖

在看 compose_sop 输出时发现最后一行异常：

```
rework  trigger=deployed  success=development_done
```

`rework` 的 `trigger_on: acceptance_rejected` 被 `_relink_stage_transitions` 改成了前一个 stage (`deploy`) 的 `success_status: deployed`。于是一旦 ticket 真的跑到 `deployed`（终态）会**重新触发 rework** → `success=development_done` → engine_compile → …（第二层循环）。

这个 bug 原本会被 #1 bug 先触发挡住，**但只要修掉 #1，#2 就会暴露**。所以必须一起修。

## 二、三连修

### 2.1 Commit `4fc24bc` — rework 分支 trigger 覆盖

**问题**：`_relink_stage_transitions` 盲目覆盖所有 stage 的 `trigger_on` 为前一个 stage 的 `success_status`。对线性 pipeline 正确，但 `rework` 是**分支目标**（只通过 `reject_goto` 进入），不该参与线性链。

**修复**：
- `_core.yaml` 的 rework 加 `branch_only: true` 标记
- `_relink_stage_transitions` 跳过 `branch_only: true` 的 stage（不覆盖其 trigger，也不把其 success 传给下一个 stage）

```python
for i, s in enumerate(result):
    if s.get("branch_only"):
        continue   # 不参与线性链
    ...
```

修复后 `rework.trigger=acceptance_rejected` 正确保留。

### 2.2 Commit `a8b7a9a` — orchestrator 止血 + BLOCKED 状态

**问题**：所有 5 个 Agent 都有 `return {"status": "error", ...}` 的 fallthrough，但 orchestrator 一律忽略。

**修复**：
1. `models.py` 加 `TicketStatus.BLOCKED = "blocked"`
2. `_handle_agent_result` 最开头识别 `result.get("status") == "error"`：
   - 写 ticket.status = blocked
   - 记 error 日志 + 发 `ticket_blocked` SSE
   - return（不走后面的 agent_name 分支）
3. `blocked` 不在任何 transition_rules 里 → poll 不拾取 → 循环停
4. 新增 `POST /tickets/{id}/unblock?target_status=...` 供前端手动解除

这个修复是**系统性**的：不管哪个 Agent 未实现哪个 action，都会被挡住。

### 2.3 Commit `66b3cec` — 中档 AI 诊断（中档方案）

用户要求："现在所有卡流程的，能智能化处理了吗？暂停，AI 自动查找原因，预警"。

给三档方案：已有（仅 blocked）/ 中档（blocked + AI 诊断 + 预警）/ 完整（再加 zombie/long-pending/retry-exhausted 检测 + dashboard）。用户选**中档**。

**实施**：

1. **`actions/diagnose_ticket.py` — DiagnoseTicketAction**

   输入 context：
   ```python
   {
     "ticket_title", "ticket_description",
     "project_traits",
     "current_status",      # 进入 blocked 前的状态
     "failed_agent", "failed_action", "failed_error_message",
     "recent_logs",         # 最近 10 条 ticket_logs
     "sop_stage",           # 该 action 对应的 SOP 阶段定义
   }
   ```

   输出 `diagnosis` JSON：
   ```json
   {
     "symptom": "现象一句话",
     "root_cause": "定位到 agent/action/配置项/文件",
     "severity": "low|medium|high",
     "suggested_actions": [
       {"action": "具体动作", "who": "developer|system|config", "priority": 1}
     ],
     "confidence": 0.0,
     "related_files": [...],
     "created_at": "ISO"
   }
   ```

   LLM system prompt 强调"根因要具体（到 action/agent/配置项/文件），建议动作必须可操作"；JSON 解析失败降级为只回显 error message（confidence=0.2）。

2. **DB 迁移**：`tickets.diagnosis TEXT`（JSON 列）

3. **orchestrator 串联**：`_handle_agent_result` 的 error 分支 return 前 `asyncio.create_task(self._diagnose_blocked_ticket(...))` fire-and-forget。`_diagnose_blocked_ticket` 方法负责：
   - 拉 project traits → compose_sop → 定位 sop_stage
   - 拉最近 10 条 logs
   - 跑 DiagnoseTicketAction
   - 写回 `tickets.diagnosis`
   - 记 "diagnose" 日志 + 发 `ticket_diagnosed` SSE

4. **手动复诊 API**：`POST /api/projects/{pid}/tickets/{tid}/diagnose`
   - 从最近一条 error 日志提取 agent/action/error
   - 调用同一个 `_diagnose_blocked_ticket`

5. **前端**：
   - 卡片：blocked 状态加 🚧 角标 + 红色样式
   - 状态下拉：补 blocked 选项
   - 抽屉：紧跟"操作"按钮的 🩺 诊断 panel（严重度颜色条 + 现象 + 根因 + 建议动作列表 + 相关文件 + 置信度 + 诊断时间）
   - 操作按钮：blocked 态加 "🔄 重新诊断" + "↪️ 解除 blocked"
   - SSE：订阅 `ticket_blocked` / `ticket_diagnosed` / `ticket_unblocked` 推 toast + refreshBoard

**成本**：每次进 blocked 跑 1 次 LLM（~1-2k tokens）。blocked 不重复触发诊断（除非用户手动点）。

## 三、Bug 叠加分析

这三个问题**叠加**才造成死机。抽开看每个单独的影响：

| bug | 单独影响 | 被其他 bug 遮盖的表现 |
|---|---|---|
| `rework` trigger 覆盖 | ticket 真的跑到 deployed 后无限循环 | 很难触发（要先过 engine_compile、review、accept、deploy 全部成功），#1 bug 会先拦住 |
| orchestrator 无视 error | 未实现 action 导致 ticket 原地循环（#1） | 让"DevAgent 开发完成、未自测"看起来像死机 |
| fragments 声明未实现 action | 本该"静止"等待实现，但被 #1 bug 放大成死循环 | 若 orchestrator 正确处理，这就是 dev-note 2026-04-24-pipeline §4.1 预期的"停在 engine_compile" |

**教训**：
- 每个 bug 单独看都不算致命（都有兜底意图），叠加后才是灾难。这是典型的 Swiss cheese model。
- 写"兜底返回 error"时，下游**必须**真的处理 error，否则兜底变漏洞
- `_relink` 这种"简化后续工作"的流程类函数，要对每一种 stage 形态（线性/分支/循环目标）明确表态

## 四、顺带加的后续能力

现在系统有了显式的"卡壳"语义（BLOCKED 状态 + 诊断元数据），可以在上面叠能力：

| 可做 | 说明 | 复杂度 |
|---|---|---|
| 失败库查同类 | 新工单 blocked 时，根据 root_cause 关键词查 failure_library 找同类案例推给用户 | 低 |
| 检测 zombie | `*_in_progress` 超 N 分钟未变 → 自动转 blocked + 诊断 | 中 |
| 诊断升级成 Action 可执行 | 诊断里的 suggested_actions 若是 "重试 / 跳过 / 切换 agent"，加个按钮一键执行 | 中 |
| 跨项目统计 | 统计哪些 action 最常 blocked → 反推应优先实现哪些 Agent.action | 低（一个 SQL） |

当前没做，只记 TODO。
