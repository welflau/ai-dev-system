## 背景

昨天（04-21）完成 MagicAI 对标 5/7 + 盲审修复 P0 后，主线剩最后一项 ⭐⭐⭐：Self-Consistency。这是论文 (Wang et al. 2022) 方法——对 CoT 类任务跑多次投票选最一致答案。我们的结构化输出变体：N 次高温（0.8）跑同一 prompt，Critic LLM 低温（0.2）选 best。

**适用场景**：主观发散任务。本项目有两个明确候选：
- **DecomposeAction（拆单）**：同一需求不同 LLM 调用可能拆出完全不同数量/粒度的工单
- **DesignArchitectureAction（架构）**：架构决策没客观最优，一次输出不稳

论文实测 +17-18%；主观任务预计 +25%。

## 实现

### 核心 helper `backend/actions/voting.py`

```python
async def fill_with_consistency(
    node_factory: Callable[[], ActionNode],
    req: str,
    llm,
    *,
    n: int = 3,
    temperature: float = 0.8,
    max_tokens: int = 4000,
    task_desc: str = "",
) -> Tuple[ActionNode, List[ActionNode], Dict]:
    """并行 N 候选 → Judge → (best, all, judge_info)"""
```

流程：
1. `asyncio.gather` 并行 N 个 ActionNode.fill（`return_exceptions=True` 防单失败连累整体）
2. 过滤 `instruct_content` 非空的成功候选
3. 0 个 → raise；1 个 → 跳过 judge；>1 个 → LLM judge 低温选 best
4. Judge 失败 → fallback 返回候选 0 + `fallback: true` 标记

**Judge prompt 4 条判分标准**：完整性 / 可执行性 / 与任务契合度 / 内部一致性（不矛盾）。输出 `{best_index, reasoning}`。

### 集成（两个 Action 几乎对称）

```python
sop_cfg = context.get("sop_config") or {}
if sop_cfg.get("self_consistency"):
    best_node, all_nodes, judge_info = await fill_with_consistency(
        _new_node, req=req_context, llm=llm_client,
        n=sop_cfg.get("consistency_n", 3),
        temperature=sop_cfg.get("consistency_temperature", 0.8),
        max_tokens=...,
        task_desc=f"为「{title}」...",
    )
    vote_info = {"stage": ..., "n_candidates": ..., "best_index": ..., "reasoning": ..., "fallback": ...}
else:
    node = _new_node()
    await node.fill(req=req_context, llm=llm_client, max_tokens=...)

# 产出带 _consistency_vote 给 orchestrator 记日志
```

失败时自动降级到单次调用，不阻塞流程。

### SOP yaml

`architecture` stage 加了配置示范（默认 false）：
```yaml
config:
  self_consistency: false
  consistency_n: 3
  consistency_temperature: 0.8
```

### Orchestrator 留痕

两处 `_log(action='consistency_vote')`：
- 架构：`_handle_agent_result` ArchitectAgent 分支
- 拆单：`handle_requirement` 成功分支

## 关键设计点

### 1. ContextVar 并发安全

**顾虑**：Skills 用 `contextvars.ContextVar _current_skills` 在 BaseAgent.run_action 里 set，`asyncio.gather` 并行任务会不会读不到？

**结论**：Python asyncio 原生支持 ContextVar copy—— 每个 task 启动时自动拷贝 context。3 个并发候选都能读到同一 skills_prompt。

单测 Test 5 明确覆盖：手动 set `SKILL_MARKER_XYZ` → gather 3 candidates → 3 个 fill 调用里都 captured 到同一个值。

### 2. 成本控制

- 默认 **opt-in**（sop yaml 默认 `self_consistency: false`），不开不多花钱
- N=3 + 1 judge = **~3.2× token**（并行 + judge 短），**~1.5× 耗时**（并行所以不是线性翻倍）
- 建议只给 architecture 这类低频 stage 开；不建议 decompose 默认开（每需求都触发累积快）

### 3. 失败路径丰富

| 失败点 | 处理 |
|---|---|
| 某个候选 fill 异常 | `return_exceptions=True` 捕获，不影响其他候选 |
| 所有候选都失败 | raise ValueError，Action 内 catch 后降级到单次调用 |
| Judge LLM 失败 | fallback 返回候选 0 + `fallback: true` 标记给日志 |
| Judge 返回越界 index | 归 0 + warning |
| Judge JSON 解析失败 | 同 fallback |
