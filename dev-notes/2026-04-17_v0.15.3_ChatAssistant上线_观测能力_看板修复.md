## 二、P3：默认化 + Fallback 兜底

### 改造
- `CHAT_USE_AGENT` 默认 `false` → `true`
- `chat_with_ai` 改为：
  ```python
  if settings.CHAT_USE_AGENT:
      try:
          return await _chat_via_agent(...)
      except HTTPException:  # 业务错误直抛
          raise
      except Exception:       # 其他任何异常 → 降级
          logger.warning("新路径失败，降级...")
          # 落一条 action=chat_fallback 到 ticket_logs
  return await _chat_via_legacy(...)
  ```
- 旧路径抽成独立函数 `_chat_via_legacy`（保留作 fallback，P4b 再删）
- 失败事件落 `ticket_logs.action='chat_fallback', level='warning'`，方便统计 new-path 失败率

### 为什么不直接切刀
新路径首次上线总有边界情况，留 fallback 可以：
1. 用户感知不到任何异常
2. 运维通过 `SELECT * FROM ticket_logs WHERE action='chat_fallback'` 看失败模式
3. 观察期过后 P4b 再清理

## 三、眼验时暴露的 bug：`tool_input=[]` 三层防御

### 现象
用户问"那些需求是合并不到 Develop 了吗" → AI 回复"请求失败：'list' object is not a mapping"。之后无论问什么，回复都变成同一串分支列表 markdown。

### 根因链
1. LLM 对**无参数工具**（如 `git_list_branches`，`required=[]`）偶尔返回 `tool_use.input=[]` 而非 `{}`
2. `_ChatToolExecutor.execute` 里 `{**tool_input}` 遇到 list 就炸 → `TypeError: 'list' object is not a mapping`
3. 即使 execute 不炸，把原始 `[]` 塞回 history 作为第 2 轮请求给 Anthropic，Anthropic schema 校验不接受 `input:[]` → 返回 **400 Bad Request**
4. 第 2 轮挂 → `max_rounds` 兜底返回第 1 轮的 tool 结果文案 → 用户看起来"所有消息回复都一样"

### 三层防御
```
┌──────────────────────────────────────────────┐
│ 1. llm_client.chat_with_tools 追加 history  │
│    前 sanitize tool_use.input 非 dict → {}  │  → 根治 400
├──────────────────────────────────────────────┤
│ 2. _ChatToolExecutor.execute 入口 sanitize  │  → 保护 Action
├──────────────────────────────────────────────┤
│ 3. chat_with_ai try/except 外层兜底         │  → 万一失守的 SLA 保护
└──────────────────────────────────────────────┘
```

### 测试
- `_test_empty_input.py`：直接压执行器，list/None/dict 三种 input 都能正确处理
- `_test_sanitize.py`：HTTP 端到端真实 LLM 触发 `git_list_branches`（会返回 `[]`），验证第 2 轮不挂并返回自然语言回复
- `_test_fallback.py`：monkey-patch agent.chat 抛异常，验证自动降级到旧路径

## 六、Kanban 看板 bug 修复：8 个工单在两列重复显示

### 现象
用户看板截图：**测试中：8** / **测试通过：8**——同一批 8 个工单在两列都显示。

### 根因
`backend/models.py:337` 的 `BOARD_COLUMNS` 定义里，`TESTING_DONE` 状态同时出现在两个列的匹配规则中：

```python
BOARD_COLUMNS = {
    ...
    "testing": [
        TicketStatus.TESTING_IN_PROGRESS,
        TicketStatus.TESTING_DONE,    # ← bug：重复
        TicketStatus.TESTING_FAILED,
    ],
    "done": [TicketStatus.TESTING_DONE],   # ← 重复
}
```

任何 `testing_done` 工单被两列同时命中，前端渲染就出现视觉重复。

### 修复
`testing` 列去掉 `TESTING_DONE`，保留 `TESTING_IN_PROGRESS` + `TESTING_FAILED`；`done` 列继续只持 `TESTING_DONE`。

### 验证
写了一个小脚本检查每个 status 只出现在一列：
```python
from collections import defaultdict
seen = defaultdict(list)
for col, statuses in BOARD_COLUMNS.items():
    for s in statuses:
        seen[s.value].append(col)
dup = {k: v for k, v in seen.items() if len(v) > 1}
# dup == {} → 没有任何 status 被多列包含
```

API 验证：修复前 testing 和 done 各返回 8 个重复工单；修复后 testing=0, done=8。

## 十二、Pipeline 可视化由 SOP 驱动（TODO.md P1 兑现）

### 背景
今天早些时候 TODO.md 里记了一条 P1：
> 需求 Pipeline 可视化由 SOP 驱动 —— 当前 Pipeline UI 的 5 阶段在 `api/requirements.py:283` 硬编码 STAGE_DEFS，与 `sop/default_sop.yaml` 的 6 阶段不挂钩，改 SOP 不会自动更新 UI。

新增的观测 Action 又复制了一份同样的硬编码（STAGE_DEFS/PAST/RUNNING_FOR_STAGE），等于**同一份判定规则在 3 处重复**。这次一并清理。

### 设计：pipeline_view 配置节

SOP 配置里新增 `pipeline_view`，每个 UI 阶段声明：
- `sop_stages`：聚合哪些 SOP stage id
- `synthetic`：非 SOP 合成阶段（"需求分析"在 SOP 前、"合入Develop"在 SOP 后，基于需求 status 判定）
- `running_statuses`：判定"running"的工单 status 集合（这是**语义判断**，不能纯从 in_statuses/past 派生）

```yaml
pipeline_view:
  stages:
    - key: requirement_analysis
      synthetic: requirement_status

    - key: architecture
      sop_stages: [architecture]
      running_statuses: [architecture_in_progress]

    - key: development
      sop_stages: [development, acceptance, rework]
      running_statuses: [development_in_progress, acceptance_rejected]

    - key: testing
      sop_stages: [testing, fix_issues]
      running_statuses: [testing_in_progress, acceptance_passed, testing_failed]

    - key: merge_develop
      synthetic: merge
```

### `sop/loader.py` 的 `build_pipeline_stages()`

从 SOP config 自动派生：
- **Linear order**：按 SOP 声明顺序走一遍，收集所有 status 的线性序（pending → arch_* → dev_* → acc_* → test_* → deploy_*）
- **in_statuses**：每个 UI 组的 sop_stages 里所有"处理中/完成/拒绝"status；**排除返工型 sop_stage**（success 比主 stage 首个处理中 status 更早的那种，如 `fix_issues` 指向 `development_done`）
- **PAST_i**：从主 stage 的 `success_status` 往后的全部 status；**排除 reject_status 若其 reject_goto 最终导向更早位置**（如 testing_failed → fix_issues → development_done，就要从 PAST[testing] 里剔掉）
- **PRE_i**：主 stage 第一个"处理中"status 之前的所有 status

### 等价性验证

改造前抓一份 pipeline 端点 JSON baseline，改造后重新抓。**100% 等价**（仅 total_duration 因时间流逝有合法差异）。

单元测试层面：
```

  [✓] architecture in_statuses / PAST / PRE
  [✓] development in_statuses / PAST / PRE
  [✓] testing in_statuses / PAST / PRE
  [✓] merge_develop in_statuses

- `id` 全局唯一

- `pipeline_view.stages[].synthetic` 只能是 `requirement_status` / `merge`
- `pipeline_view.stages[].sop_stages` 必须引用存在的 SOP id

```
stages[0] 缺少必填字段: action
stages[0] 缺少必填字段: trigger_on
stages[0] 缺少必填字段: success_status
