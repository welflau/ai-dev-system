SOP 和 UI 完全**不联动**，改一处要改三处，极易忘。

### 新增：`pipeline_view` 配置节

在 `sop/default_sop.yaml` 底部加 **UI 视图定义**，每个 UI 阶段声明聚合规则：

```yaml
pipeline_view:
  stages:
    - key: requirement_analysis
      synthetic: requirement_status        # 不走工单 status
    - key: architecture
      sop_stages: [architecture]           # 聚合哪些 SOP stage id
      running_statuses: [architecture_in_progress]
    - key: development
      sop_stages: [development, acceptance, rework]
      running_statuses: [development_in_progress, acceptance_rejected]
    - key: testing
      sop_stages: [testing, fix_issues]
      running_statuses: [testing_in_progress, acceptance_passed, testing_failed]
    - key: merge_develop
      synthetic: merge                     # 所有工单完成后触发
```

三类字段含义：
- `sop_stages`：聚合哪些 SOP id 进本 UI 分组
- `synthetic`：非 SOP 合成阶段（SOP 前的"需求分析"、SOP 后的"合入 Develop"）
- `running_statuses`：判定"此 UI 阶段正在跑"的工单 status 集合（语义判断，不能纯数学派生）

### 新增：`sop/loader.py:build_pipeline_stages()`

从 SOP config 全自动派生：
- **Linear order**：按 SOP 声明顺序走一遍，建 status 线性序（pending → arch_* → dev_* → acc_* → test_* → deploy_*）
- **in_statuses**：聚合每个 UI 组的 sop_stages，但**排除返工型 stage**（其 `success_status` 比主 stage 首个处理中 status 更早，如 `fix_issues` → `development_done`）
- **PAST**：从主 stage 的 `success_status` 起往后的所有 status，**排除 reject_goto 导向更早位置的 reject_status**（如 `testing_failed` → `fix_issues` → `development_done`，就从 PAST[testing] 剔除）
- **PRE**：主 stage 首个"处理中"status 之前的所有 status

派生结果 `{defs, past, pre, linear_order}` 供下游使用。

`_legacy_pipeline_stages()` 作兜底：SOP 没配 `pipeline_view` 时返回原硬编码的等价值。

### 等价性验证

写了 8 条断言逐一比较派生 vs 硬编码：in_statuses × 5 + PAST × 3 + PRE × 3 全部 **100% 等价**。

HTTP API diff：改造前后抓 `/api/projects/.../requirements/.../pipeline` 的 JSON，`stages[].key/name/status/icon/jobs/has_error/has_warning` 全匹配（只有 `total_duration` 因时间流逝合法差异）。

## 二、SOP 编辑器最小版（M1 + M2）

### 动机

Pipeline 被 SOP 驱动了，但改 SOP 还是要**手改 yaml + 重启 / reload**——用户门槛高。做一个网页编辑器：增删改查 stage + 调序 + 热重载。

为控制工期：
- **M1 后端写 API**（0.5 天）
- **M2 前端 stage 列表编辑**（0.5 天）
- **M3 Pipeline 聚合规则编辑 / M4 diff 预览 / 历史回滚** → 暂不做

### 后端（M1）

新增 2 个接口，扩展 `backend/main.py`：

**`GET /api/sop/config`** 返回完整原始配置：
```json
{
  "name": "...",
  "version": "2.0",
  "description": "...",
  "global": { "max_retries": 5, "zombie_timeout_seconds": 120 },
  "stages": [ ... ],
  "pipeline_view": { "stages": [ ... ] }
}
```
跟原先的 `GET /api/sop` 区别：那个只返**裁剪视图**（metadata + stages 摘要 + rules），无法原样回填编辑器。

**`PUT /api/sop/config`** 接收完整 JSON，4 步处理：
1. **校验**：调 `validate_sop_config()` → 失败返回 400 + 错误数组
2. **备份**：当前 yaml 拷贝到 `backend/sop/backups/default_sop_YYYYMMDD-HHMMSS.yaml`
3. **写 yaml**：用 `yaml.safe_dump(allow_unicode=True, sort_keys=False)` 序列化
4. **热重载**：调 `orchestrator.reload_sop()`——reload 抛错时**自动从备份还原**并重新加载

返回：
```json
{ "status": "ok", "backup": "default_sop_20260418-155503.yaml", "name": "...", "stages_count": 6 }
```

**`backend/sop/validator.py`** 新增校验器：
- 必填字段：`id / agent / action / trigger_on / success_status`
- `id` 全局唯一
- `reject_goto` 必须指向已存在的 stage id
- `pass_threshold` 必须是数字
- `pipeline_view.stages[].synthetic` 只能是 `requirement_status` / `merge`
- `pipeline_view.stages[].sop_stages` 必须引用存在的 SOP id
- `pipeline_view.stages[]` 必须二选一：`synthetic` 或 `sop_stages`

### 前端（M2）

**UI 改造**（`frontend/index.html` + `frontend/app.js`）：

SOP 配置页右上新增 4 个按钮（编辑模式下才显示后 3 个）：
- `✏️ 编辑` / `✅ 退出编辑`（状态切换）
- `+ 新增阶段`（只在编辑模式）
- `↻ 放弃修改`（只在编辑模式，二次确认）
- `💾 保存并重载`（只在编辑模式）

编辑模式下表格多一列操作：**↑ ↓ ✏️ 🗑️**
- ↑↓：本地 draft 数组 swap，立即重渲染
- ✏️：弹 modal 表单（10 字段：id/中文名/agent/action/trigger_on/success_status/reject_status/reject_goto/description/pass_threshold）
- 🗑️：二次确认后删除

**客户端校验**：保存前检查必填空/id 重复。

**模式管理**：全局 `_sopEditMode` + `_sopDraft` 深拷贝。放弃修改 = 丢 draft + 重新 GET。保存成功 = 清 draft + 退出编辑。

### 安全性设计

- yaml 备份**永不丢失**（加 `.gitignore` 防止备份进 git 但留在本地）
- reload 失败**自动回滚**上一版备份
- 校验器**在磁盘写入前**拦截错误（不让坏配置落地）
