## 背景

用户在 AI 助手里说"数字时钟文字改成红色"，助手没弹出需求确认卡片，只口头说"让我查看当前实现情况"然后就结束了——既没创建需求，也没实际执行任何查询。排查后发现其实是**两个独立问题叠加**，顺藤摸瓜做了一次体系化改造，为后续 ChatAssistant Agent 化铺路。

## 一、两个线上问题及修复

### 1.1 "光说不做"——AI 声称要查却没发指令

**现象**：AI 回复里出现"让我查看 XX"、"让我先检查 YY"，但回复结尾没有对应的 `[ACTION:...]` 指令，系统当然也不会执行。用户看到一条"预告型"消息就结束。

**根因**：系统提示词列出了 `[ACTION:GIT_LOG]` 等文本协议的使用规则，但没有强制"凡说到就必须做到"。LLM 分成了两步思维——第一轮解释我要做什么、等下一轮再查——可本系统没有"自动下一轮"机制。

**修复**：`api/chat.py` 的 `_build_system_prompt` 新增"说到做到"规则：
- 回复中出现"让我查看/检查/看看/查一下 XX"等措辞时，**必须**在同一轮末尾附上对应 `[ACTION:...]`
- **一次回复最多一个 action**（避免触发下方问题 1.2）

### 1.2 解析器贪婪正则——两个 action 一起发就全挂

**现象**：AI 偶尔会在一条回复里连发两个 `[ACTION:...]` 块，后端日志：

```
[chat] WARNING  无法解析操作数据: {"limit": 20}
[/ACTION]
...
[ACTION:GIT_LIST_BRANCHES]
{}
```

两个 Action 都不执行。

**根因**：`_parse_and_execute_action` 的正则是：

```python
pattern = r'\[ACTION:(\w+)\]\s*(\{.*\})\s*\[/ACTION\]'
#                                ^^^^^ 贪婪
match = re.search(pattern, response, re.DOTALL)
```

贪婪 `\{.*\}` + `re.DOTALL` 下，从第一个 `[ACTION:GIT_LOG]` 里的 `{` 一路吃到**最后一个** `[/ACTION]` 前的 `}`，把中间的文字和第二个 Action 都吞进 JSON 内容——json.loads 当然炸。

**修复**：改为非贪婪 `\{.*?\}`，同时修了 `_parse_global_action` 里同样的 pattern（两处都有）。

```python
pattern = r'\[ACTION:(\w+)\]\s*(\{.*?\})\s*\[/ACTION\]'
```

非贪婪 + 必须以 `[/ACTION]` 结尾的组合会自动回溯扩展，嵌套 JSON 也能正确匹配。直接 Python 实测 `{"limit": 20}\n[/ACTION]\n\n[ACTION:X]\n{}\n[/ACTION]` 输入下，第一个 Action 干净捕获。

## 二、chat.py 是 Agent 吗？—— 架构对标分析

结合用户提问梳理：`api/chat.py` 在本系统架构里**是个伪 Agent**。

| 要素 | BaseAgent 子类（DevAgent 等） | chat.py 当前 |
|---|---|---|
| 继承 `BaseAgent` | ✅ | ❌（FastAPI router） |
| 注册到 `agent_registry` | ✅（6 个） | ❌ |
| Action 类（独立 `run()`） | ✅ 每能力一个 Action | ❌ 硬编码在 `_execute_*` 函数 |
| 状态机 / ReactMode | ✅ SINGLE/BY_ORDER/REACT | ❌ 单次请求-响应 |
| watch / subscribe | ✅ `watch_actions` | ❌ |
| Memory | ✅ `self._memory` | ❌ 靠 `chat_messages` 表临时拼 history |

它实际是：**路由 → 拼 prompt → 调一次 llm → 正则抓 `[ACTION:XXX]` → 分发到 `_execute_*`**。日志里虽然挂了 `agent_type="ChatAssistant"` 的牌子，但那只是 log 标签，没有架构实体。

这个伪 Agent 模式直接导致：
- 解析器脆弱（上面 1.2 的 bug 就是副产物）
- prompt 里 `## 操作指令格式` 大段说明每次都占 token
- 能力散落（`_execute_generate_document` 等事实上是 Action，但没继承 `ActionBase`，不能单测、不能被其他 Agent 复用）

## 三、迁移方案：ChatAssistant Agent 化

完整方案文档：`docs/20260417_01_ChatAssistant_Agent化迁移方案.md`（一并提交）。

### 3.1 核心思路
- **Action 是一等公民**：chat 的每个能力独立为 `ActionBase` 子类
- **ChatAssistantAgent 继承 BaseAgent**，`react_mode = REACT`
- **用 Anthropic 原生 tool_use 替代文本协议**：调 `llm_client.chat_with_tools`（代码已有，一直未被调用）
- **路由层瘦身**：`chat.py` 只管 HTTP、上下文、消息持久化

### 3.2 "权限即 Action 集合"
Agent 化的一个关键价值——**AI 助手能做什么，完全由 `action_classes` 列表决定**：

```python
class ChatAssistantAgent(BaseAgent):
    action_classes = [GitLogAction, ConfirmRequirementAction, ...]
```

LLM 只看得到这些工具的 schema，其他能力压根不知道存在。加/去能力 = 改一行，不再需要同步 prompt + parser 白名单两处。还能按环境裁剪——比如生产环境排掉 `GitMergeAction` 防误合并。

### 3.3 分 4 阶段实施
- **P0**：tool_use 代理兼容性验证（见第四节，已完成）
- **P1**：抽 Action，不接入 Agent，保留旧路径（本次完成）
- **P2**：新建 `ChatAssistantAgent`，feature flag 双轨运行
- **P3**：切换到 tool_use 为默认路径
- **P4**：清理旧正则协议和 `_execute_*` 包装

## 四、P0：tool_use 代理兼容性验证

本系统 `llm_client.chat_with_tools` 代码早就写好（`llm_client.py:447`），但**从未被实际调用过**（全仓库 grep 只有定义和文档）。接入前先验证企业代理 `api-skynetyu.woa.com/anthropic` 对 `tools` 字段的兼容性——前阵子 `context_management: Extra inputs are not permitted` 的前科让人担心新字段会不会一样被拒。

**验证脚本**：`backend/_test_tool_use.py`
- 定义 1 个假工具 `get_weather(city)`，返回固定 `{temp: "22°C", weather: "晴"}`
- 用户消息："帮我查一下北京现在的天气怎么样？"
- 期望：tool_use block → 执行器回 tool_result → LLM 基于结果出自然语言回复

**结果**：完整跑通。2 轮完成，两次 HTTP 200，`stop_reason` 正确触发 `tool_use` → `end_turn`，LLM 最终输出"温度 22°C、晴……"基于工具结果的自然回复。

**结论**：
- 代理对 Anthropic 原生 `tools` 字段完全兼容
- 之前的 `context_management` 报错是 Claude Code 客户端发给代理的请求里带的，与本后端无关
- 迁移方案 P1→P3 按原路推进，不需要改降级路线

### 5.2 Action 模板
每个 Action 继承 `ActionBase`，实现四件：
1. `name` / `description` / `run(context)` —— 原有契约
2. `tool_schema` —— 新增，Anthropic tool-use 格式；P2 接入 `chat_with_tools` 时自动拼入 `tools` 参数

```python
class GitLogAction(ActionBase):
    @property
    def name(self): return "git_log"

    @property
    def tool_schema(self):
        return {
            "name": "git_log",
            "description": "...",
            "input_schema": {...},
        }

    async def run(self, context):
        project_id = context["project_id"]
        limit = min(int(context.get("limit") or 10), 20)
        ...
        return ActionResult(success=True, data={...})
```

### 5.3 chat.py 对接改造
两处调用点迁移：

1. **`/confirm-create` 端点**（用户点击"确认创建"按钮）：
   ```python
   # 旧：await _execute_create_requirement(project_id, {...})
   # 新：await CreateRequirementAction().run({"project_id": ..., ...})
   ```

2. **`_parse_and_execute_action` 分发器**：40 行 `if/elif` 链瘦身为 12 项 dict 映射：
   ```python
   _ACTION_MAP = {
       "CONFIRM_REQUIREMENT": ConfirmRequirementAction,
       "GIT_LOG": GitLogAction,
       ...
   }
   action_cls = _ACTION_MAP.get(action_type)
   action_result = await action_cls().run({"project_id": project_id, **action_data})
   return action_result.data
   ```

`GENERATE_DOCUMENT` 的自定义 `---` 格式预解析器也一并改调 `GenerateDocumentAction`。

### 5.4 保留旧代码——P4 再删
`_execute_*` 旧函数全部保留在 chat.py 里但**无调用方**——这是刻意的，降低 P1 回退成本；等 P2/P3 新路径稳定后 P4 一次性清理。
