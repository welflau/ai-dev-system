## 一、Phase C'' — SOP 接入 orchestrator（commit `d19453e`）

### 1.1 问题
上轮 Phase C 做了 `compose_sop(traits, ticket_type)`，但只有 Preview API 调用它展示。orchestrator 的 `self.transition_rules` 仍然在启动时从 `default_sop.yaml` 一次性加载，**所有项目共用**。意味着即使 `TestFPS` 项目 traits 里有 `engine:ue5, category:game`，真正调度工单时也跑 web 默认流程（architecture → development → code_review → acceptance → deploy），没有 `engine_compile` / `play_test` 阶段。

### 1.2 改造

**关键数据结构**：

```python
class OrchestratorAgent:
    def __init__(self):
        ...
        # 默认规则（兜底：无 traits / 旧项目）
        self.transition_rules = self._load_transition_rules()
        # v0.17 Phase C''：项目级缓存
        self._project_rules_cache: Dict[Tuple[project_id, traits_tuple], Dict] = {}
```

**核心方法**：

```python
async def _get_rules_for_project(self, project_id: str, traits_json=None) -> Dict:
    # 1. 拉 traits
    traits = json.loads(traits_json or '[]')

    # 2. 空 traits → 回退 default（完全兼容旧项目）
    if not traits:
        return self.transition_rules

    # 3. 有 traits → cache lookup
    cache_key = (project_id, tuple(sorted(traits)))
    if cache_key in self._project_rules_cache:
        return self._project_rules_cache[cache_key]

    # 4. compose → 缓存
    sop_config = compose_sop(traits=traits, ticket_type=None)
    rules = sop_to_transition_rules(sop_config)
    self._project_rules_cache[cache_key] = rules
    return rules
```

### 1.3 Dispatch 路径改造

三个调用点从 `self.transition_rules.get(status)` 改为 `(await self._get_rules_for_project(project_id)).get(status)`：

| 位置 | 用途 |
|---|---|
| `_poll_once` (line 272) | 获取 "actionable_statuses" 给 SQL `WHERE IN (...)` |
| 主 dispatch (line 1020) | 根据 ticket 当前 status 查下一步规则 |
| `_publish_ticket_ready` (line 1203) | 判断 ticket 更新后是否还能继续流转 |
| `process_ticket` 级联 (line 1598) | 续跑到下一阶段 |

**Poll 的 actionable_statuses**：之前只取 `transition_rules.keys()`（默认 7 个状态），现在 `_get_all_actionable_statuses()` 跨项目 union —— 默认 + 所有活跃项目 compose 出来的 rules keys。保证 `engine_compile_passed` / `play_test_passed` 这类"只出现在某些项目"的状态也会被轮询扫到。

### 1.4 失效机制

```python
def invalidate_project_rules(self, project_id: str):
    to_remove = [k for k in self._project_rules_cache if k[0] == project_id]
    for k in to_remove:
        del self._project_rules_cache[k]
```

Phase E 的"编辑 traits"会调用这个。`reload_sop()`（SOP yaml 改动热重载）也清空整个 cache。

               engine:ue5, lang:cpp, genre:fps, multiplayer] → SOP 7 条规则

## 二、Phase D — 导入链路自动探测 traits（commit `74a07d1`）

### 2.1 思路

对话式新建项目靠 LLM 从 user_text 推 traits，有歧义就反问。**导入现有 Git repo** 走不同路径：用户已经有 repo，系统应该能从文件特征自动推断（像 Cursor / VSCode 给语言识别那样），然后让用户采纳或修正。

MagicAI 用 4 个 Godot 模板做关键词评分，我们用 ~20 条**文件特征规则**扫 repo 给出带证据的 traits 候选。

### 2.2 Detector 规则设计（24 条）

每条规则：

```python
{
    "glob": ["*.uproject"],                     # 文件 glob
    "dep_any_of": ["react"],                    # 可选：package.json deps 分析
    "exists_also": ["app.wxss"],                # 可选：伴随文件必须存在
    "contents_has": ["<canvas"],                # 可选：文件内容子串
    "emit": ["engine:ue5", "category:game"],    # 命中后产出的 traits
    "confidence": 0.98,                         # 置信度
    "evidence": "找到 {file}（UE5 工程文件）",     # 证据模板
}
```

**规则类别**：

| 类别 | 规则数 | 示例 |
|---|---|---|
| 游戏引擎 | 5 | `*.uproject` → engine:ue5；`project.godot` → engine:godot4；`*.uasset` (弱信号) → engine:ue5 |
| 小程序 | 1 | `miniprogram/app.json` + `app.wxss` → platform:wechat |
| 前端框架 | 5 | `package.json` 分别按 react / vue / angular / svelte / react-native 的依赖 |
| 后端（Node/Python） | 5 | express/koa/fastify；fastapi/django/flask |
| 语言 | 5 | Rust (Cargo.toml)、Go (go.mod)、TypeScript (tsconfig.json)、Java (pom.xml/gradle)、Python (pyproject.toml) |
| 其他 | 3 | HTML5 canvas 游戏、.git 目录、Perforce 配置 |

**每条 confidence 校准过**：
- `*.uproject` / `project.godot`：0.98 —— 硬证据，几乎没误报
- `package.json 依赖 react`：0.92 —— 强信号但可能是依赖 A 又依赖 B 的间接依赖
- `*.uasset` 单独出现：0.75 —— 可能是别人项目的残留 asset
- `index.html` 含 `<canvas>`：0.65 —— 普通网页也可能有 canvas

### 2.3 关键工程细节

**1. 子目录扫描**（避免 monorepo 漏检）：

原本只扫根目录，像 `ai-dev-system` 这种 `backend/requirements.txt` 的结构就命中不到。改成扫根目录 + 第一层非 `.` 开头子目录：

```python
search_scopes = [root] + [d for d in root.iterdir()
                          if d.is_dir() and not d.name.startswith(".")]
```

不继续递归（避免扫大仓库过慢）。

**2. 依赖解析**（`_extract_deps`）：
- `package.json` → json.loads 取 dependencies/devDependencies/peerDependencies
- `requirements.txt` → 按 `==` / `>=` / `<` / `[` 切出包名
- `pyproject.toml` → 简单文本搜索关键字（不引 tomllib 避免版本兼容问题）

**3. Preset 反查**（`_match_preset_by_traits`）：

跟 PresetMatcher 完全不同：PresetMatcher 根据 user_text 关键词评分推荐 preset；这里根据**已探测到的 traits 集合**去 presets.yaml 里找交集最多的 preset。

**关键保护**：preset 的"核心 traits"（platform:* / category:* / engine:*）必须是探测结果的子集，否则不推荐。避免出现"探测到 `platform:web`，但推了 `ue5-game` preset（含 `platform:desktop`）"这种荒谬推荐。

**4. warnings 生成**：
- 未探测到 `platform:*` → "建议用户手选"
- 未探测到 `category:*` → "建议用户手选（app / game / service）"
- `category:game` 但缺 `engine:*` → "看起来是游戏项目但未识别到引擎"
- 多个 engine 同时命中 → "可能是残留文件"

### 2.7 发现的局限

Monorepo 项目（同一根目录下同时有 frontend/backend）会得到混合 traits，`_match_preset_by_traits` 有时推荐会偏。未来可考虑：
- 探测到多框架时 warning 提示"看起来是 monorepo，建议分拆 2 个项目"
- 或者返回多个候选 preset 让用户选

当前设计是"给用户候选，他来决定"，所以暂不优化。

## 三、全链路现状

v0.17 到本轮为止的完整闭环：

```
【新建项目】
User 对话  → ChatAssistant 反问规则 → LLM 填 traits → confirm_project
                                                            ↓
                                               前端确认卡片（含 Preview）
                                                            ↓
                                               用户确认 → CreateProjectAction
                                                            ↓
                                               projects.traits 入库

【导入项目】 (Phase D 新)
User 选本地文件夹 → ProjectTypeDetectorAction 扫文件
                         ↓
                   detected_traits + suggested_preset + evidence + warnings
                         ↓
                   前端 detectInfo 卡片展示
                         ↓
                   用户点导入 → create_project + 写 traits 入库

【调度】 (Phase C'' 新)
用户建工单 → orchestrator._poll_once (拉所有 actionable_statuses)
                        ↓
              pick ticket → _get_rules_for_project(project_id)
                        ↓
              按 project.traits compose_sop → 产出 transition_rules
                        ↓
              rule = rules.get(current_status) → dispatch 到对应 Agent.action
                        ↓
              Skill 三层过滤（按 traits）+ ActionNode 执行
                        ↓
              next_status → 级联下一轮

【展示】 (Phase C')
Preview API: POST /api/projects/preview-assembly
             返回 {sop_stages, agents_active, skills_by_agent, mcps, warnings, suggestions}
```

**每一步都用 traits 驱动**，SOP / Skill / Agent 都按项目类型动态派生，**现有 web 项目零回归**（空 traits → 全部回退 default）。
