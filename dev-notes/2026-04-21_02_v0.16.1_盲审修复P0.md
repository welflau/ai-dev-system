## 一、P0.1 — ProductAgent 拆单注入代码上下文

### 问题

TODO 描述："不看已有代码就拆单"。

实际情况（探索后）：
- `DecomposeAction.run()` 的 prompt 模板**早就有** `existing_files` / `existing_code` / `project_summary` 3 个槽位
- `ProductAgent._analyze_and_decompose()` 也会在 `existing_files` 非空时先跑 `SummarizeCodeAction` 生成 `project_summary` 并注入 context
- 问题出在 **orchestrator.py:530**：
  ```python
  result = await agent.execute("analyze_and_decompose", {
      "title": requirement["title"],
      "description": requirement["description"],
      "priority": requirement["priority"],
  })
  ```
  只传了 3 个字段，**跳过了 `_build_context()`**，上游配好的槽位永远填不上。
- 实测表现：已经有 `index.html` 的项目，拆单 LLM 会拆出"创建 index.html"这种和架构冲突的工单。

### 修法

`orchestrator.py:530` 前加 10 行：

```python
from memory import AgentMemory
try:
    code_ctx = await AgentMemory(project_id).get_code_context()
    existing_files = code_ctx.get("file_list", [])
    existing_code = code_ctx.get("code", {})
except Exception as e:
    logger.warning("decompose 读代码上下文失败: %s", e)
    existing_files, existing_code = [], {}

result = await agent.execute("analyze_and_decompose", {
    "title": requirement["title"],
    "description": requirement["description"],
    "priority": requirement["priority"],
    "project_id": project_id,
    "existing_files": existing_files,
    "existing_code": existing_code,
})
```

就是这么简单——**10 行改动一次性接通已经写好的整条链路**：AgentMemory 读代码 → SummarizeCodeAction 生成项目摘要 → DecomposeAction 拿到完整上下文。之前的工作并没有白做，只是缺了这条"最后一公里"的注入。

## 二、P0.2 — ReviewAgent 抽成独立 SOP 阶段

### 问题

TODO 描述："完全盲审，不读代码"。

实际情况（探索后）：
- `CodeReviewAction` 本身**完全正确**：读 `dev_result.files` + `existing_code`，用 ActionNode，读 SOP `max_code_chars` 配置。不是盲审。
- 但 `ReviewAgent` 类是个空壳（`agents/review.py` 只有 7 行），orchestrator 从未直接调用它
- `CodeReviewAction` 是作为 **TestAgent 的 Phase 2** 在跑（`agents/test.py:72-78`）
- 结果：代码审查变成"测试过程的副作用"而非一等 SOP 阶段；`ReviewAgent` 类长期"写了不用"

和用户讨论后，选了"抽成独立 SOP 阶段"路线（另一条是保持现状只更新 TODO）。

### 修法

**新流程**：
```
旧:  pending → architecture → development → acceptance → testing → done
新:  pending → architecture → development → code_review → acceptance → testing → done
                                                ↑ 新增
```

**保守门控策略**（增量安全）：
- ReviewAgent 总是输出 `review_passed`，不阻塞后续
- 如 `score < sop_config.pass_score`（默认 7）→ 写 warning 级日志 + artifact，但不退回 rework
- 未来可加 `block_on_low_score: true` 开启硬门控

### 具体改动

1. `backend/models.py` — 加 `REVIEW_IN_PROGRESS` / `REVIEW_PASSED` 枚举；BOARD_COLUMNS["development"] 加入这两个状态（审查归入"开发中"看板列，UI 不改）；STATUS_LABELS 加中文名
2. `backend/sop/default_sop.yaml` — 新增 `code_review` stage；`acceptance.trigger_on: development_done → review_passed`；`testing.config.code_review: true → false`；`pipeline_view.development` 组加入 code_review
3. `backend/agents/review.py` — 加 `execute()` 路由 `code_review` 任务到 `run_action("code_review", ctx)`
4. `backend/orchestrator.py::_handle_agent_result` — 加 `elif agent_name == "ReviewAgent"` 分支：状态转 `review_passed`、存 `type='code_review'` artifact、低分写 warning log
5. `backend/agents/test.py` — `_code_review` 调用包在 `sop_config.code_review` flag 后，默认关闭
6. `frontend/app.js` — `allStatuses`（2 处）+ `STATUS_LABELS`（2 处）+ 看板列映射不变
