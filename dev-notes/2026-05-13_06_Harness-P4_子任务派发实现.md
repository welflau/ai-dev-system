## 二、执行流程

```
AI（ChatAssistant）调用 dispatch_subtask 工具
  ├─ 循环检测：嵌套深度 ≤ 3，否则拒绝
  ├─ 创建子 Ticket（parent_ticket_id = 父 Ticket ID，start_status 可指定）
  ├─ 父 Ticket.status → waiting_subtasks
  └─ SSE 推送 ticket_status_changed

Orchestrator._poll_once（每 5s）
  └─ 扫 waiting_subtasks 状态 → _check_subtasks_complete()
       ├─ 有未完成子 Ticket → 继续等待
       └─ 全部完成 → _resume_parent_ticket()
            └─ 父 Ticket.status → PENDING → 轮询器下一轮正常拾取
```

---

## 三、设计决策

### start_status 参数

子 Ticket 可以从任意状态开始，避免重复走完整流程：
- `pending`：从头走完整 SOP（PM → 架构 → 开发 → 测试）
- `architecture_done`：跳过规划设计，直接进开发
- `development_done`：只做测试

### 父 Ticket 状态设计

用 `waiting_subtasks` 字符串而非新增 TicketStatus enum 值，原因：
- `TicketStatus` 是硬编码 enum，改它需要同时改 SOP 规则、validate_ticket_transition 等
- `waiting_subtasks` 是 orchestrator 内部的临时中间状态，不需要进 SOP 流转图
- Orchestrator `_poll_once` 显式处理此状态，独立于 SOP 规则

### 循环检测

`_get_ancestor_ids()` 最多向上查 5 层（超出直接截断），
结合 `MAX_SUBTASK_DEPTH = 3` 防止 A→B→C→A 的循环派发。

## 四、看板展示

- 父 Ticket 处于 `waiting_subtasks` 时：黄色左边框 + ⏳ 角标
- 子 Ticket（有 `parent_ticket_id`）：蓝色左边框 + 左缩进 12px + `↳ 子任务` 角标

`waiting_subtasks` 的父 Ticket 被合并到看板的 `pending` 列显示（`board['pending']` 追加），
前端不用新增列，视觉上与待启动工单在同一栏，通过样式区分。
