## 一、为什么要做事后编辑

上两轮 A.5（对话式新建填 traits）和 D（导入自动探测）已覆盖"项目创建时"设 traits 的两条路径，但有**真实场景需要事后改**：

1. **Phase D detector 识别不全**：Monorepo / 新技术栈 / detector 规则没覆盖 → 需要用户手动补 `platform:web`
2. **对话式 LLM 填歪了**：Phase C'' 日志里就看到 TestFPS 被 LLM 填了 `genre:fps` 和 `multiplayer`（没 `:true`）这些非 taxonomy 词，需要事后清理
3. **项目类型演进**：前 2 周是原型（`lifecycle:prototype`），要迁到正式 production → 改 trait 触发不同 SOP
4. **增加功能性 trait**：用户中途决定要国际化 → 加 `i18n`，SOP 自动插入 `i18n_check` 阶段

没这条路径，上面四种情况都得删项目重建，代价太高。

## 二、后端 API

### 2.1 `GET /api/projects/{id}/traits`

轻量查询端点：

```json
{
  "project_id": "PRJ-20260416-740362",
  "project_name": "HelloWorld",
  "traits": ["platform:web", "category:app"],
  "preset_id": "webapp",
  "traits_confidence": {
    "platform:web": {"score": 1.0, "source": "manual_edit", "evidence": "..."},
    "category:app": {...}
  }
}
```

### 2.2 `PATCH /api/projects/{id}/traits`

**完整覆盖**式更新（不是增量 merge）。请求体：
```json
{
  "traits": ["platform:web", "category:app"],
  "preset_id": "webapp",           // 可选
  "traits_confidence": {...}        // 可选，不提供则按 manual_edit 补
}
```

**关键逻辑**：
1. **Normalize**：无 `:` 的 trait 补 `:true`。例：`multiplayer` → `multiplayer:true`
2. **合法性校验**：跟 `trait_taxonomy.yaml` 对比，不在词表的进 `unknowns` 列表（**警告不阻塞**，允许未来扩词表）
3. **confidence 合并策略**：
   - 用户显式提供 → 用用户的
   - 否则保留旧 trait 的原 confidence（旧 source 如 `file_detected` 继续保留）
   - 新加的 trait 默认 `source=manual_edit, score=1.0`
4. **写库 → 关键收尾**：调 `orchestrator.invalidate_project_rules(project_id)` 清掉该项目的 SOP 规则缓存

### 2.3 为什么不阻塞 unknowns

Taxonomy 是字典，但词典不可能预测所有未来维度（比如游戏项目可能要 `genre:fps` / `genre:rpg` 这种当前 taxonomy 没有的维度）。**严格校验会让 LLM / detector 的产出被硬拦截**。

妥协方案：
- **存下来**（用户/LLM 觉得有用，就让它们存着）
- **UI 上用 ⚠ 黄色 chip 标注**（让用户知道"这个词未来可能不被识别"）
- **SOP / Skill 过滤不会 match**（因为 fragment 的 required_traits 严格对比）
- **用户看到 ⚠ 了自己判断是删还是留**

将来 taxonomy 扩维度了（如加 `genre:` 前缀到词表），老项目的自定义 trait 自动升级为合法。

### 3.3 交互要点

- **本地 draft 副本** (`_projTraitsDraft` / `_projPresetDraft`)：改动先在前端修改，不立即发请求
- **每次修改（加/删 trait、切换 preset）立即调 preview API 刷新预览**，不需要保存就能看效果
- **点"💾 保存修改"才 PATCH 落库**
- **preset 下拉应用时**：把 preset 的 traits **合并进**当前 traits（不是覆盖），这样用户可以 "先选 ue5-game 再加 multiplayer:true" 这类增量操作
- **非 taxonomy trait** 用 ⚠ 黄色 chip 标注并 title 悬停提示

### 3.4 taxonomy / presets 首次加载缓存

两份数据是系统级的，首次访问编辑器时拉一次 `/api/traits/taxonomy` + `/api/traits/presets` 存到模块变量 `_projTaxonomyCache` / `_projPresetsCache`，后续编辑器切换不重复请求。

## 四、E2E 验证

### 4.1 PATCH 调用示例

```bash
curl -X PATCH /api/projects/PRJ-xxx/traits -d '{
  "traits": ["platform:web", "category:app", "framework:react", "multiplayer", "genre:fps"],
  "preset_id": "webapp"
}'

返回：
{
  "saved traits": ["platform:web", "category:app", "framework:react",
                   "multiplayer:true", "genre:fps"],   ← normalize 生效
  "unknowns": ["multiplayer:true", "genre:fps"],       ← 非 taxonomy 警告
  "preset_id": "webapp"
}
```

### 4.2 Invalidate 效果（日志证实）

PATCH 前日志（HelloWorld 空 traits）：
```
[orchestrator] 项目 PRJ-20260416 traits=[] → SOP 7 条规则（default）
```

PATCH 后 5 秒内：
```
[sop] SOP composed: 5 阶段（base 5 + fragments []）
       traits=['category:app', 'framework:react', 'genre:fps',
               'multiplayer:true', 'platform:web']
[orchestrator] 项目 PRJ-20260416 traits=[...] → SOP 5 条规则
```

**规则数从 7 → 5** 的变化确认 orchestrator 重新 compose 了 SOP。下一个工单从 `pending` 起跑时会用新规则。

### 4.3 GET 往返

PATCH 后 GET 同一项目 → `traits` / `preset_id` / `traits_confidence` 全返回一致；confidence 自动补了 `source: manual_edit`。

## 六、踩坑记录

### 6.1 BaseModel 未定义

E.1 初版把 `UpdateTraitsRequest` 写在 Phase E 段落，但 `from pydantic import BaseModel` 在后面的 Preview Assembly 段才 import。启动直接报 `NameError: BaseModel is not defined`，watchfiles 狂刷 "change detected"。

**修法**：把 `BaseModel / Field / Optional / Dict / Any` 的 import 搬到文件顶层，删掉 Preview 段里的重复 import。

**教训**：新加带 pydantic model 的段落时，先确保顶部 import 足够；或者永远把 model 定义放在文件头，不跨段。

### 6.2 `multiplayer:true` vs taxonomy 的 `multiplayer`

`trait_taxonomy.yaml` 的 `features` 维度值是 `multiplayer`（裸词），但系统各处（fragments / preset_matcher / normalize 后的用户输入）都用 `multiplayer:true` 表示"这个功能被启用"。

Normalize 规则会把用户输入的 `multiplayer` 转成 `multiplayer:true`。但 taxonomy 里查不到 `multiplayer:true`，导致 unknowns 列表里出现它。

这是**设计层的语义歧义**，不是 bug：
- Taxonomy 的值列表是"这个维度里合法的 value 词"
- 实际 trait 存储是 `dim:value` 字符串
- `features` 维度的 value 如 `multiplayer`，合法 trait 字符串应是 `features:multiplayer`
- 但历史上 fragments 和 preset_matcher 用的是 `multiplayer:true`（把 feature 名当 dim 用，用布尔作 value）

**现状**：保持现有约定继续跑（fragments / preset / 本 API 都兼容 `multiplayer:true`），但放到 v0.17.x 的 TODO：统一成 `features:multiplayer` 还是继续 `multiplayer:true`，一次性改完。

**当前不阻塞**：unknowns 警告不拦截保存，系统各层语义一致。
