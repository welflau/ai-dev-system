## 一、工单并发限制（Orchestrator 排队）

### 问题

一次性批量确认 17-95 条需求，所有工单同时进入 `asyncio.create_task()`，导致：
- `asyncio_tasks` 瞬间达到 100+，事件循环饱和
- LLM API 并发请求爆表（95 个工单 × 4 个 Agent 阶段）
- 看板刷新卡死，界面无响应

### 修复

**方案：调度时限流**（不用 Semaphore，直接在 `_poll_once` 入口控制）

新增 `_project_active: Dict[str, set]` 追踪每个项目当前活跃工单集合：

```python
self._MAX_CONCURRENT_PER_PROJECT = 3
self._project_active: Dict[str, set] = {}
```

`_poll_once()` 分发前检查空位，满了直接 `continue` 跳过（等 30 秒下次轮询补充）：

```python
active_set = self._project_active.setdefault(project_id, set())
if len(active_set) >= self._MAX_CONCURRENT_PER_PROJECT:
    continue  # 等下次轮询
active_set.add(ticket_id)
asyncio.create_task(self._poll_process(project_id, ticket_id))
```

完成时释放槽位：
```python
finally:
    self._processing.discard(ticket_id)
    self._project_active.get(project_id, set()).discard(ticket_id)
```

**效果**：asyncio 活跃任务从 100+ 降至个位数，LLM API 并发可控。

## 二、系统性能指标条（全页顶部）

### 背景

工单一多就卡，但看不到系统压力在哪。需要实时可观测。

### 实现

**后端**：新增 `GET /api/metrics` 端点（`backend/main.py`）：

| 指标 | 说明 | 危险阈值 |
|---|---|---|
| `processing` | Orchestrator 处理中工单数 | ≥10 红 |
| `asyncio_tasks` | asyncio 活跃任务数 | ≥50 红 |
| `llm_pending` | 在途 LLM 请求数 | ≥5 红 |
| `mem_mb` | 进程内存 MB | ≥1500 红 |
| `db_ms` | SQLite 响应时间 | ≥100ms 红 |
| `wal_kb` | SQLite WAL 文件大小 | ≥5MB 红 |

同时在 `LLMClient.__init__()` 添加 `_pending_requests` 计数器，`chat()` 方法进出时 +1/-1。

**前端**：在 `<header>` 下方追加一条 26px 高的固定指标条，每 5 秒自动刷新：

```
⚙️ 处理中 3  |  🔄 异步任务 10  |  🤖 LLM并发 2  |  💾 内存 92MB  |  🗄️ DB 1.1ms  |  📝 WAL 0KB
```

颜色随阈值自动变绿/黄/红，页面内容区向下偏移 26px 适配。

## 三、批量需求默认暂停状态

### 问题

批量确认卡片一次创建 19 条需求，全部立即进入 Orchestrator 队列开始分析，即使项目刚建立、开发顺序未定。

### 修复

- `ConfirmRequirementRequest` 加 `paused: bool = False` 字段
- `CreateRequirementAction` 支持 `paused` 参数，`status` 从 `"submitted"` 改为 `"paused"`
- 前端 batch 卡片调用时固定传 `paused: true`，按钮文字改为"⏸ 创建为暂停状态（手动开始）"
- 单条 `confirm_requirement` 卡片行为不变（仍 `submitted` 立即执行）

**效果**：批量确认后所有需求进入暂停列表，用户手动逐条 resume，自然形成开发顺序。

## 四、需求批量勾选确认卡片（ConfirmRequirementsBatch）

### 背景

AI 分析文档后识别出 19 条需求，每条调一次 `confirm_requirement` 工具（受 `max_react_loop=6` 限制只识别到 1 条，且是最后一条不相关的）。

### 修复

新增 `ConfirmRequirementsBatchAction`（`backend/actions/chat/confirm_requirement.py`）：
- 工具名：`confirm_requirements_batch`
- 接收 `requirements: List[{title, description, priority}]` 数组
- 返回 `confirm_requirements_batch` 类型，前端渲染单张带 checkbox 的卡片

前端新增批量卡片（每条需求一行 checkbox，默认全选）+ `doConfirmRequirementsBatch()` handler：
- 显示创建进度 `(n/total)`
- 全部以 `paused` 状态创建（不立即执行）
- 完成后刷新需求列表

System prompt 更新：
- ≥2 条需求时用 `confirm_requirements_batch`（不再逐条调单条工具）
- 文档分析也改用 batch 工具

## 五、工单依赖关系 —— 拆单 Prompt 修复

### 问题

工单关系图中所有工单平行排列，无依赖连线。Orchestrator 代码中依赖回填逻辑完整（第二遍扫描），但 `DecomposeAction` 的 prompt 从未要求 LLM 输出 `dependencies` 字段，导致 AI 永远返回 `[]`。

### 修复

`backend/actions/decompose.py` 拆单规则追加：

```
- **dependencies 必须填**：填当前工单依赖的前置工单索引列表（0-based int）
  - 无前置 → []；工单1 依赖工单0 → [0]；工单2 同时依赖0和1 → [0,1]
  - 复杂需求应形成有序链，避免所有工单平行开发互相冲突
```

ActionNode instruction 同步更新，强调必须填 dependencies。

**注**：需求间依赖（cross-requirement ordering）为后续规划项，当前以"batch 默认 paused + 手动 resume"临时替代。
