## Phase 4：流程优化（commit `8300bbd`）

### P4-2. 对抗性代码审查

**问题**：ReviewAgent 原来只有"issues"和"suggestions"两级，经常输出"代码质量良好"类敷衍结论，审查流于形式。

**改动**：

`actions/schemas.py` — `TestReviewOutput` 扩展三级分类：
```python
class TestReviewOutput(BaseModel):
    score: int = 6
    critical_issues: List[str] = []   # 🔴 严重：必须修复
    warnings: List[str] = []          # 🟡 警告：建议修复
    suggestions: List[str] = []       # 🟢 建议：可选优化
    issues: List[str] = []            # 向后兼容
```

`actions/code_review.py` — instruction 升级为对抗模式：
```
[MODE: REVIEW] 你是严苛的代码审查者，必须主动找出代码问题。
⚠️ 禁止输出「代码质量良好」「暂无问题」等敷衍结论。
每个级别（critical/warning/suggestion）至少找出 1 个具体问题。
评分规则：有 critical→ ≤5 分；只有 warning→6-7 分；只有 suggestion→8-9 分；完美→10 分。
```

审查报告格式升级：
```markdown
## 🔴 严重问题（必须修复）
- XSS 漏洞：用户输入未转义直接插入 innerHTML

## 🟡 警告（建议修复）
- 缺少空值检查：data.items 可能为 null

## 🟢 建议（可选优化）
- 函数名 handleData 语义不够清晰
```

### P4-3. Smart Probe 需求清晰度评分

**问题**：模糊需求直接进流水线，到开发后期才发现关键信息缺失，大量返工。

**改动**：`actions/write_prd.py` 新增 `_smart_probe()` 函数，在生成 PRD 前进行 5 维度评估：

| 维度 | 满分 | 评估内容 |
|---|---|---|
| 用户价值 | 5 | 有没有说明为什么要做这个功能 |
| 范围边界 | 5 | 有没有说明不做什么 |
| 交互方式 | 5 | 有没有描述用户操作流程 |
| 数值参数 | 5 | 有没有关键数字（时间/数量/大小）|
| 关联影响 | 5 | 有没有提到与其他功能的关系 |

**处理策略**：
- **≥ 20 分**：需求清晰，直接生成 PRD
- **15-19 分**：AI 自动补充合理假设，用 `[假设]` 标注，继续流水线
- **< 15 分**：给出更强提示，要求 PRD 末尾列出「待确认事项」，不阻断流程

评分记录在 `artifacts`，可追溯分析哪类需求经常不清晰。

### P4-4. 需求规模自适应（XS/S/M/L/XL）

**问题**：改个颜色这样的小改动也要走完整 9 阶段流水线（Planner→UX→Art→Architect→Dev→…），浪费时间。

**改动**：`orchestrator.py` 拆单后根据 `complexity + ticket_count` 决定初始状态：

| 规模 | 条件 | 跳过阶段 | 初始状态 |
|---|---|---|---|
| XS | `simple` 或 1 个工单 | 跳过 Planner + Architect | `architecture_done`（直接开发）|
| S | `medium` 且 ≤ 2 个工单 | 跳过 Planner | `planning_done`（直接架构）|
| M/L/XL | 其他 | 完整流水线 | `pending` |

**效果**：
```
改个颜色 → XS → 直接进 DevAgent，节省 3 个阶段
加个小按钮 → S → 从 ArchitectAgent 开始，节省 1 个阶段
新增背包系统 → M → 完整流水线 9 阶段
```
