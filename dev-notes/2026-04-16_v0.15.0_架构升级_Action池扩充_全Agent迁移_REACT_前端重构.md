### 三、PlanCodeChangeAction（最重要的新 Action）

**解决问题**：之前每次 DevAgent 都全文件重写，新需求覆盖旧功能。

**设计**（移植自 MetaGPT `WriteCodePlanAndChange` 思想）：
```
Phase 1: 规划变更
  → LLM 分析：哪些文件要新建、哪些要修改、哪些不动
  → 输出 CodeChangePlan（Pydantic Schema）

Phase 2: 逐文件执行
  → 新建文件：LLM 生成完整内容（参考已有代码风格）
  → 修改文件：LLM 输入原文件 + 改动描述 → 输出修改后的完整文件
  → 不改的文件：跳过

回退：规划失败 → 降级到 WriteCodeAction 全文件生成
```

**DevAgent 智能分发**：
```python
if existing_code:
    # 有已有代码 → PlanCodeChange 精准增量
    code_result = await self.run_action("plan_code_change", context)
else:
    # 空项目 → WriteCode 全新生成
    code_result = await self.run_action("write_code", context)
```

### 四、产品验收改为运行效果截图

**之前**：读代码片段发给 LLM → "代码截断/不完整" → 打回

**现在**：
1. 启动 HTTP server 运行页面
2. Playwright / Chrome headless 截图
3. 截图 + 需求描述发给 LLM
4. 基于运行效果判断是否通过
5. 截图保存到工单文档目录

### 五、REACT 模式落地

`BaseAgent._react_with_think` 实现：
1. `_think()`: LLM 根据当前进度选择下一个 Action 或 "done"
2. `_act()`: 执行选中的 Action
3. 循环最多 `max_react_loop` 次

三种模式总结：
- **SINGLE**：orchestrator 指定 action（默认，最安全）
- **BY_ORDER**：按 action_classes 顺序依次执行
- **REACT**：LLM 动态选择下一步（自主决策）

### 六、前端可视化重构

**Agent API 重写**：
- `/api/agents` 从注册中心动态读取（不再依赖静态 AGENT_REGISTRY）
- 返回实际 Action 信息（名称/描述/Schema 类型/ActionNode 标识）

**Agent 配置页**：
- 列表：Action 标签 + ReactMode + 完成/异常统计
- 详情弹窗：执行模式 / Watch / Action 池（含 Schema 信息）
- 替代旧的 prompt 模板展示

**Agent 监控页**：
- Actions / Mode / Watch 信息展示
- ActionNode 蓝色标识

| Bug | 根因 | 修复 |
