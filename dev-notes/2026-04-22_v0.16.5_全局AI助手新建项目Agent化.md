# 开发日志 — 2026-04-22（深夜场）

## 版本
- v0.16.4 → **v0.16.5**

## 主题
**清掉聊天栈里最后一条 legacy 文本协议路径**：全局 AI 助手（项目列表页右侧抽屉）新建项目从 raw LLM + 正则抠 `[ACTION:CREATE_PROJECT]` 迁到 `ChatAssistantAgent.chat_global()` + Anthropic tool_use。同步补上两步式确认卡片（LLM 产草稿 → 用户确认 → 后端落库），对齐项目内聊天"需求/BUG 确认"的交互模式。

配套设计文档：`docs/20260422_03_全局AI助手新建项目链路分析与Agent化方案.md`（同 commit 入库）。

---

## 背景

今早梳理完 v0.16.4 的工单 AI 对话 bug 后，用户问"现在 AI 助手新建项目的构建路由是怎样的？"。回溯代码发现：

| 聊天类型 | 入口 | 路径 | 确认交互 |
|---|---|---|---|
| **项目内**聊天 (`/api/projects/{id}/chat`) | `chat_with_ai()` | v0.15.2 已迁 `CHAT_USE_AGENT` → `_chat_via_agent` + tool_use | ✅ 需求/BUG 有确认卡片 |
| **全局**聊天 (`/api/chat`，项目列表页) | `global_chat_with_ai()` | **raw LLM + 正则抠 `[ACTION:CREATE_PROJECT]`** | ❌ prompt 里写"等用户确认"，但后端收到就直接建 |

这是唯一还在生产跑的 legacy 文本协议路径。问题除了架构不一致外还有几个实际影响：
1. LLM 如果不听话（漏输出确认阶段），用户的"帮我看看能不能建"可能被直接当成"现在建" → 不可逆操作
2. 正则 `\{.*?\}` 贪婪截断，JSON 稍微不标准就 `logger.warning("无法解析")` 静默吞，用户看 AI 说"好的"但项目没建
3. Skills / MCP / Failure Library 的能力完全绕过
4. `CHAT_USE_AGENT` 标志对它不生效——没给这条路径做分支

用户确认"继续上面的全局 AI 的"后，按早前写的 `20260422_03_...md` 方案三阶段实施。

## 二、Phase 2 — 后端接 Agent + tool_use

### 2.1 扩展 `ChatAssistantAgent`（`backend/agents/chat_assistant.py`）

- `action_classes` 加 `ConfirmProjectAction`
- 新增白名单常量 `_GLOBAL_CHAT_TOOLS = {"confirm_project"}`——全局聊天只暴露这一个工具
- `_exposed_tool_schemas(scope)` 加 `scope` 参数：
  - `scope="project"` 排除 `_GLOBAL_CHAT_TOOLS`（项目里不能再创项目）+ 带 MCP
  - `scope="global"` 只保留 `_GLOBAL_CHAT_TOOLS`，不带 MCP（MCP 工具大多也依赖 project 上下文）
- `_ChatToolExecutor.__init__` 的 `project_id` 改成 `Optional[str]`；ctx 注入时按 None 判断
- `_TYPE_PRIORITY` 补 `confirm_project: 1`（Tier 1 需用户确认）、`project_created: 3`
- 新增 `chat_global(user_message, images, history, projects_brief)` 方法
- 新增 `_build_global_system_prompt(projects_brief)`——不挂具体项目，告诉 LLM "你只有一个工具 confirm_project，只有用户明确说建项目时才调"

### 2.2 新端点 `/api/chat/confirm-create-project`

挂在 `global_chat_router`（对比项目内的 `/api/projects/{id}/chat/confirm-create-requirement`）：

```python
class ConfirmCreateProjectRequest(BaseModel):
    name: str = Field(..., min_length=1)
    description: str = Field(default="")
    tech_stack: str = Field(default="")
    git_remote_url: str = Field(..., min_length=1)
    local_repo_path: str = Field(default="")

@global_chat_router.post("/confirm-create-project")
async def confirm_create_project(req: ConfirmCreateProjectRequest):
    """用户点击"确认创建"按钮后真正落库。"""
    from actions.chat.create_project import CreateProjectAction
    result = await CreateProjectAction().run(req.model_dump())
    if not result.success:
        raise HTTPException(400, result.data.get("message") or result.error or "创建失败")
    return result.data
```

Pydantic 层挡掉空 name / 空 git_url，不用到 Action 里才发现。

### 2.3 `global_chat_with_ai` 双轨分支

照抄项目内聊天 `chat_with_ai` 的 fallback 模式：

```python
@global_chat_router.post("", response_model=GlobalChatResponse)
async def global_chat_with_ai(req: GlobalChatRequest):
    from config import settings
    if settings.CHAT_USE_AGENT:
        try:
            return await _global_chat_via_agent(req)
        except HTTPException:
            raise
        except Exception as e:
            logger.warning("全局新路径失败，降级到旧 [ACTION:CREATE_PROJECT]: %s: %s", type(e).__name__, e)
            # 埋点进 ticket_logs.action="chat_agent_fallback"
    return await _global_chat_legacy(req)
```

`_global_chat_via_agent` 从 `agent_registry.get_registry()` 拿 `ChatAssistantAgent` 实例，调 `agent.chat_global(...)`；`_global_chat_legacy` 就是原来那个函数改了个名。

### 2.4 踩的一个坑

第一次跑时 `_global_chat_via_agent` 抛 `ImportError: cannot import name 'agent_registry' from 'agent_registry'`——我顺手写 `from agent_registry import agent_registry`，但那个模块其实只导出 `get_registry()` 函数。fallback 兜住了（用户收到的是 legacy 路径的回复），然后看日志发现 → 改成 `import agent_registry as _ar; _ar.get_registry().get("ChatAssistant")`，和项目内聊天那条路径的写法一致。

Smoke test 绿之后的典型请求：

```
[POST /api/chat, body: {"message":"create a project named demo with git url https://github.com/foo/demo.git"}]
→ reply: "我为您准备了新项目草稿：\n\n**项目名称：** demo\n**Git 远程仓库：** ..."
→ action: {"type":"confirm_project","name":"demo","git_remote_url":"...","tech_stack":"","description":"","local_repo_path":""}
```

### 3.2 两个处理器

`doConfirmProject(cardId)`：
- 读 `card.dataset.*` 组装 body
- `POST /api/chat/confirm-create-project`
- 成功：左边条变绿、标题改"✅ 项目已创建"、按钮区替换为"进入项目 →"链接；showToast + 1.2s 后 `showProjectDetail(project_id)` 跳转

## 收益与下一步

### 直接收益

- **整条聊天栈不再有 legacy `[ACTION:XXX]` 文本协议**（`CHAT_USE_AGENT=true` 默认开下）
- **两步式创建**：LLM 不会再"一不小心"直接建项目——必须经过用户点"确认创建"按钮
- Skills 自动生效：`git-workflow` Skill 现在注入到 `confirm_project` 的 system prompt（之前 raw LLM 路径完全接不到）
- Reflexion / Failure Library 设施自动接入
- Fallback 设计：Agent 路径抛异常时降级到老路，用户无感知，同时 `ticket_logs.action="chat_agent_fallback"` 埋点可统计失败率

### 后续可选项

- **观察 1-2 周**后（dev-note `20260422_03` 里建议的节奏），如果新路径失败率 < 1%，移除 `_global_chat_legacy` + `_parse_global_action` + `_execute_create_project` wrapper，清掉 `[ACTION:CREATE_PROJECT]` 协议
- 消 `docs/TODO.md` 的「ChatAssistantAgent 默认化（P3/P4 清理旧 [ACTION:XXX] 协议）」这一栏
- Failure Library 接入需要补一段：`CreateProjectAction` 的 Git 失败（clone 超时、URL 错等）应该记到 failure_library 供后续自学
