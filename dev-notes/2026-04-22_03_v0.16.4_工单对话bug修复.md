## Bug 2 — 工单 AI 对话 tab 报 `content.replace is not a function`

### 现象

点 AI 助手 → 工单 tab → 选一个工单 → 面板出 `❌ 加载失败: content.replace is not a function`，一条消息也渲染不出。

老数据的工单一切正常，**只有** v0.15.2 之后（`CHAT_USE_AGENT` 默认开、走 tool_use）新产生的工单对话会翻车。

### 根因

翻调用链：

```
frontend/app.js:5924   loadTicketConversations(ticketId)
  ↓ GET /api/projects/{id}/chat/ticket/{ticket_id}/conversations
backend/api/chat.py:818  get_ticket_conversations()
  ↓ SELECT messages FROM llm_conversations ...
  ↓ json.loads(conv["messages"])
  ↓ 循环找 role=="user" 的消息，取 m.get("content", "")   ← 🔥
  ↓ user_msg[:500] if user_msg else ...
  ↓ 返回给前端
frontend/app.js:6727  formatChatContent(content)
  ↓ content.replace(/\[ACTION:\w+\]/..., '')             ← 💥 content 是 Array 不是 String
```

`llm_conversations.messages` 里存的是 **Anthropic Messages API 原始格式**。当 Agent 用 tool_use 跟 LLM 交互时，一条 user 消息的 `content` 字段会是**多段 block 的 list**：

```python
{
  "role": "user",
  "content": [
    {"type": "text", "text": "..."},
    {"type": "tool_result", "tool_use_id": "x", "content": "..."},
  ]
}
```

后端直接 `m.get("content", "")` 拿到这个 list，`user_msg[:500]` 对 list 切片仍然返回 list，JSON 序列化成数组给前端；`formatChatContent` 第一行 `content.replace(...)` → `Array.prototype.replace` 不存在 → 整个函数抛错 → loader 的 catch 兜底显示"加载失败"。

老工单（v0.15.2 之前）的 content 都是纯字符串，所以没事。

### 修法

后端加一个 `_content_to_display_text()` helper，把 Anthropic 多段 content 展平成纯文本；两个相关端点（`/ticket/{id}/conversations` 和统一 Feed 的 `/tickets/conversations`）都走这个 helper。前端 `formatChatContent` 开头也加一层类型兜底，防御以后类似的脏数据。

后端 `backend/api/chat.py`：

```python
def _content_to_display_text(content: Any) -> str:
    """把 LLM 消息的 content 字段展平成纯文本，供前端对话 feed 展示。

    Anthropic 格式下 content 可能是：
    - str：直接返回
    - list of blocks：每个 block 是 {"type": "text"|"tool_use"|"tool_result", ...}
      - text block → 取 "text"
      - tool_use block → "[tool: xxx(...)]"
      - tool_result block → "[tool_result: ...]"
    其他类型兜底为 str()，永不返回非字符串。
    """
    if isinstance(content, str):
        return content
    if isinstance(content, list):
        parts = []
        for block in content:
            if not isinstance(block, dict):
                parts.append(str(block))
                continue
            btype = block.get("type")
            if btype == "text":
                parts.append(block.get("text", ""))
            elif btype == "tool_use":
                parts.append(f"[tool: {block.get('name', '?')}(…)]")
            elif btype == "tool_result":
                inner = block.get("content", "")
                inner_text = _content_to_display_text(inner) if not isinstance(inner, str) else inner
                parts.append(f"[tool_result] {inner_text}")
            else:
                parts.append(str(block))
        return "\n".join(p for p in parts if p)
    if content is None:
        return ""
    return str(content)
```

两个调用点（`:852`、`:782`）从 `m.get("content", "")` 改成 `_content_to_display_text(m.get("content", ""))`。

前端 `frontend/app.js:6727`：

```diff
  function formatChatContent(content) {
      if (!content) return '';
+     // 兜底：content 可能是数组/对象（tool_use 消息的多段 content），先转字符串
+     if (typeof content !== 'string') {
+         try { content = JSON.stringify(content); }
+         catch { content = String(content); }
+     }
      content = content.replace(/\[ACTION:\w+\][\s\S]*?\[\/ACTION\]/g, '');
      ...
  }
```

### 为什么后端+前端都改

后端才是根因，理论上改后端就够。前端那层兜底纯粹是**防御性**：

- 历史 `llm_conversations` 表里还可能有未来版本引入的其他奇怪 content 结构
- `formatChatContent` 还被聊天气泡、action detail、Agent 群聊等 5 处地方调用，其中任一上游传脏数据都会让整个气泡渲染崩

前端一层 3 行的 `typeof !== 'string'` guard 成本极低，收益是"无论后端传什么都不再炸"。

## 踩坑小记：uvicorn --reload 没响应

改完 `api/chat.py` 后发现 `/api/skills` 正常（新路由），但 `/api/chat/.../conversations` 仍然走老代码。查 `server.log` 发现：
- 17:08:02 watchfiles 最后一次触发 `Reloading...`
- 我的编辑在 17:09:24
- 之后再改 / touch / 甚至改 main.py 内容加时间戳，都没触发新的 reload

watchfiles 可能因为前段时间我清 `__pycache__` 时的 "159 changes detected" 风暴进了抑制状态（或 uvicorn 的 reload filter 内部状态乱了）。

解决：直接 `Stop-Process -Id <worker_pid>` 杀掉 reloader 的 spawn 子进程，父 reloader 会自动重新 spawn 一个 worker，新 worker 走一遍 import 全流程拿到最新代码。对比"kill 整个后端重启"，这种方式保留了 reloader 本体和它的启动打印、事件总线初始化等状态不丢。

记一下，以后再碰到"文件改了但 reload 不响应"先试这招。
