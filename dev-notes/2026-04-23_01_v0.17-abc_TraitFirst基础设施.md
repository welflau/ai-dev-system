## 二、Phase B — Skill 重构（4.5d 计划 → 实际 ~1h）

### B.1 SkillLoader 三层过滤

```
Layer 1: inject_to: [Agent]        Agent 级 —— 谁需要这块知识
```

## 三、Phase C — SOP/Agent/Action（3.5d 计划 → 实际 ~1.5h）

### C.1 SOP fragments 组合器（最大一块）

`backend/sop/_core.yaml` —— 5 阶段通用骨架：
```
architecture → development → code_review → acceptance → deploy
```

`backend/sop/fragments/` 6 个条件性 stage：

| fragment | 触发条件 | 插入点 | priority |
|---|---|---|---|
| smoke_test | `any_of:[category:service, category:app]` + ticket feature/refactor | after:development | 90 |
| engine_compile | `any_of:[engine:ue5/ue4/godot4/godot3/unity]` | after:development | 95 |
| asset_gen | `all_of:[category:game]` + ticket:feature | **before:development** | 100 |
| play_test | `all_of:[category:game]` | after:development | 85 |
| multiplayer_stress | `all_of:[multiplayer:true]` + ticket:feature | **after:play_test** | 50 |
| i18n_check | `any_of:[i18n]` | after:development | 80 |

`sop_loader.compose(traits, ticket_type)` 实现：
- 扫 fragments → 按 `required_traits` / `required_ticket_type` 过滤
- **ASC priority 排序**（低优先级先插入，高优先级后插入挤走低优先级 → 最终高优先级更靠近锚点，先跑）
- **fixed-point 多轮插入**（解决 fragment-to-fragment 锚点：multiplayer_stress 依赖 play_test 先存在）
- `_relink_stage_transitions` 自动重建 `trigger_on` 链（fragment 宣称的 trigger_on 被上一阶段的 success_status 覆盖，避免孤立触发状态）

**验证**：Godot 多人游戏项目完整链：
```
architecture → asset_gen → development → engine_compile → play_test →
multiplayer_stress → code_review → acceptance → deploy
```
`trigger_on` 全部连通 ✓

## 四、关键 bug / 踩坑记录

### 4.1 PresetMatcher `blanket penalty` bug
**现象**："Unity 3D 射击游戏" 本该推荐 unity-game，结果 score=-7 被过滤。

**根因**：一条 conflict_rule 只声明 `user_mentions: ["3D"]` + `preset_has_keywords_like: ["2D"]`，但早期实现没处理 `preset_has_keywords_like` 字段 → fallthrough 到"对所有 preset 扣 -20"的 blanket penalty。

**修法**：
1. 实现 `preset_has_keywords_like` 分支
2. **加防御**：`_rule_applies` 开头检查"必须至少有一个 preset 侧条件"，否则记 warning 跳过

### 4.2 Fragment 插入顺序 bug
**现象**：UE5 项目预期 `engine_compile (priority 95) → play_test (priority 85)`，实际反了：`play_test → engine_compile`。

**根因**：原本按 priority DESC 排序先插入高优先级，但 `insert(idx+1, stage)` 会把**后插入的**挤到锚点+1 位置 → 高优先级先插入反而被低优先级挤开。

**修法**：改成 ASC 排序。低优先级先插入（远离锚点），高优先级最后插入（紧挨锚点）。

### 4.3 Fragment-to-fragment 锚点依赖
**现象**：`multiplayer_stress` 的锚点是 `play_test`（另一个 fragment），一次性循环插入时 play_test 还没插入 → "锚点未找到"。

**修法**：改成 **fixed-point 多轮迭代**，每轮只插入锚点已存在的 fragment，重复到无变化（或全部剩余都找不到锚点）为止。

### 4.4 `trigger_on` 链断裂
**现象**：fragment 宣称自己的 `trigger_on: smoke_test_pending`，但插入后发现没人产出这个 status → orchestrator 永不触发该阶段。

**修法**：`_relink_stage_transitions()` 自动把每个 stage 的 `trigger_on` 设成**上一个 stage 的 success_status**（除第一个保留 `pending`）。fragment 自己宣称的 `trigger_on` 被覆盖。
