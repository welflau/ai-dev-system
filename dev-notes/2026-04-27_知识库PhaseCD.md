## Phase C：知识自进化（KnowledgeDistillAction）

### 触发逻辑（`backend/orchestrator.py`）

在 acceptance_passed 流程（ProductAgent 验收通过后）追加 fire-and-forget 调用：
```python
_fire_knowledge_distill(project_id, ticket_id)
```

`_fire_knowledge_distill`（模块级函数）：
- 检查 ticket_logs 中是否有 `action IN ('reject', 'error', 'reflection')` 记录
- 有失败-修复循环才触发（无 → 跳过，节省 LLM 成本）
- `asyncio.create_task(_run())` 后台异步，不阻塞主流程

### KnowledgeDistillAction（`backend/actions/knowledge_distill.py`）

**Hermes CLASS-FIRST 原则**：
- 记录通用错误类型和修复模式，不记录单次具体实例
- 优先 append 到已有文档，不随意新建

**流程**：
1. 读工单 title/type/module
2. 读最近 15 条关键 ticket_logs（reject/error/reflection/accept/complete）
3. 格式化事件摘要（只含错误描述、根因、修复策略，不含代码）
4. 列出现有知识库文件（让 LLM 优先追加）
5. `llm_client.chat_json(temperature=0.2)` → `{skip, filename, section_title, content, action}`
6. 安全：filename 只能在 `已知问题/`、`FAQ/`、`规范/` 三目录下
7. 写文件 + 更新 knowledge_index FTS5

## Phase D：工单历史检索

### tickets_fts 虚拟表（`backend/database.py`）

```sql
CREATE VIRTUAL TABLE IF NOT EXISTS tickets_fts USING fts5(
    search_text,
    ticket_id UNINDEXED,
    project_id UNINDEXED,
    tokenize="trigram"
);
```

**不用触发器**：代码层主动维护（`_upsert_tickets_fts`），只在关键时机更新，避免频繁 I/O。

**写入时机**（`backend/orchestrator.py`）：
- Reflexion 完成后：`search_text = title + description + root_cause + specific_changes`
  → 历史搜索能找到具体错误类型（如 UCapsuleComponent、OnRep_Score）

**startup 补录**（`backend/main.py`）：
- 扫描所有 completed 状态的工单（79 个），含 reflection root_cause，写入 tickets_fts

### SearchTicketHistoryAction（`backend/actions/chat/search_ticket_history.py`）

FTS5 MATCH 查询，返回：
- 工单标题、匹配片段（`**高亮**`）
- Reflexion 的 root_cause（最有价值：上次 AI 对该问题的根因分析）
- 工单状态、时间

注册到 `ChatAssistantAgent.action_classes`。

### Trigram tokenizer 说明

- SQLite FTS5 默认 unicode61 tokenizer 对中文分词效果差（无空格边界）
- 改用 `tokenize="trigram"`：将文本切成 3 字符 N-gram，支持中文和英文部分匹配
- 限制：query 至少 3 个字符才有结果（单字/双字搜索不支持）
- 英文用例：`UCap` 能匹配 `UCapsuleComponent` ✓
- 中文用例：`架构设` 能匹配 `架构设计文档` ✓
