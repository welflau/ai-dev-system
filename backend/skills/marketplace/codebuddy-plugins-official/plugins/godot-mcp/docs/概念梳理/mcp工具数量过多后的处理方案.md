## 第二步：数量精简策略（把几十个工具压缩到个位数/十位数，避免上下文爆炸）
你最大的问题不是内部不好维护，而是**几十个工具全暴露给大模型会占满上下文，还会让大模型选错工具**。结合你「Skill做中枢调度、全自然语言触发」的场景，用四个方法就能把对外暴露的工具数量砍60%-80%，还不损失任何能力：

### 1. 先切分「对外暴露」和「内部自用」，砍一半数量
不是所有MCP工具都需要暴露给外部大模型/Agent——**只有需要顶层大模型直接调用的工具才需要注册到MCP对外列表，只给你自己的Skill内部调用的工具，完全不需要暴露**。

举个例子：
- 需要对外暴露：`search_godot_knowledge`（用户问经验大模型需要直接调用）、`initialize_godot_project`（顶层Agent需要触发初始化）
- 完全不需要对外暴露：`download_godot_editor`、`parse_error_stack`、`scan_existing_modules`——这些都是你的Skill内部编排的时候自己调用的，顶层大模型根本不需要知道它的存在，直接不注册到MCP对外工具列表，一下子就能砍掉一半数量。

### 2. 同域工具合理聚合，不瞎拆原子工具
很多时候你会把一个操作拆成好多个工具（比如新建节点/删除节点/修改节点拆成三个），其实完全可以把**同一个操作对象、同类型的动作聚合成一个工具**，用`action`参数区分，语义反而更清晰，工具数量直接降下来。

反例（过度拆分）：
`create_node`/`delete_node`/`modify_node`/`move_node` → 4个工具

正例（合理聚合）：
```typescript
// 聚合为1个工具，参数清晰，大模型不会搞错
{
  name: "operate_node",
  description: "对Godot场景中的节点执行增删改移动操作",
  parameters: {
    action: {
      type: "string",
      enum: ["create", "delete", "modify", "move"],
      description: "要执行的节点操作类型"
    },
    scenePath: { type: "string", description: "节点所在场景路径" },
    // ...其他公共参数
  }
}
```
👉 注意：不要过度聚合，只聚合**同一个业务域、同一个操作对象**的工具，不同域的工具（比如环境部署和节点操作）绝对不要塞一起，否则大模型会乱传参数。

按这个规则聚合，通常10个原子工具能压缩到3-4个，语义还更统一。

### 3. 门面模式：端到端能力对外只暴露一个，内部保持拆分
针对流程固定的端到端能力，对外只暴露一个高层门面工具，内部还是拆成原子工具维护，兼顾「对外精简」和「对内模块化」，非常适合你的场景：

举个例子：
- 对外（给大模型/顶层Agent）只暴露一个`initialize_godot_project`工具
- 内部这个门面工具自动调度你拆分好的`check_current_env`→`download_godot_editor`→`copy_template`→`init_config`四个原子工具
- 对外只显示1个工具，内部还是按拆分好的维护，改其中一步不会影响其他，完美。

这个方法特别适合你一键部署、自动Debug这类端到端能力，对外非常干净，内部维护不受影响。

### 4. 动态按需加载：当前场景只加载当前需要的工具
MCP原生支持运行时动态注册/注销工具，结合你的Skill中枢，完全可以做到**什么阶段加载什么工具，永远只给大模型暴露当前需要的几个工具**：

举个完整的动态加载流程：
| 用户输入 | 当前Skill场景 | 只加载对应标签的工具 | 暴露数量 |
|----------|---------------|------------------------|----------|
| 打开空白项目说「帮我做一个游戏」 | 初始化阶段 | 只加载`env/init`标签的工具 | 1-2个（门面聚合后甚至只有1个） |
| 初始化完成，进入开发阶段 | 开发阶段 | 注销env工具，加载`project/edit/debug`标签的工具 | 8-12个，完全不会多 |
| 用户问「怎么设计这个游戏的数值表」 | 经验咨询阶段 | 只加载`knowledge`标签的工具 | 2-3个 |

这个方法彻底解决了“总共有几十个工具，上下文装不下”的问题——不管你内部有多少工具，当前上下文永远只有不到15个，tokens占用非常少，大模型也不会选错。

## 第三步：Skill层配合标签做调度优化
因为所有调度都是你的Skill做中枢，你可以在Skill的系统提示里加一行标签索引，让Skill（或者顶层大模型）快速定位到需要的工具，完全不会乱：
```
你是Godot开发助手，调用工具时按分组选择：
- 需要初始化环境/项目 → 选择env分组的工具
- 需要修改项目内容/创建资源 → 选择project分组的工具
- 需要启动调试/获取运行日志 → 选择debug分组的工具
- 需要查询开发经验/分析现有项目 → 选择knowledge分组的工具
```

就算内部工具多，Skill也能按标签快速过滤，不会在几十个工具里瞎找。
