| Skill | 负责需求拆解、开发规范约束、Debug循环编排：<br>用户说“做一个按空格跳跃的横版主角”，Skill会基于Godot开发规范，把大需求拆解成「新建CharacterBody2D节点→添加碰撞→绑定输入→写移动跳跃逻辑」的步骤，再一步步调度MCP执行；开发完成后自动发起Debug循环：调度MCP启动游戏→拿到错误日志→分析错误修改代码→重新启动，直到运行成功，整个循环完全由Skill控制，不需要用户干预。<br>同时Skill沉淀Godot特有的语法、API规范，避免大模型写出不符合GDScript/Godot规则的错误代码。 |

#### 3. 游戏开发经验（策划、数值、架构、设计理念）
经验类能力不是纯静态知识，需要结合当前项目适配，所以分工如下：
| 模块 | 具体职责 |
|------|----------|
| MCP | 封装确定性的经验检索能力：<br>`search_godot_development_knowledge`（按关键词检索内置的经验库，返回对应策划案模板、数值表规范、架构方案原文）<br>`generate_experience_outline`（按游戏类型输出对应经验框架，比如输入“战棋”就输出战棋数值表的标准模板）<br>检索、生成模板都是固定流程，放MCP复用。 |
| Skill | 负责经验的适配和解读：<br>用户说“帮我设计这个战棋游戏的数值表”，Skill先理解需求，调用MCP拿到战棋数值设计的通用规范，再结合当前项目的规模（是demo还是完整游戏）、已有项目结构，调整出适配当前项目的数值表结构，既可以输出给用户看，也可以直接调度MCP把数值表文件生成到项目里。<br>碰到设计理念、架构选择类问题，Skill也会结合用户项目的情况给出推荐，比如100行以内的demo就不会推荐复杂的ECS架构，这个决策由Skill做，MCP只负责提供原始经验内容。 |

#### 4. 游戏内容堆叠经验（增量开发、避免重复、架构合理）
这个是「经验分析+执行」结合的能力，分工刚好契合分层：
| 模块 | 具体职责 |
|------|----------|
| MCP | 封装确定性的项目分析和操作：<br>`scan_existing_modules`（扫描当前项目所有已有脚本、节点、资源，输出所有可复用模块的功能、接口清单，固定扫描逻辑）<br>`check_duplicate_implementation`（对比新增需求和已有模块，输出匹配度，告诉你哪里已经有实现可以复用）<br>`get_current_architecture_spec`（提取当前项目的目录规范、命名规则、架构约定，输出标准化结果）<br>所有扫描、匹配都是固定流程，放MCP。 |
| Skill | 负责增量开发的规则约束和决策：<br>用户说“给这个游戏加一个道具系统”，Skill首先调度MCP扫描现有项目，拿到可复用模块和现有架构规范，然后基于内容堆叠的经验做决策：哪些现有模块可以复用（比如已经有道具基类就不用重新写）、新增内容放在哪个目录符合现有架构、怎么改不破坏原有接口，最后再调度MCP做新增/修改操作，从规则层面避免重复开发和架构混乱。 |
