# 4.20 设计方案：Skill 内置连接 & Debug 自动循环

## 一、需求概述

### 需求1：打通插件内的 Skill 连接
- Skills 文件夹随插件安装直接可用，无需 command 下载到本地项目
- Skill 维护「自然语言 → command」的映射（意图路由）
- 清除 command 中对 skill 的调用，防止循环依赖

### 需求2：构造 Debug 循环
- Godot 编辑器运行时主动/自动发送报错回 IDE
- IDE 侧 AI 自动分析并修改代码

---

## 二、需求1 详细设计：Skill 内置连接

### 2.1 现状分析

| 层级 | 现状 | 问题 |
|------|------|------|
| Skills 存放 | 插件根目录 `skills/` | 可正常加载，无需改动 |
| 部署方式 | `skill` command 将 skills 拷贝到项目 `.codebuddy/skills/` | 多余步骤，用户安装插件即应拥有全部 skill |
| 调用方向 | AI 根据 trigger 匹配 skill → skill 内部调用 MCP tools | 正常 |
| command→skill | 经排查 Godot addon 内无 skill 引用 | 需检查 markdown command 文件中是否有引用 |

### 2.2 方案：路由 Skill 与功能 Skill 分离

#### 2.2.1 核心概念

将 skill 分为两类，存放在不同目录中：

| 类型 | 目录 | 职责 |
|------|------|------|
| **路由 Skill** | `skills/routing/` | 仅做自然语言→command 的映射引导，自身不含功能逻辑 |
| **功能 Skill** | `skills/level-0~3/` | 包含实际功能知识，由 AI 直接使用 |

```
skills/
├── routing/                  # ← 新增，纯路由映射
│   ├── project-init.md       #    项目初始化路由
│   ├── environment-setup.md  #    环境搭建路由
│   └── debug-fix.md          #    调试修复路由
├── godot-dev/                # 功能 skill（不变）
├── level-0-foundation/
├── level-1-generators/
├── level-2-modules/
└── level-3-orchestrators/
```

**分离原则**：路由 skill 只负责判断条件、选择 command 并触发执行，绝不包含游戏开发知识或代码生成逻辑。功能 skill 绝不直接触发 command，避免循环。

#### 2.2.2 路由 Skill 格式

路由 skill 是极轻量的 markdown 文件，仅包含条件判断和 command 映射：

```markdown
<!-- skills/routing/project-init.md -->
---
name: project-init-router
type: routing
triggers:
  - 创建项目
  - 新建游戏
  - 生成一个xx游戏
  - 搭建项目
---

# 项目初始化路由

## 判断条件
1. 检查用户系统是否安装了 Godot（执行 `status` command 获取环境信息）
2. 检查当前目录是否已有 `project.godot`

## 路由规则
| 条件 | 执行 command |
|------|-------------|
| 无 Godot 环境 | `env`（全环境安装） |
| 有 Godot，无项目 | `new`（创建项目） |
| 已有项目 | 不触发 command，交由功能 skill 处理 |

## 注意
- 仅负责引导到正确的 command，不做任何代码生成
- command 执行完毕后流程结束，后续功能由功能 skill 接管
```

#### 2.2.3 路由决策流程

```
用户输入 "生成一个FPS游戏"
  ↓
AI 匹配到 routing/project-init.md（triggers 命中）
  ↓
按路由规则检查本地状态
  ├─ 无 Godot → 自动执行 command:env
  ├─ 有 Godot 无项目 → 自动执行 command:new
  └─ 有项目 → 跳过路由，进入功能 skill（level-2/level-3）
  ↓
command 执行完毕 → AI 继续使用功能 skill 完成游戏开发
```

关键点：路由 skill 只在**必要的前置 command**（env/new/setup）需要执行时介入，执行完即退出，不参与后续功能流程。

#### 2.2.4 Skill 目录直接生效

**不再需要 `skill` command 部署步骤**。插件安装后 `skills/` 目录直接作为 AI 的知识库：

- IDE 插件（如 Codebuddy）读取插件根目录 `skills/` 下所有文件
- `routing/` 下的文件用于意图识别和 command 引导
- `level-0~3/` 下的文件用于功能执行
- 两类 skill 互不调用

#### 2.2.5 清除 command→skill 循环调用

需要检查并修改的文件：

| 文件 | 检查项 | 操作 |
|------|--------|------|
| `commands/skill.md` | 整个 command 的职责变更 | 改为"列出/查看可用 skill"的查询命令，移除部署逻辑 |
| `commands/new.md` | 是否调用 skill deploy | 移除 skill 部署步骤 |
| `commands/env.md` | 是否调用 skill deploy | 移除 skill 部署步骤 |
| `commands/setup.md` | 是否调用 skill install | 移除 skill 安装步骤 |

#### 2.2.6 防循环保证

```
routing skill → command → MCP tool → Godot    ✅ 允许
功能 skill   → MCP tool → Godot               ✅ 允许
routing skill → 功能 skill                     ❌ 禁止（路由完成即退出）
功能 skill   → command                         ❌ 禁止
command      → skill（任何类型）                ❌ 禁止
```

### 2.3 实施步骤

1. **修改 Skill frontmatter 格式** — 所有 `SKILL.md` 添加 `intent-mapping` 和 `conditions`
2. **改造 `skill` command** — 从"部署"变为"查询/浏览"
3. **清理 command 中的 skill 部署逻辑** — 逐文件排查移除
4. **编写意图路由 skill** — 新建 `skills/intent-router/SKILL.md` 作为顶层路由
5. **测试** — 验证"生成一个XX游戏"能正确路由到 env/new/skill

## 三、需求2 详细设计：Debug 自动循环

### 3.1 现状分析

| 能力 | 现状 | 缺口 |
|------|------|------|
| 脚本验证 | `validate_script` / `validate_all_scripts` 命令已有 | 需要手动调用 |
| 错误详情 | `get_script_errors` 可获取带上下文的错误 | 未自动推送 |
| 运行时错误 | 无捕获机制 | 需要监听 Godot 输出 |
| 自动修复 | 无 | 需要构建 错误→分析→修复 闭环 |

### 3.2 架构设计

```
┌──────────────────────────────────────────────────────────┐
│                    Debug 循环架构                         │
│                                                          │
│  Godot Editor                        MCP Server → IDE    │
│  ┌──────────┐                       ┌──────────────┐     │
│  │ 错误监听器 │──── WebSocket ──────→│ debug_tools  │     │
│  │          │   (push_error事件)     │              │     │
│  │ · 编译错误 │                      │ · 接收错误    │     │
│  │ · 运行时错误│                      │ · 格式化推送  │     │
│  │ · 控制台输出│                      │ · 触发AI分析  │     │
│  └──────────┘                       └──────────────┘     │
│       ↑                                    │             │
│       │                                    ↓             │
│  ┌──────────┐                       ┌──────────────┐     │
│  │ 脚本修改  │←── apply_fix ────────│ AI 分析修复   │     │
│  └──────────┘                       └──────────────┘     │
└──────────────────────────────────────────────────────────┘
```

### 3.3 Godot 侧实现：错误监听器

#### 3.3.1 新增 `debug_commands.gd`

在 `addons/godot_mcp/commands/` 下新增 `debug_commands.gd`，职责：

```gdscript
# 主要命令
- start_debug_monitor    # 启动错误监听
- stop_debug_monitor     # 停止错误监听
- get_debug_output       # 获取运行日志（主动拉取）
- get_runtime_errors     # 获取运行时错误

# 监听源
- EditorInterface 的脚本编译错误
- 自定义 logger 劫持 push_error / push_warning / print
- 游戏运行时的 output 捕获
```

#### 3.3.2 错误监听机制

```gdscript
# 方案 A：轮询式（简单可靠）
var _error_buffer: Array = []
var _monitor_timer: Timer

func start_debug_monitor():
    _monitor_timer = Timer.new()
    _monitor_timer.wait_time = 2.0  # 每2秒检查
    _monitor_timer.timeout.connect(_check_errors)
    add_child(_monitor_timer)
    _monitor_timer.start()

func _check_errors():
    # 1. 检查脚本编译错误
    var script_errors = _validate_all_scripts()
    # 2. 检查编辑器日志
    var log_errors = _read_godot_log()
    # 3. 有新错误则推送
    if _has_new_errors(script_errors + log_errors):
        _push_errors_to_server(script_errors + log_errors)
```

```gdscript
# 方案 B：事件驱动式（更高效，推荐）
func start_debug_monitor():
    # 监听脚本变更后自动验证
    EditorInterface.get_resource_filesystem().filesystem_changed.connect(_on_fs_changed)
    # 监听游戏运行状态
    EditorInterface.get_editor_plugin().main_screen_changed.connect(_on_screen_changed)

func _on_fs_changed():
    # 文件系统变更时，延迟验证所有脚本
    await get_tree().create_timer(0.5).timeout
    _validate_and_push()
```

#### 3.3.3 Godot 日志读取

```gdscript
func _read_godot_log() -> Array:
    var log_path = "user://logs/godot.log"
    # 也可以用 OS.get_user_data_dir() 拼接
    var file = FileAccess.open(log_path, FileAccess.READ)
    if not file:
        return []
    # 从上次读取位置继续读
    file.seek(_last_log_position)
    var new_content = file.get_as_text()
    _last_log_position = file.get_length()
    # 解析错误行
    return _parse_log_errors(new_content)
```

### 3.4 MCP Server 侧实现：Debug Tools

#### 3.4.1 新增 `debug_tools.ts`

在 `server/src/tools/` 下新增：

```typescript
// 新增 MCP Tools
- start_debug_monitor    // 启动 Godot 侧监听
- stop_debug_monitor     // 停止监听
- get_debug_errors       // 主动拉取当前错误
- apply_fix              // 将修复方案应用到文件（调用已有 script tools）
- run_debug_cycle        // 完整的 检测→分析→修复→验证 循环
```

#### 3.4.2 Debug 循环 Tool 实现

```typescript
// run_debug_cycle 伪代码
async function runDebugCycle(params: { max_iterations: number }) {
    let iteration = 0;
    const maxIter = params.max_iterations || 3;
    
    while (iteration < maxIter) {
        // Step 1: 获取当前错误
        const errors = await getDebugErrors();
        if (errors.length === 0) {
            return { status: "success", message: "所有错误已修复", iterations: iteration };
        }
        
        // Step 2: 返回错误信息给 AI（通过 MCP response）
        // AI 在 IDE 侧分析并调用 edit_script / apply_fix
        return {
            status: "errors_found",
            errors: errors,
            iteration: iteration,
            suggestion: "请分析以上错误并调用 edit_script 修复"
        };
        // IDE 侧 AI 修复后再次调用 run_debug_cycle
    }
    
    return { status: "max_iterations", message: `达到最大迭代次数 ${maxIter}` };
}
```

### 3.5 Debug 循环工作流

#### 3.5.1 主动模式（用户触发）

```
用户: "帮我检查并修复项目中的所有错误"
  ↓
AI 调用 MCP tool: run_debug_cycle
  ↓
Godot 返回错误列表
  ↓
AI 分析每个错误，调用 edit_script 修复
  ↓
AI 再次调用 validate_all_scripts 验证
  ↓
如仍有错误，继续循环（最多 N 次）
  ↓
返回修复报告
```

#### 3.5.2 自动模式（文件保存触发）

```
用户在 IDE 保存文件
  ↓
Godot 文件系统监听触发
  ↓
自动验证变更文件
  ↓
发现错误 → 通过 WebSocket 推送 debug_event
  ↓
MCP Server 收到事件 → 转为 notification 推给 IDE
  ↓
IDE 侧 AI 自动介入分析修复
  ↓
修复后再次触发验证（闭环）
```

#### 3.5.3 游戏运行时模式

```
用户点击 Godot "运行"
  ↓
debug_monitor 开始捕获运行时输出
  ↓
检测到 push_error / 崩溃日志
  ↓
推送 runtime_error 事件
  ↓
AI 分析运行时错误 + 调用栈
  ↓
提供修复建议或直接修改代码
```

### 3.6 MCP Server 事件推送（Notification）

需要扩展现有 WebSocket 通信协议，新增服务端主动推送能力：

```typescript
// server/src/utils/godot_connection.ts 扩展
class GodotConnection {
    // 现有：request-response 模式
    async sendCommand(command: string, params: object): Promise<any>;
    
    // 新增：监听 Godot 主动推送的事件
    onEvent(eventType: string, handler: (data: any) => void): void;
}

// 事件类型
type DebugEventType = 
    | "errors_detected"      // 发现编译错误
    | "runtime_error"        // 运行时错误
    | "game_started"         // 游戏开始运行
    | "game_stopped"         // 游戏停止
    | "script_validated";    // 脚本验证完成
```

## 五、风险与注意事项

1. **循环调用风险**：意图路由中 skill 不能再调用 command 去触发其他 skill，路由必须是单向的（`用户 → router → command/skill → MCP tool`）
2. **日志解析可靠性**：Godot 日志格式可能因版本不同而变化，需做兼容处理
3. **性能**：debug monitor 的轮询/监听不能影响编辑器性能，建议使用事件驱动+防抖
4. **最大迭代限制**：debug 循环必须有上限（默认3次），防止 AI 修复引入新错误导致无限循环
5. **WebSocket 协议兼容**：新增事件推送需向下兼容现有 request-response 通信
