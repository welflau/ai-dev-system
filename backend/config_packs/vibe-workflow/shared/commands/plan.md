# /plan — Wave 执行计划

当用户执行此命令时，严格按以下步骤执行。**严禁在此阶段编写任何业务代码。**

## Step 0: 加载上下文

**优先读取 `DESIGN_BRIEF.md`**：
- 状态为 APPROVED → 直接使用选定路径，不重复路径对比。
- 状态为 IDLE/不存在 → 独立规划模式，要求用户提供需求描述后，输出 2 条路径供选择。

读取 `CURRENT_PLAN.md`（检查未完成前置任务），了解现有代码能力。

## Step 1: 生成 Wave 计划

输出到 `CURRENT_PLAN.md`，格式：

```markdown
# CURRENT PLAN

> 状态：🟢 IN PROGRESS | 更新时间：{日期}

## NOT in scope
- [延后的工作] — 原因

## What already exists
- [现有代码/模块] — 复用 or 重建

---

## Wave 0: 前置任务（串行）
共享类型、接口协议等被多模块依赖的基础变更。

- [ ] **[路径]** 任务描述

---

## Wave 1: 核心开发（并行）

### Agent A: 角色 · `目录/`
目录范围：`目录/`
涉及已有文件：`file1`, `file2`

- [ ] 任务 1
- [ ] 任务 2

### Agent B: 角色 · `目录/`
目录范围：`目录/`

- [ ] 任务 1

---

## Wave 2: 集成验证（串行）

- [ ] 运行测试确认通过
- [ ] E2E 验收
```

## 并行拆分硬性条件（全部满足才可并行）

1. 各 Agent 目录范围**无交集**
2. 各 Agent **不互相依赖**对方产出
3. 每个 Agent 任务**自包含**，可独立验证

不满足时退化为串行。

## 红线

- **所有任务项必须使用 `- [ ]` checkbox 格式**，这是 `/execute` 进度追踪的依赖格式
- 每个并行 Agent 的目录范围**必须明确标注**
- 严禁编写或修改任何业务代码

## 输出确认

> "计划已更新至 `CURRENT_PLAN.md`。下一步：运行 `/execute` 开始执行。"
