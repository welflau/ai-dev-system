# {{project_name}} — 代码质量规范

## Agent 行为准则

**编辑完整性**：每次编辑文件前重新读取，编辑后验证变更已正确应用。

**重命名安全**：重命名函数/类型/变量时，分别搜索：直接调用、类型层引用、字符串字面量、动态 import、re-export、测试文件。默认假设一次搜索漏了东西。

**单一事实来源**：不靠复制数据修显示问题。只能有一个源头，其他地方从它读取。

**破坏性操作安全**：删除文件前确认无其他引用；撤销变更前确认不会丢失未保存工作。

**资深开发者标准**：不满足于创可贴式修补。架构有问题就结构性修复，不过度设计。

**汇报前先验证**：说"完成了"之前，重新读一遍所有改过的内容，明确说明验证了什么。

**失败恢复**：同一个修复尝试两次都没生效，停下来重新读相关代码，找出心智模型错在哪里。

## 代码简洁性

**清晰优于巧妙**。目标是易于快速阅读和理解，不追求优雅的复杂性。

问自己：
- 我是否真正解决了手头的问题？
- 我是否过度考虑了可能的未来用例？
- 我是否至少考虑了 1 个其他解决方案？

## 代码注释规范

注释应解释"**为什么**"而不是"**是什么**"。代码本身应足够清晰说明它做了什么。

**何时写注释**：
- 从代码中无法直接看出的业务决策或领域逻辑
- 权宜之计或临时方案，需要解释为什么这样做
- 不明显的性能优化
- 重要的安全考虑

**何时不写注释**：
```
// ❌ 显而易见的注释
const user = await getUser(userId); // 获取用户

// ✅ 解释为什么
// 需要在获取时间段之前先获取可用性，因为时区转换依赖于用户配置的规则
const availability = await getAvailability(userId);
```

## Git 提交规范

Commit message 使用 Conventional Commits 格式：

```
<type>(<scope>): <描述>
```

| 前缀 | 用途 |
|------|------|
| `feat` | 新功能 |
| `fix` | Bug 修复 |
| `refactor` | 重构 |
| `docs` | 文档变更 |
| `test` | 测试相关 |
| `chore` | 构建/工具/依赖 |
| `wip` | 阶段性 checkpoint |

示例：
```
feat(api): 新增用户认证接口
fix(chat): 修复 SSE 连接断开未重连的问题
refactor(db): 统一数据库连接池管理
```
