# 建议书输出格式规范

审计完成后需要输出两部分：**对话中的完整建议书** + **写入 `CURRENT_PLAN.md` 的待办事项**。

---

## 一、对话输出模板

在对话中展示完整建议书，**必须**严格遵循以下结构，不得省略任何章节。

```markdown
# 🔍 代码重构建议书

> 生成时间: YYYY-MM-DD | 扫描范围: [路径] | 审计维度: [全部 / 指定维度]

---

## 🚨 审计摘要

| 指标 | 数值 |
|------|------|
| 扫描范围 | `[目录路径]` |
| 扫描文件数 | [N] 个 |
| 发现问题数 | [X] 个 |
| P0 (红线/故障) | [n] 个 |
| P1 (准则/维护) | [n] 个 |
| P2 (异味/优化) | [n] 个 |

## 📊 问题分布

| 维度 | P0 | P1 | P2 | 合计 |
|------|:--:|:--:|:--:|:----:|
| 跨包耦合 | - | - | - | - |
| 类型安全 | - | - | - | - |
| 状态管理 | - | - | - | - |
| 代码重复 | - | - | - | - |
| 复杂度 | - | - | - | - |
| 架构红线 | - | - | - | - |
| **合计** | **-** | **-** | **-** | **-** |

## 📋 详细任务单

### P0 — 必须立即修复

| ID | 模块 | 维度 | 问题描述 | 重构建议 (Before → After) | 预期收益 |
|:---|:-----|:-----|:---------|:-------------------------|:---------|
| R-001 | `path/to/file.ts` | 架构红线 | [具体问题] | [具体方案] | [收益] |

### P1 — 当前迭代修复

| ID | 模块 | 维度 | 问题描述 | 重构建议 (Before → After) | 预期收益 |
|:---|:-----|:-----|:---------|:-------------------------|:---------|
| R-002 | `path/to/file.ts` | 类型安全 | [具体问题] | [具体方案] | [收益] |

### P2 — 择机处理

| ID | 模块 | 维度 | 问题描述 | 重构建议 (Before → After) | 预期收益 |
|:---|:-----|:-----|:---------|:-------------------------|:---------|
| R-003 | `path/to/file.ts` | 复杂度 | [具体问题] | [具体方案] | [收益] |

---

## 🛠️ 自动重构指令

| 指令 | 说明 |
|------|------|
| `修复 R-XXX` | 对单条建议生成修复代码 |
| `修复全部 P0` | 批量修复所有 P0 问题 |
| `修复维度 [维度代号]` | 批量修复指定维度的所有问题 |
| `详情 R-XXX` | 展示某条问题的详细代码上下文 |
```

---

## 二、CURRENT_PLAN.md 写入模板

在对话输出建议书之后，**必须**使用 `replace_in_file` 将审计结果写入 `CURRENT_PLAN.md`。

### 写入规则

1. **插入位置**：在 `## ⚠️ 遗留待办` 章节**之前**插入新的里程碑章节
2. **更新状态**：将文件顶部的「当前状态」从 `✅ 空闲` 改为 `🔧 活跃开发 — 代码重构审计`
3. **更新日期**：将 `最后更新` 日期改为当天
4. **里程碑代号**：使用 `R` 前缀（Refactor），编号接续项目已有的里程碑（查看 CHANGELOG.md 确定下一个编号）
5. **checkbox 格式**：每条待办使用 `- [ ]` 格式，方便后续打勾标记完成
6. **按优先级分组**：P0 / P1 / P2 各一个子章节，空的优先级不展示

### 写入模板

```markdown
## 🔍 R[N]: 代码重构审计 (v0.X.0)

> 审计时间: YYYY-MM-DD | 扫描范围: [路径] | 问题总数: [X] 个（P0: ?, P1: ?, P2: ?）

### P0 — 必须立即修复

- [ ] **R-001** `path/to/file.ts` — [问题摘要] → [重构方向]
- [ ] **R-002** `path/to/file.ts` — [问题摘要] → [重构方向]

### P1 — 当前迭代修复

- [ ] **R-003** `path/to/file.ts` — [问题摘要] → [重构方向]
- [ ] **R-004** `path/to/file.ts` — [问题摘要] → [重构方向]

### P2 — 择机处理

- [ ] **R-005** `path/to/file.ts` — [问题摘要] → [重构方向]

---
```

### 写入示例

假设审计发现 3 个问题，`CURRENT_PLAN.md` 应变为：

```markdown
# 当前开发计划

> 基于 Monorepo 架构的 AI 小游戏生成平台
> 最后更新：2026-03-15

**当前状态**：🔧 活跃开发 — 代码重构审计 R3

---

> 📦 **已归档里程碑与版本记录**请查阅 [`CHANGELOG.md`](CHANGELOG.md)（当前版本：v0.11.0）

---

## 🔍 R3: 代码重构审计 (v0.12.0)

> 审计时间: 2026-03-15 | 扫描范围: `apps/` + `packages/` 全局 | 问题总数: 3 个（P0: 1, P1: 1, P2: 1）

### P0 — 必须立即修复

- [ ] **R-001** `apps/web/src/services/llm.ts` — 前端直接调用 LLM API → 改为通过后端代理

### P1 — 当前迭代修复

- [ ] **R-002** `apps/server/src/routes/chat.ts` — 路由中直接操作 session state → 抽取到 Service 层

### P2 — 择机处理

- [ ] **R-003** `apps/server/src/agents/codegenAgent.ts` — 函数超 80 行 → 拆分为独立的校验、生成、后处理步骤

---

## ⚠️ 遗留待办
...（原有内容保持不变）
```

## 三、格式规则（通用）

1. **ID 编号**：全局递增，格式为 `R-001`、`R-002`，不按维度重置
2. **模块路径**：使用相对于项目根目录的路径，用反引号包裹
3. **重构建议**：必须给出具体的 Before → After 方向（类名 / 函数名 / 文件拆分），不能只写「优化一下」
4. **预期收益**：用一句话说明收益（降低耦合 / 消除 any / 提高可测试性等）
5. **空维度**：如果某个维度没有问题，在问题分布表中填 `0`，详细任务单中不列出该维度
6. **空优先级**：`CURRENT_PLAN.md` 中如果某个优先级没有问题，不展示该子章节
7. **排序**：P0 在前，同级按维度顺序排列
8. **修复联动**：用户在 `CURRENT_PLAN.md` 看到某条待办后，可以输入 `修复 R-XXX` 触发自动修复；修复完成后将对应的 `- [ ]` 改为 `- [x]`
